# Story 2.3 — Save types (Flash en EEPROM)

**Epic:** 02 - Memory Bus

**Doel:** Naast SRAM ook Flash en EEPROM save types ondersteunen.

## Zig concepten

### Waarom meerdere save types?

GBA-cartridges gebruiken verschillende technologieën om save data op te slaan. Een emulator moet weten welk type de game verwacht:

```
Type          Grootte         Adres            Gebruikt door
SRAM          32 KB           0x0E000000       Simpele games
Flash 64KB    64 KB           0x0E000000       Veel games (Pokémon Ruby)
Flash 128KB   128 KB          0x0E000000       Enkele games
EEPROM        512B of 8KB     via DMA          Veel games (Zelda: Minish Cap)
```

### Auto-detectie via strings in de ROM

Games bevatten een string die het save type aangeeft. Door de ROM te doorzoeken kun je het automatisch detecteren:

```zig
const SaveType = enum {
    none,
    sram,
    flash64,
    flash128,
    eeprom512,
    eeprom8k,
};

fn detect_save_type(rom: []const u8) SaveType {
    // Zoek naar bekende strings in de ROM
    if (mem_contains(rom, "SRAM_V")) return .sram;
    if (mem_contains(rom, "FLASH_V") or mem_contains(rom, "FLASH512_V")) return .flash64;
    if (mem_contains(rom, "FLASH1M_V")) return .flash128;
    if (mem_contains(rom, "EEPROM_V")) return .eeprom512;  // grootte later bepalen
    return .none;
}
```

### std.mem.indexOf — zoeken in bytes

```zig
const std = @import("std");

fn mem_contains(haystack: []const u8, needle: []const u8) bool {
    return std.mem.indexOf(u8, haystack, needle) != null;
}
```

`std.mem.indexOf` doorzoekt een slice en geeft de positie van de eerste match terug (of `null`).

### Flash emulatie — command sequences

Flash geheugen is complexer dan SRAM. Je kunt niet zomaar schrijven — je moet specifieke **command sequences** sturen:

```zig
const Flash = struct {
    data: [128 * 1024]u8 = [_]u8{0xFF} ** (128 * 1024),
    state: FlashState = .ready,
    bank: u1 = 0,           // voor 128KB: 2 banken van 64KB

    const FlashState = enum {
        ready,
        cmd1,       // ontvangen: 0x5555 = 0xAA
        cmd2,       // ontvangen: 0x2AAA = 0x55
        write_byte, // klaar om één byte te schrijven
        erase,      // wacht op erase-commando
        bank_switch,
        id_mode,
    };

    pub fn write8(self: *Flash, address: u32, value: u8) void {
        const offset = address & 0xFFFF;

        switch (self.state) {
            .ready => {
                if (offset == 0x5555 and value == 0xAA) {
                    self.state = .cmd1;
                }
            },
            .cmd1 => {
                if (offset == 0x2AAA and value == 0x55) {
                    self.state = .cmd2;
                }
            },
            .cmd2 => {
                if (offset == 0x5555) {
                    switch (value) {
                        0x90 => self.state = .id_mode,   // chip identify
                        0x80 => self.state = .erase,     // begin erase
                        0xA0 => self.state = .write_byte, // begin write
                        0xB0 => self.state = .bank_switch,
                        0xF0 => self.state = .ready,     // reset
                        else => self.state = .ready,
                    }
                }
            },
            .write_byte => {
                // Flash write: kan alleen bits van 1→0 zetten (niet andersom)
                const bank_offset = @as(u32, self.bank) * 0x10000;
                self.data[bank_offset + offset] &= value;
                self.state = .ready;
            },
            // ... andere states
            else => self.state = .ready,
        }
    }

    pub fn read8(self: *Flash, address: u32) u8 {
        const offset = address & 0xFFFF;
        if (self.state == .id_mode) {
            // Retourneer manufacturer/device ID
            return switch (offset) {
                0x0000 => 0x62,  // Sanyo manufacturer ID
                0x0001 => 0x13,  // device ID
                else => 0,
            };
        }
        const bank_offset = @as(u32, self.bank) * 0x10000;
        return self.data[bank_offset + offset];
    }
};
```

### State machines — commando's afhandelen

Flash emulatie is een **state machine**: het gedrag hangt af van welke commando's eerder zijn ontvangen. Dit patroon zie je vaak in hardware-emulatie:

```
ready → ontvang 0xAA op 0x5555 → cmd1
cmd1  → ontvang 0x55 op 0x2AAA → cmd2
cmd2  → ontvang commando op 0x5555 → actie uitvoeren → ready
```

In Zig modelleer je dit met een `enum` voor de huidige state en een `switch` op die state.

### EEPROM — seriële communicatie via DMA

EEPROM is anders: het communiceert bit-voor-bit via een speciaal adres. De game stuurt een **bitstream** via DMA:

```zig
const Eeprom = struct {
    data: [8192]u8 = [_]u8{0xFF} ** 8192,
    state: EepromState = .idle,
    address: u14 = 0,         // 6-bit of 14-bit adres
    bit_count: u8 = 0,
    buffer: u64 = 0,          // opgebouwd uit individuele bits
    bus_width: enum { unknown, small, large } = .unknown,

    const EepromState = enum {
        idle,
        read_address,
        read_data,
        write_address,
        write_data,
        write_finish,
    };
};
```

**Hoe EEPROM werkt:**
1. Game stuurt via DMA een reeks bits: `1 1` (read) of `1 0` (write) + adres
2. Bij read: EEPROM stuurt 64 bits data terug (8 bytes)
3. Bij write: game stuurt 64 bits data, EEPROM slaat het op

De bus-width (6-bit of 14-bit adres) wordt automatisch bepaald door het aantal bits in het eerste commando.

### Save data naar schijf schrijven

Om saves te bewaren tussen sessies, schrijf je de save data naar een `.sav` bestand:

```zig
fn save_to_file(self: *GamePak, allocator: std.mem.Allocator) !void {
    // Vervang .gba extensie door .sav
    const sav_path = try std.fmt.allocPrint(allocator, "{s}.sav", .{
        self.rom_path[0 .. self.rom_path.len - 4],
    });
    defer allocator.free(sav_path);

    const file = try std.fs.cwd().createFile(sav_path, .{});
    defer file.close();

    switch (self.save_type) {
        .sram => try file.writeAll(&self.sram),
        .flash64 => try file.writeAll(self.flash.data[0..65536]),
        .flash128 => try file.writeAll(&self.flash.data),
        .eeprom512 => try file.writeAll(self.eeprom.data[0..512]),
        .eeprom8k => try file.writeAll(&self.eeprom.data),
        .none => {},
    }
}
```

## Stap-voor-stap

### Stap 1: `src/core/gamepak.zig` — save type detectie

Het eerste probleem: hoe weet je welk save type een game gebruikt? Er is geen standaard header-veld dat dit aangeeft — een ontwerpbeslissing van Nintendo die elke emulator-ontwikkelaar hoofdpijn bezorgt. De oplossing: zoek naar bekende strings in de ROM data.

```zig
pub const SaveType = enum {
    none,
    sram,
    flash64,
    flash128,
    eeprom,
};

pub fn detectSaveType(rom: []const u8) SaveType {
    if (containsString(rom, "SRAM_V")) return .sram;
    if (containsString(rom, "FLASH1M_V")) return .flash128;
    if (containsString(rom, "FLASH512_V")) return .flash64;
    if (containsString(rom, "FLASH_V")) return .flash64;
    if (containsString(rom, "EEPROM_V")) return .eeprom;
    return .none;
}

fn containsString(haystack: []const u8, needle: []const u8) bool {
    return std.mem.indexOf(u8, haystack, needle) != null;
}
```

**Waarom save type detectie via strings lastig is:** Nintendo definieerde geen standaard header-veld voor het save type. In plaats daarvan bevatten game-ROM's strings zoals `"SRAM_V"` of `"FLASH_V"` als onderdeel van de save-bibliotheek die de game linkt. Dit is niet 100% betrouwbaar — homebrew-ROM's missen deze strings soms, en sommige games bevatten meerdere strings. Maar voor de overgrote meerderheid van commerciële games werkt het.

**Waarom de volgorde van detectie belangrijk is:** `"FLASH_V"` is een substring van `"FLASH512_V"` en zou dus ook matchen op 512K Flash ROM's. Door `"FLASH1M_V"` en `"FLASH512_V"` **eerst** te checken, voorkom je dat een 128 KB Flash-game foutief als 64 KB gedetecteerd wordt.

**Waarom EEPROM-grootte niet hier bepalen?** EEPROM kan 512 bytes of 8 KB zijn, maar de string `"EEPROM_V"` zegt niet welke. De grootte wordt pas duidelijk bij het eerste DMA-commando (aan de hand van de adresbreedte: 6-bit = 512B, 14-bit = 8 KB). Dat lossen we op in de EEPROM state machine.

Roep `detectSaveType` aan in `GamePak.load()`:

```zig
pub fn load(allocator: std.mem.Allocator, rom_path: []const u8) !GamePak {
    // ... ROM laden ...
    const save_type = detectSaveType(rom);
    std.debug.print("Save type gedetecteerd: {}\n", .{save_type});

    return GamePak{
        .rom = rom,
        .allocator = allocator,
        .rom_path = rom_path,
        .save_type = save_type,
        .sram = [_]u8{0} ** (32 * 1024),
        .flash = Flash{},
        .eeprom = Eeprom{},
    };
}
```

> **Tip:** Print het gedetecteerde save type bij het laden. Dit is enorm handig bij debuggen — als een game niet goed saved, weet je meteen of het detectie-probleem is of een implementatie-bug.

### Stap 2: `src/core/gamepak.zig` — SRAM implementatie

SRAM is het simpelste save type: gewoon een blok van 32 KB dat byte-voor-byte gelezen en geschreven kan worden, net als gewoon RAM.

Voeg een SRAM-array toe aan de `GamePak` struct:

```zig
pub const GamePak = struct {
    rom: []const u8,
    allocator: std.mem.Allocator,
    rom_path: []const u8,
    save_type: SaveType,
    sram: [32 * 1024]u8,
    flash: Flash,
    eeprom: Eeprom,

    pub fn readSave(self: *const GamePak, address: u32) u8 {
        return switch (self.save_type) {
            .sram => self.sram[address & 0x7FFF],
            .flash64, .flash128 => self.flash.read8(address),
            else => 0,
        };
    }

    pub fn writeSave(self: *GamePak, address: u32, value: u8) void {
        switch (self.save_type) {
            .sram => self.sram[address & 0x7FFF] = value,
            .flash64, .flash128 => self.flash.write8(address, value),
            else => {},
        }
    }
};
```

**Waarom SRAM zo simpel is:** SRAM (Static RAM) gedraagt zich als gewoon geheugen — elke byte is direct lees- en schrijfbaar zonder commando's. Op echte hardware wordt het gevoed door een batterij in het cartridge. In de emulator is het gewoon een array van 32 KB.

**Waarom `& 0x7FFF` voor SRAM?** SRAM is 32 KB = 0x8000 bytes. Het mask `0x7FFF` (= 32768 - 1) beperkt de offset tot 15 bits, waardoor de SRAM automatisch mirrort als een game een hoger adres probeert te benaderen.

### Stap 3: `src/core/bus.zig` — save regio aansluiten op 0x0E000000

Breid de `read8` en `write8` functies in de Bus uit met regio 0x0E:

```zig
pub fn read8(self: *const Bus, address: u32) u8 {
    const region = (address >> 24) & 0xF;
    return switch (region) {
        // ... bestaande regio's ...
        0x0E => self.gamepak.readSave(address),
        else => 0,
    };
}

pub fn write8(self: *Bus, address: u32, value: u8) void {
    const region = (address >> 24) & 0xF;
    switch (region) {
        // ... bestaande regio's ...
        0x0E => self.gamepak.writeSave(address, value),
        else => {},
    }
}
```

**Waarom `read8` voor saves `*const Bus` kan houden maar `write8` `*Bus` moet zijn?** Bij SRAM muteert een save-write alleen de GamePak data, niet de Bus zelf. Maar `self.gamepak` is een pointer, en om via die pointer te schrijven moet de Bus zelf mutable zijn (Zig's borrow checker vereist dit). Voor `readSave` is `*const` voldoende omdat we alleen lezen.

> **Let op:** SRAM en Flash delen adresregio 0x0E. Er is nooit een game die beide tegelijk gebruikt — het save type bepaalt welke actief is. Daarom routeert `readSave`/`writeSave` op basis van `save_type`.

### Stap 4: `src/core/flash.zig` — de Flash state machine

Flash geheugen is fundamenteel anders dan SRAM. Je kunt niet zomaar naar een adres schrijven — de chip verwacht specifieke **command sequences** voordat hij data accepteert.

**Waarom heeft Flash een state machine nodig?** Flash-chips zijn fysiek anders dan SRAM. Het schrijven en wissen van data vereist hoge voltages die intern gegenereerd worden door de chip. Om onbedoelde writes te voorkomen, moet software een specifieke reeks commando's sturen. De chip houdt intern bij welk commando-stadium hij verwacht — dat is precies een state machine. Als emulator moeten we dat gedrag nabootsen.

```zig
const std = @import("std");

pub const Flash = struct {
    data: [128 * 1024]u8 = [_]u8{0xFF} ** (128 * 1024),
    state: State = .ready,
    bank: u1 = 0,

    pub const State = enum {
        ready,
        cmd1,
        cmd2,
        write_byte,
        erase,
        bank_switch,
        id_mode,
    };
```

**Waarom `0xFF` als initiële waarde?** Gewist ("erased") Flash geheugen bevat `0xFF`, niet `0x00`. Dit is een eigenschap van de onderliggende NAND/NOR technologie: wissen zet alle bits op 1. Als je met `0x00` initialiseert, detecteren sommige games dat als "corrupte save data" en weigeren te laden.

**Waarom een `bank: u1`?** 128 KB Flash heeft twee banken van 64 KB elk. De chip kan maar één bank tegelijk adresseren (het adresbereik is maar 16 bits = 64 KB). Door een bank-switch commando te sturen, wissel je tussen bank 0 en bank 1. Het type `u1` kan precies de waarden 0 en 1 bevatten — perfect voor een bank-selector.

Implementeer de write-functie als state machine:

```zig
    pub fn write8(self: *Flash, address: u32, value: u8) void {
        const offset: u17 = @truncate(address & 0xFFFF);

        switch (self.state) {
            .ready => {
                if (offset == 0x5555 and value == 0xAA) {
                    self.state = .cmd1;
                }
            },
            .cmd1 => {
                if (offset == 0x2AAA and value == 0x55) {
                    self.state = .cmd2;
                } else {
                    self.state = .ready; // Ongeldige sequentie → reset
                }
            },
            .cmd2 => {
                if (offset == 0x5555) {
                    switch (value) {
                        0x90 => self.state = .id_mode,
                        0x80 => self.state = .erase,
                        0xA0 => self.state = .write_byte,
                        0xB0 => self.state = .bank_switch,
                        0xF0 => self.state = .ready,
                        else => self.state = .ready,
                    }
                } else {
                    self.state = .ready;
                }
            },
            .write_byte => {
                const bank_offset = @as(u32, self.bank) * 0x10000;
                // Flash write: kan alleen bits van 1→0 zetten
                self.data[bank_offset + offset] &= value;
                self.state = .ready;
            },
            .erase => {
                if (offset == 0x5555 and value == 0x10) {
                    // Full chip erase
                    @memset(&self.data, 0xFF);
                    self.state = .ready;
                } else if (value == 0x30) {
                    // Sector erase (4 KB sector)
                    const bank_offset = @as(u32, self.bank) * 0x10000;
                    const sector_start = bank_offset + (@as(u32, offset) & 0xF000);
                    @memset(self.data[sector_start..][0..0x1000], 0xFF);
                    self.state = .ready;
                } else {
                    self.state = .ready;
                }
            },
            .bank_switch => {
                if (offset == 0x0000) {
                    self.bank = @truncate(value & 1);
                }
                self.state = .ready;
            },
            .id_mode => {
                // In ID mode: 0xF0 op 0x5555 om terug te gaan naar ready
                if (offset == 0x5555 and value == 0xF0) {
                    self.state = .ready;
                }
            },
        }
    }
```

**Waarom `&= value` bij write_byte?** Flash kan alleen bits van 1 naar 0 zetten, niet andersom. De `&=` (AND-assign) operator doet precies dat: als een bit in `value` 0 is, wordt het bit in `data` ook 0. Als een bit in `value` 1 is, blijft het bestaande bit ongewijzigd. Om bits terug naar 1 te krijgen, moet je eerst een erase doen (wat alles op 0xFF zet).

**Waarom sector erase en niet byte-voor-byte wissen?** Flash hardware kan niet individuele bytes wissen — het kan alleen hele sectoren (4 KB blokken) tegelijk wissen. Dit is een fysieke beperking van de technologie. Games die saves opslaan moeten hier rekening mee houden: ze wissen een heel sector en schrijven dan de nieuwe data.

Implementeer de read-functie:

```zig
    pub fn read8(self: *const Flash, address: u32) u8 {
        const offset: u17 = @truncate(address & 0xFFFF);
        if (self.state == .id_mode) {
            return switch (offset) {
                0x0000 => 0x62, // Sanyo manufacturer ID
                0x0001 => 0x13, // Device ID
                else => 0,
            };
        }
        const bank_offset = @as(u32, self.bank) * 0x10000;
        return self.data[bank_offset + offset];
    }
};
```

**Waarom manufacturer/device ID's?** Games sturen het `0x90` (identify) commando om te bepalen welke Flash chip aanwezig is. Aan de hand van het antwoord weten ze de grootte en het erase-protocol. Als je de verkeerde ID's retourneert, kan een game concluderen dat er geen Flash aanwezig is en weigeren te saven. De Sanyo ID's (`0x62`, `0x13`) werken voor de meeste games.

> **Valkuil:** De command-adressen `0x5555` en `0x2AAA` lijken willekeurig, maar zijn bewust gekozen door Flash-chipfabrikanten om onbedoelde writes te voorkomen. Het patroon is altijd: `0xAA` naar `0x5555`, dan `0x55` naar `0x2AAA`, dan het commando naar `0x5555`.

### Stap 5: `src/core/eeprom.zig` — serieel protocol via bitstream

EEPROM is het meest complexe save type. In tegenstelling tot SRAM en Flash communiceert EEPROM niet via gewone geheugen-reads/writes, maar via een serieel bitstream protocol.

```zig
const std = @import("std");

pub const Eeprom = struct {
    data: [8192]u8 = [_]u8{0xFF} ** 8192,
    state: State = .idle,
    address: u14 = 0,
    bit_count: u8 = 0,
    buffer: u64 = 0,
    bus_width: BusWidth = .unknown,
    read_pos: u8 = 0,

    pub const BusWidth = enum {
        unknown,
        small,  // 6-bit adres, 512 bytes (64 blokken van 8 bytes)
        large,  // 14-bit adres, 8 KB (1024 blokken van 8 bytes)
    };

    pub const State = enum {
        idle,
        get_command,
        read_address,
        read_data,
        read_dummy,
        write_address,
        write_data,
        write_finish,
    };
```

**Waarom EEPROM zo anders werkt:** EEPROM-chips zijn goedkoper dan SRAM maar hebben minder pinnen. In plaats van een volledige adres- en databus hebben ze maar één datalijn. Alle communicatie gaat bit-voor-bit via die ene lijn. De GBA stuurt bits via DMA naar een speciaal adres, en leest bits terug via hetzelfde adres.

**Waarom `bus_width: .unknown`?** De EEPROM-grootte is niet te bepalen uit de ROM-string — `"EEPROM_V"` zegt niets over 512B vs 8 KB. Het verschil zit in de adresbreedte: 512-byte EEPROM gebruikt 6-bit adressen, 8 KB gebruikt 14-bit adressen. De emulator bepaalt dit automatisch bij het eerste commando: als er meer dan 9 bits in een read-commando zitten (2 command bits + 6 adres bits + 1 dummy bit), is het een 14-bit EEPROM.

Implementeer de write-functie (bits ontvangen van de game):

```zig
    pub fn write1(self: *Eeprom, bit: u1) void {
        switch (self.state) {
            .idle => {
                // Eerste bit van een commando: altijd 1
                if (bit == 1) {
                    self.state = .get_command;
                    self.bit_count = 0;
                }
            },
            .get_command => {
                // Tweede bit: 1 = read, 0 = write
                self.bit_count = 0;
                self.address = 0;
                self.buffer = 0;
                if (bit == 1) {
                    self.state = .read_address;
                } else {
                    self.state = .write_address;
                }
            },
            .read_address, .write_address => {
                // Adresbits verzamelen, MSB first
                self.address = (self.address << 1) | @as(u14, bit);
                self.bit_count += 1;

                const addr_bits = self.getAddressBits();
                if (self.bit_count >= addr_bits) {
                    if (self.state == .read_address) {
                        self.loadReadBuffer();
                        self.state = .read_dummy;
                        self.bit_count = 0;
                    } else {
                        self.state = .write_data;
                        self.bit_count = 0;
                    }
                }
            },
            .read_dummy => {
                // Na het adres verwacht EEPROM 1 dummy bit
                self.state = .read_data;
                self.read_pos = 0;
            },
            .write_data => {
                // 64 data bits ontvangen, MSB first
                self.buffer = (self.buffer << 1) | @as(u64, bit);
                self.bit_count += 1;
                if (self.bit_count >= 64) {
                    self.storeWriteBuffer();
                    self.state = .write_finish;
                    self.bit_count = 0;
                }
            },
            .write_finish => {
                // Game stuurt een afsluitende 0-bit
                self.state = .idle;
            },
            .read_data => {},
        }
    }
```

**Waarom `write1` i.p.v. `write8`?** EEPROM communiceert bit-voor-bit. De game schrijft naar een speciaal adres, maar alleen bit 0 van de geschreven waarde telt. De bus-laag extraheert dat bit en roept `write1` aan:

```zig
// In bus.zig, bij EEPROM-detectie:
self.gamepak.eeprom.write1(@truncate(value & 1));
```

Implementeer de read-functie (bits terugsturen naar de game):

```zig
    pub fn read1(self: *Eeprom) u1 {
        if (self.state == .read_data) {
            // 64 bits data terugsturen, MSB first, plus 4 leading 0-bits
            if (self.read_pos < 4) {
                self.read_pos += 1;
                return 0; // 4 dummy bits
            }
            const data_bit = self.read_pos - 4;
            self.read_pos += 1;
            if (data_bit < 64) {
                return @truncate((self.buffer >> @intCast(63 - data_bit)) & 1);
            }
            if (self.read_pos >= 68) {
                self.state = .idle;
            }
            return 0;
        }
        // Niet in read-modus: return ready bit (1 = klaar)
        return 1;
    }
```

Helper-functies voor de buffer:

```zig
    fn getAddressBits(self: *const Eeprom) u8 {
        return switch (self.bus_width) {
            .unknown => 6, // Ga uit van small, detecteer later
            .small => 6,
            .large => 14,
        };
    }

    fn loadReadBuffer(self: *Eeprom) void {
        // Detecteer bus_width bij eerste gebruik
        if (self.bus_width == .unknown) {
            self.bus_width = if (self.bit_count > 6) .large else .small;
        }

        const byte_offset = @as(u32, self.address) * 8;
        self.buffer = 0;
        var i: u32 = 0;
        while (i < 8) : (i += 1) {
            if (byte_offset + i < self.data.len) {
                self.buffer = (self.buffer << 8) | self.data[byte_offset + i];
            }
        }
    }

    fn storeWriteBuffer(self: *Eeprom) void {
        if (self.bus_width == .unknown) {
            self.bus_width = if (self.bit_count > 6 + 64) .large else .small;
        }

        const byte_offset = @as(u32, self.address) * 8;
        var i: u32 = 0;
        while (i < 8) : (i += 1) {
            if (byte_offset + i < self.data.len) {
                self.data[byte_offset + i] = @truncate(self.buffer >> @intCast((7 - i) * 8));
            }
        }
    }
};
```

**Waarom blokken van 8 bytes?** EEPROM wordt geadresseerd in blokken, niet in individuele bytes. Elk blok is 8 bytes (64 bits) — precies de grootte van één read/write operatie. Adres 0 = bytes 0-7, adres 1 = bytes 8-15, enzovoort.

> **Valkuil:** EEPROM data wordt MSB-first (most significant bit first) verstuurd, terwijl de rest van de GBA little-endian (LSB first) is. Dit is een veelvoorkomende bug: als je de bitvolgorde omkeert, worden saves onleesbaar.

### Stap 6: `src/core/bus.zig` — EEPROM routering

EEPROM is bijzonder omdat het niet op het gewone 0x0E adres zit, maar via een speciaal adres communiceert aan het einde van de ROM-ruimte:

```zig
// In read8:
0x0D => {
    if (self.gamepak.save_type == .eeprom) {
        // EEPROM reads op de bovenste adressen van de 0x0D regio
        return @as(u8, self.gamepak.eeprom.read1());
    }
    return self.gamepakRead(address); // Anders gewoon ROM mirror
},

// In write8:
0x0D => {
    if (self.gamepak.save_type == .eeprom) {
        self.gamepak.eeprom.write1(@truncate(value & 1));
    }
},
```

**Waarom 0x0D en niet 0x0E?** EEPROM-games gebruiken DMA om te communiceren met het EEPROM-adres. Het exacte adres hangt af van de ROM-grootte: voor ROM's ≤ 16 MB is het `0x0D000000`, voor grotere ROM's is het het laatste adres in de ROM-regio. De meeste games gebruiken regio 0x0D. Dit is anders dan SRAM/Flash die op 0x0E zitten.

> **Let op:** De EEPROM-detectie op de bus vereist later verfijning. In werkelijkheid wordt EEPROM gedetecteerd via DMA-transfers naar specifieke adressen. Voor nu is routering op regio 0x0D een goede start.

### Stap 7: `src/core/gamepak.zig` — save data opslaan en laden

Om saves te bewaren tussen sessies, schrijven we de data naar een `.sav` bestand:

```zig
pub fn saveToDisk(self: *const GamePak) !void {
    const sav_path = try self.makeSavPath();
    defer self.allocator.free(sav_path);

    const file = try std.fs.cwd().createFile(sav_path, .{});
    defer file.close();

    switch (self.save_type) {
        .sram => try file.writeAll(&self.sram),
        .flash64 => try file.writeAll(self.flash.data[0..65536]),
        .flash128 => try file.writeAll(&self.flash.data),
        .eeprom => {
            const size: usize = switch (self.eeprom.bus_width) {
                .small => 512,
                .large, .unknown => 8192,
            };
            try file.writeAll(self.eeprom.data[0..size]);
        },
        .none => {},
    }
}

pub fn loadFromDisk(self: *GamePak) !void {
    const sav_path = try self.makeSavPath();
    defer self.allocator.free(sav_path);

    const file = std.fs.cwd().openFile(sav_path, .{}) catch return;
    defer file.close();

    switch (self.save_type) {
        .sram => {
            _ = try file.readAll(&self.sram);
        },
        .flash64 => {
            _ = try file.readAll(self.flash.data[0..65536]);
        },
        .flash128 => {
            _ = try file.readAll(&self.flash.data);
        },
        .eeprom => {
            _ = try file.readAll(&self.eeprom.data);
        },
        .none => {},
    }
}

fn makeSavPath(self: *const GamePak) ![]u8 {
    // Vervang .gba extensie door .sav
    const base = if (std.mem.endsWith(u8, self.rom_path, ".gba"))
        self.rom_path[0 .. self.rom_path.len - 4]
    else
        self.rom_path;

    return try std.fmt.allocPrint(self.allocator, "{s}.sav", .{base});
}
```

**Waarom `loadFromDisk` stilletjes faalt als het bestand niet bestaat?** De eerste keer dat een game gedraaid wordt, is er nog geen `.sav` bestand. De `catch return` op `openFile` vangt de "bestand niet gevonden" fout netjes op — de save data blijft op de initiële `0xFF` waarden staan. Dit is correct: gewist save geheugen is `0xFF`.

**Waarom de extensie vervangen i.p.v. eraan toevoegen?** Convention in de emulator-wereld is dat `game.gba` een save bestand `game.sav` krijgt. Als je `game.gba.sav` zou gebruiken, herkent geen enkele andere emulator het bestand. Door de extensie te vervangen ben je compatibel met andere emulators.

> **Tip:** Roep `saveToDisk` niet bij elke write aan — dat zou de performance vermoorden. Bewaar het voor momenten zoals: de emulator wordt afgesloten, of elke paar seconden op een timer. Sommige emulators saven alleen als er een write heeft plaatsgevonden sinds de laatste save (dirty flag).

```zig
test "Flash write command sequence" {
    var flash = Flash{};

    // Stuur de write command sequence
    flash.write8(0x0E005555, 0xAA);  // Stap 1
    try std.testing.expectEqual(Flash.State.cmd1, flash.state);

    flash.write8(0x0E002AAA, 0x55);  // Stap 2
    try std.testing.expectEqual(Flash.State.cmd2, flash.state);

    flash.write8(0x0E005555, 0xA0);  // Write commando
    try std.testing.expectEqual(Flash.State.write_byte, flash.state);

    // Schrijf een byte naar adres 0x100
    flash.write8(0x0E000100, 0x42);
    try std.testing.expectEqual(Flash.State.ready, flash.state);

    // Verifieer dat de byte geschreven is
    try std.testing.expectEqual(@as(u8, 0x42), flash.read8(0x0E000100));
}

test "Flash write kan alleen bits 1→0 zetten" {
    var flash = Flash{};

    // Schrijf 0b10101010
    flash.write8(0x0E005555, 0xAA);
    flash.write8(0x0E002AAA, 0x55);
    flash.write8(0x0E005555, 0xA0);
    flash.write8(0x0E000000, 0xAA); // 0b10101010

    // Probeer 0b01010101 te schrijven — alleen overlap-bits worden 0
    flash.write8(0x0E005555, 0xAA);
    flash.write8(0x0E002AAA, 0x55);
    flash.write8(0x0E005555, 0xA0);
    flash.write8(0x0E000000, 0x55); // 0b01010101

    // Resultaat: 0xAA & 0x55 = 0x00 (alle bits naar 0)
    try std.testing.expectEqual(@as(u8, 0x00), flash.read8(0x0E000000));
}

test "Flash sector erase zet sector op 0xFF" {
    var flash = Flash{};

    // Schrijf iets naar sector 0
    flash.write8(0x0E005555, 0xAA);
    flash.write8(0x0E002AAA, 0x55);
    flash.write8(0x0E005555, 0xA0);
    flash.write8(0x0E000000, 0x42);

    // Sector erase command sequence
    flash.write8(0x0E005555, 0xAA);
    flash.write8(0x0E002AAA, 0x55);
    flash.write8(0x0E005555, 0x80); // Erase commando
    flash.write8(0x0E005555, 0xAA);
    flash.write8(0x0E002AAA, 0x55);
    flash.write8(0x0E000000, 0x30); // Sector 0 erase

    // Sector moet terug op 0xFF staan
    try std.testing.expectEqual(@as(u8, 0xFF), flash.read8(0x0E000000));
}

test "Flash ID mode" {
    var flash = Flash{};
    flash.write8(0x0E005555, 0xAA);
    flash.write8(0x0E002AAA, 0x55);
    flash.write8(0x0E005555, 0x90);

    try std.testing.expectEqual(@as(u8, 0x62), flash.read8(0x0E000000)); // Manufacturer
    try std.testing.expectEqual(@as(u8, 0x13), flash.read8(0x0E000001)); // Device
}
```

**Waarom zo veel tests voor Flash?** De Flash state machine is het meest foutgevoelige deel van de save implementatie. Één verkeerde state-transitie en saves werken niet meer. Door elke commando-sequentie apart te testen, weet je precies waar het misgaat als er iets breekt.

> **Valkuil:** Flash-adressen in de tests gebruiken het volledige GBA-adres (0x0E00xxxx), maar de Flash `write8` functie maskt met `& 0xFFFF`. Zorg dat je consistent bent — ofwel mask je in de Flash zelf, ofwel in de bus-laag die de Flash aanroept. Kies één plek en documenteer het.
