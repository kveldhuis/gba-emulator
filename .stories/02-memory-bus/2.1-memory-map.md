# Story 2.1 — Memory map implementeren

**Epic:** 02 - Memory Bus

**Doel:** De GBA memory map modelleren zodat reads/writes naar het juiste geheugen gaan.

## Zig concepten

### Bit shifting — het kerngereedschap van emulatie

Stel je een rij van 32 lampjes voor, elk aan (1) of uit (0). Dat is een `u32`. Bit shifting schuift alle lampjes naar links of rechts.

```zig
const x: u32 = 0b00001111;  // binair: 4 bits aan

const y = x << 4;  // shift LEFT 4 posities
// y = 0b11110000   (de bits zijn 4 plekken naar links geschoven)

const z = x >> 2;  // shift RIGHT 2 posities  
// z = 0b00000011   (de bits zijn 2 plekken naar rechts geschoven)
```

**Waarom is dit handig?** In de GBA bepaalt een deel van een adres welk geheugen je aanpreekt. Het adres `0x03001234` heeft op bits [27:24] de waarde `0x3` (= IWRAM). Door te shiften haal je dat eruit:

```zig
const address: u32 = 0x03001234;
const region = (address >> 24) & 0xF;  // shift 24 naar rechts, pak onderste 4 bits
// region = 0x3 → dit is IWRAM!
```

### Bitwise AND (`&`) — bits maskeren

`&` vergelijkt twee getallen bit voor bit: alleen als BEIDE bits 1 zijn, is het resultaat 1.

```zig
  0b11010110
& 0b00001111    // "mask": alleen de onderste 4 bits doorlaten
= 0b00000110    // de bovenste bits zijn "gemaskeerd" (weggefilterd)
```

Veelgebruikt patroon: `(waarde >> shift) & mask` om een stukje bits uit een groter getal te halen.

```zig
const address: u32 = 0x03001234;

// Offset binnen IWRAM: alleen de onderste 15 bits (IWRAM is 32KB = 0x8000)
const offset = address & 0x7FFF;
// offset = 0x1234
```

### Waarom `& 0x7FFF` voor 32KB?

32KB = 32768 bytes = `0x8000`. De adressen lopen van `0x0000` tot `0x7FFF`. Dat zijn precies 15 bits. `0x7FFF` in binair is `0b111111111111111` — 15 enen. Door `& 0x7FFF` te doen pak je alleen de onderste 15 bits, oftewel de offset binnen dat geheugenblok.

Handig patroon: voor een blok van N bytes is het mask `N - 1` (mits N een macht van 2 is).
- 16 KB → `& 0x3FFF`
- 32 KB → `& 0x7FFF`
- 256 KB → `& 0x3FFFF`
- 1 KB → `& 0x3FF`

### Switch op waarden

```zig
fn read8(address: u32) u8 {
    const region = (address >> 24) & 0xF;
    return switch (region) {
        0x02 => self.ewram[address & 0x3FFFF],   // 256 KB
        0x03 => self.iwram[address & 0x7FFF],     // 32 KB
        0x05 => self.palette[address & 0x3FF],    // 1 KB
        0x06 => self.vram[address & 0x17FFF],     // 96 KB
        0x07 => self.oam[address & 0x3FF],        // 1 KB
        else => 0,  // onbekende regio
    };
}
```

### Little-endian: hoe 16/32-bit reads werken

De GBA is **little-endian**: de minst significante byte staat op het laagste adres.

```
Adres:   0x100  0x101  0x102  0x103
Bytes:   0x78   0x56   0x34   0x12

read8(0x100)  → 0x78
read16(0x100) → 0x5678
read32(0x100) → 0x12345678
```

In Zig bouw je een 16-bit waarde uit twee bytes:
```zig
fn read16(address: u32) u16 {
    const lo: u16 = self.read8(address);
    const hi: u16 = self.read8(address + 1);
    return (hi << 8) | lo;
    // hi wordt 8 posities naar links geschoven, dan ge-OR'd met lo
    // bijv: (0x56 << 8) | 0x78 = 0x5600 | 0x78 = 0x5678
}
```

### Bitwise OR (`|`) — bits combineren

`|` is het tegenovergestelde van `&`: als MINSTENS ÉÉN bit 1 is, is het resultaat 1.

```zig
  0b11000000
| 0b00001111
= 0b11001111    // bits van beide kanten worden gecombineerd
```

Perfect om twee helften samen te voegen (zoals hi en lo bytes hierboven).

## Context — GBA memory map

```
0x00000000 - 0x00003FFF  BIOS (16 KB, read-only)
0x02000000 - 0x0203FFFF  EWRAM (256 KB, extern werkgeheugen)
0x03000000 - 0x03007FFF  IWRAM (32 KB, intern werkgeheugen, snel)
0x04000000 - 0x040003FE  I/O Registers
0x05000000 - 0x050003FF  Palette RAM (1 KB)
0x06000000 - 0x06017FFF  VRAM (96 KB)
0x07000000 - 0x070003FF  OAM (1 KB, sprite attributen)
0x08000000 - 0x09FFFFFF  Game Pak ROM (max 32 MB)
0x0E000000 - 0x0E00FFFF  Game Pak SRAM (save data)
```

## Stap-voor-stap

### Stap 1: `src/core/bus.zig` — de Bus struct definiëren

Maak eerst de directory `src/core/` aan en daarin `bus.zig`. De Bus struct bevat arrays voor elk geheugengebied:

```zig
const std = @import("std");

pub const Bus = struct {
    bios: [16 * 1024]u8 = undefined,
    ewram: [256 * 1024]u8 = undefined,
    iwram: [32 * 1024]u8 = undefined,
    palette: [1024]u8 = undefined,
    vram: [96 * 1024]u8 = undefined,
    oam: [1024]u8 = undefined,
    io: [1024]u8 = undefined,

    pub fn init() Bus {
        var bus = Bus{};
        // Zet al het geheugen op 0 bij initialisatie
        @memset(&bus.bios, 0);
        @memset(&bus.ewram, 0);
        @memset(&bus.iwram, 0);
        @memset(&bus.palette, 0);
        @memset(&bus.vram, 0);
        @memset(&bus.oam, 0);
        @memset(&bus.io, 0);
        return bus;
    }
};
```

**Waarom `= undefined` bij de veld-declaraties?** Zig vereist dat elk struct-veld een default waarde heeft als je `Bus{}` wilt schrijven zonder alle velden op te geven. Met `= undefined` zeg je: "dit veld heeft nog geen waarde". Dat is OK zolang je het later initialiseert — wat we in `init()` doen met `@memset`.

**Waarom `@memset` in init() i.p.v. `= [_]u8{0} ** N`?** Grote arrays initialiseren met `[_]u8{0} ** N` genereert enorme hoeveelheden comptime data. `@memset` is runtime en efficiënter voor grote blokken geheugen.

**Waarom aparte arrays per geheugengebied?** De GBA heeft geen aaneengesloten adresruimte — er zitten "gaten" tussen de regio's. Door ze apart te houden hoef je geen 256 MB array te alloceren voor de hele adresruimte.

> **Tip:** De GBA heeft geen GamePak ROM/SRAM velden in de Bus — die komen in Story 2.2 via een pointer naar GamePak. Voeg ze nu nog niet toe.

### Stap 2: `src/core/bus.zig` — `read8` met adres decodering

Dit is het hart van de memory bus: een adres vertalen naar het juiste geheugengebied.

```zig
pub fn read8(self: *const Bus, address: u32) u8 {
    const region = (address >> 24) & 0xF;
    return switch (region) {
        0x00 => self.bios[address & 0x3FFF],
        0x02 => self.ewram[address & 0x3FFFF],
        0x03 => self.iwram[address & 0x7FFF],
        0x04 => self.io[address & 0x3FF],
        0x05 => self.palette[address & 0x3FF],
        0x06 => self.readVram(address),
        0x07 => self.oam[address & 0x3FF],
        else => 0, // Open bus — later verfijnen
    };
}
```

**Waarom `(address >> 24) & 0xF` voor adres decodering?**

De GBA-adresbus is 28 bits breed (bits [27:0]). De bovenste 4 bits (bits [27:24]) bepalen de geheugenregio. Door 24 posities naar rechts te shiften en met `& 0xF` te maskeren isoleer je precies die 4 bits. Dit is veel sneller dan een reeks `if (address >= 0x02000000 and address < 0x03000000)` vergelijkingen — de CPU doet dit in één of twee klokcycli. Bovendien bootst het de hardware na: de echte GBA decodeert adressen met dezelfde techniek.

**Waarom masking met `&` voor de offset?**

Het mask `& 0x3FFFF` voor EWRAM (256 KB) haalt alleen de onderste 18 bits op — dat is de offset binnen EWRAM. Maar het doet nog iets: **automatische mirroring**. Als een game adres `0x02040000` probeert te lezen (voorbij de 256 KB grens), geeft `& 0x3FFFF` netjes offset `0x00000` terug. De echte GBA doet dit ook — het geheugen herhaalt ("mirrors") zich. Door masking implementeer je dit gedrag gratis.

**Waarom VRAM apart?** VRAM is 96 KB, geen macht van 2. Het laatste blok van 32 KB mirrors naar de bovenste 32 KB, wat extra logica vereist. Vandaar een aparte functie:

```zig
fn readVram(self: *const Bus, address: u32) u8 {
    // VRAM is 96 KB. Offset modulo 128 KB, maar 64-96 KB range
    // mirrors naar 32-64 KB.
    var offset = address & 0x1FFFF; // 128 KB mask
    if (offset >= 0x18000) {
        // Mirror: 96-128 KB range spiegelt naar 64-96 KB (= 32 KB blok)
        offset -= 0x8000;
    }
    return self.vram[offset];
}
```

> **Valkuil:** Vergeet de VRAM mirroring niet. Veel games leunen hierop. Als je simpelweg `address & 0x17FFF` doet, werkt dat voor de eerste 96 KB maar crasht bij mirror-adressen.

### Stap 3: `src/core/bus.zig` — `write8` met bescherming

De write-functie lijkt op `read8`, maar met een cruciale toevoeging: sommige regio's zijn read-only.

```zig
pub fn write8(self: *Bus, address: u32, value: u8) void {
    const region = (address >> 24) & 0xF;
    switch (region) {
        0x00 => {}, // BIOS is read-only — write wordt genegeerd
        0x02 => self.ewram[address & 0x3FFFF] = value,
        0x03 => self.iwram[address & 0x7FFF] = value,
        0x04 => self.io[address & 0x3FF] = value,
        0x05 => self.palette[address & 0x3FF] = value,
        0x06 => self.writeVram(address, value),
        0x07 => self.oam[address & 0x3FF] = value,
        else => {}, // Writes naar onbekende regio's negeren
    }
}
```

**Waarom BIOS-writes negeren i.p.v. een error?** De GBA hardware negeert writes naar read-only gebieden stilletjes. Sommige games schrijven per ongeluk naar deze adressen. Als je een error gooit, crasht de game.

```zig
fn writeVram(self: *Bus, address: u32, value: u8) void {
    var offset = address & 0x1FFFF;
    if (offset >= 0x18000) {
        offset -= 0x8000;
    }
    self.vram[offset] = value;
}
```

### Stap 4: `src/core/bus.zig` — `read16` en `read32` (little-endian)

De ARM7TDMI processor in de GBA is **little-endian**: het minst significante byte staat op het laagste adres. We bouwen grotere reads op uit meerdere `read8` calls:

```zig
pub fn read16(self: *const Bus, address: u32) u16 {
    const aligned = address & ~@as(u32, 1); // forceer 2-byte alignment
    const lo: u16 = self.read8(aligned);
    const hi: u16 = self.read8(aligned + 1);
    return (hi << 8) | lo;
}

pub fn read32(self: *const Bus, address: u32) u32 {
    const aligned = address & ~@as(u32, 3); // forceer 4-byte alignment
    const b0: u32 = self.read8(aligned);
    const b1: u32 = self.read8(aligned + 1);
    const b2: u32 = self.read8(aligned + 2);
    const b3: u32 = self.read8(aligned + 3);
    return (b3 << 24) | (b2 << 16) | (b1 << 8) | b0;
}
```

**Waarom little-endian belangrijk is:** De ARM7TDMI is little-endian. Als je de byte-volgorde omkeert, lees je verkeerde instructies en data. Dit is de #1 oorzaak van "mijn emulator doet raar"-bugs bij beginners.

**Waarom alignment forceren?** De GBA hardware negeert de onderste bits bij 16-bit en 32-bit reads. Een `read16(0x03)` leest eigenlijk van `0x02`. Dit is een eigenschap van de ARM-architectuur. Als je dit niet doet, lezen sommige games verkeerde data.

**Waarom opbouwen uit `read8` i.p.v. direct uit arrays lezen?** Door `read8` te hergebruiken krijg je automatisch correcte adresdecodering en mirroring voor alle reads. Later kun je optimaliseren door direct uit arrays te lezen, maar dit is een veilig startpunt.

> **Tip:** Cast de `u8` resultaten naar het bredere type (`u16` of `u32`) **voordat** je shift. Zig staat geen shift toe als het resultaat niet past in het brontype — `@as(u8, 0xFF) << 8` is een compile error.

### Stap 5: `src/core/bus.zig` — `write16` en `write32` (little-endian)

Het omgekeerde van multi-byte reads: splits de waarde op in losse bytes.

```zig
pub fn write16(self: *Bus, address: u32, value: u16) void {
    const aligned = address & ~@as(u32, 1);
    self.write8(aligned, @truncate(value));           // onderste 8 bits
    self.write8(aligned + 1, @truncate(value >> 8));  // bovenste 8 bits
}

pub fn write32(self: *Bus, address: u32, value: u32) void {
    const aligned = address & ~@as(u32, 3);
    self.write8(aligned, @truncate(value));
    self.write8(aligned + 1, @truncate(value >> 8));
    self.write8(aligned + 2, @truncate(value >> 16));
    self.write8(aligned + 3, @truncate(value >> 24));
}
```

**Waarom `@truncate` gebruiken?** Zig staat impliciete narrowing conversies niet toe. `value` is een `u16`, maar `write8` verwacht een `u8`. `@truncate` zegt expliciet: "pak de onderste 8 bits". Dit is veiliger dan in C waar truncatie stilletjes gebeurt.

### Stap 6: `src/core/bus.zig` — unit tests

Zig heeft ingebouwde test-ondersteuning. Voeg tests toe onderaan het bestand:

```zig
test "IWRAM read/write round-trip" {
    var bus = Bus.init();
    bus.write8(0x03000000, 0x42);
    try std.testing.expectEqual(@as(u8, 0x42), bus.read8(0x03000000));
}

test "IWRAM mirroring" {
    var bus = Bus.init();
    bus.write8(0x03000000, 0xAB);
    // 0x03008000 = 0x03000000 + 32KB → mirrors terug naar 0x03000000
    try std.testing.expectEqual(@as(u8, 0xAB), bus.read8(0x03008000));
}

test "BIOS is read-only" {
    var bus = Bus.init();
    bus.write8(0x00000000, 0xFF);
    try std.testing.expectEqual(@as(u8, 0), bus.read8(0x00000000));
}

test "16-bit little-endian read" {
    var bus = Bus.init();
    bus.write8(0x02000000, 0x78);  // lo byte
    bus.write8(0x02000001, 0x56);  // hi byte
    try std.testing.expectEqual(@as(u16, 0x5678), bus.read16(0x02000000));
}

test "32-bit little-endian read" {
    var bus = Bus.init();
    bus.write8(0x02000000, 0x78);
    bus.write8(0x02000001, 0x56);
    bus.write8(0x02000002, 0x34);
    bus.write8(0x02000003, 0x12);
    try std.testing.expectEqual(@as(u32, 0x12345678), bus.read32(0x02000000));
}

test "VRAM mirroring 96KB boundary" {
    var bus = Bus.init();
    bus.write8(0x06000000, 0xAA);
    // Adres 0x06018000 (= 96 KB) moet mirroren naar 0x06010000 (= 64 KB)
    // want het bovenste 32 KB blok mirrors
    try std.testing.expectEqual(@as(u8, 0xAA), bus.read8(0x06020000));
}

test "unknown region returns 0" {
    var bus = Bus.init();
    try std.testing.expectEqual(@as(u8, 0), bus.read8(0x10000000));
}
```

**Waarom nu al tests?** Tests zijn je vangnet. De memory bus raakt alles in de emulator — CPU, PPU, DMA. Als hier een bug zit, is die later bijna onmogelijk te vinden. Test nu elke regio, mirroring, little-endian reads, en read-only bescherming.

Draai de tests met `zig build test` (nadat je in `build.zig` een test-stap hebt toegevoegd):

```zig
// In build.zig, na de exe definitie:
const unit_tests = b.addTest(.{
    .root_module = b.createModule(.{
        .root_source_file = b.path("src/core/bus.zig"),
        .target = target,
        .optimize = optimize,
    }),
});
const run_tests = b.addRunArtifact(unit_tests);
const test_step = b.step("test", "Run unit tests");
test_step.dependOn(&run_tests.step);
```

> **Valkuil:** Vergeet niet `*const Bus` te gebruiken voor `self` in read-functies (niet `*Bus`). Zig maakt onderscheid tussen mutable en immutable pointers. Reads muteren de Bus niet, dus `*const` is correcter en laat de compiler beter optimaliseren.
