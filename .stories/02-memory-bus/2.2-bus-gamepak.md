# Story 2.2 — Bus verbinden met GamePak

**Epic:** 02 - Memory Bus

**Doel:** ROM data beschikbaar maken via de bus op 0x08000000+.

## Zig concepten

### Pointers — verwijzingen naar data

Een pointer is een verwijzing naar data die ergens anders leeft. Nodig wanneer meerdere dingen dezelfde data moeten gebruiken.

```zig
const Bus = struct {
    gamepak: *GamePak,  // * = pointer naar een GamePak die elders bestaat
    ewram: [256 * 1024]u8,
    // ...
};
```

Waarom een pointer? De `Bus` hoeft de ROM data niet te *bezitten* — de `GamePak` bestaat al, en de `Bus` moet er alleen bij kunnen.

```zig
var pak = try GamePak.load(allocator, "rom.gba");
var bus = Bus{ .gamepak = &pak, ... };  // & = "neem het adres van"
```

### Meerdere structs samenvoegen — de GBA struct

```zig
const GBA = struct {
    cpu: Cpu,
    bus: Bus,
    gamepak: GamePak,

    pub fn init(allocator: std.mem.Allocator, rom_path: []const u8) !GBA {
        var gba = GBA{
            .gamepak = try GamePak.load(allocator, rom_path),
            .cpu = Cpu.init(),
            .bus = undefined,  // wordt zo gezet
        };
        gba.bus = Bus.init(&gba.gamepak);
        return gba;
    }
};
```

### @import — bestanden koppelen

```zig
// In gba.zig:
const Cpu = @import("cpu.zig").Cpu;
const Bus = @import("bus.zig").Bus;
const GamePak = @import("gamepak.zig").GamePak;
```

Elk `.zig` bestand is automatisch een module. Je importeert specifieke types/functies eruit.

## Bestanden

- `src/core/bus.zig` — voeg `gamepak: *GamePak` toe aan Bus struct
  - Reads van 0x08000000-0x09FFFFFF lezen uit `gamepak.rom`
  - Reads voorbij ROM-grootte geven 0 terug
- `src/core/gba.zig` — de GBA struct die Bus + GamePak + (later) CPU samenvoegt
  - `pub fn init(allocator: std.mem.Allocator, rom_path: []const u8) !GBA`

## Stap-voor-stap

### Stap 1: `src/core/gamepak.zig` — ROM laden vanuit een bestand

De GamePak is het cartridge dat de ROM bevat. Het eerste wat we nodig hebben is het laden van een `.gba` bestand in geheugen.

```zig
const std = @import("std");

pub const GamePak = struct {
    rom: []const u8,
    allocator: std.mem.Allocator,
    rom_path: []const u8,

    pub fn load(allocator: std.mem.Allocator, rom_path: []const u8) !GamePak {
        const file = try std.fs.cwd().openFile(rom_path, .{});
        defer file.close();

        const rom = try file.readToEndAlloc(allocator, 32 * 1024 * 1024); // max 32 MB

        return GamePak{
            .rom = rom,
            .allocator = allocator,
            .rom_path = rom_path,
        };
    }

    pub fn deinit(self: *GamePak) void {
        self.allocator.free(self.rom);
    }
};
```

**Waarom een allocator meegeven?** Zig heeft geen globale allocator zoals `malloc` in C. Je geeft expliciet aan welke allocator geheugen beheert. Dit maakt het makkelijk om in tests een andere allocator te gebruiken (bijv. `testing.allocator` die geheugenleaks detecteert).

**Waarom `readToEndAlloc` met een limiet van 32 MB?** GBA ROM's zijn maximaal 32 MB. Door een limiet mee te geven bescherm je tegen corrupte of te grote bestanden. `readToEndAlloc` leest het hele bestand in één keer — dat is wat we willen, want de CPU moet willekeurig in de ROM kunnen lezen.

**Waarom `[]const u8` als type voor `rom`?** Een slice `[]const u8` is een pointer + lengte. De `const` geeft aan dat de ROM data niet gewijzigd mag worden — ROM is immers read-only. De lengte is belangrijk: we moeten weten hoe groot de ROM is om out-of-bounds reads te detecteren.

> **Tip:** Bewaar `rom_path` in de struct. In Story 2.3 heb je het pad nodig om het `.sav` bestand naast de ROM op te slaan.

### Stap 2: `src/core/bus.zig` — GamePak pointer toevoegen aan Bus

Nu verbinden we de Bus met de GamePak via een pointer. De Bus *bezit* de GamePak niet — hij verwijst ernaar.

```zig
const GamePak = @import("gamepak.zig").GamePak;

pub const Bus = struct {
    // ... bestaande velden ...
    gamepak: *GamePak,

    pub fn init(gamepak: *GamePak) Bus {
        var bus = Bus{
            .gamepak = gamepak,
            // ... overige velden initialiseren ...
        };
        @memset(&bus.ewram, 0);
        // ... etc
        return bus;
    }
};
```

**Waarom een pointer (`*GamePak`) en geen waarde (`GamePak`)?** De GamePak bevat de volledige ROM data (tot 32 MB). Als de Bus een waarde zou bevatten, zou die bij elke kopie meegekopieerd worden. Met een pointer deelt de Bus een referentie naar de GamePak die ergens anders leeft (namelijk in de GBA struct). Bovendien heeft de GBA struct straks zowel de Bus als de GamePak — als de Bus de GamePak zou *bevatten*, heb je een circulair ownership probleem.

### Stap 3: `src/core/bus.zig` — ROM reads in de switch toevoegen

Breid `read8` en `write8` uit met de GamePak ROM regio's:

```zig
pub fn read8(self: *const Bus, address: u32) u8 {
    const region = (address >> 24) & 0xF;
    return switch (region) {
        // ... bestaande regio's (0x00 - 0x07) ...
        0x08, 0x09 => self.gamepakRead(address),  // ROM mirror 0
        0x0A, 0x0B => self.gamepakRead(address),  // ROM mirror 1
        0x0C, 0x0D => self.gamepakRead(address),  // ROM mirror 2
        else => 0,
    };
}

fn gamepakRead(self: *const Bus, address: u32) u8 {
    // Offset binnen de ROM: mask met 0x1FFFFFF (32 MB = 25 bits)
    const offset = address & 0x1FFFFFF;
    if (offset < self.gamepak.rom.len) {
        return self.gamepak.rom[offset];
    }
    return 0; // Reads voorbij ROM-grootte geven 0 terug
}
```

**Waarom drie mirror-regio's (0x08-0x09, 0x0A-0x0B, 0x0C-0x0D)?** De GBA hardware mapt dezelfde ROM op drie opeenvolgende adresblokken. Sommige games en het BIOS gebruiken de mirrors (0x0A en 0x0C) voor waitstate-configuratie — elk mirror heeft verschillende timing. Voor correctheid moeten alle drie dezelfde data teruggeven.

**Waarom regio's in paren (0x08 + 0x09)?** Elke regio is 16 MB (24 bits). De ROM is max 32 MB, dus heeft twee 16 MB-regio's nodig. `0x08` dekt `0x08000000-0x08FFFFFF` en `0x09` dekt `0x09000000-0x09FFFFFF`.

**Waarom 0 teruggeven voorbij de ROM-grootte?** De echte GBA geeft "open bus" waarden terug (de laatst gelezen waarde op de bus). `0` is een goede start — de meeste games lezen niet voorbij de ROM. Open bus emulatie kun je later toevoegen als accuracy-verbetering.

**Waarom geen write naar ROM?** ROM is read-only. De `write8` switch heeft geen entries voor 0x08-0x0D. Writes naar ROM worden stilletjes genegeerd, net als op echte hardware.

### Stap 4: `src/core/gba.zig` — de GBA struct als verbindend geheel

De GBA struct is de top-level container die alle componenten samenvoegt. Dit is het "eigenaarschap"-patroon: de GBA struct *bezit* de data, en geeft pointers door aan wie het nodig heeft.

```zig
const std = @import("std");
const Bus = @import("bus.zig").Bus;
const GamePak = @import("gamepak.zig").GamePak;

pub const Gba = struct {
    bus: Bus,
    gamepak: GamePak,

    pub fn init(allocator: std.mem.Allocator, rom_path: []const u8) !Gba {
        var gba = Gba{
            .gamepak = try GamePak.load(allocator, rom_path),
            .bus = undefined, // wordt hieronder geïnitialiseerd
        };
        gba.bus = Bus.init(&gba.gamepak);
        return gba;
    }

    pub fn deinit(self: *Gba) void {
        self.gamepak.deinit();
    }
};
```

**Waarom `bus = undefined` en dan apart initialiseren?** Dit is het "twee-staps initialisatie" patroon in Zig. Het probleem: `Bus.init()` verwacht een pointer naar de GamePak, maar die leeft *in dezelfde struct*. Je kunt geen pointer nemen naar een veld van een struct die je nog aan het bouwen bent. Oplossing: maak de struct eerst met `bus = undefined`, en initialiseer de bus daarna — dan heb je een stabiel adres voor `&gba.gamepak`.

**Waarom niet `bus: Bus = undefined` als default?** Dat zou het ook oplossen, maar het is minder expliciet. Door het in de `init` functie te doen, is het duidelijk dat `bus` altijd geïnitialiseerd wordt voordat de struct gebruikt wordt.

> **Valkuil:** Let op dat je de `Gba` struct niet verplaatst nadat `Bus.init()` is aangeroepen. De bus bevat een pointer naar `gba.gamepak` — als je de gba struct kopieert of verplaatst, wijst die pointer naar ongeldig geheugen. Retourneer de Gba vanuit `init()` en sla hem op in een `var` die niet verplaatst wordt.

### Stap 5: `src/main.zig` — alles aan elkaar knopen

Update `main.zig` om de GBA struct te gebruiken en een ROM te laden:

```zig
const std = @import("std");
const Gba = @import("core/gba.zig").Gba;
const platform = @import("platform/renderer.zig");

pub fn main() !void {
    var gpa = std.heap.GeneralPurposeAllocator(.{}){};
    defer _ = gpa.deinit();
    const allocator = gpa.allocator();

    // Pak ROM-pad uit command line argumenten
    const args = try std.process.argsAlloc(allocator);
    defer std.process.argsFree(allocator, args);

    if (args.len < 2) {
        std.debug.print("Gebruik: gba-emu <rom.gba>\n", .{});
        return;
    }

    var gba = try Gba.init(allocator, args[1]);
    defer gba.deinit();

    // Lees de eerste instructie om te verifiëren dat ROM laden werkt
    const entry = gba.bus.read32(0x08000000);
    std.debug.print("ROM entry point: 0x{X:0>8}\n", .{entry});

    var renderer = platform.Renderer.init();
    defer renderer.deinit();

    while (platform.Renderer.processEvents()) {
        renderer.present();
    }
}
```

**Waarom `GeneralPurposeAllocator`?** Dit is de standaard allocator voor toepassingen in Zig. Hij detecteert geheugenleaks in debug-modus (als `gpa.deinit()` wordt aangeroepen). In release-modus is hij efficiënt. Later kun je overwegen om `std.heap.page_allocator` te gebruiken voor betere performance, maar GPA is prima voor nu.

**Waarom de entry point printen?** GBA ROM's beginnen met een ARM branch-instructie (meestal `0xEA00002E` of vergelijkbaar). Als je dit ziet in de output, weet je dat ROM laden en bus reads correct werken. Als je `0x00000000` ziet, is er iets mis.

### Stap 6: unit tests voor ROM reads

Voeg tests toe in `bus.zig` of een apart testbestand:

```zig
test "GamePak ROM read via bus" {
    // Maak een kleine nep-ROM
    var rom_data = [_]u8{ 0x78, 0x56, 0x34, 0x12 };

    var gamepak = GamePak{
        .rom = &rom_data,
        .allocator = std.testing.allocator,
        .rom_path = "test.gba",
    };

    var bus = Bus.init(&gamepak);

    // Lees individuele bytes
    try std.testing.expectEqual(@as(u8, 0x78), bus.read8(0x08000000));
    try std.testing.expectEqual(@as(u8, 0x56), bus.read8(0x08000001));

    // 32-bit little-endian read
    try std.testing.expectEqual(@as(u32, 0x12345678), bus.read32(0x08000000));

    // ROM mirror: 0x0A000000 leest dezelfde data
    try std.testing.expectEqual(@as(u8, 0x78), bus.read8(0x0A000000));

    // Read voorbij ROM-grootte geeft 0
    try std.testing.expectEqual(@as(u8, 0), bus.read8(0x08000010));
}
```

**Waarom een nep-ROM in de test?** Je wilt niet afhankelijk zijn van een echt ROM-bestand voor tests. Door een kleine array te gebruiken, test je de bus-logica zonder bestandsoperaties. Dit maakt de tests snel en reproduceerbaar.

> **Tip:** Als je `zig build test` draait en een "pointer to stack memory" error krijgt, controleer dan dat je `gamepak` als `var` declareert (niet `const`), zodat je er een mutable pointer naar kunt nemen.
