# Story 17.2 — Cycle-Exact PPU

**Epic:** 17 - Accuracy — Lage Impact

**Doel:** PPU rendering op pixel-niveau in plaats van scanline-niveau.

## Waarom normaal niet nodig?

De meeste GBA-games veranderen PPU registers alleen tijdens HBlank of VBlank. Scanline-accurate rendering (één keer per scanline renderen) is dan voldoende. Maar een handvol games wijzigen registers **midden in een scanline** — zonder CPU-level timing voor de PPU mis je die effecten.

**Waarom is scanline-accurate rendering de standaard?** Het is een uitstekende balans tussen nauwkeurigheid en performance. Een scanline renderen kost je renderer 1 functie-aanroep per 240 pixels. Cycle-exact rendering zou 240 aanroepen per scanline vereisen (of meer), wat ~240× meer overhead geeft. Aangezien 99% van de GBA-games geen mid-scanline effecten gebruikt, is die overhead niet gerechtvaardigd als standaard.

**Waarom voegt cycle-exact PPU enorm veel complexiteit toe?** In een scanline-accurate renderer render je een hele lijn in één keer: lees de registers, bereken 240 pixels, klaar. In een cycle-exacte renderer moet je bij *elke* write naar een PPU-register controleren of de PPU midden in een scanline zit, en zo ja, het al gerenderde stuk "afsluiten" en verder gaan met de nieuwe waarden. Dit verweeft de CPU- en PPU-emulatie op cycle-niveau, wat de code significant complexer maakt.

## Wanneer wel nodig?

- Games die registers wijzigen via exacte CPU-timing (geen DMA/interrupt)
- Extreme raster effects die niet op HBlank-grenzen vallen
- "Beam racing" technieken

In de praktijk zijn dit zeer weinig GBA-games. De meeste emulators (inclusief mGBA) zijn scanline-accurate, niet pixel-accurate.

## Concept

In plaats van één `render_scanline()` call per lijn, track je **op welk pixel de PPU is** en render je in stukjes:

```zig
const Ppu = struct {
    // Bestaande velden...
    dot: u16 = 0,  // huidige pixel binnen de scanline (0-307)

    /// Advance PPU tot een bepaald cycle-nummer
    fn advance_to(self: *Ppu, target_cycle: u64, bus: *Bus) void {
        while (self.current_cycle < target_cycle) {
            // Render één dot (pixel)
            if (self.scanline < 160 and self.dot < 240) {
                self.render_pixel(bus, self.dot, self.scanline);
            }

            self.dot += 1;
            self.current_cycle += 4;  // 1 dot = 4 cycles

            if (self.dot >= 308) {
                self.dot = 0;
                self.scanline += 1;
                if (self.scanline >= 228) {
                    self.scanline = 0;
                }
            }
        }
    }

    fn render_pixel(self: *Ppu, bus: *Bus, x: u16, y: u16) void {
        // Lees PPU registers op dit exacte moment
        // Render één pixel met de huidige instellingen
        const color = self.compute_pixel(bus, x, y);
        self.framebuffer[@as(u32, y) * 240 + x] = color;
    }
};
```

### Trade-off: snelheid vs accuraatheid

```
Scanline-accurate:   render 240 pixels per keer, ~60× sneller
Pixel-accurate:      render 1 pixel per keer, maximale accuraatheid

De meeste emulators kiezen scanline-accurate als standaard
met optionele pixel-accurate mode voor problematische games.
```

### Lazy rendering — het beste van beide werelden

Een slimmere aanpak: render normaal per scanline, maar als een register **midden in een scanline** geschreven wordt, render je het stuk tot dat punt en ga je verder met de nieuwe instellingen:

```zig
fn write_ppu_register(self: *Ppu, bus: *Bus, address: u32, value: u16) void {
    if (self.scanline < 160 and self.dot > 0 and self.dot < 240) {
        // Mid-scanline write! Render alles tot nu toe met oude waarden
        self.flush_scanline_partial(bus, self.dot);
    }

    // Nu pas de register-waarde updaten
    self.set_register(address, value);
}
```

Dit geeft pixel-level accuraatheid alleen wanneer nodig, zonder de constant overhead.

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — Dot counter toevoegen aan PPU struct

De eerste stap is het bijhouden van de exacte positie van de PPU binnen een scanline. Elke "dot" (pixel-klokpuls) is 4 CPU-cycles.

```zig
const Ppu = struct {
    // Bestaande velden...
    scanline: u16 = 0,

    // Nieuwe velden voor cycle-exact tracking:
    dot: u16 = 0,              // huidige pixel-positie (0-307)
    current_cycle: u64 = 0,    // absolute cycle-teller
    last_rendered_dot: u16 = 0, // tot waar we al gerenderd hebben op deze scanline

    // ... overige velden ...
};
```

**Waarom 308 dots per scanline?** Een GBA-scanline duurt 1232 CPU-cycles. Bij 4 cycles per dot is dat 308 dots. De eerste 240 dots zijn zichtbare pixels, dots 240-307 zijn de HBlank-periode. Het totale frame is 228 scanlines × 308 dots = 280896 cycles per frame, wat overeenkomt met ~59.7275 Hz.

**Waarom `last_rendered_dot` bijhouden?** Dit is essentieel voor de "lazy rendering" optimalisatie (stap 4). Wanneer een register midden in een scanline verandert, moet je weten welk deel al gerenderd is. `last_rendered_dot` vertelt je: "alles tot dit punt is al getekend met de oude waarden."

**Tip:** De dot counter is niet hetzelfde als de x-coördinaat. Dot 0 = pixel 0, maar dot 240 is al in HBlank (geen zichtbare pixel). Wees consistent in het onderscheid.

### Stap 2: `src/core/ppu.zig` — `advance_to()` functie: PPU synchroniseren met CPU

In plaats van de PPU elke cycle te updaten, synchroniseer je de PPU met de CPU wanneer dat nodig is (bijv. bij een register-write of aan het einde van een CPU-stap).

```zig
fn advance_to(self: *Ppu, target_cycle: u64, bus: *Bus) void {
    while (self.current_cycle < target_cycle) {
        if (self.scanline < 160 and self.dot < 240) {
            // Zichtbare pixel: render het
            self.render_pixel(bus, self.dot, self.scanline);
        }

        self.dot += 1;
        self.current_cycle += 4;  // 1 dot = 4 CPU cycles

        // Einde van de scanline?
        if (self.dot >= 308) {
            self.dot = 0;
            self.last_rendered_dot = 0;
            self.scanline += 1;

            if (self.scanline >= 228) {
                self.scanline = 0;  // nieuw frame
            }
        }

        // HBlank start bij dot 240
        if (self.dot == 240) {
            self.handle_hblank_start(bus);
        }
    }
}
```

**Waarom "advance to" in plaats van "tick per cycle"?** De CPU kan in een enkele instructie 1-20 cycles verbruiken. In plaats van de PPU 20 keer aan te roepen (eens per cycle), roep je `advance_to()` aan met het doel-tijdstip. De PPU verwerkt dan intern alle dots tot dat punt. Dit is efficiënter en geeft hetzelfde resultaat.

**Waarom de PPU synchroniseren bij register writes?** Dit is het kernidee: je hoeft de PPU niet elke cycle bij te houden. Je synchroniseert alleen wanneer iets verandert dat de PPU-output beïnvloedt. Bij een write naar bijv. BG0HOFS roep je eerst `ppu.advance_to(current_cpu_cycle)` aan, zodat alle pixels tot nu toe gerenderd zijn met de oude waarde. Daarna update je het register, en volgende pixels gebruiken de nieuwe waarde.

### Stap 3: `src/core/ppu.zig` — `render_pixel()` per dot

Deze functie rendert exact één pixel met de huidige PPU-instellingen.

```zig
fn render_pixel(self: *Ppu, bus: *Bus, x: u16, y: u16) void {
    // Lees PPU registers op dit exacte moment
    const bg_color = self.compute_bg_pixel(bus, x, y);
    const obj_color = self.compute_obj_pixel(bus, x, y);

    // Priority sorting en blending (zelfde logica als scanline renderer)
    const final_color = self.composite_single_pixel(bg_color, obj_color, x);

    // Schrijf naar framebuffer
    self.framebuffer[@as(u32, y) * 240 + x] = final_color;
}
```

**Waarom is dit zoveel trager dan scanline rendering?** Bij scanline rendering kun je veel werk delen: je berekent eenmalig welke tiles zichtbaar zijn, welke palette geladen is, etc. Bij per-pixel rendering moet je dit voor elke pixel opnieuw doen (of cachen en invalideren bij register-changes). De overhead van 240 functie-aanroepen per scanline (in plaats van 1) is significant.

**Waarom niet gewoon de scanline-renderer hergebruiken?** Dat kan deels, maar de scanline-renderer maakt aannames die bij per-pixel rendering niet gelden: bijv. dat de scroll offset constant is voor de hele lijn. Bij mid-scanline writes verandert de scroll offset halverwege, en de scanline-renderer zou de hele lijn opnieuw moeten renderen. De per-pixel renderer heeft dit probleem niet.

### Stap 4: `src/core/ppu.zig` — Lazy rendering: het beste van beide werelden

De slimste aanpak is een hybride: render normaal per scanline, maar schakel over naar per-pixel wanneer een mid-scanline write gedetecteerd wordt.

```zig
fn write_ppu_register(self: *Ppu, bus: *Bus, address: u32, value: u16, current_cycle: u64) void {
    // Synchroniseer de PPU tot het huidige tijdstip
    self.advance_to(current_cycle, bus);

    // Check of we midden in een zichtbare scanline zitten
    if (self.scanline < 160 and self.dot > 0 and self.dot < 240) {
        // Mid-scanline write! Flush het al gerenderde deel
        self.flush_partial_scanline(bus, self.last_rendered_dot, self.dot);
        self.last_rendered_dot = self.dot;
    }

    // Update het register met de nieuwe waarde
    self.set_register(address, value);
}

fn flush_partial_scanline(self: *Ppu, bus: *Bus, from_dot: u16, to_dot: u16) void {
    // Render pixels from_dot tot to_dot met de HUIDIGE (oude) register-waarden
    var x = from_dot;
    while (x < to_dot) : (x += 1) {
        self.render_pixel(bus, x, self.scanline);
    }
}
```

**Waarom is lazy rendering zo effectief?** In 99% van de gevallen worden PPU-registers niet midden in een scanline geschreven. Dan kost de `write_ppu_register()` functie bijna niets extra: alleen een check `dot > 0 and dot < 240` die false is (want de write gebeurt in HBlank of VBlank). Alleen wanneer er daadwerkelijk een mid-scanline write is, wordt het duurdere per-pixel pad genomen.

**Waarom "flush" tot het huidige dot?** Alle pixels tot `self.dot` zijn nog met de *oude* register-waarden getekend (of moeten dat nog worden). Door te flushen tot het huidige dot, garandeer je dat die pixels correct zijn. Pixels na het huidige dot worden later gerenderd met de *nieuwe* waarden.

### Stap 5: `src/core/ppu.zig` — Optionele mode met runtime-toggle

Maak cycle-exact rendering een optie die de gebruiker kan in- of uitschakelen.

```zig
const RenderMode = enum { scanline, cycle_exact };

const Ppu = struct {
    render_mode: RenderMode = .scanline,  // standaard: scanline-accurate

    fn end_of_scanline(self: *Ppu, bus: *Bus) void {
        if (self.render_mode == .scanline) {
            // Traditionele scanline rendering (snel)
            if (self.scanline < 160) {
                self.render_full_scanline(bus);
            }
        } else {
            // Cycle-exact: pixels zijn al gerenderd via advance_to()
            // Flush eventueel het resterende deel van de scanline
            if (self.scanline < 160 and self.last_rendered_dot < 240) {
                self.flush_partial_scanline(bus, self.last_rendered_dot, 240);
            }
        }
    }
};
```

**Waarom een runtime toggle?** De meeste spelers willen de snelheid van scanline rendering. Maar als een specifieke game mid-scanline effecten gebruikt en er visueel fout uitziet, kan de gebruiker overschakelen naar cycle-exact mode. Dit is dezelfde aanpak die mGBA en andere emulators gebruiken: een "accuracy vs performance" instelling.

**Tip:** In de lazy rendering aanpak (stap 4) is het onderscheid tussen de twee modes kleiner: de lazy renderer gedraagt zich als scanline-accurate wanneer er geen mid-scanline writes zijn, en schaalt automatisch op naar cycle-exact wanneer nodig. De toggle zou dan alleen de `advance_to()` overhead in- of uitschakelen.

**Valkuil:** In cycle-exact mode moet *elke* write naar een PPU-gerelateerd adres (niet alleen de "bekende" registers) door `write_ppu_register()` gaan. Dit omvat writes naar Palette RAM (0x05000000), VRAM (0x06000000), en OAM (0x07000000). Een mid-scanline write naar palette RAM (bijv. om een kleur te veranderen) heeft hetzelfde effect als een register-write.

## Acceptatiecriteria

- Mid-scanline register changes zijn zichtbaar
- Performance is acceptabel (lazy rendering als optimalisatie)
- Kan als optionele mode aan/uit gezet worden
