# Story 17.1 — Serial Communication, Link Cable & GPIO Sensoren

**Epic:** 17 - Accuracy — Lage Impact

**Doel:** Seriële communicatie en cartridge-sensoren emuleren.

## Context

## Waarom belangrijk?

De GBA heeft een seriële poort voor multiplayer en cartridges kunnen GPIO-sensoren bevatten. Hoewel de meeste single-player games prima werken zonder serial-emulatie, zijn er twee redenen om dit te implementeren:

**Waarom serial emuleren?** Sommige games controleren bij het opstarten of de seriële poort beschikbaar is, of proberen een link-partner te detecteren. Als je emulator niet antwoordt op serial I/O reads, kan de game hangen in een "wachten op verbinding" scherm of crashen. Een minimale stub die "geen verbinding" retourneert, lost dit op.

**Waarom GPIO-sensoren?** Een handvol cartridges bevat speciale hardware: de Boktai-serie heeft een zonnesensor, WarioWare Twisted heeft een tilt-sensor (gyroscoop), en Drill Dozer heeft een rumble-motor. Zonder sensor-emulatie zijn deze games onspeelbaar — niet vanwege crashes, maar omdat de game-invoer ontbreekt.

### Seriële communicatie (Link Cable)

```
Registers:
SIOCNT    (0x04000128) — Serial I/O control
SIODATA8  (0x0400012A) — 8-bit seriële data
SIODATA32 (0x04000120) — 32-bit seriële data
SIOMLT    (0x04000120) — Multiplayer data (4 spelers)
RCNT      (0x04000134) — GPIO mode select
```

Modes:
- **Normal**: 8-bit of 32-bit, master/slave
- **Multi-player**: tot 4 GBA's, elk stuurt 16 bits per transfer
- **UART**: standaard serieel (niet door games gebruikt)
- **GPIO**: direct pin control (gebruikt voor RTC, sensoren)

**Minimale emulatie:** de meeste single-player games werken als je:
1. SIOCNT reads beantwoordt met "geen verbinding"
2. Transfer-complete bit direct zet na een transfer start

```zig
const Serial = struct {
    siocnt: u16 = 0,

    pub fn read(self: *Serial, address: u32) u16 {
        return switch (address) {
            0x04000128 => self.siocnt & ~@as(u16, 0x80),  // bit 7 (start) altijd 0 = klaar
            else => 0,
        };
    }

    pub fn write(self: *Serial, address: u32, value: u16) void {
        switch (address) {
            0x04000128 => {
                self.siocnt = value;
                if (value & 0x80 != 0) {
                    // Transfer gestart → meteen als klaar markeren
                    self.siocnt &= ~@as(u16, 0x80);
                    // Optioneel: trigger serial IRQ als enabled
                }
            },
            else => {},
        }
    }
};
```

### Solar sensor (Boktai)

De solar sensor levert een analoge waarde (0-255) die zonlicht vertegenwoordigt:

```zig
const SolarSensor = struct {
    value: u8 = 0,     // 0=donker, 255=fel zonlicht
    counter: u8 = 0,   // interne teller
    clocking: bool = false,

    /// Game stuurt clock pulses, sensor retourneert bits
    fn clock(self: *SolarSensor) u1 {
        if (!self.clocking) return 1;
        self.counter +%= 1;
        // Retourneer 0 als counter < value, anders 1
        // Dit simuleert een ADC (analog-to-digital converter)
        return if (self.counter < self.value) 0 else 1;
    }
};
```

De speler zou de "zonlicht-intensiteit" via een UI-slider of toets kunnen instellen.

### Tilt sensor (WarioWare Twisted)

```zig
const TiltSensor = struct {
    x: i16 = 0,   // -512 tot +512 (links/rechts kantelen)
    y: i16 = 0,   // -512 tot +512 (voor/achter kantelen)

    // Mapped naar 0x080000C4-C8 via GPIO reads
    // Waarden omzetten naar ADC output (10-bit)
    fn read_x(self: *TiltSensor) u16 {
        return @intCast(@as(i32, self.x) + 0x3A0);  // center offset
    }
};
```

Input zou van muis-beweging of gamepad-gyro kunnen komen.

## Stap-voor-stap

### Stap 1: `src/core/serial.zig` — Minimale serial stub registers

Het belangrijkste is dat reads van serial-registers een bruikbare waarde retourneren. De meeste single-player games checken alleen SIOCNT.

```zig
const Serial = struct {
    siocnt: u16 = 0,
    siodata8: u8 = 0,
    siodata32: u32 = 0,
    rcnt: u16 = 0,

    pub fn read(self: *Serial, address: u32) u16 {
        return switch (address) {
            0x04000120 => @truncate(self.siodata32),         // SIODATA32 low
            0x04000122 => @truncate(self.siodata32 >> 16),   // SIODATA32 high
            0x04000128 => self.siocnt & ~@as(u16, 0x80),     // SIOCNT: bit 7 altijd 0
            0x0400012A => @as(u16, self.siodata8),            // SIODATA8
            0x04000134 => self.rcnt,                          // RCNT
            else => 0,
        };
    }
};
```

**Waarom bit 7 van SIOCNT altijd 0?** Bit 7 is het "start/busy" bit. Wanneer de game een transfer start (bit 7 = 1), controleert het daarna of de transfer klaar is (bit 7 = 0). Door bit 7 altijd als 0 te retourneren, denkt de game dat de transfer onmiddellijk is afgerond. Er is geen verbinding, dus er zijn geen bytes om te versturen.

**Waarom niet simpelweg alle reads als 0 retourneren?** SIOCNT bevat ook de mode-selectie bits en interrupt enable bits. Als de game deze schrijft en terugleest, verwacht het dezelfde waarden terug (minus het start bit). Door het hele register op te slaan en alleen bit 7 te maskeren, werkt dit correct.

### Stap 2: `src/core/serial.zig` — Transfer-complete logica

Wanneer de game een transfer start, markeer deze direct als voltooid en trigger optioneel een interrupt.

```zig
pub fn write(self: *Serial, address: u32, value: u16, irq: *Irq) void {
    switch (address) {
        0x04000128 => {
            self.siocnt = value;
            if (value & 0x80 != 0) {
                // Transfer gestart → direct als klaar markeren
                self.siocnt &= ~@as(u16, 0x80);

                // Trigger serial IRQ als enabled (bit 14)
                if (value & 0x4000 != 0) {
                    irq.request(.serial);
                }
            }
        },
        0x04000120 => self.siodata32 = (self.siodata32 & 0xFFFF0000) | value,
        0x04000122 => self.siodata32 = (self.siodata32 & 0x0000FFFF) | (@as(u32, value) << 16),
        0x04000134 => self.rcnt = value,
        else => {},
    }
}
```

**Waarom de interrupt triggeren?** Sommige games wachten niet actief op het start-bit, maar gebruiken een interrupt om te detecteren wanneer een transfer klaar is. Als je de interrupt niet triggert, hangt de game in een wacht-loop. Door de interrupt direct te triggeren (samen met het clearen van het start-bit), verloopt alles zoals de game verwacht.

**Waarom "direct klaar"?** Op echte hardware duurt een serial transfer een aantal clock cycles, afhankelijk van de baudrate. Maar omdat er geen verbinding is (en dus geen data om te versturen), is "direct klaar" het veiligste gedrag. De game krijgt lege data terug, wat het interpreteert als "geen verbinding".

### Stap 3: `src/core/gpio.zig` — GPIO framework met sensor-dispatch

Als je in story 16.1 al een GPIO-module hebt, breid deze uit met een dispatch-mechanisme voor verschillende sensortypen.

```zig
const GpioDevice = union(enum) {
    none,
    rtc: Rtc,
    solar: SolarSensor,
    tilt: TiltSensor,
    rumble: RumbleMotor,
};

const Gpio = struct {
    data: u4 = 0,
    direction: u4 = 0,
    control: u1 = 0,
    device: GpioDevice = .none,

    pub fn write(self: *Gpio, address: u32, value: u8) void {
        switch (address) {
            0x080000C4 => {
                self.data = @truncate(value);
                switch (self.device) {
                    .rtc => |*rtc| rtc.write_pins(self.data, self.direction),
                    .solar => |*solar| solar.write_pins(self.data, self.direction),
                    .tilt => |*tilt| tilt.write_pins(self.data, self.direction),
                    .rumble => |*rumble| rumble.set(self.data & 1 == 1),
                    .none => {},
                }
            },
            0x080000C6 => self.direction = @truncate(value),
            0x080000C8 => self.control = @truncate(value),
            else => {},
        }
    }
};
```

**Waarom een tagged union?** Elke cartridge heeft maximaal één type GPIO-device. Een `union(enum)` is de idiomatische Zig-manier om dit te modelleren: het kan een RTC, een solar sensor, een tilt sensor, of niets zijn. De `switch` op de union garandeert dat je elk geval afhandelt, en de compiler waarschuwt als je een variant vergeet.

**Waarom alle sensoren achter dezelfde GPIO-interface?** Op hardware-niveau communiceren alle sensoren via dezelfde 3-4 GPIO-pinnen op de cartridge-connector. Het verschil zit in het *protocol* dat over die pinnen gaat. Door een gemeenschappelijke `write_pins()` interface te definiëren, kun je het sensortype los koppelen van de GPIO-logica.

### Stap 4: `src/core/solar.zig` / `src/core/tilt.zig` — Optionele sensor-implementaties

Deze sensoren zijn optioneel maar maken specifieke games speelbaar.

```zig
// Solar sensor (Boktai): simuleert zonlicht-intensiteit
const SolarSensor = struct {
    value: u8 = 128,    // 0=donker, 255=fel (instelbaar via UI)
    counter: u8 = 0,    // interne ADC-teller
    old_clk: u1 = 0,    // voor clock edge detectie

    fn write_pins(self: *SolarSensor, data: u4, direction: u4) void {
        _ = direction;
        const clk = @as(u1, @truncate(data >> 0));
        const cs = @as(u1, @truncate(data >> 2));

        if (cs == 0) {
            self.counter = 0;
            return;
        }

        // Rising edge van clock: increment counter
        if (self.old_clk == 0 and clk == 1) {
            self.counter +%= 1;
        }
        self.old_clk = clk;
    }

    fn output_bit(self: *SolarSensor) u1 {
        // ADC: retourneer 0 als counter < value, anders 1
        return if (self.counter < self.value) 0 else 1;
    }
};
```

**Waarom een ADC-simulatie?** De echte solar sensor levert een analoog signaal dat door een ADC (Analog-to-Digital Converter) op de cartridge wordt omgezet. De game stuurt clock-pulsen en telt hoeveel pulsen nodig zijn voordat het signaal van 0 naar 1 flipt. Hoe meer pulsen, hoe donkerder. Door `value` (de "zonlicht-intensiteit") in te stellen via een UI-slider, kan de speler het gedrag simuleren.

**Waarom is de tilt sensor complexer?** De tilt sensor (ADXL202) levert X en Y waarden via een pulsbreedte-modulatie (PWM) protocol. De game meet de breedte van de puls om de kantelhoek te bepalen. In een emulator kun je de input koppelen aan muisbewegingen of gamepad-gyroscoop.

**Tip:** Begin met alleen de RTC (story 16.1) en voeg andere sensoren later toe. De solar sensor is het makkelijkst na de RTC, omdat het protocol simpeler is (alleen clock-pulsen tellen).
