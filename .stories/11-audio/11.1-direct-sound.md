# Story 11.1 — APU basis en Direct Sound

**Epic:** 11 - Audio (APU)

**Doel:** Audio output via SDL, DirectSound kanalen.

## Zig concepten

### Hoe werkt digitale audio?

Audio is een golf die door de lucht beweegt. Om dit digitaal weer te geven, neem je duizenden keren per seconde een "snapshot" van de golfhoogte — dit zijn **samples**. Hoe meer samples per seconde (sample rate), hoe beter de kwaliteit.

De GBA gebruikt 8-bit signed samples: waarden van -128 tot +127, waar 0 stilte is.

### FIFO — First In, First Out buffer

DirectSound gebruikt twee FIFO buffers (A en B). Een FIFO is als een wachtrij: data die er het eerst ingaat, komt er het eerst uit.

**Waarom FIFO buffers?** Audio moet op een vast tempo worden afgespeeld — elke ~30 microseconden een nieuw sample. Als de CPU elk sample individueel zou moeten aanleveren, zou hij niets anders meer kunnen doen. De FIFO ontkoppelt *productie* van *consumptie*: DMA vult de buffer in bulk (16 samples per keer), en de hardware leest er één sample per timer-tick uit. Zolang de buffer niet leegraakt, speelt de audio vloeiend af — ook als de CPU even bezig is met iets anders.

```zig
const Fifo = struct {
    buffer: [32]i8 = [_]i8{0} ** 32,  // 32 samples
    read_pos: u5 = 0,                  // waar lezen we?
    write_pos: u5 = 0,                 // waar schrijven we?
    count: u6 = 0,                     // hoeveel samples in de buffer?

    fn push(self: *Fifo, sample: i8) void {
        if (self.count >= 32) return;  // buffer vol
        self.buffer[self.write_pos] = sample;
        self.write_pos +%= 1;  // +%= is wrapping add: 31 + 1 = 0
        self.count += 1;
    }

    fn pop(self: *Fifo) i8 {
        if (self.count == 0) return 0;  // buffer leeg
        const sample = self.buffer[self.read_pos];
        self.read_pos +%= 1;
        self.count -= 1;
        return sample;
    }
};
```

### Wrapping arithmetic — `+%=`

Een FIFO buffer is circulair: na positie 31 ga je terug naar positie 0. In Zig gebruik je `+%=` (wrapping add) zodat dit automatisch gebeurt:

```zig
var pos: u5 = 31;  // u5 kan 0-31 opslaan
pos +%= 1;          // 31 + 1 wraps naar 0
// Normaal zou pos += 1 een overflow error geven in Zig!
```

### Array initialisatie met `**`

```zig
// Maak een array van 32 nullen:
const buffer: [32]i8 = [_]i8{0} ** 32;
// [_]i8{0} = één element: 0
// ** 32 = herhaal 32 keer
```

### Hoe DirectSound werkt op de GBA

1. De game schrijft audio samples naar FIFO A/B via het I/O register (0x040000A0 / 0x040000A4)
2. DMA vult de FIFO automatisch bij wanneer die leeg raakt
3. Timer 0 of 1 bepaalt wanneer het volgende sample wordt afgespeeld
4. Bij elke timer-tick wordt een sample uit de FIFO gehaald en naar de speaker gestuurd

**Waarom DMA?** Zonder DMA zou de CPU bij elke timer-tick handmatig een sample uit het geheugen moeten laden en naar de FIFO moeten schrijven — duizenden keren per seconde. DMA (Direct Memory Access) doet dit automatisch: wanneer de FIFO bijna leeg is, kopieert de DMA-controller zelfstandig een blok samples van het ROM/RAM naar de FIFO, zonder de CPU te onderbreken. Zo kan de CPU zich richten op gamelogica, physics en rendering terwijl de audio op de achtergrond doorloopt. Dit is essentieel voor een 16 MHz processor die al weinig rekenkracht heeft.

```zig
const DirectSound = struct {
    fifo_a: Fifo = .{},
    fifo_b: Fifo = .{},
    volume_a: u1 = 0,  // 0=50%, 1=100%
    volume_b: u1 = 0,
    timer_select_a: u1 = 0,  // welke timer stuurt FIFO A? (0=TM0, 1=TM1)
    timer_select_b: u1 = 0,
    enable_right_a: u1 = 0,
    enable_left_a: u1 = 0,
    enable_right_b: u1 = 0,
    enable_left_b: u1 = 0,

    fn on_timer_overflow(self: *DirectSound, timer_id: u1) i16 {
        var sample: i16 = 0;

        // FIFO A
        if (self.timer_select_a == timer_id) {
            const s = @as(i16, self.fifo_a.pop());
            sample += if (self.volume_a == 1) s else s >> 1;
        }

        // FIFO B
        if (self.timer_select_b == timer_id) {
            const s = @as(i16, self.fifo_b.pop());
            sample += if (self.volume_b == 1) s else s >> 1;
        }

        return sample;
    }
};
```

### SDL audio via AudioStream

SDL3 gebruikt een stream-gebaseerd audio model. Je opent een audio device stream met een callback:

**Waarom sample rate conversie nodig is:** De GBA draait audio typisch op ~32768 Hz (CPU klok / 512), maar SDL opent standaard een audio device op 44100 of 48000 Hz. Als je GBA-samples direct naar SDL stuurt zonder conversie, speelt de audio te snel af — alles klinkt als chipmunks. Je moet dus samples "oprekken" van 32768 Hz naar de SDL sample rate, bijvoorbeeld door tussenliggende samples te interpoleren.

```zig
// SDL3 vraagt om samples via een AudioStream callback
fn audio_callback(userdata: ?*anyopaque, stream: ?*c.SDL_AudioStream, additional_amount: c_int, total_amount: c_int) callconv(.c) void {
    _ = total_amount;
    const apu: *Apu = @ptrCast(@alignCast(userdata));
    const sample_count = @as(usize, @intCast(additional_amount)) / 2;  // 16-bit samples

    var buffer: [1024]i16 = undefined;
    for (0..sample_count) |i| {
        buffer[i] = apu.get_next_sample();
    }

    _ = c.SDL_PutAudioStreamData(stream, &buffer, additional_amount);
}
```

### `callconv(.c)` — C calling convention

SDL is een C library. Wanneer SDL jouw Zig functie aanroept, moet die de C-standaard volgen voor hoe argumenten worden doorgegeven. `callconv(.c)` vertelt de Zig compiler dit. Let op: vanaf Zig 0.15 zijn calling conventions lowercase (`.c` in plaats van `.C`).

In SDL3 is de audio callback signature gewijzigd: je krijgt een `SDL_AudioStream` pointer en een `additional_amount` (hoeveel bytes SDL nodig heeft). Je schrijft data naar de stream met `SDL_PutAudioStreamData`.

## Stap-voor-stap

### Stap 1: `src/core/apu.zig` — FIFO struct implementeren

Begin met het hart van DirectSound: de circulaire FIFO buffer. Dit is een op zichzelf staand onderdeel dat je los kunt testen.

```zig
const Fifo = struct {
    buffer: [32]i8 = [_]i8{0} ** 32,
    read_pos: u5 = 0,
    write_pos: u5 = 0,
    count: u6 = 0,

    fn push(self: *Fifo, sample: i8) void {
        if (self.count >= 32) return;
        self.buffer[self.write_pos] = sample;
        self.write_pos +%= 1;
        self.count += 1;
    }

    fn pop(self: *Fifo) i8 {
        if (self.count == 0) return 0;
        const sample = self.buffer[self.read_pos];
        self.read_pos +%= 1;
        self.count -= 1;
        return sample;
    }

    fn reset(self: *Fifo) void {
        self.read_pos = 0;
        self.write_pos = 0;
        self.count = 0;
    }
};
```

**Waarom 32 samples?** Dit is een hardware-specificatie van de GBA — de FIFO buffers zijn exact 32 bytes groot. Niet meer, niet minder. Games zijn geschreven met deze limiet in gedachten: DMA wordt getriggerd om de buffer bij te vullen voordat hij leegraakt.

**Waarom `u5` voor posities?** Een `u5` kan waarden 0–31 opslaan, precies de indices van een 32-element array. Bij `+%= 1` (wrapping add) loopt positie 31 automatisch over naar 0 — je krijgt circulair gedrag gratis van het type systeem, zonder `% 32` te hoeven schrijven. Dit is een kenmerkend Zig-patroon: gebruik het type om invarianten af te dwingen.

**Waarom `u6` voor count?** De count moet waarden 0–32 kunnen bevatten (de buffer kan 0 tot 32 samples bevatten). Een `u5` gaat maar tot 31, dus we hebben één extra bit nodig → `u6`.

**Tip:** Voeg ook een `reset()` functie toe. Games resetten de FIFO wanneer ze een nieuw geluid starten — zonder reset hoor je oude samples als artefacten.

**Valkuil:** Als `pop()` wordt aangeroepen op een lege buffer, geef dan 0 (stilte) terug, niet een error. In de echte hardware produceert een lege FIFO gewoon stilte. Een panic zou de hele emulator crashen elke keer dat de audio even niet snel genoeg wordt gevuld.

### Stap 2: `src/core/apu.zig` — DirectSound struct opzetten

De DirectSound struct beheert beide FIFO kanalen en hun configuratie. De configuratie komt uit het **SOUNDCNT_H** register (`0x04000082`).

```zig
const DirectSound = struct {
    fifo_a: Fifo = .{},
    fifo_b: Fifo = .{},
    // SOUNDCNT_H velden (0x04000082)
    volume_a: u1 = 0,        // bit 2:  0=50%, 1=100%
    volume_b: u1 = 0,        // bit 3:  0=50%, 1=100%
    enable_right_a: u1 = 0,  // bit 8:  FIFO A naar rechter speaker
    enable_left_a: u1 = 0,   // bit 9:  FIFO A naar linker speaker
    timer_select_a: u1 = 0,  // bit 10: 0=Timer0, 1=Timer1
    enable_right_b: u1 = 0,  // bit 12: FIFO B naar rechter speaker
    enable_left_b: u1 = 0,   // bit 13: FIFO B naar linker speaker
    timer_select_b: u1 = 0,  // bit 14: 0=Timer0, 1=Timer1

    fn write_soundcnt_h(self: *DirectSound, value: u16) void {
        self.volume_a = @truncate(value >> 2);
        self.volume_b = @truncate(value >> 3);
        self.enable_right_a = @truncate(value >> 8);
        self.enable_left_a = @truncate(value >> 9);
        self.timer_select_a = @truncate(value >> 10);
        self.enable_right_b = @truncate(value >> 12);
        self.enable_left_b = @truncate(value >> 13);
        self.timer_select_b = @truncate(value >> 14);

        // Bit 11: reset FIFO A, Bit 15: reset FIFO B
        if (value & (1 << 11) != 0) self.fifo_a.reset();
        if (value & (1 << 15) != 0) self.fifo_b.reset();
    }

    fn read_soundcnt_h(self: *const DirectSound) u16 {
        return @as(u16, self.volume_a) << 2 |
            @as(u16, self.volume_b) << 3 |
            @as(u16, self.enable_right_a) << 8 |
            @as(u16, self.enable_left_a) << 9 |
            @as(u16, self.timer_select_a) << 10 |
            @as(u16, self.enable_right_b) << 12 |
            @as(u16, self.enable_left_b) << 13 |
            @as(u16, self.timer_select_b) << 14;
    }
};
```

**Waarom aparte velden in plaats van één `u16`?** Je zou het hele register als één getal kunnen opslaan en steeds bitmaskers gebruiken. Maar losse velden maken de code leesbaarder: `self.volume_a` is duidelijker dan `(self.soundcnt_h >> 2) & 1`. Bovendien gebruik je de meeste velden individueel in de mix-logica — dan is directe toegang sneller en minder foutgevoelig.

**Waarom `@truncate`?** Na het shiften van een `u16` is het resultaat nog steeds een `u16`. `@truncate` snijdt het af tot `u1` — het behoudt alleen het laagste bit. Zig vereist dat je expliciet bent over dit soort conversies (geen impliciete narrowing).

**Tip:** De reset-bits (11 en 15) zijn write-only en "self-clearing" — ze worden niet opgeslagen. Bij het schrijven naar SOUNDCNT_H check je of deze bits gezet zijn en reset je de bijbehorende FIFO, maar je slaat de bits zelf niet op.

### Stap 3: `src/core/apu.zig` — Timer-overflow koppelen aan FIFO

Dit is waar de audio tot leven komt. Elke keer dat de geselecteerde timer overloopt, wordt een sample uit de FIFO gehaald en naar de output gestuurd.

```zig
fn on_timer_overflow(self: *DirectSound, timer_id: u1) struct { left: i16, right: i16 } {
    var left: i16 = 0;
    var right: i16 = 0;

    // FIFO A
    if (self.timer_select_a == timer_id) {
        const raw = @as(i16, self.fifo_a.pop());
        const s = if (self.volume_a == 1) raw else raw >> 1;
        if (self.enable_left_a == 1) left += s;
        if (self.enable_right_a == 1) right += s;
    }

    // FIFO B
    if (self.timer_select_b == timer_id) {
        const raw = @as(i16, self.fifo_b.pop());
        const s = if (self.volume_b == 1) raw else raw >> 1;
        if (self.enable_left_b == 1) left += s;
        if (self.enable_right_b == 1) right += s;
    }

    return .{ .left = left, .right = right };
}
```

**Waarom timer overflow = sample afspelen?** De timer telt omhoog met de CPU-klok (of een prescaler daarvan). Wanneer hij overloopt (van max naar 0), is er precies één "tick" verstreken. De game stelt de timer zo in dat hij overloopt met de gewenste sample rate. Bijvoorbeeld: voor 32768 Hz audio stel je de timer in op `0x10000 - (16777216 / 32768) = 0xFE00`. Elke overflow = "speel het volgende sample af".

**Hoe DMA-refill werkt:** Wanneer een FIFO sample wordt uitgelezen en de buffer bijna leeg is (minder dan 16 samples over), stuurt de hardware een signaal naar de DMA-controller. DMA kopieert dan 4 words (= 16 bytes = 16 samples) van het bronadres in het geheugen naar de FIFO. Dit gebeurt volledig in hardware — je emulator moet dit DMA-verzoek simuleren.

```zig
// In je timer overflow handler:
if (self.fifo_a.count <= 16) {
    // Trigger DMA1 voor FIFO A refill
    dma.trigger(.fifo_a);
}
if (self.fifo_b.count <= 16) {
    // Trigger DMA2 voor FIFO B refill
    dma.trigger(.fifo_b);
}
```

**Valkuil:** De DMA-trigger drempel is belangrijk. Als je te laat triggert (bij count == 0), krijg je stilte-gaten. Als je te vroeg triggert, kan de DMA data overschrijven die nog niet is afgespeeld. De hardware triggert wanneer een sample wordt geconsumeerd en de FIFO minder dan 16 samples heeft — emuleer dit zo nauwkeurig mogelijk.

**Tip:** Return zowel een `left` als `right` sample. DirectSound heeft per kanaal aparte enable-bits voor links en rechts — een geluid kan in één oor spelen (panning), in beide, of uitgeschakeld zijn.

### Stap 4: `src/platform/audio.zig` — SDL audio stream initialiseren

Nu koppelen we de GBA-audio aan de host-computer via SDL3.

```zig
const std = @import("std");
const c = @cImport({
    @cInclude("SDL3/SDL.h");
});
const Apu = @import("../core/apu.zig").Apu;

pub const Audio = struct {
    stream: *c.SDL_AudioStream,

    pub fn init(apu: *Apu) Audio {
        if (!c.SDL_Init(c.SDL_INIT_AUDIO)) {
            @panic("SDL audio init failed");
        }

        const spec = c.SDL_AudioSpec{
            .format = c.SDL_AUDIO_S16,  // 16-bit signed
            .channels = 2,               // stereo
            .freq = 48000,               // host sample rate
        };

        const stream = c.SDL_OpenAudioDeviceStream(
            c.SDL_AUDIO_DEVICE_DEFAULT_PLAYBACK,
            &spec,
            audio_callback,
            apu,  // userdata — pointer naar de Apu
        ) orelse @panic("Cannot open audio stream");

        _ = c.SDL_ResumeAudioStreamDevice(stream);

        return Audio{ .stream = stream };
    }

    pub fn deinit(self: *Audio) void {
        c.SDL_DestroyAudioStream(self.stream);
    }
};
```

**Waarom `SDL_AUDIO_S16` (16-bit signed)?** De GBA heeft 8-bit samples, maar we mixen twee kanalen (FIFO A + B). Twee 8-bit samples opgeteld kunnen een waarde tot ±254 bereiken — dat past niet meer in 8 bits. 16-bit geeft ons genoeg "headroom" voor mixing zonder clipping. Later, als je ook PSG-kanalen toevoegt (Story 11.2), heb je nog meer headroom nodig.

**Waarom 48000 Hz?** Dit is de standaard sample rate van de meeste moderne audio hardware. SDL kan intern resampling doen, maar het is beter om zelf te resampling voor meer controle (zie Stap 6).

**Waarom stereo (2 kanalen)?** DirectSound heeft aparte links/rechts enable-bits. Zelfs als je begint met mono-output, is het verstandig om meteen stereo op te zetten — anders moet je later de hele audio pipeline aanpassen.

**Tip:** Vergeet `SDL_ResumeAudioStreamDevice` niet! SDL3 start audio streams standaard in een gepauzeerde staat. Zonder resume hoor je niets en denk je dat je code niet werkt.

### Stap 5: `src/platform/audio.zig` — Audio callback implementeren

De callback is het hart van de audio pipeline. SDL roept deze functie aan wanneer het audio device klaar is voor nieuwe samples.

```zig
fn audio_callback(
    userdata: ?*anyopaque,
    stream: ?*c.SDL_AudioStream,
    additional_amount: c_int,
    total_amount: c_int,
) callconv(.c) void {
    _ = total_amount;
    const apu: *Apu = @ptrCast(@alignCast(userdata));

    // additional_amount is in bytes; elke sample is 2 bytes (i16), stereo = 4 bytes per frame
    const frame_count = @as(usize, @intCast(additional_amount)) / 4;

    var buffer: [2048]i16 = undefined;  // max 1024 stereo frames
    for (0..frame_count) |i| {
        const sample = apu.sample();  // geeft .left en .right terug
        buffer[i * 2] = sample.left;
        buffer[i * 2 + 1] = sample.right;
    }

    const bytes = frame_count * 4;
    _ = c.SDL_PutAudioStreamData(stream, &buffer, @intCast(bytes));
}
```

**Waarom een callback model (pull-based)?** SDL gebruikt een "pull" model: het audio device vraagt om data wanneer het klaar is, in plaats van dat jij data "pusht" wanneer je wilt. Dit heeft twee voordelen:
1. **Geen buffer-beheer aan jouw kant** — SDL bepaalt wanneer en hoeveel data het nodig heeft
2. **Geen timing-problemen** — je hoeft niet zelf bij te houden wanneer de audio buffer leeg raakt

**Hoe userdata veilig casten:** De `userdata` parameter is `?*anyopaque` (een optionele void pointer, equivalent aan C's `void*`). Met `@ptrCast(@alignCast(...))` zet je het terug naar `*Apu`. Dit is een onvermijdelijk patroon bij C-interop: C kent geen generics, dus callbacks werken altijd met void pointers.

**Valkuil: de callback draait op een aparte thread!** SDL roept de audio callback aan vanuit een achtergrond-thread. Dit betekent:
- **Geen allocaties** in de callback (allocators zijn meestal niet thread-safe)
- **Wees voorzichtig met gedeelde data** — als de emulator-thread en de audio-thread tegelijk de FIFO lezen/schrijven, kun je corrupte data krijgen
- **Eenvoudige oplossing voor nu:** gebruik `@atomicLoad` en `@atomicStore` voor de FIFO count, of accepteer dat er af en toe een sample verkeerd kan zijn (in de praktijk is dit nauwelijks hoorbaar)
- **Betere oplossing later:** een lock-free ring buffer tussen de emulator en de audio thread

### Stap 6: `src/core/apu.zig` — Sample rate conversie

De GBA produceert samples op ~32768 Hz, maar SDL verwacht 48000 Hz. Zonder conversie speelt de audio ~1.46x te snel af.

**Waarom is conversie nodig?** Stel je voor dat de GBA 32768 samples per seconde produceert, maar SDL verwacht er 48000. Als je elk GBA-sample één keer doorstuurt, ben je in 32768/48000 = 0.68 seconden door een seconde aan GBA-audio — alles klinkt versneld en te hoog. Je moet extra samples "verzinnen" om het gat te vullen.

**Eenvoudige aanpak — nearest-neighbor resampling:**

```zig
const Resampler = struct {
    // Verhouding: hoeveel GBA-samples per SDL-sample
    // gba_rate / sdl_rate = 32768 / 48000 ≈ 0.6827
    ratio: f64,
    position: f64 = 0.0,
    last_sample_left: i16 = 0,
    last_sample_right: i16 = 0,

    fn init(gba_rate: u32, sdl_rate: u32) Resampler {
        return .{
            .ratio = @as(f64, @floatFromInt(gba_rate)) / @as(f64, @floatFromInt(sdl_rate)),
        };
    }

    /// Geeft true terug als er een nieuw GBA-sample nodig is
    fn next_output_sample(self: *Resampler) struct { need_new: bool, left: i16, right: i16 } {
        self.position += self.ratio;
        if (self.position >= 1.0) {
            self.position -= 1.0;
            return .{ .need_new = true, .left = self.last_sample_left, .right = self.last_sample_right };
        }
        // Herhaal het vorige sample (nearest-neighbor)
        return .{ .need_new = false, .left = self.last_sample_left, .right = self.last_sample_right };
    }

    fn feed(self: *Resampler, left: i16, right: i16) void {
        self.last_sample_left = left;
        self.last_sample_right = right;
    }
};
```

**Betere aanpak — lineaire interpolatie:**

```zig
fn interpolate(self: *Resampler) struct { left: i16, right: i16 } {
    // position is het fractie-deel: hoe ver zijn we tussen twee GBA-samples?
    const frac: f64 = self.position;
    const left = @as(i16, @intFromFloat(
        @as(f64, @floatFromInt(self.prev_left)) * (1.0 - frac) +
        @as(f64, @floatFromInt(self.last_sample_left)) * frac
    ));
    const right = @as(i16, @intFromFloat(
        @as(f64, @floatFromInt(self.prev_right)) * (1.0 - frac) +
        @as(f64, @floatFromInt(self.last_sample_right)) * frac
    ));
    return .{ .left = left, .right = right };
}
```

Lineaire interpolatie "mengt" twee opeenvolgende samples op basis van hoe ver je ertussen zit. Dit geeft gladdere audio dan nearest-neighbor, dat harde sprongen kan veroorzaken.

**Valkuil:** Verkeerde sample rate conversie is de #1 oorzaak van rare audio-problemen:
- **Te hoge ratio** → audio speelt te langzaam af, klinkt diep en vervormd
- **Te lage ratio** → audio speelt te snel af, klinkt hoog als chipmunks
- **Geen conversie** → audio speelt op de verkeerde snelheid én de buffer raakt vol of leeg, wat crackling veroorzaakt

**Tip:** Begin met nearest-neighbor (simpeler te debuggen). Schakel pas over naar interpolatie als de basisfunctionaliteit werkt. Het verschil is subtiel — de meeste GBA-games gebruiken relatief lage kwaliteit audio waar nearest-neighbor prima voor volstaat.
