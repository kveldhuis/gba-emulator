# Story 13.1 — BIOS HLE (High-Level Emulation)

**Epic:** 13 - BIOS

**Doel:** GBA BIOS functies emuleren zonder het echte BIOS ROM nodig te hebben.

## Zig concepten

### Wat is het BIOS?

Het BIOS (Basic Input/Output System) is een klein stukje software ingebouwd in de GBA. Het bevat handige utility functies die games kunnen aanroepen, zoals wiskundige functies (delen, wortel) en geheugenoperaties (kopiëren, decompressie).

### SWI — Software Interrupt

Games roepen BIOS functies aan met de `SWI` instructie (Software Interrupt). Dit is vergelijkbaar met een functieaanroep, maar dan naar de BIOS:

```zig
// In ARM mode: SWI nummer zit in bits [23:8]
// In Thumb mode: SWI nummer zit in bits [7:0]
fn handle_swi(self: *Cpu, comment: u8) void {
    switch (comment) {
        0x00 => self.swi_soft_reset(),
        0x02 => self.swi_halt(),
        0x05 => self.swi_vblank_intr_wait(),
        0x06 => self.swi_div(),
        0x08 => self.swi_sqrt(),
        0x0B => self.swi_cpu_set(),
        0x0C => self.swi_cpu_fast_set(),
        0x11 => self.swi_lz77_uncomp(),
        else => {},  // onbekende SWI, negeer
    }
}
```

**Waarom moeten SWI nummers exact kloppen?** Games hebben specifieke SWI nummers hard-gecodeerd in hun machinecode. Als een game `SWI 0x06` uitvoert, verwacht die *precies* de Div-functie. Als jouw emulator een ander nummer toewijst aan Div, of SWI 0x06 niet afhandelt, dan krijgt de game verkeerde rekenresultaten terug — of crasht hij. Er is geen onderhandeling: het SWI nummer *is* de API. Dit is gedocumenteerd door Nintendo en elke GBA game vertrouwt erop.

**Waarom maakt ARM vs Thumb mode uit voor het SWI nummer?** De SWI instructie codeert het "comment field" (het functienummer) op een andere plek afhankelijk van de CPU-modus:

- **ARM mode** (32-bit instructies): het SWI nummer zit in bits [23:8] van de instructie. De onderste 8 bits worden genegeerd door de GBA BIOS, maar het nummer dat je nodig hebt zit in bits [23:16] — effectief byte 2 van de instructie.
- **Thumb mode** (16-bit instructies): het SWI nummer zit in bits [7:0] — de onderste byte van de instructie.

De GBA gebruikt in de praktijk altijd het bovenste byte (bits [23:16]) in ARM mode, wat overeenkomt met de onderste byte (bits [7:0]) in Thumb mode. Daarom kun je in beide gevallen hetzelfde 8-bit nummer extraheren en doorgeven aan `handle_swi`. Als je de verkeerde bits pakt, roep je de verkeerde BIOS functie aan — met onvoorspelbare gevolgen.

### HLE vs LLE

- **LLE** (Low-Level Emulation): je laadt het echte BIOS ROM bestand en laat de CPU het uitvoeren. Nauwkeurig maar je hebt het bestand nodig.
- **HLE** (High-Level Emulation): je schrijft de BIOS functies zelf in Zig. Sneller en geen BIOS bestand nodig.

**Waarom kiezen we voor HLE in dit project?** Er zijn vier sterke redenen:

1. **Geen auteursrechtproblemen.** Het echte GBA BIOS is auteursrechtelijk beschermd door Nintendo. Het distribueren of downloaden ervan is illegaal. Met HLE schrijf je je eigen implementatie — volledig legaal en je emulator werkt out-of-the-box zonder dat gebruikers een BIOS-bestand moeten zoeken.

2. **Snellere uitvoering.** Bij LLE moet de geëmuleerde ARM CPU elke BIOS-instructie één voor één uitvoeren — honderden ARM-instructies voor een simpele deelfunctie. Bij HLE voer je native Zig-code uit die direct compileert naar efficiënte x86/ARM64 machinecode. Een `@divTrunc` in Zig is één CPU-instructie; het emuleren van de BIOS Div-routine kost honderden geëmuleerde cycli.

3. **Makkelijker te debuggen.** Als een game crasht na een BIOS-aanroep, kun je bij HLE gewoon een breakpoint zetten in je Zig-code, logging toevoegen, of de input/output van registers printen. Bij LLE zou je door geëmuleerde ARM assembly moeten stappen — veel lastiger.

4. **De meeste games gebruiken maar een handvol functies.** Van de ~40 BIOS functies gebruiken de meeste games er slechts 5-10. Je hoeft dus niet alles te implementeren — begin met Div, Halt, CpuSet en LZ77, en voeg de rest toe wanneer een specifieke game erom vraagt.

### Parameters via registers

BIOS functies ontvangen parameters en geven resultaten terug via CPU-registers (R0-R3).

**Waarom registers in plaats van de stack?** Dit is de ARM calling convention voor BIOS-aanroepen (de BIOS Calling Convention, kortweg "BIOS ABI"). Er zijn twee praktische redenen:

1. **Snelheid.** Registers zitten *in* de CPU — lezen en schrijven kost nul wachtcycli. De stack daarentegen ligt in RAM-geheugen, wat op de GBA 2-3 keer langzamer is. Voor een veelgebruikte functie als Div maakt dat verschil.

2. **Eenvoud.** De SWI-instructie schakelt de CPU naar Supervisor mode en springt naar de BIOS. Met registers als interface hoeft de BIOS geen stackframe op te zetten of te ontleden — de parameters staan al klaar in R0-R3, en het resultaat wordt simpelweg teruggeschreven naar dezelfde registers. Dit maakt zowel de BIOS-implementatie als jouw HLE-code eenvoudiger.

In de praktijk: R0-R3 zijn de "argument/return" registers. R0 bevat meestal de eerste parameter en het eerste resultaat. Sommige functies (zoals Div) gebruiken meerdere output-registers (R0=quotiënt, R1=rest, R3=abs(quotiënt)).

```zig
// SWI 0x06: Div — signed integer divisie
// Input: R0 = teller (numerator), R1 = noemer (denominator)
// Output: R0 = quotient, R1 = rest, R3 = abs(quotient)
fn swi_div(self: *Cpu) void {
    const numerator: i32 = @bitCast(self.regs[0]);
    const denominator: i32 = @bitCast(self.regs[1]);

    if (denominator == 0) {
        // Division by zero — GBA BIOS gedrag
        return;
    }

    const quotient = @divTrunc(numerator, denominator);
    const remainder = @rem(numerator, denominator);

    self.regs[0] = @bitCast(quotient);
    self.regs[1] = @bitCast(remainder);
    self.regs[3] = @bitCast(if (quotient < 0) -quotient else quotient);
}
```

### @divTrunc en @rem — integer divisie in Zig

Zig heeft geen `/` operator voor signed integers (omdat het gedrag bij negatieve getallen onduidelijk is). In plaats daarvan gebruik je expliciete functies:

```zig
// @divTrunc: deelt en rondt af richting 0
@divTrunc(7, 2)    // = 3
@divTrunc(-7, 2)   // = -3  (niet -4!)

// @rem: rest na deling (zelfde teken als teller)
@rem(7, 2)         // = 1
@rem(-7, 2)        // = -1
```

### SWI 0x02: Halt — CPU pauzeren

Halt stopt de CPU tot een interrupt binnenkomt:

```zig
fn swi_halt(self: *Cpu) void {
    self.halted = true;
    // In de main loop:
    // if (cpu.halted) { skip instructie, check interrupts }
}
```

### SWI 0x0B: CpuSet — geheugen kopiëren/vullen

```zig
// R0 = source address, R1 = dest address, R2 = length/mode
fn swi_cpu_set(self: *Cpu, bus: *Bus) void {
    const src = self.regs[0];
    const dst = self.regs[1];
    const control = self.regs[2];

    const count = control & 0x1FFFFF;          // bits [20:0]: aantal woorden
    const fixed_source = (control >> 24) & 1;  // bit 24: 0=copy, 1=fill
    const word_size = (control >> 26) & 1;     // bit 26: 0=16-bit, 1=32-bit

    const size: u32 = if (word_size == 1) 4 else 2;

    var s = src;
    var d = dst;
    for (0..count) |_| {
        if (size == 4) {
            bus.write32(d, bus.read32(s));
        } else {
            bus.write16(d, bus.read16(s));
        }

        if (fixed_source == 0) s += size;  // copy: advance source
        // fill: source blijft hetzelfde
        d += size;
    }
}
```

### SWI 0x11: LZ77 decompressie

Veel GBA games comprimeren data met LZ77. De BIOS kan dit decomprimeren:

```zig
fn swi_lz77_uncomp(self: *Cpu, bus: *Bus) void {
    var src = self.regs[0];
    var dst = self.regs[1];

    // Header: 4 bytes (type + decompressed size)
    const header = bus.read32(src);
    const decompressed_size = header >> 8;
    src += 4;

    var written: u32 = 0;
    while (written < decompressed_size) {
        // Lees flag byte: elke bit bepaalt of volgende data
        // een letterlijke byte (0) of een referentie (1) is
        const flags = bus.read8(src);
        src += 1;

        for (0..8) |bit| {
            if (written >= decompressed_size) break;

            if ((flags >> @intCast(7 - bit)) & 1 == 1) {
                // Referentie: kopieer eerder geschreven data
                const ref = bus.read16(src);
                src += 2;
                const length = ((ref >> 4) & 0xF) + 3;
                const offset = ((ref & 0xF) << 8 | (ref >> 8)) + 1;

                for (0..length) |_| {
                    bus.write8(dst, bus.read8(dst - offset));
                    dst += 1;
                    written += 1;
                }
            } else {
                // Letterlijke byte: kopieer direct
                bus.write8(dst, bus.read8(src));
                src += 1;
                dst += 1;
                written += 1;
            }
        }
    }
}
```

## Stap-voor-stap

### Stap 1: `src/core/bios.zig` — SWI dispatcher opzetten

Maak een nieuw bestand `src/core/bios.zig` aan met de centrale `handleSwi` functie. Dit is het startpunt: wanneer de CPU een SWI-instructie decodeert, roept hij deze functie aan.

**Waarom een apart bestand?** Separation of concerns. De CPU-decoder (`cpu.zig`) is al complex genoeg — die hoeft niet te weten *hoe* elke BIOS-functie werkt. Door de BIOS-logica in een apart bestand te zetten, houd je `cpu.zig` overzichtelijk en kun je BIOS-functies onafhankelijk testen en uitbreiden.

```zig
const std = @import("std");
const Cpu = @import("cpu.zig").Cpu;
const Bus = @import("bus.zig").Bus;

pub fn handleSwi(cpu: *Cpu, bus: *Bus, comment: u8) void {
    switch (comment) {
        0x00 => swiSoftReset(cpu),
        0x02 => swiHalt(cpu),
        0x05 => swiVBlankIntrWait(cpu, bus),
        0x06 => swiDiv(cpu),
        0x08 => swiSqrt(cpu),
        0x0B => swiCpuSet(cpu, bus),
        0x0C => swiCpuFastSet(cpu, bus),
        0x11 => swiLz77UnComp(cpu, bus),
        else => {
            std.log.warn("Onbekende SWI: 0x{X:0>2}", .{comment});
        },
    }
}
```

**Hoe roept de CPU dit aan?** In je CPU-decoder, wanneer je een SWI-instructie tegenkomt:

```zig
// In cpu.zig, bij het decoderen van SWI:
// ARM mode: comment = (instruction >> 16) & 0xFF
// Thumb mode: comment = instruction & 0xFF
const comment: u8 = if (self.in_thumb_mode)
    @truncate(instruction & 0xFF)
else
    @truncate((instruction >> 16) & 0xFF);

bios.handleSwi(self, self.bus, comment);
```

**Waarom `(instruction >> 16) & 0xFF` voor ARM mode?** De SWI-instructie in ARM mode heeft het "comment field" in bits [23:0] — dat zijn 24 bits. Maar de GBA BIOS kijkt alleen naar de bovenste 8 bits daarvan (bits [23:16]). Dit is bits [23:16] van de volledige instructie, ofwel `(instruction >> 16) & 0xFF`. In Thumb mode is de instructie maar 16 bits breed en zit het nummer simpelweg in bits [7:0].

> **Tip:** Log onbekende SWI nummers met `std.log.warn`. Wanneer een game niet werkt, vertelt deze log precies welke BIOS-functies je nog moet implementeren. Dit is een van de krachtigste debugtools tijdens de ontwikkeling van een emulator.

### Stap 2: `src/core/bios.zig` — SWI 0x06 Div implementeren

Dit is de meest gebruikte BIOS-functie. Bijna elke game die wiskunde doet (en dat zijn ze allemaal) gebruikt SWI 0x06 voor signed integer divisie.

**Waarom biedt de BIOS een Div-functie aan?** De ARM7TDMI processor in de GBA heeft *geen hardware divide instructie*. Delen moet in software gebeuren. Nintendo heeft daarom een geoptimaliseerde deelroutine in de BIOS gezet die games kunnen hergebruiken.

```zig
fn swiDiv(cpu: *Cpu) void {
    const numerator: i32 = @bitCast(cpu.regs[0]);
    const denominator: i32 = @bitCast(cpu.regs[1]);

    if (denominator == 0) {
        // De echte GBA BIOS crasht niet bij deling door nul,
        // maar geeft ongedefinieerde waarden terug.
        // Sommige games triggeren dit per ongeluk — niet crashen!
        std.log.warn("SWI Div: deling door nul!", .{});
        return;
    }

    const quotient = @divTrunc(numerator, denominator);
    const remainder = @rem(numerator, denominator);

    cpu.regs[0] = @bitCast(quotient);
    cpu.regs[1] = @bitCast(remainder);
    cpu.regs[3] = @bitCast(if (quotient < 0) -quotient else quotient);
}
```

**Waarom `@bitCast`?** CPU-registers zijn `u32` (unsigned), maar Div werkt met *signed* getallen. `@bitCast` herinterpreteerd de bits zonder ze te veranderen: het getal `0xFFFFFFFF` als `u32` is 4294967295, maar als `i32` is het -1. Dit is exact wat we nodig hebben — de game heeft -1 in R0 gezet, en wij moeten het als -1 behandelen.

**Waarom `@divTrunc` en `@rem`?** Zig weigert de `/` en `%` operators voor signed integers omdat het afrondgedrag bij negatieve getallen verschilt per programmeertaal. De GBA BIOS rondt af richting nul (truncation), wat overeenkomt met `@divTrunc`. De rest heeft hetzelfde teken als de teller, wat overeenkomt met `@rem`.

**Edge case: deling door nul.** De echte GBA BIOS gaat in een oneindige lus bij deling door nul (het hangt). Sommige emulators geven gewoon 0 terug, anderen loggen een waarschuwing. Het belangrijkste is dat je emulator *niet crasht* — sommige games triggeren dit per ongeluk in edge cases.

> **Tip:** Test met bekende waarden om je implementatie te verifiëren:
> - `7 / 2 = 3` rest `1`, abs = `3`
> - `-7 / 2 = -3` rest `-1`, abs = `3`
> - `7 / -2 = -3` rest `1`, abs = `3`
> - `-7 / -2 = 3` rest `-1`, abs = `3`

### Stap 3: `src/core/bios.zig` — SWI 0x08 Sqrt implementeren

SWI 0x08 berekent de integer vierkantswortel: gegeven een getal in R0, geeft het de afgeronde wortel terug in R0.

**Waarom hebben games een wortel nodig?** Veel GBA games berekenen afstanden (bijv. "is de vijand binnen schootafstand?"). De afstand tussen twee punten is `sqrt(dx*dx + dy*dy)`. Zonder hardware floating point moet dit in integers, en de BIOS biedt een geoptimaliseerde integer sqrt.

**Simpele aanpak — iteratief:**

```zig
fn swiSqrt(cpu: *Cpu) void {
    const input = cpu.regs[0];

    // Simpel: tel op tot n*n > input
    var result: u32 = 0;
    while ((result + 1) * (result + 1) <= input) {
        result += 1;
    }

    cpu.regs[0] = result;
}
```

Dit werkt, maar is traag voor grote getallen (tot 65536 iteraties voor een 32-bit input).

**Betere aanpak — binair zoeken:**

```zig
fn swiSqrt(cpu: *Cpu) void {
    const input = cpu.regs[0];

    if (input == 0) {
        cpu.regs[0] = 0;
        return;
    }

    var low: u32 = 1;
    var high: u32 = 0xFFFF; // sqrt(0xFFFFFFFF) < 0x10000
    if (high > input) high = input;

    while (low <= high) {
        const mid = low + (high - low) / 2;
        if (mid <= input / mid) { // vermijd overflow: mid*mid kan > u32 max
            low = mid + 1;
        } else {
            high = mid - 1;
        }
    }

    cpu.regs[0] = high;
}
```

**Waarom `mid <= input / mid` in plaats van `mid * mid <= input`?** Overflow-preventie. Als `mid` groot is (bijv. 70000), dan is `mid * mid` = 4.900.000.000 — dat past niet in een `u32`. Door te delen in plaats van te vermenigvuldigen vermijd je dit probleem.

> **Tip:** De GBA BIOS Sqrt retourneert de *afgeronde naar beneden* wortel. Dus `sqrt(8)` = 2, niet 3. Test dit!

### Stap 4: `src/core/bios.zig` — SWI 0x0B/0x0C CpuSet en CpuFastSet

Dit zijn de werkpaarden van de GBA: geheugen kopiëren en vullen. Games gebruiken ze constant — VRAM leegmaken, sprites laden, achtergronden kopiëren.

**Verschil tussen CpuSet en CpuFastSet:**

| | CpuSet (0x0B) | CpuFastSet (0x0C) |
|---|---|---|
| Woordgrootte | 16-bit of 32-bit (instelbaar) | Altijd 32-bit |
| Count | Exact aantal | Moet veelvoud van 8 zijn |
| Snelheid | Normaal | Sneller (geoptimaliseerd voor grote blokken) |

**CpuSet implementatie:**

```zig
fn swiCpuSet(cpu: *Cpu, bus: *Bus) void {
    var src = cpu.regs[0];
    var dst = cpu.regs[1];
    const control = cpu.regs[2];

    const count = control & 0x1FFFFF;          // bits [20:0]
    const fixed_source = (control >> 24) & 1;  // bit 24: 0=copy, 1=fill
    const word_size = (control >> 26) & 1;     // bit 26: 0=halfword, 1=word

    const size: u32 = if (word_size == 1) 4 else 2;

    for (0..count) |_| {
        if (size == 4) {
            bus.write32(dst, bus.read32(src));
        } else {
            bus.write16(dst, bus.read16(src));
        }

        if (fixed_source == 0) src += size; // copy: bron schuift mee
        // fill: bron blijft staan (steeds dezelfde waarde)
        dst += size;
    }
}
```

**Waarom bestaat de "fill" modus (fixed_source)?** Dit wordt gebruikt om geheugen te vullen met één waarde. Bijvoorbeeld: VRAM leegmaken (vullen met nullen), of een achtergrondkleur instellen. In fill-modus leest de functie steeds van hetzelfde bronadres — effectief schrijft het dezelfde waarde naar een heel geheugengebied.

**CpuFastSet implementatie:**

```zig
fn swiCpuFastSet(cpu: *Cpu, bus: *Bus) void {
    var src = cpu.regs[0];
    var dst = cpu.regs[1];
    const control = cpu.regs[2];

    // CpuFastSet werkt altijd met 32-bit woorden
    var count = control & 0x1FFFFF;
    const fixed_source = (control >> 24) & 1;

    // Count MOET een veelvoud van 8 zijn — rond naar boven af
    count = (count + 7) & ~@as(u32, 7);

    for (0..count) |_| {
        bus.write32(dst, bus.read32(src));
        if (fixed_source == 0) src += 4;
        dst += 4;
    }
}
```

**Waarom moet de count een veelvoud van 8 zijn bij CpuFastSet?** De echte BIOS-implementatie kopieert intern 8 woorden per iteratie (een "burst" van 32 bytes). Dit is geoptimaliseerd voor de GBA's geheugenbus die het meest efficiënt is bij opeenvolgende toegangen. Als de count geen veelvoud van 8 is, rondt de BIOS naar boven af — je moet dit gedrag nabootsen, anders kopieer je te weinig data.

> **Valkuil:** CpuFastSet vereist dat zowel bron- als doeladres 4-byte aligned zijn. Als een game een niet-aligned adres doorgeeft, kan het gedrag onvoorspelbaar zijn. In de praktijk doen de meeste games dit correct, maar hou er rekening mee bij debugging.

### Stap 5: `src/core/bios.zig` — SWI 0x02/0x05 Halt en VBlankIntrWait

Deze functies laten de CPU slapen tot er een interrupt binnenkomt. Ze zijn cruciaal voor zowel correctheid als prestaties van je emulator.

**SWI 0x02: Halt**

```zig
fn swiHalt(cpu: *Cpu) void {
    cpu.halted = true;
}
```

Simpel, maar met grote impact. Wanneer `halted == true`, stopt de CPU met instructies uitvoeren. In je main loop:

```zig
// In de hoofdlus van de emulator:
if (cpu.halted) {
    // Sla instructie-uitvoering over.
    // Spring direct naar het volgende geplande event (bijv. HBlank, VBlank).
    scheduler.advance_to_next_event();
} else {
    cpu.step();
}

// Bij het afhandelen van een interrupt:
if (interrupt_pending and ime_enabled) {
    cpu.halted = false; // CPU wordt weer wakker
    cpu.handle_interrupt();
}
```

**Waarom is Halt belangrijk voor emulatorprestaties?** Zonder Halt-ondersteuning "spint" de CPU in een lege loop — hij voert duizenden NOP-achtige instructies uit terwijl hij wacht op een interrupt. Dat is verspilde rekentijd op je echte computer. Door te detecteren dat de CPU gehalted is en direct naar het volgende event te springen, bespaar je enorm veel werk. Dit kan het verschil zijn tussen 60 FPS en 600 FPS.

**SWI 0x05: VBlankIntrWait**

VBlankIntrWait is een combinatie van twee acties: het instellen van de juiste interrupt-vlaggen en dan halteren.

```zig
fn swiVBlankIntrWait(cpu: *Cpu, bus: *Bus) void {
    // 1. Zet de VBlank-interrupt aan in IE (Interrupt Enable)
    const ie = bus.read16(0x0400_0200);
    bus.write16(0x0400_0200, ie | 0x0001); // bit 0 = VBlank interrupt

    // 2. Zorg dat IME (Interrupt Master Enable) aan staat
    bus.write16(0x0400_0208, 1);

    // 3. Wis de VBlank-vlag in IntrCheckFlag (BIOS RAM op 0x0300_7FF8)
    // Dit adres wordt door de interrupt handler op 1 gezet als VBlank is opgetreden
    bus.write16(0x0300_7FF8, 0);

    // 4. Halt de CPU
    cpu.halted = true;
}
```

**Waarom gebruiken games VBlankIntrWait?** De VBlank-periode (wanneer het scherm niet getekend wordt) is het veilige moment om VRAM te updaten. Games wachten hierop met VBlankIntrWait. Het is efficiënter dan Halt alleen, omdat het ook de juiste interrupts configureert.

> **Valkuil:** VBlankIntrWait werkt samen met de interrupt handler. De interrupt handler moet de vlag op adres `0x0300_7FF8` zetten wanneer een VBlank-interrupt optreedt. Als je dit vergeet, blijft de game eeuwig hangen in de halt-toestand. Veel "game hangt bij opstarten"-bugs komen door een incompleet VBlankIntrWait.

**Hoe integreer je dit met je scheduler?** Wanneer de CPU gehalted is:

1. Bereken het tijdstip van het eerstvolgende event (bijv. volgende HBlank, timer overflow, of VBlank).
2. Zet `current_cycle` direct naar dat tijdstip — sla alle tussenliggende cycli over.
3. Verwerk het event. Als het een interrupt triggert en IME+IE staan goed, maak de CPU wakker.

### Stap 6: `src/core/bios.zig` — SWI 0x11 LZ77 decompressie

Veel GBA games comprimeren hun grafische data (sprites, achtergronden, tilemaps) met LZ77 om ruimte te besparen op het 32MB cartridge. De BIOS kan deze data in-place decomprimeren naar WRAM of VRAM.

**Hoe werkt LZ77?**

LZ77 is een simpel maar effectief compressie-algoritme. Het idee: als een reeks bytes eerder al voorkwam, verwijs je terug naar die eerdere plek in plaats van de bytes opnieuw op te slaan.

**De structuur van LZ77-gecomprimeerde data op de GBA:**

```
[Header: 4 bytes] [Flag byte] [Data blokken...] [Flag byte] [Data blokken...] ...
```

- **Header** (4 bytes): byte 0 = type (moet `0x10` zijn voor LZ77), bytes 1-3 = gedecomprimeerde grootte (24 bits, little-endian).
- **Flag byte**: 8 bits, elk bit beschrijft het volgende datablok:
  - Bit = `0`: het volgende blok is een **letterlijke byte** (1 byte, kopieer direct).
  - Bit = `1`: het volgende blok is een **referentie** (2 bytes: lengte + offset naar eerder geschreven data).

**Voorbeeld doorlopen:**

Stel je hebt de gecomprimeerde bytes: `0x00, 0x41, 0x42, 0x43, 0x44, 0x45, 0x46, 0x47, 0x48`

- Flag byte = `0x00` = `0b00000000` → alle 8 blokken zijn letterlijke bytes.
- De output is: `A B C D E F G H` (de ASCII-waarden van 0x41-0x48).

Nu een flag byte van `0x80` = `0b10000000`:
- Bit 7 = 1 → eerste blok is een referentie.
- De 2 referentiebytes coderen een lengte en offset: "kopieer 5 bytes vanaf 3 posities terug".
- De overige 7 bits = 0 → letterlijke bytes.

```zig
fn swiLz77UnComp(cpu: *Cpu, bus: *Bus) void {
    var src = cpu.regs[0];
    var dst = cpu.regs[1];

    const header = bus.read32(src);
    src += 4;

    // Check type — onderste 8 bits moeten 0x10 zijn (LZ77)
    const compression_type = header & 0xFF;
    if (compression_type != 0x10) {
        std.log.warn("LZ77: onverwacht type 0x{X:0>2}, verwacht 0x10", .{compression_type});
        return;
    }

    const decompressed_size = header >> 8;
    var written: u32 = 0;

    while (written < decompressed_size) {
        const flags = bus.read8(src);
        src += 1;

        // Verwerk 8 blokken per flag byte (MSB eerst)
        for (0..8) |bit| {
            if (written >= decompressed_size) break;

            if ((flags >> @intCast(7 - bit)) & 1 == 1) {
                // Referentie: 2 bytes → lengte + offset
                const byte1 = bus.read8(src);
                const byte2 = bus.read8(src + 1);
                src += 2;

                const length: u32 = ((@as(u32, byte1) >> 4) & 0xF) + 3;
                const offset: u32 = ((@as(u32, byte1) & 0xF) << 8 | @as(u32, byte2)) + 1;

                for (0..length) |_| {
                    if (written >= decompressed_size) break;
                    bus.write8(dst, bus.read8(dst - offset));
                    dst += 1;
                    written += 1;
                }
            } else {
                // Letterlijke byte: kopieer direct
                bus.write8(dst, bus.read8(src));
                src += 1;
                dst += 1;
                written += 1;
            }
        }
    }
}
```

**Waarom `+ 3` bij de lengte?** Een referentie is alleen nuttig als het *korter* is dan de data die het vervangt. Een referentie kost 2 bytes. Een letterlijke byte kost 1 byte. Dus een referentie is pas efficiënt als het minstens 3 bytes vervangt. Daarom is de minimale referentielengte 3, en slaat het compressie-formaat `length - 3` op. Bij decompressie tel je er 3 bij op.

**Waarom `+ 1` bij de offset?** Offset 0 zou betekenen "kopieer van de huidige schrijfpositie" — maar daar is nog niets geschreven (of het is de byte die je nu aan het schrijven bent). Dat is nutteloos. Offset 1 betekent "kopieer de laatst geschreven byte", wat wél zinvol is (herhalende patronen). Daarom slaat het formaat `offset - 1` op, en tel je er 1 bij op.

> **Valkuil:** De header bevat het compressietype in de onderste 8 bits. Check dat dit `0x10` is — andere waarden (zoals `0x20` voor Huffman of `0x30` voor Run-Length) vereisen een ander decompressie-algoritme. Als je dit niet checkt en de data is geen LZ77, zal je decompressor willekeurig geheugen overschrijven.

> **Tip:** LZ77 decompressie is een goede kandidaat voor unit tests. Maak een klein gecomprimeerd blok met de hand, decompress het, en controleer of de output klopt. Dit is veel sneller dan een hele game opstarten om te testen.
