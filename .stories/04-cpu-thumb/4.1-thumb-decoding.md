# Story 4.1 — Thumb instructie decodering

**Epic:** 04 - CPU — Thumb Instructies

**Doel:** 16-bit Thumb instructies decoderen via een comptime lookup table.

## Waarom Thumb?

De ARM7TDMI processor in de GBA ondersteunt twee instructiesets: ARM (32-bit) en Thumb (16-bit). Thumb bestaat om een specifiek probleem op te lossen: **code size**.

- **16-bit instructies = halve code size** — een Thumb programma neemt ~65% van de ruimte in van hetzelfde ARM programma. Dit is cruciaal voor de GBA: de ROM is beperkt en 16-bit breed.
- **Betere cache/bus performance** — de GBA Game Pak ROM bus is slechts 16-bit breed. Een ARM instructie (32-bit) kost dus 2 bus-reads, terwijl een Thumb instructie (16-bit) in 1 read past. Thumb code draait daardoor **sneller** vanuit ROM.
- **Thumb is een subset van ARM** — het kan minder (alleen R0-R7, altijd flags zetten, beperktere operaties), maar voor de meeste code is dat genoeg.

In de praktijk draait ~90% van GBA game-code in Thumb mode. Alleen performance-kritieke routines (die vanuit het snelle IWRAM draaien) gebruiken ARM.

## Zig concepten

### Verschil met ARM decodering

Thumb instructies zijn 16-bit in plaats van 32-bit. De decodering is simpeler:
- Geen condition codes (behalve bij conditional branch)
- De bovenste 8 bits (of soms 6) bepalen het instructietype
- Lookup table van 256 entries (op bits [15:8]) is voldoende

**Waarom 256 entries?** Bij ARM gebruiken we 4096 entries (12 bits). Bij Thumb zijn 8 bits genoeg omdat het instructieformaat compacter is — de bovenste 8 bits bevatten altijd genoeg informatie om het formaat te bepalen. 8 bits = 2⁸ = 256 mogelijke waarden. Dit geeft O(1) decodering: één array-lookup in plaats van een keten van if/else checks.

```zig
const ThumbHandler = *const fn (*Cpu, *Bus, u16) void;

const thumb_lut = comptime blk: {
    var table: [256]ThumbHandler = undefined;
    for (0..256) |i| {
        table[i] = decode_thumb(@intCast(i));
    }
    break :blk table;
};

pub fn execute_thumb(cpu: *Cpu, bus: *Bus, instruction: u16) void {
    const index = instruction >> 8;
    thumb_lut[index](cpu, bus, instruction);
}
```

### Thumb formaten overzicht

De bovenste bits bepalen het formaat:

```
Bits [15:13]  Bits [15:8] range   Format
000           00-07               1: Move shifted register
001           08-1F               2: Add/Sub  &  3: Mov/Cmp/Add/Sub immediate
010000        40-43               4: ALU operations
010001        44-47               5: Hi register / BX
01001         48-4F               6: PC-relative load
0101          50-5F               7/8: Load/store register offset
011           60-7F               9: Load/store immediate offset
1000          80-8F               10: Load/store halfword
1001          90-9F               11: SP-relative load/store
1010          A0-AF               12: Load address
1011          B0-BF               13/14: SP offset / Push-Pop
1100          C0-CF               15: Multiple load/store
1101          D0-DF               16: Conditional branch / SWI
11100         E0-E7               18: Unconditional branch
11110/11111   F0-FF               19: Long branch with link
```

### Comptime functie voor de decode tabel

```zig
fn decode_thumb(index: u8) ThumbHandler {
    return switch (index) {
        0x00...0x07 => move_shifted_register,
        0x08...0x0F => add_sub,
        0x10...0x1F => mov_cmp_add_sub_imm,
        0x20...0x3F => mov_cmp_add_sub_imm,
        0x40...0x43 => alu_operations,
        0x44...0x47 => hi_register_bx,
        0x48...0x4F => pc_relative_load,
        // ... etc
        else => thumb_undefined,
    };
}
```

### Range switch syntax in Zig

```zig
switch (value) {
    0x00...0x07 => {},  // match 0 t/m 7
    0x08...0x0F => {},  // match 8 t/m 15
    else => {},
}
```

`...` in een switch is een range: alle waarden van links t/m rechts.

## Stap-voor-stap

### Stap 1: `src/core/thumb.zig` — bestand aanmaken met imports

Maak het Thumb module bestand aan. Dit wordt de tegenhanger van `arm.zig` voor 16-bit instructies.

```zig
const std = @import("std");
const Cpu = @import("cpu.zig").Cpu;
const Bus = @import("bus.zig").Bus;
```

**Waarom een apart bestand?** Thumb en ARM zijn twee verschillende instructiesets met eigen decoderingslogica. Door ze te scheiden houd je elk bestand behapbaar (~300-500 regels) en kun je ze onafhankelijk testen. De `cpu.zig` roept de juiste module aan op basis van de T-bit in het CPSR.

### Stap 2: `ThumbHandler` type definiëren — de function pointer

```zig
pub const ThumbHandler = *const fn (*Cpu, *Bus, u16) void;
```

Dit is het type voor elke handler-functie in de lookup table. Let op de verschillen met de ARM handler:
- **`u16`** in plaats van `u32` — Thumb instructies zijn 16-bit
- Verder identiek: elke handler krijgt een pointer naar de CPU, de Bus, en de ruwe instructie

**Waarom function pointers?** Elke entry in de lookup table is een pointer naar de functie die dat specifieke instructieformaat afhandelt. Zo hoef je bij het uitvoeren geen switch/if meer te doen — je springt direct naar de juiste code.

### Stap 3: `decode_thumb()` comptime functie — het hart van de decodering

Deze functie wordt **alleen tijdens compilatie** aangeroepen om de lookup table te vullen. Voor elke mogelijke waarde van bits [15:8] (0-255) retourneert hij de juiste handler.

```zig
fn decode_thumb(index: u8) ThumbHandler {
    return switch (index) {
        // Format 1: Move shifted register (LSL/LSR/ASR met immediate)
        0x00...0x07 => move_shifted_register,

        // Format 2: Add/Subtract (register en 3-bit immediate)
        0x08...0x0F => add_sub,   // onderscheid: 0x18-0x1F bij sommige implementaties

        // Format 3: Mov/Cmp/Add/Sub met 8-bit immediate
        0x10...0x1F => mov_cmp_add_sub_imm,
        0x20...0x3F => mov_cmp_add_sub_imm,

        // Format 4: ALU operaties (16 operaties op lage registers)
        0x40...0x43 => alu_operations,

        // Format 5: Hi register operaties en BX
        0x44...0x47 => hi_register_bx,

        // Format 6: PC-relative load
        0x48...0x4F => pc_relative_load,

        // Formats 7 & 8: Load/store met register offset
        0x50...0x5F => load_store_register_offset,

        // Format 9: Load/store met immediate offset
        0x60...0x7F => load_store_immediate_offset,

        // Format 10: Load/store halfword
        0x80...0x8F => load_store_halfword,

        // Format 11: SP-relative load/store
        0x90...0x9F => sp_relative_load_store,

        // Format 12: Load address (PC of SP + offset)
        0xA0...0xAF => load_address,

        // Formats 13 & 14: SP offset aanpassen / Push-Pop
        0xB0...0xBF => sp_offset_push_pop,

        // Format 15: Multiple load/store (LDMIA/STMIA)
        0xC0...0xCF => multiple_load_store,

        // Format 16: Conditional branch
        // Format 17: SWI (0xDF)
        0xD0...0xDF => conditional_branch_swi,

        // Format 18: Unconditional branch
        0xE0...0xE7 => unconditional_branch,

        // Format 19: Long branch with link (BL, twee-instructie sequentie)
        0xF0...0xFF => long_branch_link,

        // Ongebruikte ranges
        else => thumb_undefined,
    };
}
```

**Waarom `0x00...0x07` voor Format 1?** Kijk naar de bit-layout: Format 1 heeft `000` op bits [15:13]. De bovenste 8 bits (bits [15:8]) lopen dan van `0000_0000` (0x00) tot `0000_0111` (0x07). Elk range in de switch correspondeert exact met de mogelijke waarden van bits [15:8] voor dat formaat.

**Tip:** Controleer dat alle 256 waarden (0x00-0xFF) gedekt zijn. Zig's `switch` met `else` vangt de rest op, maar het is goed om te verifiëren dat je geen formaat mist. Een veelgemaakte fout is dat ranges overlappen of gaten hebben.

### Stap 4: `thumb_lut` — de comptime lookup table genereren

```zig
pub const thumb_lut: [256]ThumbHandler = comptime blk: {
    @setEvalBranchQuota(512);
    var table: [256]ThumbHandler = undefined;
    for (0..256) |i| {
        table[i] = decode_thumb(@intCast(i));
    }
    break :blk table;
};
```

**Waarom `@setEvalBranchQuota`?** Zig beperkt standaard het aantal stappen dat comptime code mag uitvoeren (om oneindige loops te voorkomen). Met 256 iteraties × de switch in `decode_thumb` kun je tegen het limiet aanlopen. `@setEvalBranchQuota(512)` verhoogt het limiet.

**Waarom `comptime`?** De tabel wordt volledig **tijdens compilatie** berekend. In het uiteindelijke binary staat een vaste array van 256 function pointers — geen runtime overhead. Dit is hetzelfde patroon als bij de ARM lookup table, maar kleiner (256 vs 4096 entries).

### Stap 5: `execute_thumb()` — de publieke entry point

```zig
pub fn execute_thumb(cpu: *Cpu, bus: *Bus, instruction: u16) void {
    const index: u8 = @truncate(instruction >> 8);
    thumb_lut[index](cpu, bus, instruction);
}
```

**Waarom `instruction >> 8`?** We gebruiken de bovenste 8 bits (bits [15:8]) als index. Door 8 posities naar rechts te shiften, worden bits [15:8] de waarde 0-255.

**Waarom `@truncate`?** Na de shift is het resultaat nog steeds een `u16`. `@truncate` converteert het naar `u8` zodat het als array-index werkt. Zig vereist expliciete type-conversies — er zijn geen impliciete casts.

### Stap 6: stub handlers aanmaken — placeholder per formaat

Maak voor elk formaat een stub die later wordt ingevuld:

```zig
fn move_shifted_register(cpu: *Cpu, bus: *Bus, instruction: u16) void {
    _ = .{ cpu, bus, instruction };
    @panic("TODO: Thumb Format 1 - Move shifted register");
}

fn add_sub(cpu: *Cpu, bus: *Bus, instruction: u16) void {
    _ = .{ cpu, bus, instruction };
    @panic("TODO: Thumb Format 2 - Add/Sub");
}

// ... herhaal voor elk formaat ...

fn thumb_undefined(cpu: *Cpu, bus: *Bus, instruction: u16) void {
    _ = .{ cpu, bus, instruction };
    @panic("Undefined Thumb instruction");
}
```

**Waarom `_ = .{ cpu, bus, instruction };`?** Zig geeft een compile error als je functie-parameters niet gebruikt. Met `_ = .{ ... }` markeer je ze expliciet als "bewust ongebruikt". Dit is nodig voor stubs die later worden ingevuld.

**Waarom `@panic` in plaats van niets doen?** Een stille no-op zou bugs verbergen. Als je per ongeluk een nog niet-geïmplementeerde instructie uitvoert, wil je dat direct zien. `@panic` stopt het programma met een duidelijke foutmelding.

### Stap 7: integratie in `cpu.zig` — T-bit check toevoegen

In de `step()` functie van de CPU moet je de T-bit checken om te beslissen of ARM of Thumb decodering gebruikt wordt:

```zig
const thumb = @import("thumb.zig");
const arm = @import("arm.zig");

pub fn step(self: *Cpu, bus: *Bus) void {
    if (self.cpsr.t == 1) {
        // Thumb mode: 16-bit instructies, PC loopt met +2
        const instruction = bus.read16(self.r[15] - 4);
        thumb.execute_thumb(self, bus, instruction);
        self.r[15] +%= 2;
    } else {
        // ARM mode: 32-bit instructies, PC loopt met +4
        const instruction = bus.read32(self.r[15] - 8);
        arm.execute_arm(self, bus, instruction);
        self.r[15] +%= 4;
    }
}
```

**Waarom `self.r[15] - 4` bij Thumb?** De ARM7TDMI heeft een 3-stage pipeline (fetch → decode → execute). Als een instructie wordt uitgevoerd, wijst PC al 2 instructies verder. Bij Thumb (16-bit = 2 bytes per instructie) is dat `PC + 4`. Dus om de huidige instructie te lezen: `PC - 4`.

**Waarom `+%= 2`?** De `+%=` operator is wrapping addition — het voorkomt overflow-panics als de PC bij het maximum komt. In de praktijk zal dit niet gebeuren, maar Zig's standaard `+=` op unsigned integers zou panicken bij overflow.
