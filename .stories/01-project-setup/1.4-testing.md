# Story 1.4 — Testen met Zig's ingebouwde test framework

**Epic:** 01 - Project Setup & Infrastructuur

**Doel:** Van begin af aan tests schrijven voor emulator-componenten.

## Zig concepten

### Test blocks — tests naast je code

In Zig schrijf je tests direct in hetzelfde bestand als je code. Elk `test` blok is een onafhankelijke test.

**Waarom tests in hetzelfde bestand?** In veel talen heb je aparte test-bestanden (bijv. `test_bus.py`). Zig kiest een andere aanpak: tests staan *naast* de code die ze testen. Het voordeel: als je een functie aanpast, zijn de tests letterlijk een paar regels verderop. Geen zoektocht door een apart test-bestand. De compiler verwijdert alle `test` blokken uit release builds, dus ze kosten niks in productie.

```zig
const std = @import("std");
const expect = std.testing.expect;

fn add(a: u32, b: u32) u32 {
    return a +% b;
}

test "add normale waarden" {
    try expect(add(2, 3) == 5);
}

test "add wrapping overflow" {
    try expect(add(0xFFFFFFFF, 1) == 0);
}
```

- `test "naam"` — definieert een test met een beschrijvende naam
- `std.testing.expect(conditie)` — faalt als de conditie `false` is
- `try` voor `expect` — omdat expect een error teruggeeft bij falen

### Tests draaien

```bash
# Draai alle tests in het hele project:
zig build test

# Draai tests van één specifiek bestand:
zig test src/core/bus.zig
```

Om `zig build test` te laten werken, moet je een test step toevoegen in `build.zig`:

```zig
pub fn build(b: *std.Build) void {
    const target = b.standardTargetOptions(.{});
    const optimize = b.standardOptimizeOption(.{});

    // ... exe setup (bestaande code) ...

    // Test step
    const unit_tests = b.addTest(.{
        .root_module = b.createModule(.{
            .root_source_file = b.path("src/main.zig"),
            .target = target,
            .optimize = optimize,
        }),
    });
    const run_unit_tests = b.addRunArtifact(unit_tests);
    const test_step = b.step("test", "Run unit tests");
    test_step.dependOn(&run_unit_tests.step);
}
```

### expectEqual — betere foutmeldingen

`expect` geeft alleen "test failed", maar `expectEqual` laat zien wat je verwachtte vs. wat je kreeg:

```zig
test "memory read/write" {
    var bus = Bus.init();
    bus.write8(0x03000000, 0x42);
    try std.testing.expectEqual(@as(u8, 0x42), bus.read8(0x03000000));
    // Bij falen: "expected 0x42, found 0x00"
}
```

### Test patronen voor een emulator

**Patroon 1: Memory bus tests**

```zig
test "IWRAM read/write roundtrip" {
    var bus = Bus.init();
    bus.write32(0x03000000, 0x12345678);
    try std.testing.expectEqual(@as(u32, 0x12345678), bus.read32(0x03000000));
}

test "BIOS is read-only" {
    var bus = Bus.init();
    bus.write8(0x00000000, 0xFF);  // schrijf naar BIOS
    try std.testing.expectEqual(@as(u8, 0), bus.read8(0x00000000));  // ongewijzigd
}

test "little-endian read16" {
    var bus = Bus.init();
    bus.write8(0x03000000, 0x78);  // lage byte
    bus.write8(0x03000001, 0x56);  // hoge byte
    try std.testing.expectEqual(@as(u16, 0x5678), bus.read16(0x03000000));
}
```

**Patroon 2: CPU instructie tests**

```zig
test "ARM MOV R0, #42" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    // MOV R0, #42 = 0xE3A0002A
    // cond=AL(0xE), opcode=MOV(0xD), S=0, Rd=R0, imm=42
    const instruction: u32 = 0xE3A0002A;
    arm.execute(&cpu, &bus, instruction);
    try std.testing.expectEqual(@as(u32, 42), cpu.r[0]);
}

test "ARM ADD sets zero flag" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    cpu.r[0] = 5;
    cpu.r[1] = 0;
    // SUBS R2, R0, R0 (R2 = R0 - R0 = 0, zet Z flag)
    const instruction: u32 = 0xE0502000;
    arm.execute(&cpu, &bus, instruction);
    try std.testing.expectEqual(@as(u1, 1), cpu.cpsr.z);
}
```

**Patroon 3: Packed struct roundtrip**

```zig
test "PSR bitCast roundtrip" {
    const psr = PSR{
        .mode = 0x1F, .t = 0, .f = 0, .i = 0,
        ._reserved = 0, .v = 1, .c = 0, .z = 1, .n = 0,
    };
    const raw: u32 = @bitCast(psr);
    const back: PSR = @bitCast(raw);
    try std.testing.expectEqual(@as(u1, 1), back.z);
    try std.testing.expectEqual(@as(u1, 1), back.v);
    try std.testing.expectEqual(@as(u5, 0x1F), back.mode);
}
```

### Tip: test met bekende instructie-encodings

De ARM Architecture Reference Manual bevat exacte bit-patronen voor elke instructie. Gebruik die als test-input. Voorbeeld:

```
MOV R0, #1    → 0xE3A00001
ADD R0, R1, R2 → 0xE0810002
B #8          → 0xEA000000
```

Je kunt ook hex-waarden van bestaande ROMs gebruiken om te verifiëren dat je decoder correct werkt.

## Stap-voor-stap

### Stap 1: `build.zig` — test step toevoegen

Voeg de test-configuratie toe aan je bestaande `build.zig`, ná de executable setup:

```zig
pub fn build(b: *std.Build) void {
    const target = b.standardTargetOptions(.{});
    const optimize = b.standardOptimizeOption(.{});

    // ... bestaande exe setup ...

    // === Test step toevoegen ===
    const unit_tests = b.addTest(.{
        .root_module = b.createModule(.{
            .root_source_file = b.path("src/main.zig"),
            .target = target,
            .optimize = optimize,
        }),
    });
    const run_unit_tests = b.addRunArtifact(unit_tests);
    const test_step = b.step("test", "Run unit tests");
    test_step.dependOn(&run_unit_tests.step);
}
```

**Waarom `b.addTest` in plaats van `b.addExecutable`?**
- `addTest` bouwt een speciaal test-binary dat alle `test` blokken in je code vindt en uitvoert. Het is geen gewone executable — het draait alleen de tests en rapporteert het resultaat.

**Waarom weer `b.createModule(...)` met `root_source_file`?**
- Net als bij de executable, moet je in Zig 0.15+ een module aanmaken. De `root_source_file` verwijst naar `src/main.zig`, en Zig volgt automatisch alle `@import`-ketens om tests in alle geïmporteerde bestanden te ontdekken.

**Let op:** Zig vindt alleen tests in bestanden die (direct of indirect) geïmporteerd worden vanuit `root_source_file`. Als je een nieuw bestand toevoegt maar het nergens importeert, worden de tests in dat bestand **niet** meegenomen. Zorg dus altijd dat elk bestand ergens in de import-keten zit.

**Tip: meerdere test-targets**

Als je later bestanden hebt die niet vanuit `main.zig` bereikbaar zijn (bijv. standalone utilities), kun je meerdere test-targets toevoegen:
```zig
const extra_tests = b.addTest(.{
    .root_module = b.createModule(.{
        .root_source_file = b.path("src/core/bus.zig"),
        .target = target,
        .optimize = optimize,
    }),
});
const run_extra_tests = b.addRunArtifact(extra_tests);
test_step.dependOn(&run_extra_tests.step); // zelfde test_step hergebruiken
```

### Stap 2: `src/core/bus.zig` — eerste tests schrijven

Begin met de meest fundamentele tests: memory read/write roundtrips. Deze tests valideren dat de memory bus correct werkt — de basis waar alles op bouwt.

```zig
const std = @import("std");
const expect = std.testing.expect;
const expectEqual = std.testing.expectEqual;

// ... Bus struct en implementatie ...

test "IWRAM read/write roundtrip u8" {
    var bus = Bus.init();
    bus.write8(0x03000000, 0x42);
    try expectEqual(@as(u8, 0x42), bus.read8(0x03000000));
}

test "IWRAM read/write roundtrip u32" {
    var bus = Bus.init();
    bus.write32(0x03000000, 0x12345678);
    try expectEqual(@as(u32, 0x12345678), bus.read32(0x03000000));
}
```

**Waarom `expectEqual` in plaats van `expect`?**
- `expect(a == b)` geeft bij falen alleen "expected true, found false" — nutteloos voor debugging.
- `expectEqual(expected, actual)` geeft: "expected 0x42, found 0x00" — je ziet meteen wat er mis is.

**Waarom `@as(u8, 0x42)` als type-hint?**
- Zig's type inference kan soms niet bepalen welk type een literal heeft. `0x42` kan een `u8`, `u16`, `u32`, of `comptime_int` zijn. `@as(u8, 0x42)` vertelt de compiler expliciet: "dit is een u8". Zonder deze hint krijg je een "ambiguous" compilerfout bij `expectEqual`.

**Waarom `try` voor `expectEqual`?**
- `expectEqual` geeft een error terug als de waarden niet gelijk zijn. Dat is hoe Zig's test framework weet dat een test gefaald is. De `try` propageert die error naar het test framework.

### Stap 3: `src/core/bus.zig` — edge cases testen

Test de grenzen van je implementatie. Dit zijn de tests die bugs vroeg opvangen:

```zig
test "BIOS regio is read-only" {
    var bus = Bus.init();
    bus.write8(0x00000000, 0xFF); // probeer naar BIOS te schrijven
    try expectEqual(@as(u8, 0), bus.read8(0x00000000)); // waarde ongewijzigd
}

test "little-endian read16 van twee losse bytes" {
    var bus = Bus.init();
    bus.write8(0x03000000, 0x78); // lage byte eerst (little-endian)
    bus.write8(0x03000001, 0x56); // hoge byte
    try expectEqual(@as(u16, 0x5678), bus.read16(0x03000000));
}

test "little-endian write16 leest als twee bytes" {
    var bus = Bus.init();
    bus.write16(0x03000000, 0xABCD);
    try expectEqual(@as(u8, 0xCD), bus.read8(0x03000000)); // lage byte
    try expectEqual(@as(u8, 0xAB), bus.read8(0x03000001)); // hoge byte
}
```

**Waarom little-endian tests zo belangrijk zijn:**
- De GBA (ARM7TDMI) is een **little-endian** processor: de laagste byte staat op het laagste adres. Als je dit verkeerd implementeert, ziet elke multi-byte waarde er gespiegeld uit. Dit soort bugs zijn later extreem moeilijk te vinden, omdat alles "bijna" werkt.

**Waarom BIOS read-only testen?**
- Het BIOS-geheugen (0x00000000-0x00003FFF) is op de echte GBA alleen-lezen. Als je emulator hier schrijfacties toestaat, kunnen ROMs per ongeluk het BIOS overschrijven — wat tot bizarre crashes leidt.

### Stap 4: tests draaien en verifiëren

```bash
# Draai alle tests
zig build test

# Verwachte output bij succes:
# All 5 passed.

# Bij een falende test zie je:
# Test [1/5] test "IWRAM read/write roundtrip u8"...
# FAIL: expected 0x42, found 0x00
# Tests failed.
```

**Tip: één specifiek bestand testen tijdens development**

Als je aan een enkel bestand werkt, is het sneller om alleen dat bestand te testen:
```bash
zig test src/core/bus.zig
```
Dit compileert en runt alleen de tests in `bus.zig` (en zijn imports), zonder het hele project te bouwen. Veel snellere feedback-loop.

**Tip: tests als documentatie**

Elke test is ook documentatie van hoe een component hoort te werken. Geef tests beschrijvende namen die het *gedrag* beschrijven, niet de implementatie:
- ✅ `"little-endian read16 van twee losse bytes"` — beschrijft wat er moet gebeuren
- ❌ `"test read16"` — zegt niks over het verwachte gedrag

### Stap 5: tests importeerbaar maken via `main.zig`

Zorg dat alle modules met tests bereikbaar zijn vanuit de import-keten van `main.zig`:

```zig
// In src/main.zig — voeg toe voor test-discovery:
test {
    _ = @import("core/bus.zig");
    // Voeg hier toekomstige modules toe:
    // _ = @import("core/cpu.zig");
    // _ = @import("core/arm.zig");
}
```

**Waarom een `test` blok in `main.zig` met `_ = @import(...)`?**
- Zig's test runner vindt alleen tests in bestanden die vanuit de `root_source_file` geïmporteerd worden. Als `bus.zig` niet direct of indirect door `main.zig` geïmporteerd wordt in de normale code, worden de tests overgeslagen.
- Door een `test` blok met imports toe te voegen, worden die bestanden *alleen* tijdens het testen meegenomen — ze beïnvloeden de normale executable niet.
- De `_ =` vertelt Zig: "importeer dit bestand (zodat de tests gevonden worden), maar gebruik de waarde niet."
