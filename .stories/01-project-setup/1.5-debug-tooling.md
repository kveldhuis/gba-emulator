# Story 1.5 — Debug tooling en logging

**Epic:** 01 - Project Setup & Infrastructuur

**Doel:** Debug-hulpmiddelen inrichten zodat je problemen kunt opsporen wanneer een ROM niet werkt.

## Zig concepten

### Conditional compilation met comptime

Je wilt debug-output alleen in debug-builds, niet in release.

**Waarom is dit cruciaal voor een emulator?** Een emulator voert miljoenen instructies per seconde uit. Zelfs een simpele `if (trace_enabled)` check op *runtime* kost cycles. Met `comptime` evaluatie beslist de *compiler* of de debug-code meegenomen wordt — in release builds bestaat de code letterlijk niet, alsof je het nooit geschreven hebt.

```zig
const builtin = @import("builtin");
const is_debug = builtin.mode == .Debug;

fn log(comptime fmt: []const u8, args: anytype) void {
    if (comptime is_debug) {
        std.debug.print(fmt, args);
    }
    // In release builds bestaat deze code niet eens — de compiler gooit het weg
}
```

### std.log — gestructureerd loggen

Zig heeft een ingebouwd log-systeem met niveaus:

```zig
const std = @import("std");
const log = std.log.scoped(.cpu);

// Gebruik:
log.debug("PC={x:0>8} instr={x:0>8}", .{ cpu.r[15], instruction });
log.warn("onbekende instructie: {x:0>8}", .{instruction});
log.err("illegale geheugen-access op {x:0>8}", .{address});
```

- `.debug` — detailinfo, alleen zichtbaar in debug-builds
- `.warn` — mogelijke problemen
- `.err` — fouten
- `scoped(.cpu)` — prefix "[cpu]" voor alle berichten van deze module

### Format specifiers voor hex output

```zig
std.debug.print("{x:0>8}\n", .{@as(u32, 0x1234)});
// Output: "00001234"
// x = hexadecimaal
// 0>8 = pad met nullen, rechts uitgelijnd, 8 tekens breed
```

### CPU instruction tracing

De krachtigste debug-tool voor een emulator: log elke instructie die de CPU uitvoert.

```zig
const trace_enabled = false;  // zet op true om tracing aan te zetten

pub fn step(self: *Cpu, bus: *Bus) void {
    if (comptime trace_enabled) {
        self.print_state();
    }

    // ... normale fetch/decode/execute ...
}

fn print_state(self: *Cpu) void {
    // Registers
    std.debug.print("R0={x:0>8} R1={x:0>8} R2={x:0>8} R3={x:0>8}\n", .{
        self.r[0], self.r[1], self.r[2], self.r[3],
    });
    std.debug.print("R4={x:0>8} R5={x:0>8} R6={x:0>8} R7={x:0>8}\n", .{
        self.r[4], self.r[5], self.r[6], self.r[7],
    });
    std.debug.print("R12={x:0>8} SP={x:0>8} LR={x:0>8} PC={x:0>8}\n", .{
        self.r[12], self.r[13], self.r[14], self.r[15],
    });

    // Flags
    const cpsr: u32 = @bitCast(self.cpsr);
    std.debug.print("CPSR={x:0>8} [{}{}{}{}{}]\n", .{
        cpsr,
        @as(u8, if (self.cpsr.n == 1) 'N' else '-'),
        @as(u8, if (self.cpsr.z == 1) 'Z' else '-'),
        @as(u8, if (self.cpsr.c == 1) 'C' else '-'),
        @as(u8, if (self.cpsr.v == 1) 'V' else '-'),
        @as(u8, if (self.cpsr.t == 1) 'T' else '-'),
    });
}
```

### Comptime booleans als feature flags

```zig
// Bovenaan cpu.zig:
const config = struct {
    const trace_cpu = false;       // log elke instructie
    const trace_memory = false;    // log elke memory read/write
    const trace_interrupts = true; // log interrupt events
    const break_on_undefined = true; // stop bij onbekende instructie
};

// In je code:
if (comptime config.trace_memory) {
    log.debug("read8({x:0>8}) = {x:0>2}", .{ address, value });
}
```

Omdat dit `comptime` is, wordt de code in release builds volledig weggeoptimaliseerd — geen runtime overhead.

### Vergelijken met test ROMs

Er bestaan speciale test-ROMs die systematisch CPU-instructies testen:

```
Populaire GBA test ROMs:
- arm.gba / thumb.gba     — ARM/Thumb instructie tests
- memory.gba              — memory timing tests
- hello.gba               — simpelste homebrew (tekst op scherm)
- tonc demos              — graphics mode demos (Mode 3, tiles, sprites)
```

**Debug-workflow:**
1. Draai een test ROM
2. Als het niet werkt: zet CPU tracing aan
3. Vergelijk de trace output met een bekende goede emulator (bijv. mGBA)
4. Zoek het punt waar jouw output afwijkt
5. Fix de instructie of het geheugengedrag

### Breakpoints in code

Zig heeft `@breakpoint()` — stopt de CPU op dat punt (als je in een debugger zit):

```zig
if (instruction == 0xDEADBEEF) {
    @breakpoint();  // trigger debugger breakpoint
}
```

Handig om te stoppen op een specifieke instructie of geheugen-adres.

### unreachable — onbereikbare code markeren

```zig
fn decode(instruction: u32) void {
    switch (get_type(instruction)) {
        .data_processing => handle_dp(instruction),
        .branch => handle_branch(instruction),
        // ... alle types ...
        _ => unreachable,  // als dit bereikt wordt, is er een bug in de decoder
    }
}
```

`unreachable` crasht het programma met een duidelijke foutmelding. Gebruik dit voor situaties die nooit mogen voorkomen — het helpt je bugs sneller te vinden.

**Waarom `unreachable` en niet een `@panic("...")`?**
- `unreachable` vertelt de compiler dat dit pad *onmogelijk* is. In debug-builds genereert dit een crash met locatie-informatie. In release-builds is het **undefined behavior** — de compiler optimaliseert alsof dit pad niet bestaat, wat snellere code oplevert.
- `@panic("...")` crasht altijd met je bericht, ook in release builds. Gebruik `@panic` als het pad *mogelijk* is maar niet *gewenst*; gebruik `unreachable` als het pad een bug aanduidt.

## Stap-voor-stap

### Stap 1: `src/core/cpu.zig` — comptime config flags definiëren

Begin met een config struct bovenaan het bestand. Dit wordt het centrale punt voor alle debug-opties:

```zig
const std = @import("std");
const log = std.log.scoped(.cpu);

const config = struct {
    const trace_cpu = false;           // log elke instructie
    const break_on_undefined = true;   // stop bij onbekende instructie
};
```

**Waarom `comptime` constanten in een struct?**
- Door de flags in een `const struct` te zetten, worden het **comptime bekende waarden**. De compiler evalueert `if (comptime config.trace_cpu)` tijdens compilatie en verwijdert de hele `if`-body als de flag `false` is. Er is letterlijk **nul runtime overhead** — de debug-code bestaat niet in het uiteindelijke binary.
- Een alternatief zou `var trace_cpu: bool = false` zijn (runtime toggle), maar dat kost een branch bij elke instructie — miljoenen keren per seconde. Voor een emulator is dat onacceptabel.

**Waarom `std.log.scoped(.cpu)`?**
- `scoped(.cpu)` prefix alle logberichten met `[cpu]`. Wanneer je straks ook `.bus`, `.ppu`, en `.timer` scopes hebt, kun je in de output snel filteren welke component een bericht genereert. Veel handiger dan overal `std.debug.print("[cpu] ...")` te schrijven.

### Stap 2: `src/core/cpu.zig` — `print_state()` implementeren

Dit is je belangrijkste debug-tool: een snapshot van de complete CPU-staat.

```zig
fn print_state(self: *Cpu) void {
    // Registers R0-R7 (meest gebruikte registers)
    std.debug.print("R0={x:0>8} R1={x:0>8} R2={x:0>8} R3={x:0>8}\n", .{
        self.r[0], self.r[1], self.r[2], self.r[3],
    });
    std.debug.print("R4={x:0>8} R5={x:0>8} R6={x:0>8} R7={x:0>8}\n", .{
        self.r[4], self.r[5], self.r[6], self.r[7],
    });
    // Speciale registers
    std.debug.print("R12={x:0>8} SP={x:0>8} LR={x:0>8} PC={x:0>8}\n", .{
        self.r[12], self.r[13], self.r[14], self.r[15],
    });

    // CPSR flags
    const cpsr: u32 = @bitCast(self.cpsr);
    std.debug.print("CPSR={x:0>8} [{}{}{}{}{}]\n", .{
        cpsr,
        @as(u8, if (self.cpsr.n == 1) 'N' else '-'),
        @as(u8, if (self.cpsr.z == 1) 'Z' else '-'),
        @as(u8, if (self.cpsr.c == 1) 'C' else '-'),
        @as(u8, if (self.cpsr.v == 1) 'V' else '-'),
        @as(u8, if (self.cpsr.t == 1) 'T' else '-'),
    });
}
```

**Waarom `{x:0>8}` als format specifier?**
- `x` = hexadecimaal (kleine letters)
- `0>8` = pad links met nullen tot 8 tekens breed
- Dit geeft altijd `00001234` in plaats van `1234`, waardoor kolommen mooi uitgelijnd blijven en je snel patronen in de output kunt herkennen.

**Waarom `@bitCast(self.cpsr)` naar `u32`?**
- Het CPSR (Current Program Status Register) is gemodelleerd als een **packed struct** met individuele bit-velden (n, z, c, v, t, mode, etc.). `@bitCast` converteert dit naar een ruwe `u32` zodat je de volledige waarde in hex kunt printen. Dit is handig om te vergelijken met referentie-emulators die het CPSR als één getal tonen.

**Waarom `@as(u8, if (...) 'N' else '-')`?**
- De `if` expressie geeft een `comptime_int` terug (want `'N'` en `'-'` zijn character literals). `@as(u8, ...)` cast dit naar een `u8` zodat de format string het als karakter kan printen.

### Stap 3: `src/core/cpu.zig` — tracing integreren in de step-functie

```zig
pub fn step(self: *Cpu, bus: *Bus) void {
    // === TRACE: log CPU state vóór elke instructie ===
    if (comptime config.trace_cpu) {
        self.print_state();
    }

    // Fetch instructie op huidige PC
    const instruction = bus.read32(self.r[15]);

    // === TRACE: log de opgehaalde instructie ===
    if (comptime config.trace_cpu) {
        log.debug("PC={x:0>8} instr={x:0>8}", .{ self.r[15], instruction });
    }

    // Decode en execute
    self.execute(bus, instruction);
}
```

**Waarom tracing *vóór* de execute?**
- Als een instructie een crash veroorzaakt, wil je weten welke instructie dat was. Door de state te loggen *voordat* je de instructie uitvoert, is de laatste log-regel altijd de instructie die het probleem veroorzaakte.

**Waarom `log.debug` in plaats van `std.debug.print`?**
- `std.log.scoped(.cpu)` prefix automatisch met `[cpu]` en respecteert het log-niveau. In een release build wordt `.debug` niveau automatisch uitgefilterd. `std.debug.print` gaat altijd naar stderr, ongeacht build mode.

### Stap 4: `src/core/bus.zig` — memory access logging toevoegen

Voeg vergelijkbare config en logging toe aan de memory bus:

```zig
const std = @import("std");
const log = std.log.scoped(.bus);

const config = struct {
    const trace_memory = false; // log elke memory read/write
};

// In read8:
pub fn read8(self: *Bus, address: u32) u8 {
    const value = self.internal_read8(address);
    if (comptime config.trace_memory) {
        log.debug("read8({x:0>8}) = {x:0>2}", .{ address, value });
    }
    return value;
}

// In write8:
pub fn write8(self: *Bus, address: u32, value: u8) void {
    if (comptime config.trace_memory) {
        log.debug("write8({x:0>8}, {x:0>2})", .{ address, value });
    }
    self.internal_write8(address, value);
}
```

**Waarom memory tracing apart van CPU tracing?**
- Memory tracing genereert *enorm veel* output (elke instructie doet minstens één read). Door het apart te schakelen, kun je gericht debuggen: CPU tracing aan voor instructie-problemen, memory tracing aan voor geheugen-mapping issues.

**Waarom `{x:0>2}` voor bytes en `{x:0>8}` voor adressen?**
- Een byte is maximaal 2 hex-tekens (`FF`), een 32-bit adres is maximaal 8 hex-tekens (`FFFFFFFF`). Door de padding consistent te houden, is de output makkelijk scanbaar en vergelijkbaar met referentie-emulators.

### Stap 5: onbekende instructies afhandelen met `unreachable` en `@breakpoint()`

```zig
fn decode(self: *Cpu, bus: *Bus, instruction: u32) void {
    const opcode = get_opcode(instruction);
    switch (opcode) {
        .data_processing => self.handle_dp(bus, instruction),
        .branch => self.handle_branch(instruction),
        // ... bekende instructies ...
        _ => {
            log.err("onbekende instructie: {x:0>8} op PC={x:0>8}", .{
                instruction, self.r[15],
            });
            if (comptime config.break_on_undefined) {
                @breakpoint(); // trigger debugger breakpoint
            }
            unreachable; // in release builds: undefined behavior (crash)
        },
    }
}
```

**Waarom `@breakpoint()` vóór `unreachable`?**
- `@breakpoint()` stuurt een debug-interrupt naar het OS. Als je in een debugger zit (bijv. `lldb` of GDB), stopt het programma precies op dat punt en kun je de CPU-staat inspecteren.
- `unreachable` vertelt de compiler: "dit pad mag nooit bereikt worden." In debug-builds crasht het met een duidelijke foutmelding. In release-builds is het **undefined behavior** — de compiler mag aannemen dat het niet voorkomt.
- De combinatie geeft je het beste van beide werelden: debugger-ondersteuning in development, en een harde crash als safety-net.

**Waarom `log.err` voor onbekende instructies?**
- Zelfs als `@breakpoint()` niet actief is (geen debugger), wil je altijd weten *welke* instructie het probleem veroorzaakte. De `log.err` output gaat naar stderr en is zichtbaar in alle build modes.

### Stap 6: debug-workflow testen

Verifieer dat je debug-tooling werkt:

```bash
# 1. Zet trace_cpu op true in src/core/cpu.zig
# 2. Bouw en draai met een test ROM:
zig build run -- roms/test.gba

# Verwachte output (voorbeeld):
# [cpu] PC=08000000 instr=EA00002E
# R0=00000000 R1=00000000 R2=00000000 R3=00000000
# R4=00000000 R5=00000000 R6=00000000 R7=00000000
# R12=00000000 SP=03007F00 LR=00000000 PC=08000000
# CPSR=0000001F [-----]

# 3. Vergelijk met mGBA's trace output om te controleren of de waarden kloppen

# 4. Zet trace_cpu weer op false als je klaar bent
```

**Tip: output naar bestand pipen voor analyse**

CPU traces genereren duizenden regels per seconde. Pipe de output naar een bestand om rustig te analyseren:
```bash
zig build run -- roms/test.gba 2> trace.log
# Dan in een andere terminal:
head -100 trace.log          # eerste 100 regels bekijken
grep "onbekende" trace.log   # zoek naar problemen
```

**Waarom `2>` en niet `>`?**
- `std.debug.print` en `std.log` schrijven naar **stderr** (file descriptor 2), niet stdout. `2>` redirect alleen stderr naar het bestand. `>` zou stdout redirecten, en dan mis je alle output.

## Acceptatiecriteria

- CPU state kan geprint worden in leesbaar formaat
- Trace logging is aan/uit te zetten via comptime flags (geen runtime overhead als uit)
- Onbekende instructies geven een duidelijke waarschuwing
