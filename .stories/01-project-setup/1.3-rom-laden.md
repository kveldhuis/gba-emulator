# Story 1.3 — ROM laden vanaf command line

**Epic:** 01 - Project Setup & Infrastructuur

**Doel:** Een .gba ROM-bestand inlezen in geheugen.

## Zig concepten

### Error handling met `!` en `try`

Zig heeft geen exceptions. In plaats daarvan kan een functie een **error union** teruggeven: ofwel een waarde, ofwel een error.

**Waarom geen exceptions?** Exceptions in talen als Java/Python hebben een verborgen control flow — je weet niet welke functies kunnen gooien zonder de documentatie te lezen. In Zig is falen *zichtbaar in het type*: als er een `!` staat, kan het falen. Geen `!`? Dan is succes gegarandeerd.

```zig
// Het ! teken betekent: "deze functie kan falen"
fn load(path: []const u8) !GamePak {
    // try = probeer dit, als het faalt geef de error meteen door
    const file = try std.fs.cwd().openFile(path, .{});
    defer file.close();  // wordt ALTIJD uitgevoerd, ook als readToEndAlloc faalt

    const data = try file.readToEndAlloc(allocator, max_size);
    return GamePak{ .rom = data };
}

// Bij het aanroepen:
const pak = load("rom.gba") catch |err| {
    std.debug.print("Kon ROM niet laden: {}\n", .{err});
    return;
};
```

**Waarom `defer file.close()` direct na `openFile`?**
Zonder `defer` zou je bij elke mogelijke error handmatig `file.close()` moeten aanroepen. `defer` garandeert opruiming ongeacht hoe de functie eindigt — het is Zig's veiligheidsnet tegen resource leaks.

### Slices — `[]u8` vs arrays

```zig
// Array: vaste grootte, bekend bij compilatie
var buffer: [1024]u8 = undefined;

// Slice: verwijzing naar een stuk geheugen, grootte bekend at runtime
// Ideaal voor ROM data (je weet pas hoe groot de ROM is als je 'm leest)
var rom: []u8 = undefined;  // pointer + lengte
rom.len;  // hoeveel bytes
rom[0];   // eerste byte
```

### Allocators — geheugen aanvragen

Zig doet geen verborgen geheugentoewijzingen. Je moet expliciet aangeven waar geheugen vandaan komt.

**Waarom geen automatisch geheugenmanagement?** In talen als Python of Java regelt een garbage collector het geheugen — handig, maar onvoorspelbaar (GC-pauzes). In C doe je `malloc`/`free`, maar vergeet je makkelijk een `free` (memory leak) of doe je een dubbele `free` (crash). Zig's allocator-systeem is expliciet maar veilig: je *ziet* waar geheugen vandaan komt, en `defer` helpt je het altijd op te ruimen.

```zig
const allocator = std.heap.page_allocator;

// Geheugen aanvragen (kan falen, dus try)
const data = try allocator.alloc(u8, file_size);

// Geheugen vrijgeven
defer allocator.free(data);
```

### Command line arguments

```zig
pub fn main() !void {
    var args = std.process.args();
    _ = args.next();  // skip programmanaam
    const rom_path = args.next() orelse {
        std.debug.print("Gebruik: gba-emu <rom.gba>\n", .{});

```

`orelse` is voor optionele waarden: "als er geen waarde is, doe dit".

### Struct velden met vaste arrays

```zig
const GamePak = struct {
    rom: []u8,           // dynamische grootte (de ROM data)
    title: [12]u8,       // altijd 12 bytes (game titel in header)
    game_code: [4]u8,    // altijd 4 bytes

    pub fn load(allocator: std.mem.Allocator, path: []const u8) !GamePak {
        // ...
    }
};
```

### GBA ROM header

De eerste 192 bytes (0x00-0xBF) van een .gba bestand bevatten de header:

```
Offset  Grootte  Inhoud
0x00    4        ARM entry point instructie (branch naar start van game)
0x04    156      Nintendo logo (wordt gecheckt door BIOS)
0xA0    12       Game titel (ASCII)
0xAC    4        Game code (ASCII, bijv. "AXVE" voor Pokemon Ruby)
0xB0    2        Maker code
0xBD    1        Header checksum
```

Je kunt de titel lezen met:
```zig
// Kopieer 12 bytes vanaf offset 0xA0
var title: [12]u8 = undefined;
@memcpy(&title, rom[0xA0..0xAC]);
```

## Stap-voor-stap

### Stap 1: `src/main.zig` — return type naar `!void` veranderen

Voordat we bestanden gaan lezen, moet `main()` errors kunnen teruggeven:

```zig
const std = @import("std");
const platform = @import("platform/renderer.zig");

pub fn main() !void {  // was: void
    var args = std.process.args();
    _ = args.next(); // skip programmanaam (altijd het eerste argument)
    const rom_path = args.next() orelse {
        std.debug.print("Gebruik: gba-emu <rom.gba>\n", .{});
        return;
    };

    std.debug.print("ROM path: {s}\n", .{rom_path});

    // ... rest van main
}
```

**Waarom `!void` in plaats van `void`?**
- Het `!` teken maakt er een **error union** van: de functie kan óf succesvol afsluiten (`void`), óf een error teruggeven. Zodra je ergens `try` gebruikt in een functie, *moet* die functie een `!` in zijn return type hebben — anders compileert het niet.

**Waarom `_ = args.next()` voor de programmanaam?**
- Net als in C is het eerste argument altijd de naam van het programma zelf (bijv. `./zig-out/bin/gba-emu`). We willen het ROM-pad, dat is het *tweede* argument. De `_ =` syntax vertelt Zig expliciet: "ik weet dat ik deze waarde weggooi" — zonder dit geeft de compiler een "unused variable" error.

**Waarom `orelse` hier?**
- `args.next()` geeft een **optionele waarde** terug (`?[]const u8`): er is misschien een argument, maar misschien ook niet. `orelse` is Zig's manier om met optionele waarden om te gaan: "als er een waarde is, gebruik die; anders voer dit blok uit."

**Let op:** `{s}` vs `{}` in format strings:
- `{s}` is nodig voor string-slices (`[]const u8`). Als je `{}` gebruikt, krijg je de pointer-representatie in plaats van de tekst.

### Stap 2: `src/core/gamepak.zig` — de GamePak struct definiëren

Maak eerst de directory `src/core/` aan, en dan het bestand:

```zig
const std = @import("std");

pub const GamePak = struct {
    rom: []u8,
    title: [12]u8,
    game_code: [4]u8,
    allocator: std.mem.Allocator,

    pub fn load(allocator: std.mem.Allocator, path: []const u8) !GamePak {
        // Stap 3 en 4 vullen dit in
    }

    pub fn deinit(self: *GamePak) void {
        self.allocator.free(self.rom);
    }
};
```

**Waarom slaan we de `allocator` op in de struct?**
- In Zig moet je geheugen vrijgeven met *dezelfde allocator* waarmee je het hebt aangevraagd. Door de allocator op te slaan in de struct, kan `deinit()` altijd correct opruimen — zelfs als de aanroepende code een andere allocator heeft.

**Waarom `rom: []u8` (slice) en `title: [12]u8` (array)?**
- De ROM-data is variabel in grootte (1KB tot 32MB), dus gebruiken we een **slice** — een pointer + lengte die at runtime bepaald worden.
- De titel is *altijd* 12 bytes in de GBA header (offset 0xA0-0xAB), dus een **vaste array** is hier efficiënter en duidelijker.

### Stap 3: `src/core/gamepak.zig` — bestand inlezen in `load()`

```zig
pub fn load(allocator: std.mem.Allocator, path: []const u8) !GamePak {
    // Open het bestand
    const file = try std.fs.cwd().openFile(path, .{});
    defer file.close(); // sluit het bestand ALTIJD, ook bij errors

    // Lees de volledige inhoud in geheugen
    const rom = try file.readToEndAlloc(allocator, 32 * 1024 * 1024); // max 32 MB

    // Valideer minimale grootte
    if (rom.len < 0xC0) {
        allocator.free(rom);
        return error.RomTooSmall;
    }

    // Parse de header (wordt ingevuld in Stap 4)
    var title: [12]u8 = undefined;
    @memcpy(&title, rom[0xA0..0xAC]);

    var game_code: [4]u8 = undefined;
    @memcpy(&game_code, rom[0xAC..0xB0]);

    return GamePak{
        .rom = rom,
        .title = title,
        .game_code = game_code,
        .allocator = allocator,
    };
}
```

**Waarom `defer file.close()`?**
- `defer` garandeert dat het bestand gesloten wordt, ongeacht hoe de functie eindigt — ook als een latere `try` faalt. Zonder `defer` zou je bij elke mogelijke error handmatig `file.close()` moeten aanroepen. Dit is Zig's alternatief voor `try/finally` in andere talen.

**Waarom `readToEndAlloc` met een limiet van 32 MB?**
- GBA ROMs zijn maximaal 32 MB (256 Mbit). De limiet voorkomt dat een corrupt of verkeerd bestand al je geheugen opeet. Als het bestand groter is dan 32 MB, geeft `readToEndAlloc` een error terug.

**Waarom `try` voor elke operatie?**
- `try` is Zig's manier om errors te "propageren": als `openFile` faalt (bijv. bestand bestaat niet), stopt de hele `load()` functie en geeft de error door aan de aanroeper. Zonder `try` zou je elke error handmatig moeten afvangen met `catch`.

**Let op:** Na `if (rom.len < 0xC0)` moeten we `allocator.free(rom)` aanroepen:
- De `defer file.close()` sluit het bestand, maar het ROM-geheugen is al gealloceerd door `readToEndAlloc`. Als we een error teruggeven zonder `free`, lekt dat geheugen.

### Stap 4: `src/core/gamepak.zig` — header parsen

De GBA ROM header zit in de eerste 192 bytes (0x00-0xBF):

```zig
// Kopieer 12 bytes vanaf offset 0xA0 (game titel)
var title: [12]u8 = undefined;
@memcpy(&title, rom[0xA0..0xAC]);

// Kopieer 4 bytes vanaf offset 0xAC (game code)
var game_code: [4]u8 = undefined;
@memcpy(&game_code, rom[0xAC..0xB0]);
```

**Waarom `@memcpy` en niet gewoon `=`?**
- In Zig kun je niet zomaar een slice aan een array toewijzen met `=`. `@memcpy` kopieert byte-voor-byte van de bron (een slice van de ROM) naar de bestemming (de vaste array). Dit is een **builtin functie** — herkenbaar aan het `@` teken.

**Waarom `undefined` als initiële waarde?**
- `undefined` vertelt Zig: "ik ga deze waarde meteen overschrijven, geef me geen default." In debug-builds vult Zig `undefined` met `0xAA` zodat je het opmerkt als je per ongeluk een ongeïnitialiseerde waarde leest.

**Tip: titel trimmen voor weergave**

GBA titels worden vaak opgevuld met null-bytes of spaties. Om ze netjes te printen:
```zig
const trimmed = std.mem.trimRight(u8, &pak.title, &[_]u8{ 0, ' ' });
std.debug.print("Game: {s}\n", .{trimmed});
```

### Stap 5: `src/main.zig` — alles aan elkaar knopen

```zig
const std = @import("std");
const platform = @import("platform/renderer.zig");
const GamePak = @import("core/gamepak.zig").GamePak;

pub fn main() !void {
    // Command line argument lezen
    var args = std.process.args();
    _ = args.next(); // skip programmanaam
    const rom_path = args.next() orelse {
        std.debug.print("Gebruik: gba-emu <rom.gba>\n", .{});
        return;
    };

    // ROM laden
    var gpa = std.heap.GeneralPurposeAllocator(.{}){};
    defer _ = gpa.deinit();
    const allocator = gpa.allocator();

    var pak = GamePak.load(allocator, rom_path) catch |err| {
        std.debug.print("Kon ROM niet laden: {}\n", .{err});
        return;
    };
    defer pak.deinit();

    const trimmed = std.mem.trimRight(u8, &pak.title, &[_]u8{ 0, ' ' });
    std.debug.print("Geladen: {s} ({d} bytes)\n", .{ trimmed, pak.rom.len });

    // Renderer starten
    var renderer = platform.Renderer.init();
    defer renderer.deinit();

    while (platform.Renderer.processEvents()) {
        renderer.present();
    }
}
```

**Waarom `GeneralPurposeAllocator` en niet `page_allocator`?**
- `page_allocator` vraagt geheugen aan in hele OS-pagina's (4KB blokken) — simpel maar verspillend voor kleine allocaties, en het kan memory leaks niet detecteren.
- `GeneralPurposeAllocator` (GPA) is een debug-allocator die:
  - Memory leaks detecteert (rapporteert bij `deinit()`)
  - Double-frees opspoort
  - Out-of-bounds writes detecteert
- Perfect tijdens development. Later kun je eventueel switchen naar een snellere allocator.

**Waarom `catch |err|` in plaats van `try`?**
- `try` propageert de error naar boven (de aanroeper). Maar in `main()` willen we een **gebruiksvriendelijke foutmelding** tonen, niet een stack trace. `catch |err|` vangt de error op zodat we een nette boodschap kunnen printen.

**Waarom twee `defer` statements?**
- `defer pak.deinit()` ruimt het ROM-geheugen op
- `defer renderer.deinit()` ruimt SDL resources op
- Zig voert defers uit in **omgekeerde volgorde** (LIFO): renderer wordt eerst opgeruimd, dan de GamePak. Dit is logisch: de renderer kan afhankelijk zijn van de ROM-data, dus die moet als eerste stoppen.
