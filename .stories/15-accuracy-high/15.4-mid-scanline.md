# Story 15.4 — Mid-scanline Effecten (HBlank DMA)

**Epic:** 15 - Accuracy — Hoge Impact

**Doel:** Register-wijzigingen tijdens HBlank correct afhandelen voor raster effects.

## Waarom belangrijk?

Veel GBA-games veranderen PPU registers **per scanline** via HBlank DMA of HBlank interrupts. Dit creëert visuele effecten die met simpele per-frame rendering niet werken:

```
Voorbeelden:
- Wavey achtergronden: BG scroll offset verandert elke scanline (Golden Sun water)
- Kleur-cycling: palette entries veranderen per scanline (Castlevania sunrise)
- Mode 7-achtig: affine parameters veranderen per scanline (Mario Kart-stijl)
- Fade-in/out: brightness verandert per scanline (zachte overgang)
```

**Waarom is HBlank DMA cruciaal?** De CPU alleen is te langzaam om per scanline registers te veranderen — een interrupt afhandelen kost ~20-30 cycles, en de HBlank-periode is maar ~272 cycles. DMA daarentegen kan direct na de HBlank starten zonder interrupt-overhead, en een transfer van 1-2 woorden kost maar ~4-6 cycles. Daarom gebruiken games DMA voor per-scanline effecten: het is snel genoeg om honderden keren per frame betrouwbaar te vuren.

**Waarom werkt per-frame rendering niet?** Als je het hele scherm in één keer rendert met de register-waarden van frame-begin, zie je geen per-scanline variatie. Je rendert dan 160 scanlines met dezelfde scroll offset, terwijl de game elke scanline een andere offset instelt. Het resultaat: een statische achtergrond in plaats van een golvend wateroppervlak.

## Context

De truc: games zetten HBlank DMA op, die elke HBlank automatisch nieuwe waarden naar PPU registers kopieert. Elke scanline krijgt zo andere instellingen.

```
Scanline 0:  BG0HOFS = 0    ← DMA schrijft waarde[0]
Scanline 1:  BG0HOFS = 2    ← DMA schrijft waarde[1]
Scanline 2:  BG0HOFS = 5    ← DMA schrijft waarde[2]
...
Resultaat: de achtergrond "golft" verticaal
```

## Zig concepten

### Per-scanline register snapshots

De simpelste aanpak: render elke scanline met de registers **zoals ze op dat moment staan**:

```zig
fn run_scanline(self: *Ppu, bus: *Bus, dma: *DmaController, scheduler: *Scheduler) void {
    // 1. Render de zichtbare pixels met HUIDIGE register-waarden
    if (self.scanline < 160) {
        self.render_scanline(bus);
    }

    // 2. Ga naar HBlank
    self.set_hblank(true);

    // 3. Trigger HBlank DMA — dit kan registers veranderen voor de VOLGENDE scanline
    dma.check_and_transfer(.hblank);

    // 4. Trigger HBlank interrupt (als enabled)
    if (self.dispstat.hblank_irq == 1) {
        irq.request(.hblank);
    }

    // 5. Advance naar volgende scanline
    self.scanline += 1;
    self.set_hblank(false);
}
```

**Volgorde is cruciaal:** render EERST, dan DMA. Zo ziet de volgende scanline de nieuwe waarden.

### HBlank DMA met repeat

HBlank DMA's herhalen automatisch elke scanline (als repeat bit aan staat):

```zig
fn check_hblank_dma(self: *DmaController, bus: *Bus) void {
    for (0..4) |i| {
        const ch = &self.channels[i];
        if (ch.control.enabled == 1 and ch.control.timing == 0b10) { // HBlank timing
            ch.execute_transfer(bus);

            // Bij repeat + dest_increment_reload: reset dest naar origineel adres
            if (ch.control.repeat == 1) {
                if (ch.control.dest_control == 0b11) {
                    ch.internal_dest = ch.dest;  // reload destination
                }
                ch.internal_count = ch.count;  // reload count
            } else {
                ch.control.enabled = 0;  // eenmalig: klaar
            }
        }
    }
}
```

### Voorbeeld: wavey achtergrond setup

```zig
// Game setup (eenmalig):
// 1. Vul een tabel met 160 scroll-waarden (één per scanline)
const wave_table: [160]u16 = compute_wave();  // bijv. sin(y) * 4

// 2. Kopieer tabel naar geheugen
for (0..160) |i| {
    bus.write16(0x02000000 + i * 2, wave_table[i]);
}

// 3. Stel DMA3 in voor HBlank:
//    source = 0x02000000 (tabel), dest = 0x04000010 (BG0HOFS)
//    count = 1, timing = HBlank, repeat = 1, dest_control = fixed
dma.channels[3].source = 0x02000000;
dma.channels[3].dest = 0x04000010;
dma.channels[3].count = 1;
dma.channels[3].control = .{
    .timing = 0b10,       // HBlank
    .repeat = 1,          // elke HBlank opnieuw
    .dest_control = 0b10, // fixed (steeds naar BG0HOFS)
    .source_control = 0,  // increment (door de tabel)
    .enabled = 1,
    // ...
};
```

## Bestanden

- `src/core/ppu.zig` — render-then-DMA volgorde per scanline

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — Render-dan-DMA volgorde implementeren

De kernverandering is de volgorde van operaties per scanline. Dit moet precies goed zijn: eerst renderen met de *huidige* register-waarden, dan DMA uitvoeren die de registers wijzigt voor de *volgende* scanline.

```zig
fn run_scanline(self: *Ppu, bus: *Bus, dma: *DmaController) void {
    // 1. Render de zichtbare pixels met HUIDIGE register-waarden
    if (self.scanline < 160) {
        self.render_scanline(bus);
    }

    // 2. Ga naar HBlank fase
    self.set_hblank(true);

    // 3. Trigger HBlank DMA — dit kan registers veranderen
    dma.check_and_transfer(.hblank);

    // 4. Trigger HBlank interrupt (als enabled)
    if (self.dispstat.hblank_irq == 1) {
        irq.request(.hblank);
    }

    // 5. Advance naar volgende scanline
    self.scanline += 1;
    self.set_hblank(false);
}
```

**Waarom eerst renderen, dan DMA?** Op echte hardware worden pixels getekend tijdens de "draw" periode van de scanline (960 cycles). Pas daarna begint HBlank (272 cycles), waarin DMA's en interrupts vuren. Als je DMA *vóór* het renderen uitvoert, ziet scanline N de waarden die bedoeld zijn voor scanline N (in plaats van N-1), en alle effecten verschuiven één scanline naar boven.

**Waarom is de volgorde "DMA → interrupt" belangrijk?** DMA heeft hogere prioriteit dan interrupts op de GBA. Als zowel DMA als een interrupt op HBlank getriggerd worden, gaat DMA eerst. De interrupt-handler draait pas nadat de DMA klaar is. Als je dit omdraait, kan de interrupt-handler registers schrijven die direct weer overschreven worden door DMA.

**Valkuil:** Vergeet niet om DISPSTAT bit 1 (HBlank flag) te zetten vóór het triggeren van DMA en interrupts. Games en DMA-triggers kunnen deze flag lezen.

### Stap 2: `src/core/dma.zig` — HBlank DMA repeat logica implementeren

HBlank DMA's zijn bijzonder: ze vuren elke scanline opnieuw (als de repeat-bit aan staat). Na elke transfer moet de count opnieuw geladen worden.

```zig
fn check_hblank_dma(self: *DmaController, bus: *Bus) u32 {
    var total_cycles: u32 = 0;

    for (0..4) |i| {
        const ch = &self.channels[i];
        if (ch.control.enabled == 1 and ch.control.timing == 0b10) {
            total_cycles += ch.execute_transfer(bus);

            if (ch.control.repeat == 1) {
                // Repeat: herlaad count voor volgende scanline
                ch.internal_count = if (ch.count == 0) ch.max_count() else ch.count;

                // Dest reload bij control mode 3 (increment+reload)
                if (ch.control.dest_control == 0b11) {
                    ch.internal_dest = ch.dest;
                }
                // Source wordt NIET herlaad — blijft doorlopen
            } else {
                ch.control.enabled = 0;  // eenmalig: klaar
            }
        }
    }
    return total_cycles;
}
```

**Waarom wordt de source niet herlaad?** Dit is het geheim achter per-scanline effecten: de source-pointer loopt door een tabel in geheugen. Elke scanline leest de DMA het *volgende* element uit de tabel. Bij een wavey achtergrond bevat die tabel 160 scroll-waarden (één per scanline). De destination (bijv. BG0HOFS register) wordt wél herlaad — elke scanline schrijft naar hetzelfde register.

**Waarom `internal_count` herlaad als `count == 0`?** Als het count-register 0 bevat, betekent dat op de GBA het maximum aantal transfers (0x4000 voor kanaal 0-2, 0x10000 voor kanaal 3). Dit is een hardware-quirk die je correct moet afhandelen.

### Stap 3: `src/core/dma.zig` — Destination reload (control mode 3)

Destination address control mode `0b11` ("increment and reload") is speciaal ontworpen voor HBlank DMA effecten.

```zig
const dest_control = ch.control.dest_control;
switch (dest_control) {
    0b00 => ch.internal_dest += size,  // increment
    0b01 => ch.internal_dest -= size,  // decrement
    0b10 => {},                        // fixed (schrijf steeds naar hetzelfde adres)
    0b11 => {                          // increment + reload bij repeat
        ch.internal_dest += size;
        // Reload gebeurt pas bij de volgende repeat-trigger (zie stap 2)
    },
}
```

**Waarom bestaat "increment + reload"?** Stel je voor dat je per scanline 4 bytes naar VRAM wilt kopiëren (palette-cycling). De source loopt door een tabel (160 × 4 bytes), en de destination moet bij elke scanline opnieuw beginnen bij dezelfde palette-locatie. Met "fixed" (mode 2) zou het werken voor 1 byte per keer, maar niet voor meerdere bytes. "Increment + reload" incrementeert tijdens de transfer (zodat meerdere bytes naar opeenvolgende adressen gaan) en reset dan het destination-adres voor de volgende trigger.

**Valkuil:** Mode `0b11` gedraagt zich als "increment" (mode 0) tijdens de transfer zelf. Het verschil zit alleen in wat er bij een repeat-trigger gebeurt: het destination-adres wordt dan teruggeplaatst naar de originele waarde. Als je dit als "fixed" implementeert, werken multi-woord HBlank DMA's niet.

### Stap 4: `src/core/ppu.zig` — Testen met een simpel wave-effect

Om te verifiëren dat alles werkt, kun je een test schrijven die een wave-tabel opzet en controleert dat elke scanline de juiste scroll-offset krijgt.

```zig
// Test: simuleer 160 scanlines met HBlank DMA
test "hblank dma wave effect" {
    var bus = Bus.init();
    var dma = DmaController.init();
    var ppu = Ppu.init();

    // Vul een tabel met 160 scroll-waarden
    for (0..160) |i| {
        bus.write16(0x02000000 + @as(u32, @intCast(i)) * 2, @intCast(i * 2));
    }

    // Stel DMA3 in: source=tabel, dest=BG0HOFS, count=1, HBlank, repeat
    dma.channels[3].source = 0x02000000;
    dma.channels[3].dest = 0x04000010;  // BG0HOFS
    dma.channels[3].count = 1;
    dma.channels[3].control = .{ .timing = 0b10, .repeat = 1, .dest_control = 0b10, .enabled = 1 };

    // Simuleer 160 scanlines
    for (0..160) |scanline| {
        ppu.run_scanline(&bus, &dma);
        // Controleer dat BG0HOFS de juiste waarde heeft na DMA
        const expected_offset = @as(u16, @intCast(scanline * 2));
        try std.testing.expectEqual(expected_offset, bus.read16(0x04000010));
    }
}
```

**Waarom een test op dit niveau?** Mid-scanline effecten zijn notoir lastig te debuggen visueel — je ziet alleen dat "het er raar uitziet". Een unit test die de register-waarden na elke scanline controleert, is veel sneller om problemen op te sporen. Vooral off-by-one errors (scanline N krijgt de waarde van scanline N+1) zijn hiermee direct zichtbaar.

**Tip:** Test ook het geval waar de DMA source in ROM zit (wait states van toepassing) en controleer dat de totale cycle-kosten van de DMA kloppen.

## Acceptatiecriteria

- Wavey achtergronden zijn zichtbaar (scroll offset verandert per scanline)
- Palette DMA per scanline werkt (kleur-effecten)
- DMA repeat + destination reload werkt correct
- Geen off-by-one errors in scanline timing
