# Story 15.2 — Prefetch Buffer

**Epic:** 15 - Accuracy — Hoge Impact

**Doel:** De Game Pak prefetch buffer emuleren die ROM reads versnelt.

## Waarom belangrijk?

De GBA heeft een 8-halfword (16 bytes) prefetch buffer die automatisch de volgende ROM-adressen vooruit leest terwijl de CPU bezig is met interne operaties. Zonder prefetch-emulatie draaien veel games te langzaam of hangen ze — ze verwachten dat bepaalde loops in een exact aantal cycles draaien.

**Waarom bestaat de prefetch buffer?** Het is een hardware-optimalisatie van Nintendo om het snelheidsverlies van ROM wait states te verbergen. Terwijl de CPU bezig is met een berekening (die geen bus nodig heeft), staat de bus anders "stil". De prefetch buffer benut die vrije bus-tijd om alvast de volgende instructies op te halen. Dit maakt lineaire code significant sneller — in het beste geval kost een ROM-fetch maar 1 cycle in plaats van 5.

**Waarom beïnvloedt dit zoveel games?** Veel GBA-games bevatten timing-gevoelige loops, bijvoorbeeld voor audio mixing of raster effects. Deze loops zijn vaak geoptimaliseerd met de prefetch buffer in gedachten. Als je emulator geen prefetch emuleert, draaien die loops in een ander aantal cycles, waardoor audio crackt of visuele glitches ontstaan.

## Context

Als WAITCNT bit 14 (prefetch enable) aan staat:
- Terwijl de CPU intern bezig is (ALU operatie, register write), leest de prefetch buffer alvast de volgende instructies van ROM
- Als de CPU vervolgens die instructie fetched, is die al beschikbaar → 1 cycle in plaats van de volle wait state kosten
- De buffer wordt geleegd bij een branch (verkeerde instructies geprefetched)

**Waarom maar 8 halfwords?** Dit is een hardwarebeperking — de buffer is een klein FIFO-register in de ARM7TDMI bus interface. 8 halfwords (16 bytes) is genoeg voor 4 ARM-instructies of 8 Thumb-instructies. Meer zou meer chip-oppervlak kosten zonder veel voordeel, omdat branches de buffer toch regelmatig legen.

## Zig concepten

### Prefetch buffer struct

```zig
const PrefetchBuffer = struct {
    // De buffer bevat tot 8 halfwords (16 bytes) vooruit gelezen van ROM
    active: bool = false,
    head_addr: u32 = 0,     // adres van het oudste item in de buffer
    tail_addr: u32 = 0,     // adres van het volgende dat gelezen gaat worden
    cycles_remaining: u32 = 0, // cycles tot volgende halfword beschikbaar is
    count: u4 = 0,           // hoeveel halfwords in de buffer zitten (max 8)

    fn reset(self: *PrefetchBuffer) void {
        self.count = 0;
        self.cycles_remaining = 0;
    }

    fn step(self: *PrefetchBuffer, internal_cycles: u32, waitcnt: WaitCnt) void {
        if (!self.active or self.count >= 8) return;

        // Gebruik "vrije" cycles om vooruit te lezen
        var remaining = internal_cycles;
        while (remaining > 0 and self.count < 8) {
            if (self.cycles_remaining == 0) {
                // Start volgende prefetch (sequential access)
                self.cycles_remaining = get_seq_wait(self.tail_addr, waitcnt);
            }
            if (remaining >= self.cycles_remaining) {
                remaining -= self.cycles_remaining;
                self.cycles_remaining = 0;
                self.tail_addr += 2;  // halfword opgehaald
                self.count += 1;
            } else {
                self.cycles_remaining -= remaining;
                remaining = 0;
            }
        }
    }

    /// Retourneert true als het gevraagde adres in de buffer zit
    fn contains(self: *PrefetchBuffer, address: u32) bool {
        return self.count > 0 and
            address >= self.head_addr and
            address < self.head_addr + @as(u32, self.count) * 2;
    }

    /// Consumeer een halfword uit de buffer
    fn consume(self: *PrefetchBuffer, address: u32) void {
        if (self.contains(address)) {
            // Verwijder alles tot en met dit adres
            const consumed = (address - self.head_addr) / 2 + 1;
            self.count -= @intCast(consumed);
            self.head_addr = address + 2;
        }
    }
};
```

### Integratie met instruction fetch

```zig
fn fetch_cycles(self: *Cpu, bus: *Bus, address: u32) u32 {
    const region = (address >> 24) & 0xF;

    // Alleen ROM (0x08-0x0D) heeft prefetch
    if (region >= 0x08 and region <= 0x0D) {
        if (bus.prefetch.contains(address)) {
            bus.prefetch.consume(address);
            return 1;  // gratis! (al geprefetched)
        }
    }

    // Niet in buffer: volle wait state kosten + reset prefetch
    bus.prefetch.reset();
    bus.prefetch.head_addr = address + (if (self.cpsr.t == 1) @as(u32, 2) else 4);
    bus.prefetch.tail_addr = bus.prefetch.head_addr;
    return bus.access_cycles(address, .nonseq, if (self.cpsr.t == 1) .half else .word);
}
```

### Branches resetten de buffer

```zig
fn handle_branch(self: *Cpu, bus: *Bus, target: u32) void {
    self.r[15] = target;
    bus.prefetch.reset();  // geprefetchte instructies zijn nu ongeldig
}
```

## Stap-voor-stap

### Stap 1: `src/core/bus.zig` — PrefetchBuffer struct aanmaken

Begin met de datastructuur die de buffer-state bijhoudt. De prefetch buffer is een simpele FIFO die bijhoudt welke adressen al opgehaald zijn.

```zig
const PrefetchBuffer = struct {
    active: bool = false,       // is prefetch ingeschakeld via WAITCNT?
    head_addr: u32 = 0,         // adres van het oudste item in de buffer
    tail_addr: u32 = 0,         // adres van het eerstvolgende op te halen halfword
    cycles_remaining: u32 = 0,  // hoeveel cycles tot het huidige halfword klaar is
    count: u4 = 0,              // aantal halfwords momenteel in de buffer (max 8)
};
```

**Waarom `head_addr` en `tail_addr` bijhouden in plaats van een echte buffer met data?** We hoeven de data zelf niet te bufferen — de ROM-data verandert niet. We hoeven alleen te weten *of* een bepaald adres al geprefetched is. Door head en tail bij te houden weten we het adresbereik dat in de buffer zit, zonder een array van halfwords te beheren. Dit is een veelgebruikte truc in emulators: modelleer het *gedrag*, niet de exacte hardware.

**Waarom `u4` voor count?** De maximale waarde is 8, wat in 4 bits past (0-15). Zig's type-systeem helpt je hier: een `u4` kan niet per ongeluk boven 15 komen.

### Stap 2: `src/core/bus.zig` — `step()` functie: buffer vullen met vrije cycles

De CPU voert soms interne operaties uit die de bus niet gebruiken (bijv. een vermenigvuldiging, een barrel shift, of een register-naar-register move). Gedurende die "vrije" cycles kan de prefetch buffer de bus gebruiken om vooruit te lezen.

```zig
fn step(self: *PrefetchBuffer, internal_cycles: u32, waitcnt: WaitCnt) void {
    if (!self.active or self.count >= 8) return;

    var remaining = internal_cycles;
    while (remaining > 0 and self.count < 8) {
        if (self.cycles_remaining == 0) {
            // Begin met het ophalen van het volgende halfword (sequential access)
            self.cycles_remaining = get_seq_wait_for_addr(self.tail_addr, waitcnt);
        }
        if (remaining >= self.cycles_remaining) {
            remaining -= self.cycles_remaining;
            self.cycles_remaining = 0;
            self.tail_addr += 2;  // halfword succesvol opgehaald
            self.count += 1;
        } else {
            // Niet genoeg cycles om dit halfword af te ronden — bewaar voortgang
            self.cycles_remaining -= remaining;
            remaining = 0;
        }
    }
}
```

**Waarom sequential access kosten?** De prefetch buffer leest altijd opeenvolgende adressen (sequentieel vooruit). Na de eerste non-sequential access (die de CPU doet), leest de buffer elke volgende locatie als sequential — wat goedkoper is (bijv. 1 cycle in plaats van 4).

**Waarom partiële voortgang bijhouden (`cycles_remaining`)?** Stel dat een sequential ROM-read 2 cycles kost, maar de CPU maar 1 vrije cycle heeft. Dan is het halfword nog niet klaar, maar er is wel 1 cycle "werk" gedaan. Bij de volgende vrije cycles hoeft nog maar 1 cycle afgerond te worden. Zonder deze tracking zou je cycles weggooien.

**Valkuil:** De `get_seq_wait_for_addr()` helper moet het adres vertalen naar de juiste ROM wait state regio (WS0/WS1/WS2) en de bijbehorende sequential wait opzoeken in WAITCNT.

### Stap 3: `src/core/bus.zig` — `contains()` en `consume()`: buffer raadplegen

Bij een instruction fetch moet je controleren of het gewenste adres al in de buffer zit.

```zig
fn contains(self: *PrefetchBuffer, address: u32) bool {
    return self.count > 0 and
        address >= self.head_addr and
        address < self.head_addr + @as(u32, self.count) * 2;
}

fn consume(self: *PrefetchBuffer, address: u32) void {
    if (self.contains(address)) {
        const consumed = (address - self.head_addr) / 2 + 1;
        self.count -= @intCast(consumed);
        self.head_addr = address + 2;
    }
}
```

**Waarom `* 2` in contains?** Elk item in de buffer is een halfword (2 bytes). Als `head_addr` = 0x08000000 en `count` = 3, dan bevat de buffer adressen 0x08000000, 0x08000002, en 0x08000004. Het eindadres is `head + count * 2`.

**Waarom consume alles tot en met het gevraagde adres verwijdert?** De buffer is een FIFO — je kunt niet een halfword uit het midden pakken zonder de eerdere te "consumeren". In de praktijk vraagt de CPU bijna altijd het head-adres op (lineaire executie), maar na een korte voorwaartse sprong zou het een adres verderop in de buffer kunnen zijn.

**Tip:** Na consume kan `self.cycles_remaining` resetten naar 0 als de huidige fetch werd onderbroken, want het adres dat in voorbereiding was sluit nu niet meer aan bij de head.

### Stap 4: `src/core/cpu.zig` — Integratie in instruction fetch

Pas de fetch-logica aan: controleer eerst de prefetch buffer voordat je een dure ROM-read doet.

```zig
fn fetch_cycles(self: *Cpu, bus: *Bus, address: u32) u32 {
    const region = (address >> 24) & 0xF;

    // Alleen ROM-regio's (0x08-0x0D) hebben prefetch
    if (region >= 0x08 and region <= 0x0D and bus.prefetch.active) {
        if (bus.prefetch.contains(address)) {
            bus.prefetch.consume(address);
            return 1;  // al in de buffer: kost maar 1 cycle
        }
    }

    // Niet in buffer: volle wait state kosten
    // Reset de prefetch buffer en begin opnieuw vanaf het volgende adres
    bus.prefetch.reset();
    const instr_size: u32 = if (self.cpsr.t == 1) 2 else 4;
    bus.prefetch.head_addr = address + instr_size;
    bus.prefetch.tail_addr = bus.prefetch.head_addr;

    const width: AccessWidth = if (self.cpsr.t == 1) .half else .word;
    return bus.access_cycles(address, .nonseq, width);
}
```

**Waarom reset bij een cache miss?** Als het gevraagde adres niet in de buffer zit, is alles wat erin zit nutteloos (de CPU is naar een ander pad gegaan). We resetten de buffer en starten opnieuw vanaf het adres na de huidige fetch — zodat de *volgende* fetch hopelijk wel in de buffer zit.

**Waarom 1 cycle bij een hit?** Op echte hardware kost het lezen uit de interne prefetch buffer precies 1 cycle — het is een register-read, geen bus-transfer. Dit is de hele reden dat de prefetch buffer bestaat: wat normaal 4-5 cycles zou kosten, kost nu maar 1.

### Stap 5: `src/core/cpu.zig` — Buffer voeden met interne cycles en reset bij branch

Na elke instructie: voer vrije cycles door naar de prefetch buffer. Bij een branch: reset.

```zig
pub fn step(self: *Cpu, bus: *Bus) u32 {
    const fetch_cycles = self.fetch_instruction(bus);

    // Execute de instructie
    const result = self.execute(instruction, bus);

    // Interne cycles (bijv. shifter, multiplier) voeden de prefetch buffer
    if (result.internal_cycles > 0) {
        bus.prefetch.step(result.internal_cycles, bus.waitcnt);
    }

    return fetch_cycles + result.total_cycles;
}

fn handle_branch(self: *Cpu, bus: *Bus, target: u32) void {
    self.r[15] = target;
    self.branched = true;
    bus.prefetch.reset();  // alles in de buffer is nu ongeldig
}
```

**Waarom reset bij een branch?** De prefetch buffer leest altijd *vooruit* van het huidige adres. Bij een branch springt de CPU naar een compleet ander adres — de geprefetchte instructies zijn voor het oude pad en dus waardeloos. De buffer moet leeg beginnen op het nieuwe adres.

**Waarom alleen interne cycles doorvoeren?** Tijdens een bus-access (memory read/write) is de bus bezet en kan de prefetch buffer niet lezen. Alleen cycles waarin de CPU intern bezig is (ALU-operaties, register writes, barrel shifter) zijn "vrij" voor de prefetch buffer. Dit onderscheid correct maken is essentieel voor accurate timing.

**Valkuil:** Multiplies op de ARM7TDMI kosten variabel veel interne cycles (1-4 afhankelijk van de waarde). Deze cycles moeten allemaal als "intern" geteld worden en aan de prefetch buffer doorgegeven worden. Een veel voorkomende bug is om multiply-cycles als bus-cycles te tellen.

## Acceptatiecriteria

- Games die afhankelijk zijn van prefetch-timing hangen niet meer
- Prefetch buffer wordt geleegd bij branches
- WAITCNT bit 14 schakelt prefetch in/uit
- Cycle-counting is nauwkeuriger
