# Story 15.5 — Mosaic Effect

**Epic:** 15 - Accuracy — Hoge Impact

**Doel:** Het mosaic (pixelation) effect implementeren voor achtergronden en sprites.

## Waarom belangrijk?

Mosaic is een veelgebruikt visueel effect in GBA-games. Het maakt pixels groter (blokkerig), vaak gebruikt voor:
- Screen transitions (fade naar mosaic → nieuw scherm)
- Dream/flashback sequences
- Damage effects

Veel RPGs (Final Fantasy, Fire Emblem, Golden Sun) gebruiken het.

**Waarom is mosaic simpel te implementeren?** In tegenstelling tot veel andere PPU-features is mosaic puur een coördinaat-transformatie: je rondt de pixel-coördinaten af naar een veelvoud van de blokgrootte. Er is geen extra state, geen complexe interacties met de bus, en geen timing-gevoeligheid. Het is letterlijk twee delingen en twee vermenigvuldigingen per pixel. Dit maakt het een ideale "quick win" voor accuracy.

## Context

Het MOSAIC register (0x0400004C) bepaalt de blokgrootte:

```zig
const Mosaic = packed struct(u16) {
    bg_h: u4,     // BG horizontale mosaic (0 = uit, 1 = 2px breed, 15 = 16px breed)
    bg_v: u4,     // BG verticale mosaic
    obj_h: u4,    // Sprite horizontale mosaic
    obj_v: u4,    // Sprite verticale mosaic
};
```

De waarde `N` betekent: blokken van `N+1` pixels. Dus 0 = normaal (1px), 1 = 2px blokken, etc.

**Waarom aparte BG en sprite mosaic?** Games willen soms alleen sprites pixeleren (bijv. een "beschadigd" personage) terwijl de achtergrond scherp blijft, of andersom (achtergrond pixeleert bij een screen transition terwijl de cursor scherp blijft). De GBA geeft de programmeur volledige controle door horizontale en verticale mosaic apart instelbaar te maken voor zowel achtergronden als sprites.

## Zig concepten

### Hoe mosaic werkt (per-as)

Mosaic is simpel: rond de coördinaat af naar beneden op een veelvoud van de blokgrootte.

```zig
fn apply_mosaic(coord: u32, mosaic_size: u4) u32 {
    if (mosaic_size == 0) return coord;
    const block = @as(u32, mosaic_size) + 1;
    return (coord / block) * block;
    // Voorbeeld: coord=5, size=3 (blokken van 4)
    // (5 / 4) * 4 = 1 * 4 = 4
    // Pixels 4, 5, 6, 7 krijgen allemaal de kleur van pixel 4
}
```

### BG mosaic — toepassen op de broncoördinaten

Bij achtergrond-rendering pas je mosaic toe **vóór** het opzoeken van de tile:

```zig
fn render_bg_scanline(self: *Ppu, bus: *Bus, bg: u2, y: u8) void {
    const bgcnt: BgCnt = @bitCast(bus.read16(0x04000008 + @as(u32, bg) * 2));
    const mosaic_reg: Mosaic = @bitCast(bus.read16(0x0400004C));

    // Verticale mosaic: gebruik de y van het begin van het mosaic-blok
    const render_y = if (bgcnt.mosaic == 1)
        apply_mosaic(y, mosaic_reg.bg_v)
    else
        y;

    for (0..240) |screen_x| {
        // Horizontale mosaic: rond x af
        const render_x = if (bgcnt.mosaic == 1)
            apply_mosaic(@intCast(screen_x), mosaic_reg.bg_h)
        else
            @as(u32, @intCast(screen_x));

        // Gebruik render_x en render_y voor tile lookup
        // (scroll offset erbij, dan tile + pixel ophalen)
        const color = lookup_bg_pixel(bus, bg, render_x, render_y);
        self.framebuffer[@as(u32, y) * 240 + @as(u32, @intCast(screen_x))] = color;
    }
}
```

### Sprite mosaic — toepassen op sprite-lokale coördinaten

Bij sprites pas je mosaic toe op de coördinaten **binnen** de sprite:

```zig
fn render_sprite_pixel(
    bus: *Bus,
    attr2: OamAttr2,
    local_x: u32,
    local_y: u32,
    mosaic_enabled: bool,
    mosaic_h: u4,
    mosaic_v: u4,
) u8 {
    const mx = if (mosaic_enabled) apply_mosaic(local_x, mosaic_h) else local_x;
    const my = if (mosaic_enabled) apply_mosaic(local_y, mosaic_v) else local_y;

    return read_sprite_pixel(bus, attr2, mx, my);
}
```

### Mosaic interactie met andere effecten

Mosaic wordt toegepast **na** scrolling maar **vóór** blending:

```
Rendering pipeline:
1. Scroll offset berekenen
2. Mosaic toepassen op coördinaten ← hier
3. Tile/pixel opzoeken
4. Windowing check
5. Priority sorting
6. Blending/brightness
```

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — MOSAIC register uitlezen

Voeg het MOSAIC register toe aan je I/O-handling. Het register is write-only op echte hardware, maar de meeste emulators slaan de waarde op zodat de PPU-renderer erbij kan.

```zig
// In je Bus of PPU struct:
mosaic: Mosaic = @bitCast(@as(u16, 0)),

// In io_write16:
0x0400004C => {
    self.mosaic = @bitCast(value);
},
```

**Waarom write-only?** Op echte hardware retourneert een read van 0x0400004C nul (of open bus). Maar intern slaat de PPU de waarde wel op — het register stuurt de hardware-logica aan. In je emulator sla je het gewoon op als een veld, zodat de renderer het kan gebruiken.

**Tip:** De default waarde 0 betekent "geen mosaic" (blokgrootte 1×1 = normaal). Je hoeft geen speciale initialisatie te doen.

### Stap 2: `src/core/ppu.zig` — `apply_mosaic()` helper functie schrijven

Dit is de kern van het hele effect: een functie die een coördinaat afrondt naar het begin van zijn mosaic-blok.

```zig
fn apply_mosaic(coord: u32, mosaic_size: u4) u32 {
    if (mosaic_size == 0) return coord;  // 0 = uit, geen effect
    const block = @as(u32, mosaic_size) + 1;
    return (coord / block) * block;
}
```

**Waarom `(coord / block) * block`?** Dit is integer-deling gevolgd door vermenigvuldiging — het effect is "afronden naar beneden op een veelvoud van block". Voorbeeld met block=4: coördinaten 0,1,2,3 worden allemaal 0; coördinaten 4,5,6,7 worden allemaal 4. Alle pixels in een blok krijgen zo dezelfde kleur (die van de linkerbovenhoek).

**Waarom `mosaic_size + 1`?** De hardware definieert de blokgrootte als `N+1`. Een registerwaarde van 0 = blokken van 1 pixel (normaal), 1 = blokken van 2 pixels, etc. Dit is een veelvoorkomend patroon in GBA-registers: de waarde 0 betekent "minimaal" in plaats van "uit". De "uit"-check doe je met de mosaic enable-bit in BGCNT, niet met de grootte.

### Stap 3: `src/core/ppu.zig` — BG mosaic integreren in de achtergrond-renderer

Pas de bestaande BG-renderer aan om mosaic toe te passen op de broncoördinaten, *vóór* het opzoeken van de tile.

```zig
fn render_bg_scanline(self: *Ppu, bus: *Bus, bg: u2, y: u8) void {
    const bgcnt: BgCnt = @bitCast(bus.read16(0x04000008 + @as(u32, bg) * 2));
    const mosaic_enabled = bgcnt.mosaic == 1;

    // Verticale mosaic: vervang y door het begin van het mosaic-blok
    const render_y: u32 = if (mosaic_enabled)
        apply_mosaic(@as(u32, y), self.mosaic.bg_v)
    else
        @as(u32, y);

    for (0..240) |screen_x| {
        // Horizontale mosaic: vervang x door het begin van het mosaic-blok
        const render_x: u32 = if (mosaic_enabled)
            apply_mosaic(@as(u32, @intCast(screen_x)), self.mosaic.bg_h)
        else
            @as(u32, @intCast(screen_x));

        // Gebruik render_x en render_y (+ scroll offset) voor tile lookup
        const color = lookup_bg_pixel(bus, bg, render_x + scroll_x, render_y + scroll_y);
        self.framebuffer[@as(u32, y) * 240 + screen_x] = color;
    }
}
```

**Waarom toepassen vóór de tile-lookup?** Mosaic werkt door meerdere pixels dezelfde kleur te geven — de kleur van de "referentie-pixel" (linkerbovenhoek van het blok). Door de coördinaten af te ronden vóór het opzoeken van de tile, zoek je automatisch de kleur van de referentie-pixel op. Het alternatief (renderen en dan pixels kopiëren) zou meer geheugen en een extra pass vereisen.

**Waarom de mosaic-bit in BGCNT checken?** Het MOSAIC register definieert alleen de blokgrootte — het schakelt mosaic niet in. Elke achtergrondlaag heeft een individuele mosaic-bit in zijn BGCNT register. Zo kan de game mosaic aanzetten voor BG0 maar niet voor BG1, terwijl ze dezelfde blokgrootte delen.

**Valkuil:** Vergeet niet dat de scroll offset *na* mosaic-afronding wordt toegevoegd. De mosaic-afronding werkt op de schermcoördinaten, niet op de achtergrondcoördinaten. Dit verschil is subtiel maar belangrijk: als je scroll offset vóór mosaic toepast, verschuiven de mosaic-blokken mee met de achtergrond, wat niet correct is.

### Stap 4: `src/core/ppu.zig` — Sprite mosaic integreren

Bij sprites pas je mosaic toe op de *lokale* coördinaten (positie binnen de sprite), niet op de schermcoördinaten.

```zig
fn render_sprite_pixel(
    bus: *Bus,
    attr: OamAttrs,
    local_x: u32,
    local_y: u32,
    mosaic_h: u4,
    mosaic_v: u4,
) u8 {
    const mosaic_enabled = attr.attr0.mosaic == 1;
    const mx = if (mosaic_enabled) apply_mosaic(local_x, mosaic_h) else local_x;
    const my = if (mosaic_enabled) apply_mosaic(local_y, mosaic_v) else local_y;

    return read_sprite_pixel(bus, attr.attr2, mx, my);
}
```

**Waarom lokale coördinaten?** Een sprite van 32×32 pixels met mosaic_h=3 (blokken van 4) moet intern in 4×4 blokken verdeeld worden, ongeacht waar de sprite op het scherm staat. Als je schermcoördinaten zou gebruiken, zouden de mosaic-blokken vast aan het scherm zitten en zou een bewegende sprite er anders uitzien op elke positie.

**Waarom is sprite mosaic onafhankelijk van BG mosaic?** Het MOSAIC register heeft aparte velden: `obj_h`/`obj_v` voor sprites en `bg_h`/`bg_v` voor achtergronden. Een game kan sprites met grote blokken pixeleren terwijl de achtergrond fijner gepixeleerd is, of andersom.

### Stap 5: `src/core/ppu.zig` — Mosaic in de rendering pipeline plaatsen

Controleer dat mosaic op de juiste plek in de pipeline zit: na scrolling, vóór blending.

```
Rendering pipeline:
1. Scroll offset berekenen          ← achtergrondpositie bepalen
2. Mosaic toepassen op coördinaten  ← HIER: coördinaten afronden
3. Tile/pixel opzoeken              ← kleur bepalen
4. Windowing check                  ← is deze pixel zichtbaar?
5. Priority sorting                 ← welke laag is bovenop?
6. Blending/brightness              ← effecten op de finale kleur
```

**Waarom vóór windowing?** Mosaic is een coördinaat-effect, geen kleur-effect. Het beïnvloedt welke pixel je opzoekt, niet of die pixel zichtbaar is. Windowing werkt op schermcoördinaten en moet na mosaic komen — je wilt dat het window de gemosaic-te pixels afkapt, niet dat mosaic de window-grenzen beïnvloedt.

**Waarom vóór blending?** Blending werkt op de uiteindelijke kleuren van twee lagen. Mosaic heeft de kleur al bepaald (door de referentie-pixel op te zoeken). Blending neemt die kleur en mengt hem met de laag eronder. De volgorde is logisch: eerst bepalen welke kleur een pixel heeft (mosaic), dan effecten toepassen op die kleur (blending).

**Tip:** Test mosaic met een simpel geval: zet bg_h=1, bg_v=0 (alleen horizontale mosaic, blokken van 2 pixels). Elke twee horizontale pixels moeten dezelfde kleur hebben, terwijl de verticale resolutie normaal blijft.

## Acceptatiecriteria

- BG mosaic horizontaal en verticaal werkt
- Sprite mosaic werkt onafhankelijk van BG mosaic
- Mosaic grootte 0 = normaal (geen effect)
- Mosaic wordt alleen toegepast als de mosaic-bit in BG/OAM attributen aan staat
