# Story 15.3 — DMA Timing

**Epic:** 15 - Accuracy — Hoge Impact

**Doel:** DMA transfers de juiste hoeveelheid CPU-tijd laten kosten.

## Waarom belangrijk?

In story 10.1 kopiëren DMA transfers data "instant". In werkelijkheid kost elke DMA transfer cycles — en gedurende die tijd is de CPU **gepauzeerd**. Dit beïnvloedt:
- Audio timing (DMA1/2 vullen FIFO — als timing fout is, crackelt audio)
- Raster effects (HBlank DMA moet binnen HBlank passen)
- Algemene game timing

**Waarom blokkeert DMA de CPU?** De GBA heeft maar één bus — DMA en CPU kunnen niet tegelijk het geheugen gebruiken. Wanneer een DMA-transfer start, neemt de DMA-controller de bus over en moet de CPU wachten tot de transfer klaar is. Dit is geen keuze van de software, maar een hardwarebeperking.

**Waarom breekt dit games als de timing fout is?** Stel dat een game een HBlank DMA instelt die 32 halfwords kopieert. Op echte hardware kost dit ~70 cycles, wat binnen de 272 HBlank-cycles past. Als je emulator DMA als "instant" (0 cycles) behandelt, lijkt alles goed — maar de CPU krijgt dan 70 extra cycles die hij op echte hardware niet had. Die extra cycles kunnen ertoe leiden dat de CPU al met de volgende scanline bezig is voordat de PPU verwacht, wat raster effects verschuift of audio-buffers op het verkeerde moment vult.

## Context — DMA cycle kosten

```
DMA cycle kosten = 2 (startup) + N × (src_access + dst_access)

Waarbij:
- N = aantal transfers
- src_access = wait states van het bronadres
- dst_access = wait states van het doeladres
- Eerste transfer van src en dst is non-sequential, rest is sequential
```

## Zig concepten

### DMA met cycle-kosten

```zig
fn execute_transfer(self: *DmaChannel, bus: *Bus) u32 {
    const size: u32 = if (self.control.transfer_type == 1) 4 else 2;
    const width: AccessWidth = if (size == 4) .word else .half;

    var total_cycles: u32 = 2;  // DMA startup overhead
    var src = self.internal_source;
    var dst = self.internal_dest;
    const src_delta = get_addr_delta(self.control.source_control, size);
    const dst_delta = get_addr_delta(self.control.dest_control, size);

    for (0..self.internal_count) |i| {
        // Eerste access = non-sequential, rest = sequential
        const access_type: AccessType = if (i == 0) .nonseq else .seq;

        if (size == 4) {
            bus.write32(dst, bus.read32(src));
        } else {
            bus.write16(dst, bus.read16(src));
        }

        total_cycles += bus.access_cycles(src, access_type, width);
        total_cycles += bus.access_cycles(dst, access_type, width);

        src = @intCast(@as(i64, src) + src_delta);
        dst = @intCast(@as(i64, dst) + dst_delta);
    }

    self.internal_source = src;
    self.internal_dest = dst;
    return total_cycles;
}
```

### CPU pauzeren tijdens DMA

```zig
// In de main loop / scheduler:
fn check_dma(scheduler: *Scheduler, dma: *DmaController, bus: *Bus) void {
    const cycles_used = dma.check_and_transfer(bus);
    if (cycles_used > 0) {
        // DMA heeft cycles verbruikt — CPU was gepauzeerd
        scheduler.current_time += cycles_used;
    }
}
```

### Audio DMA — special timing

DMA1 en DMA2 in "special" mode vullen de audio FIFO's. Ze worden automatisch getriggerd wanneer de FIFO bijna leeg is (na een timer overflow):

```zig
fn on_timer_overflow(self: *Apu, dma: *DmaController, timer_id: u1) void {
    // Pop sample uit FIFO
    const sample = self.fifo_a.pop();

    // Als FIFO bijna leeg is (≤16 samples), trigger DMA refill
    if (self.fifo_a.count <= 16) {
        dma.trigger_audio_dma(1);  // DMA1 vult FIFO A
    }
}
```

## Stap-voor-stap

### Stap 1: `src/core/dma.zig` — Cycle-kosten berekening toevoegen aan execute_transfer

De eerste stap is om `execute_transfer()` aan te passen zodat het een cycle-count retourneert in plaats van `void`.

```zig
fn execute_transfer(self: *DmaChannel, bus: *Bus) u32 {
    const size: u32 = if (self.control.transfer_type == 1) 4 else 2;
    const width: AccessWidth = if (size == 4) .word else .half;

    var total_cycles: u32 = 2;  // DMA startup overhead: altijd 2 cycles
    var src = self.internal_source;
    var dst = self.internal_dest;

    for (0..self.internal_count) |i| {
        const access_type: AccessType = if (i == 0) .nonseq else .seq;

        if (size == 4) {
            bus.write32(dst, bus.read32(src));
        } else {
            bus.write16(dst, bus.read16(src));
        }

        total_cycles += bus.access_cycles(src, access_type, width);
        total_cycles += bus.access_cycles(dst, access_type, width);

        src = @intCast(@as(i64, src) + get_addr_delta(self.control.source_control, size));
        dst = @intCast(@as(i64, dst) + get_addr_delta(self.control.dest_control, size));
    }

    self.internal_source = src;
    self.internal_dest = dst;
    return total_cycles;
}
```

**Waarom 2 cycles startup?** Elke DMA-transfer begint met een 2-cycle "overhead" waarin de DMA-controller de bus overneemt van de CPU. Dit is een hardware-eigenschap van de ARM7TDMI bus arbiter. Zelfs een DMA van 1 woord kost minstens 2 + src_wait + dst_wait cycles.

**Waarom is de eerste access non-sequential?** Net als bij de CPU: de eerste read/write gaat naar een "nieuw" adres (de DMA-controller moet het adres op de bus zetten). Alle daaropvolgende transfers zijn sequential (opeenvolgende adressen), wat goedkoper is op ROM.

**Valkuil:** De source- en destination-access kunnen verschillende wait states hebben. Een DMA van ROM (5 cycles) naar VRAM (1 cycle) kost 6 cycles per transfer, terwijl IWRAM→IWRAM maar 2 cycles per transfer kost. Beide kosten correct berekenen is essentieel.

### Stap 2: `src/core/scheduler.zig` — CPU pauzeren tijdens DMA

De scheduler (of main loop) moet de DMA-cycles aftrekken van de beschikbare CPU-tijd.

```zig
fn check_dma(scheduler: *Scheduler, dma: *DmaController, bus: *Bus) void {
    const cycles_used = dma.check_and_transfer(bus);
    if (cycles_used > 0) {
        // DMA heeft cycles verbruikt — CPU was al die tijd gepauzeerd
        scheduler.current_time += cycles_used;
    }
}
```

**Waarom de scheduler-tijd vooruit zetten?** De CPU en PPU delen dezelfde tijdlijn. Als DMA 70 cycles kost, zijn er 70 cycles verstreken waarin de CPU niets kon doen. Door de scheduler-klok vooruit te zetten, "slaat" de CPU die 70 cycles over en komt alles (PPU-timing, timer-ticks, interrupt-timing) correct uit.

**Waarom niet simpelweg de CPU laten wachten?** In een cycle-accurate emulator zou je de CPU step-voor-step laten draaien en bij elke stap checken of DMA actief is. Maar de meeste GBA-emulators gebruiken een scheduler die events op tijdstippen plaatst. Dan is het eenvoudiger en sneller om de DMA-kosten als één blok bij de scheduler op te tellen.

### Stap 3: `src/core/dma.zig` — Non-sequential/sequential tracking per transfer

Een subtiliteit: als een DMA wisselt van source-regio naar destination-regio, is de eerste access op de *nieuwe* regio ook non-sequential.

```zig
for (0..self.internal_count) |i| {
    // Source read
    const src_access: AccessType = if (i == 0) .nonseq else .seq;
    const data = bus.read(src, width);
    total_cycles += bus.access_cycles(src, src_access, width);

    // Destination write — eerste write naar dst is ALTIJD non-seq
    // (want we wisselen van bus-richting: read → write)
    const dst_access: AccessType = if (i == 0) .nonseq else .seq;
    bus.write(dst, data, width);
    total_cycles += bus.access_cycles(dst, dst_access, width);

    src += src_delta;
    dst += dst_delta;
}
```

**Waarom is de eerste destination-write non-sequential?** Wanneer de bus wisselt van een read-operatie naar een write-operatie (of van het ene adresbereik naar een ander), moet het adres opnieuw op de bus gezet worden. Dit kost een non-sequential cycle. Na de eerste write is de bus al "ingesteld" op het destination-adresbereik, en volgende writes zijn sequential.

**Tip:** In de praktijk zijn de source en destination non-sequential/sequential patronen identiek: beide beginnen non-seq en gaan dan over naar seq. Sommige emulators vereenvoudigen dit door gewoon `i == 0` te controleren voor beide.

### Stap 4: `src/core/dma.zig` — Audio DMA speciale timing

DMA-kanalen 1 en 2 in "special" mode zijn de audio FIFO-vullers. Ze hebben een vaste transfer-grootte van 4 woorden (16 bytes = 4 samples) en worden automatisch getriggerd door een timer overflow.

```zig
fn on_timer_overflow(self: *Apu, dma: *DmaController, timer_id: u1) void {
    // Pop één sample uit de FIFO voor afspelen
    const sample = self.fifo_a.pop();
    self.current_sample_a = sample;

    // Als FIFO laag is (≤16 bytes), trigger DMA refill
    if (self.fifo_a.count <= 16) {
        dma.trigger(.{ .channel = 1, .reason = .special });
    }
}
```

**Waarom is audio DMA timing zo kritiek?** De GBA speelt audio af op een vaste sample rate (bepaald door timer 0 of 1). Elke timer overflow popt één sample uit de FIFO. Als de DMA de FIFO niet op tijd hervult (omdat de timing fout is), raakt de FIFO leeg en hoor je een "klik" of "crack" in het geluid. Als de DMA te vroeg vult, kan de FIFO overlopen en gaan samples verloren. De timing moet exact kloppen.

**Valkuil:** Audio DMA kopieert altijd 4 woorden (32 bytes), ongeacht wat er in het DMA count register staat. Dit is een hardware-eigenschap die in "special" mode geldt. Als je de count uit het register leest, krijg je het verkeerde resultaat.

## Acceptatiecriteria

- DMA transfers kosten de juiste hoeveelheid cycles
- CPU is gepauzeerd tijdens DMA
- Audio DMA timing is correct (geen crackling)
- HBlank DMA past binnen de HBlank periode
