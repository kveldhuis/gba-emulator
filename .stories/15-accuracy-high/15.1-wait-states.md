# Story 15.1 — Wait States (memory access timing)

**Epic:** 15 - Accuracy — Hoge Impact

**Doel:** Correcte cycle-kosten per geheugenregio.

## Waarom belangrijk?

De huidige emulator telt waarschijnlijk 1 cycle per memory access. Maar op echte hardware kost een ROM-read 5 cycles en een IWRAM-read 1 cycle. Games die timing-gevoelig zijn (audio, raster effects) werken alleen correct als deze kosten kloppen.

**Waarom is dit de #1 accuracy-verbetering?** Bijna élke instructie die de CPU uitvoert, bevat minstens één memory access (de instruction fetch). Als die timing niet klopt, loopt de hele emulator te snel of te langzaam. Audio crackt, raster effects verschuiven, en sommige games hangen in timing-loops die wachten op een exact aantal cycles. Wait states correct implementeren is het fundament waarop alle andere timing-verbeteringen bouwen.

## Context — GBA wait states

```
Regio              Default cycles    Opmerkingen
                   (16-bit access)
BIOS               1                 Altijd snel
EWRAM              3                 Extern, relatief traag
IWRAM              1                 Intern, snelst
I/O                1
Palette RAM        1                 +1 als PPU er ook bij wil (contention)
VRAM               1                 +1 bij contention met PPU
OAM                1                 +1 bij contention met PPU
ROM Wait State 0   5 (default)       Configureerbaar via WAITCNT
ROM Wait State 1   5 (default)       Configureerbaar
ROM Wait State 2   5 (default)       Configureerbaar
SRAM               5                 Altijd 8-bit bus
```

### 32-bit access = twee 16-bit accesses

De GBA bus is 16-bit breed voor ROM. Een 32-bit read kost daarom **twee keer** de wait state kosten (sequential + non-sequential).

**Waarom is de bus maar 16-bit?** Kostenbesparing. Een 32-bit bus naar de ROM-chip zou dubbel zoveel pinnen en verbindingen vereisen op het PCB, wat de productiekosten van elke GBA-cartridge verhoogt. Nintendo koos voor een 16-bit bus en compenseerde de snelheidsverlies met de prefetch buffer (zie story 15.2).

```
Type              Uitleg
Non-sequential    Eerste access op een nieuw adres (duurder)
Sequential        Opvolgende adressen (goedkoper, ROM kan "voorlezen")
```

**Waarom het onderscheid non-sequential/sequential?** ROM-chips bevatten een intern adres-register. Bij een sequential access hoeft alleen de lage bits van het adres te veranderen — de ROM kan al "voorbereid" zijn. Bij een non-sequential access moet het hele adres opnieuw ingesteld worden, wat extra klokcycli kost.

## Zig concepten

### WAITCNT register (0x04000204)

Games kunnen de wait states configureren. **Waarom?** Verschillende ROM-chips hebben verschillende snelheden. Een game die op een snelle ROM-chip zit, kan lagere wait states instellen om sneller te draaien. De BIOS stelt bij het opstarten een default in, maar games overschrijven dit vaak direct in hun initialisatiecode.

```zig
const WaitCnt = packed struct(u16) {
    sram_wait: u2,        // SRAM wait: 0=4, 1=3, 2=2, 3=8 cycles
    ws0_nonseq: u2,       // ROM WS0 non-sequential: 0=4, 1=3, 2=2, 3=8
    ws0_seq: u1,          // ROM WS0 sequential: 0=2, 1=1
    ws1_nonseq: u2,       // ROM WS1 non-sequential
    ws1_seq: u1,          // ROM WS1 sequential
    ws2_nonseq: u2,       // ROM WS2 non-sequential
    ws2_seq: u1,          // ROM WS2 sequential
    phi_terminal: u2,     // PHI terminal output (niet relevant)
    _unused: u1,
    prefetch: u1,         // Prefetch buffer enable (zie story 15.2)
    game_type: u1,        // 0=GBA, 1=CGB (niet relevant)
};
```

### Cycle-kosten lookup

```zig
const nonseq_wait = [4]u4{ 4, 3, 2, 8 };
const seq_wait_01 = [2]u4{ 2, 1 };
const seq_wait_2  = [2]u4{ 8, 1 };

fn access_cycles(self: *Bus, address: u32, access_type: enum { seq, nonseq }, width: enum { half, word }) u32 {
    const region = (address >> 24) & 0xF;
    const base: u32 = switch (region) {
        0x00 => 1,                        // BIOS
        0x02 => 3,                        // EWRAM
        0x03 => 1,                        // IWRAM
        0x04 => 1,                        // I/O
        0x05 => 1,                        // Palette
        0x06 => 1,                        // VRAM
        0x07 => 1,                        // OAM
        0x08, 0x09 => blk: {              // ROM WS0
            break :blk if (access_type == .nonseq)
                nonseq_wait[self.waitcnt.ws0_nonseq]
            else
                seq_wait_01[self.waitcnt.ws0_seq];
        },
        0x0A, 0x0B => blk: {              // ROM WS1
            break :blk if (access_type == .nonseq)
                nonseq_wait[self.waitcnt.ws1_nonseq]
            else
                seq_wait_01[self.waitcnt.ws1_seq];
        },
        0x0C, 0x0D => blk: {              // ROM WS2
            break :blk if (access_type == .nonseq)
                nonseq_wait[self.waitcnt.ws2_nonseq]
            else
                seq_wait_2[self.waitcnt.ws2_seq];
        },
        0x0E => nonseq_wait[self.waitcnt.sram_wait] + 1,  // SRAM
        else => 1,
    };

    // 32-bit access op 16-bit bus = 2 accesses
    if (width == .word and (region >= 0x08 and region <= 0x0D or region == 0x02)) {
        return base + seq_wait(region);  // non-seq + seq
    }
    return base;
}
```

### Labeled blocks met `blk:`

```zig
const value = blk: {
    // bereken iets complex
    break :blk result;  // "return" uit het blok met een waarde
};
```

Dit is handig in switch-arms waar je meer dan één expressie nodig hebt.

### Integratie met CPU step

```zig
pub fn step(self: *Cpu, bus: *Bus) u32 {
    // Fetch = non-sequential als er net een branch was, anders sequential
    const fetch_type: AccessType = if (self.branched) .nonseq else .seq;
    const fetch_cycles = bus.access_cycles(self.r[15], fetch_type, .word);
    self.branched = false;

    // ... decode en execute ...
    // Data accesses (LDR/STR) voegen hun eigen cycles toe

    return fetch_cycles + execute_cycles;
}
```

## Stap-voor-stap

### Stap 1: `src/core/bus.zig` — WAITCNT register definiëren als packed struct

Definieer het WAITCNT register als een `packed struct(u16)`. Dit is het eerste wat je doet, want alle cycle-berekeningen hangen af van de configuratie in dit register.

```zig
const WaitCnt = packed struct(u16) {
    sram_wait: u2,        // bits 0-1: SRAM wait cycles
    ws0_nonseq: u2,       // bits 2-3: ROM WS0 non-sequential
    ws0_seq: u1,          // bit 4: ROM WS0 sequential
    ws1_nonseq: u2,       // bits 5-6: ROM WS1 non-sequential
    ws1_seq: u1,          // bit 7: ROM WS1 sequential
    ws2_nonseq: u2,       // bits 8-9: ROM WS2 non-sequential
    ws2_seq: u1,          // bit 10: ROM WS2 sequential
    phi_terminal: u2,     // bits 11-12: niet relevant voor emulatie
    _unused: u1,          // bit 13
    prefetch: u1,         // bit 14: prefetch buffer enable
    game_type: u1,        // bit 15: 0=GBA, 1=CGB
};
```

**Waarom een packed struct?** Omdat het register bitwise gedefinieerd is door de hardware. Met een `packed struct(u16)` kun je een `u16` direct casten naar de struct met `@bitCast`, en Zig garandeert dat de velden exact op de juiste bits liggen. Geen handmatige bit-shifting nodig.

**Tip:** Voeg het veld `waitcnt: WaitCnt = @bitCast(@as(u16, 0))` toe aan je Bus struct. De default waarde 0 komt overeen met de langzaamste instellingen (4 cycles non-seq, 2 cycles seq) — dit is correct, want de BIOS stelt de echte waarden in tijdens het opstarten.

### Stap 2: `src/core/bus.zig` — lookup tabellen voor cycle-kosten aanmaken

Maak constante arrays die de bitwaarden uit WAITCNT vertalen naar daadwerkelijke cycle-aantallen.

```zig
// Non-sequential wait: 2-bit veld → 4, 3, 2, of 8 cycles
const nonseq_lut = [4]u32{ 4, 3, 2, 8 };

// Sequential wait voor WS0 en WS1: 1-bit veld → 2 of 1 cycles
const seq_lut_01 = [2]u32{ 2, 1 };

// Sequential wait voor WS2: 1-bit veld → 8 of 1 cycles
const seq_lut_2 = [2]u32{ 8, 1 };
```

**Waarom lookup tabellen en geen formule?** De mapping van bits naar cycles is niet lineair — kijk naar de non-sequential waarden: 0→4, 1→3, 2→2, 3→8. Er is geen simpele formule. Een lookup tabel is zowel sneller als duidelijker. De `[4]u32` syntax is een array van 4 elementen, geïndexeerd door de 2-bit waarde uit het register.

**Valkuil:** WS2 sequential is anders dan WS0/WS1 — de "trage" optie is 8 cycles in plaats van 2. Dit is makkelijk te missen als je ze allemaal dezelfde tabel laat delen.

### Stap 3: `src/core/bus.zig` — `access_cycles()` functie implementeren

Dit is de kernfunctie die voor elk adres en access-type het juiste aantal cycles retourneert.

```zig
pub fn access_cycles(self: *Bus, address: u32, access_type: AccessType, width: AccessWidth) u32 {
    const region = (address >> 24) & 0xF;
    const base: u32 = switch (region) {
        0x00 => 1,                                              // BIOS
        0x02 => 3,                                              // EWRAM (altijd 3)
        0x03 => 1,                                              // IWRAM (snel intern geheugen)
        0x04 => 1,                                              // I/O registers
        0x05 => 1,                                              // Palette RAM
        0x06 => 1,                                              // VRAM
        0x07 => 1,                                              // OAM
        0x08, 0x09 => if (access_type == .nonseq)               // ROM WS0
            nonseq_lut[self.waitcnt.ws0_nonseq]
        else
            seq_lut_01[self.waitcnt.ws0_seq],
        0x0A, 0x0B => if (access_type == .nonseq)               // ROM WS1
            nonseq_lut[self.waitcnt.ws1_nonseq]
        else
            seq_lut_01[self.waitcnt.ws1_seq],
        0x0C, 0x0D => if (access_type == .nonseq)               // ROM WS2
            nonseq_lut[self.waitcnt.ws2_nonseq]
        else
            seq_lut_2[self.waitcnt.ws2_seq],
        0x0E, 0x0F => nonseq_lut[self.waitcnt.sram_wait] + 1,  // SRAM
        else => 1,                                              // unmapped
    };

    // 32-bit access op 16-bit bus = twee halve accesses
    if (width == .word and is_16bit_bus(region)) {
        return base + self.seq_cycles(region);
    }
    return base;
}
```

**Waarom `(address >> 24) & 0xF`?** Het GBA geheugenmodel verdeelt het 32-bit adresbereik in regio's gebaseerd op bits 24-27. Door 24 bits naar rechts te schuiven en te masken met 0xF krijg je een getal van 0-15 dat de regio identificeert. Dit is de snelste manier om een adres te classificeren — sneller dan bereik-checks.

**Waarom twee regio's per ROM wait state (bijv. 0x08 en 0x09)?** Elke ROM wait state dekt een 32MB adresruimte, maar de GBA adresseert ze in blokken van 16MB (één nibble). ROM WS0 begint op 0x08000000 en loopt tot 0x09FFFFFF, dus zowel region 0x08 als 0x09 gebruiken dezelfde wait state configuratie.

**Valkuil:** SRAM (0x0E) is altijd 8-bit, zelfs als je een 16-bit of 32-bit read doet. De extra `+1` cycle in de code hierboven compenseert voor de 8-bit bus. Sommige bronnen noteren SRAM wait states zonder die +1 — controleer je referentiedocumentatie.

### Stap 4: `src/core/cpu.zig` — CPU step uitbreiden met cycle-tracking

Nu de bus cycles kan berekenen, moet de CPU deze gebruiken. Elke CPU-stap retourneert het totaal aantal cycles dat hij gekost heeft.

```zig
pub fn step(self: *Cpu, bus: *Bus) u32 {
    // Fetch: non-sequential na een branch, anders sequential
    const fetch_type: AccessType = if (self.branched) .nonseq else .seq;
    const width: AccessWidth = if (self.cpsr.t == 1) .half else .word;
    const fetch_cycles = bus.access_cycles(self.r[15], fetch_type, width);
    self.branched = false;

    const instruction = if (self.cpsr.t == 1)
        @as(u32, bus.read16(self.r[15]))
    else
        bus.read32(self.r[15]);

    // Decode + execute (retourneert extra cycles voor data accesses)
    const execute_cycles = self.execute(instruction, bus);

    return fetch_cycles + execute_cycles;
}
```

**Waarom `branched` bijhouden?** Na een branch wijst de program counter (R15) naar een compleet nieuw adres. De ROM-chip kan niet "voorspellen" waar we naartoe springen, dus de eerste fetch op het nieuwe adres is altijd non-sequential (duurder). Bij lineaire code-uitvoering is elke volgende fetch sequential (goedkoper), omdat de adressen opeenvolgend zijn.

**Waarom de width afhankelijk van Thumb mode?** In ARM mode zijn instructies 32-bit, in Thumb mode 16-bit. Een 32-bit fetch op de 16-bit ROM bus kost twee bus-transfers, een 16-bit fetch maar één. Dit is precies waarom Thumb code op ROM sneller draait dan ARM code — een cruciale optimalisatie die GBA-ontwikkelaars vaak gebruiken.

### Stap 5: `src/core/bus.zig` — WAITCNT schrijfbaar maken via I/O

Vergeet niet om het WAITCNT register schrijfbaar te maken in je I/O write handler:

```zig
fn io_write16(self: *Bus, address: u32, value: u16) void {
    switch (address) {
        // ... andere I/O registers ...
        0x04000204 => {
            self.waitcnt = @bitCast(value);
        },
        // ...
    }
}
```

**Waarom is schrijfbaarheid belangrijk?** Games stellen hun eigen wait states in bij het opstarten. Pokémon Ruby schrijft bijvoorbeeld `0x4317` naar WAITCNT, wat WS0 op 3/1 cycles zet (non-seq/seq). Als je WAITCNT niet schrijfbaar maakt, draait de hele game op de default (langzame) wait states, wat de timing volledig verstoort.

**Tip:** Sommige bits in WAITCNT zijn read-only of ongebruikt. Mask de waarde bij het schrijven: `self.waitcnt = @bitCast(value & 0xDFFF)` (bit 13 is ongebruikt).

**Valkuil:** Vergeet niet om ook een read-handler te implementeren. Sommige games lezen WAITCNT terug om te controleren wat ze geschreven hebben.

## Acceptatiecriteria

- CPU step retourneert het juiste aantal cycles
- WAITCNT register is schrijfbaar en beïnvloedt timing
- ROM access is trager dan IWRAM access
- 32-bit ROM reads kosten dubbel
