# Story 14.1 — Cross-platform builds (Windows, macOS, Linux)

**Epic:** 14 - Cross-Platform & Distributie

**Doel:** De emulator bouwen en distribueren voor Windows, macOS en Linux.

## Zig concepten

### Zig's ingebouwde cross-compilation

Het mooie aan Zig: cross-compilatie zit er standaard in. Je hoeft geen aparte toolchain te installeren zoals bij C/C++. Vanuit Linux kun je direct een Windows `.exe` bouwen.

**Waarom is dit bijzonder?** Bij C/C++ moet je voor elk doelplatform een aparte toolchain installeren en configureren: `mingw-w64` voor Windows, `osxcross` voor macOS, een ARM cross-compiler voor Raspberry Pi. Elke toolchain heeft eigen quirks, versieconflicten en configuratieproblemen. Zig lost dit fundamenteel op: de compiler bundelt LLVM als backend én bevat eigen libc-implementaties voor alle ondersteunde targets. Dat betekent dat `zig build -Dtarget=x86_64-windows` direct werkt zonder *iets* extra te installeren — zelfs vanuit een Linux devcontainer. Dit maakt Zig uniek geschikt voor projecten die meerdere platforms moeten ondersteunen, zoals onze emulator.

```bash
# Bouw voor je huidige platform (in devcontainer = Linux):
zig build

# Bouw voor Windows:
zig build -Dtarget=x86_64-windows

# Bouw voor macOS (Intel):
zig build -Dtarget=x86_64-macos

# Bouw voor macOS (Apple Silicon):
zig build -Dtarget=aarch64-macos

# Bouw voor Linux (ARM, bijv. Raspberry Pi):
zig build -Dtarget=aarch64-linux
```

### SDL3 als dependency bundelen

Het probleem met `linkSystemLibrary("SDL3")` is dat SDL3 op het systeem geïnstalleerd moet zijn. Voor distributie wil je SDL3 meebundelen. Zig kan dit via het build system.

**Waarom bundelen in plaats van system library?** Er zijn drie belangrijke redenen:

1. **Gebruikerservaring:** Je wilt niet dat gebruikers eerst SDL3 moeten installeren voordat ze je emulator kunnen draaien. Dat is een enorme drempel — veel mensen haken dan af.
2. **Versieconsistentie:** Verschillende systemen kunnen verschillende SDL3-versies hebben. Jij hebt getest met versie X, maar de gebruiker heeft versie Y met subtiele gedragsverschillen of bugs. Door te bundelen garandeer je exact de versie waarmee je hebt getest.
3. **Cross-compilatie:** Als je `linkSystemLibrary` gebruikt, moet SDL3 op het *bouw*systeem geïnstalleerd zijn voor het *doel*platform. Dat is bijna onmogelijk — je Linux-machine heeft geen Windows SDL3-headers. Met een Zig package dependency bouwt Zig SDL3 zelf vanuit broncode voor elk doelplatform.

```zig
// build.zig
pub fn build(b: *std.Build) void {
    const target = b.standardTargetOptions(.{});
    const optimize = b.standardOptimizeOption(.{});

    const exe = b.addExecutable(.{
        .name = "gba-emu",
        .root_module = b.createModule(.{
            .root_source_file = b.path("src/main.zig"),
            .target = target,
            .optimize = optimize,
        }),
    });

    // Optie 1: System library (werkt alleen als SDL3 geïnstalleerd is)
    // exe.root_module.linkSystemLibrary("SDL3");

    // Optie 2: SDL3 als Zig package dependency (aanbevolen!)
    // In build.zig.zon voeg je SDL3 toe als dependency
    // Dan kan Zig het zelf bouwen voor elk platform
    const sdl_dep = b.dependency("SDL3", .{
        .target = target,
        .optimize = optimize,
    });
    exe.linkLibrary(sdl_dep.artifact("SDL3"));

    exe.root_module.linkLibC();
    b.installArtifact(exe);
}
```

### SDL3 als Zig dependency toevoegen

In `build.zig.zon` voeg je SDL3 toe:

```zig
.{
    .name = .gba_emu,
    .version = "0.1.0",
    .fingerprint = 0xabcdef1234567890,
    .minimum_zig_version = "0.15.2",
    .dependencies = .{
        .SDL3 = .{
            // Zig package voor SDL3 die cross-compilatie ondersteunt
            .url = "https://github.com/andrewrk/SDL.zig/archive/<commit>.tar.gz",
            .hash = "...",
        },
    },
    .paths = .{
        "build.zig",
        "build.zig.zon",
        "src",
    },
}
```

**Let op:** `.name` moet een geldige Zig identifier zijn (geen streepjes, geen aanhalingstekens). `.fingerprint` is verplicht sinds Zig 0.15 — als je het weglaat, suggereert de compiler een waarde.

### Target detectie in code

Soms moet je code per platform aanpassen:

```zig
const builtin = @import("builtin");

// Check het huidige platform
if (builtin.os.tag == .windows) {
    // Windows-specifieke code
} else if (builtin.os.tag == .macos) {
    // macOS-specifieke code
} else {
    // Linux/andere
}

// Dit wordt COMPILE-TIME geëvalueerd
// Code voor andere platforms wordt weggeoptimaliseerd
```

### Comptime if — code die verdwijnt

```zig
const is_windows = builtin.os.tag == .windows;

// Op Windows heeft SDL een andere main() signature
pub fn main() void {
    if (comptime is_windows) {
        // Dit bestaat NIET in de Linux/macOS binary
        // De compiler gooit het weg
    }
}
```

### Paden en bestandssysteem

Windows gebruikt `\` als pad-scheider, Linux/macOS gebruiken `/`. Zig's `std.fs` regelt dit automatisch:

```zig
const std = @import("std");

fn load_rom(path: []const u8) ![]u8 {
    // std.fs werkt op alle platforms
    const file = try std.fs.cwd().openFile(path, .{});
    defer file.close();
    return try file.readToEndAlloc(allocator, max_size);
}
```

### Release builds

Voor distributie wil je een geoptimaliseerde build:

```bash
# Debug (standaard): traag, met debug info
zig build

# ReleaseSafe: snel, met safety checks (overflow, bounds, etc.)
zig build -Doptimize=ReleaseSafe

# ReleaseFast: maximale snelheid, geen safety checks
zig build -Doptimize=ReleaseFast

# ReleaseSmall: kleinste binary
zig build -Doptimize=ReleaseSmall
```

Voor een emulator is `ReleaseSafe` een goede balans: snel genoeg, maar bugs worden nog gevangen.

**Waarom ReleaseSafe voor emulators?** Emulators zijn bijzonder complexe software met enorm veel bit-manipulatie, shift-operaties en array-indexering. Denk aan het decoderen van CPU-instructies, het aanspreken van geheugenregio's via berekende adressen, en het manipuleren van hardware-registers. Dit zijn precies de plekken waar bugs zich verstoppen: een integer overflow bij een adresberekening, een off-by-one bij een geheugen-lookup, een shift die te ver gaat. `ReleaseSafe` vangt deze fouten op met runtime checks, terwijl `ReleaseFast` ze stilletjes laat passeren — wat kan leiden tot corrupte emulatie-state die pas veel later zichtbaar wordt. Het prestatieverschil tussen `ReleaseSafe` en `ReleaseFast` is voor emulators typisch minder dan 5%, dus de extra veiligheid is de minimale snelheidsverlies meer dan waard.

## Context

- Zig heeft ingebouwde cross-compilation (geen extra toolchain nodig)
- SDL3 moet gebundeld worden voor eenvoudige distributie
- Platform-specifieke code is minimaal dankzij SDL3 abstractie
- Belangrijkste targets: Windows x86_64, macOS x86_64 + aarch64, Linux x86_64

## Bestanden

- `build.zig` — target options, SDL3 dependency, release configuratie
- `build.zig.zon` — SDL3 als package dependency
- Eventueel een `build-all.sh` script om voor alle platforms te bouwen

## Stap-voor-stap

### Stap 1: `build.zig` — Target options configureren

De eerste stap is het configureren van je `build.zig` zodat het cross-compilatie en optimalisatie-opties ondersteunt. Zig's build system biedt hiervoor twee standaardfuncties die command-line opties toevoegen:

```zig
const std = @import("std");

pub fn build(b: *std.Build) void {
    // Laat de gebruiker het doelplatform kiezen via -Dtarget=...
    const target = b.standardTargetOptions(.{});

    // Laat de gebruiker het optimalisatieniveau kiezen via -Doptimize=...
    const optimize = b.standardOptimizeOption(.{});

    const exe = b.addExecutable(.{
        .name = "gba-emu",
        .root_module = b.createModule(.{
            .root_source_file = b.path("src/main.zig"),
            .target = target,
            .optimize = optimize,
        }),
    });

    b.installArtifact(exe);
}
```

**Waarom `standardTargetOptions` en `standardOptimizeOption`?** Deze functies voegen automatisch de `-Dtarget` en `-Doptimize` command-line opties toe aan je build. Zonder deze functies zou je voor elk platform een aparte build-configuratie moeten schrijven. Nu kan dezelfde `build.zig` bouwen voor elk platform — de gebruiker (of je CI/CD pipeline) kiest het doel via de command line. Als geen target wordt opgegeven, bouwt Zig voor het huidige platform (native). Als geen optimize wordt opgegeven, bouwt Zig in debug-modus.

**Waarom `target` en `optimize` aan `createModule` meegeven?** De target en optimize opties moeten doorgesluisd worden naar de module zodat de compiler weet voor welk platform en met welke optimalisatie hij moet compileren. Dit geldt niet alleen voor je eigen code, maar ook voor alle dependencies — zij krijgen dezelfde target en optimize mee, zodat alles consistent wordt gebouwd.

> **Tip:** Test altijd eerst met je native platform (`zig build` zonder `-Dtarget`) voordat je cross-compileert. Als je code niet werkt op je eigen machine, werkt het zeker niet op een ander platform. Native builds zijn ook sneller en makkelijker te debuggen.

### Stap 2: `build.zig.zon` — SDL3 als package dependency toevoegen

Nu moeten we SDL3 als dependency toevoegen zodat Zig het automatisch kan downloaden en bouwen:

```zig
.{
    .name = .gba_emu,
    .version = "0.1.0",
    .fingerprint = 0xabcdef1234567890,
    .minimum_zig_version = "0.15.2",
    .dependencies = .{
        .SDL3 = .{
            .url = "https://github.com/andrewrk/SDL.zig/archive/<commit>.tar.gz",
            .hash = "...",
        },
    },
    .paths = .{
        "build.zig",
        "build.zig.zon",
        "src",
    },
}
```

**Waarom een Zig package in plaats van een system library?** Een system library (`linkSystemLibrary`) vereist dat de bibliotheek al geïnstalleerd is op het systeem — en dan specifiek voor het *doel*platform. Op je Linux-machine heb je geen Windows-versie van SDL3 staan, dus cross-compilatie is onmogelijk met system libraries. Een Zig package dependency daarentegen bevat de broncode, en Zig compileert die broncode voor elk doelplatform. Bovendien is de versie vastgepind: iedereen die je project bouwt, gebruikt exact dezelfde SDL3-versie.

**Hoe vind je de juiste URL en hash?** Gebruik het `zig fetch` commando:

```bash
# Download het package en bereken de hash automatisch
zig fetch --save "https://github.com/andrewrk/SDL.zig/archive/<commit>.tar.gz"
```

Dit commando downloadt het archief, berekent de hash, en voegt de dependency automatisch toe aan je `build.zig.zon`. De `--save` flag zorgt ervoor dat je het niet handmatig hoeft over te typen.

**Wat is `.fingerprint`?** Sinds Zig 0.15 is een fingerprint verplicht in `build.zig.zon`. Dit is een unieke identifier voor je package die helpt bij het voorkomen van naamconflicten in de package-cache. Als je het weglaat bij een nieuwe `build.zig.zon`, geeft de compiler een foutmelding met een gesuggereerde waarde die je kunt overnemen.

**Wat is `.minimum_zig_version`?** Dit veld documenteert de minimale Zig-versie die nodig is om je project te bouwen. Het is optioneel maar aan te raden — het voorkomt verwarrende foutmeldingen als iemand met een te oude Zig-versie probeert te bouwen.

> **Valkuil:** De hash MOET exact kloppen — één bit verschil en Zig weigert het package te gebruiken. Dit is een beveiligingsmaatregel: het garandeert dat niemand de broncode heeft gewijzigd na publicatie. Gebruik altijd `zig fetch --save` om de juiste hash te krijgen in plaats van hem handmatig in te vullen.

### Stap 3: `build.zig` — SDL3 linken via dependency

Nu koppelen we de SDL3 dependency aan ons executable in `build.zig`:

```zig
const std = @import("std");

pub fn build(b: *std.Build) void {
    const target = b.standardTargetOptions(.{});
    const optimize = b.standardOptimizeOption(.{});

    const exe = b.addExecutable(.{
        .name = "gba-emu",
        .root_module = b.createModule(.{
            .root_source_file = b.path("src/main.zig"),
            .target = target,
            .optimize = optimize,
        }),
    });

    // Haal de SDL3 dependency op uit build.zig.zon
    const sdl_dep = b.dependency("SDL3", .{
        .target = target,
        .optimize = optimize,
    });

    // Link de SDL3 library aan ons executable
    exe.linkLibrary(sdl_dep.artifact("SDL3"));

    // SDL3 is geschreven in C, dus we moeten libc meelinken
    exe.root_module.linkLibC();

    b.installArtifact(exe);
}
```

**Waarom `b.dependency("SDL3", ...)`?** Deze functie zoekt de dependency met de naam `SDL3` op in je `build.zig.zon`, downloadt de broncode (als dat nog niet is gebeurd), en voert het build-script van die dependency uit. De `.target` en `.optimize` opties worden doorgegeven zodat SDL3 voor hetzelfde platform en met dezelfde optimalisatie wordt gebouwd als je eigen code.

**Waarom `linkLibrary` in plaats van `linkSystemLibrary`?** `linkSystemLibrary` zoekt naar een al-geïnstalleerde library via pkg-config of standaard systeempaden. `linkLibrary` daarentegen linkt tegen een library die door het Zig build system zelf is gebouwd — in dit geval SDL3 vanuit broncode. Het verschil is cruciaal voor cross-compilatie: `linkSystemLibrary("SDL3")` zou zoeken naar een SDL3 die gecompileerd is voor het doelplatform, en die heb je waarschijnlijk niet op je bouwmachine.

**Waarom `linkLibC()`?** SDL3 is geschreven in C en maakt gebruik van de C standard library (voor geheugenallocatie, string-operaties, enz.). Zonder `linkLibC()` krijg je linker-fouten over ontbrekende symbolen zoals `malloc`, `free`, `memcpy`, etc. Zig bundelt libc-implementaties voor alle ondersteunde platforms, dus dit werkt ook bij cross-compilatie.

> **Tip:** Zorg dat de dependency-naam in `b.dependency("SDL3", ...)` EXACT overeenkomt met de naam in `build.zig.zon` (het `.SDL3` veld onder `.dependencies`). Een typfout hier geeft een cryptische foutmelding over een onbekende dependency.

### Stap 4: `src/platform/` — Platform-specifieke code isoleren

Hoewel SDL3 het meeste platformverschil abstraheert, is er soms toch platform-specifieke code nodig. Isoleer die in een aparte directory:

```zig
// src/platform/detect.zig
const builtin = @import("builtin");
const std = @import("std");

pub const Platform = enum {
    windows,
    macos,
    linux,
    other,
};

/// Geeft het huidige platform terug (compile-time bekend)
pub const current: Platform = switch (builtin.os.tag) {
    .windows => .windows,
    .macos => .macos,
    .linux => .linux,
    else => .other,
};

/// Platform-specifieke standaard ROM-directory
pub fn getDefaultRomDir() []const u8 {
    return switch (builtin.os.tag) {
        .windows => "C:\\Users\\Public\\GBA-Emu\\roms",
        .macos => "/Users/Shared/GBA-Emu/roms",
        .linux => "/home/" ++ "user" ++ "/.local/share/gba-emu/roms",
        else => "./roms",
    };
}

/// Geeft het juiste pad-scheidingsteken voor het huidige platform
pub const path_separator = std.fs.path.sep;
```

**Waarom platform-code isoleren?** Door alle platform-specifieke logica in `src/platform/` te verzamelen, hoef je bij het toevoegen van een nieuw platform maar op één plek te kijken. De rest van je emulator-code importeert `@import("platform/detect.zig")` en hoeft niet te weten op welk platform het draait. Dit is het *separation of concerns* principe.

**Waarom minimaliseert SDL3 platformverschillen?** SDL3 abstraheert het aanmaken van vensters, audio-output, controller-input en meer. Zonder SDL3 zou je voor elk platform aparte code moeten schrijven: Win32 API voor Windows, Cocoa voor macOS, X11/Wayland voor Linux. Met SDL3 schrijf je één keer `SDL_CreateWindow()` en het werkt overal. De enige platform-specifieke code die overblijft gaat over zaken die SDL3 niet afdekt: bestandsdialogen, systeemnotificaties, of platformspecifieke standaardpaden.

**Compile-time platform detectie:** Alle `builtin.os.tag` checks worden op compile-time geëvalueerd. Dat betekent dat als je bouwt voor Windows, alle Linux/macOS code-paden door de compiler worden verwijderd — ze bestaan letterlijk niet in de uiteindelijke binary. Er is dus geen runtime overhead.

> **Tip:** Gebruik `std.fs.path` voor pad-manipulatie in plaats van handmatig strings samen te voegen. `std.fs.path.join` gebruikt automatisch het juiste scheidingsteken (`\` op Windows, `/` op Linux/macOS) en normaliseert paden correct. Dit voorkomt subtiele bugs met paden die op het ene platform werken maar op het andere niet.

### Stap 5: `build-all.sh` — Build script voor alle platforms

Maak een script dat voor alle doelplatforms bouwt, zodat je dit niet elke keer handmatig hoeft te doen:

```bash
#!/bin/bash
# build-all.sh — Bouw de emulator voor alle ondersteunde platforms

set -e  # Stop bij de eerste fout

VERSION="0.1.0"
BUILD_DIR="zig-out/release"

echo "=== GBA Emulator v${VERSION} — Cross-platform build ==="
echo ""

# Lijst van alle targets
TARGETS=(
    "x86_64-linux"
    "aarch64-linux"
    "x86_64-windows"
    "aarch64-macos"
    "x86_64-macos"
)

for target in "${TARGETS[@]}"; do
    echo "--- Bouwen voor ${target} ---"

    # Bouw met ReleaseSafe (aanbevolen voor emulators)
    zig build \
        -Dtarget="${target}" \
        -Doptimize=ReleaseSafe \
        --prefix "${BUILD_DIR}/${target}" \
        2>&1

    if [ $? -eq 0 ]; then
        echo "  ✓ ${target} succesvol gebouwd"
    else
        echo "  ✗ ${target} MISLUKT"
    fi
    echo ""
done

echo "=== Alle builds voltooid ==="
echo "Output in: ${BUILD_DIR}/"
ls -la "${BUILD_DIR}/"
```

**Waarom dit automatiseren?** Zonder script is het verleidelijk om "even snel" alleen voor je eigen platform te bouwen en de rest over te slaan. Maar cross-compilatie kan breken door code-wijzigingen die op je eigen platform prima werken — bijvoorbeeld een Linux-specifieke syscall die je per ongeluk buiten een platform-check hebt gebruikt. Een build script dat alle platforms bouwt, vangt dit meteen op.

**Waarom `--prefix` voor output-directories?** Standaard plaatst `zig build` alles in `zig-out/bin/`. Met `--prefix` kun je per platform een aparte output-directory maken. Zo overschrijft de Windows-build niet de Linux-build, en kun je alle binaries naast elkaar bewaren voor distributie.

**Waarom `set -e`?** Dit zorgt ervoor dat het script stopt bij de eerste fout. Zonder `set -e` zou het script doorgaan na een mislukte build, en zou je de foutmelding kunnen missen tussen alle output.

> **Valkuil:** Niet alle targets werken altijd — een update van SDL3 of Zig kan cross-compilatie voor een specifiek platform tijdelijk breken. Test regelmatig of alle targets nog bouwen, bij voorkeur in een CI/CD pipeline die bij elke commit alle platforms bouwt.

### Stap 6: Testen en distribueren

Na het bouwen wil je verifiëren dat de cross-compiled binaries ook echt werken, en ze klaarmaken voor distributie.

**Cross-compiled binaries testen:**

Je kunt niet zomaar een Windows `.exe` draaien op Linux, maar er zijn hulpmiddelen:

```bash
# Test de Windows binary met Wine
wine zig-out/release/x86_64-windows/bin/gba-emu.exe

# Test ARM binaries met QEMU (als je op x86_64 zit)
qemu-aarch64 zig-out/release/aarch64-linux/bin/gba-emu

# macOS binaries kun je helaas niet testen op Linux
# Daarvoor heb je een echte Mac of macOS CI nodig (bijv. GitHub Actions)
```

**Waarom testen met Wine/QEMU?** Het feit dat een binary compileert, betekent niet dat het correct werkt. Wine emuleert de Windows API op Linux, zodat je basale functionaliteit kunt testen zonder een Windows-machine. QEMU emuleert een andere CPU-architectuur, zodat je ARM-binaries kunt testen op een x86_64-machine. Beide zijn niet perfect (subtiele gedragsverschillen zijn mogelijk), maar ze vangen de meeste problemen op.

**Optimalisatieniveaus kiezen:**

| Niveau | Snelheid | Veiligheid | Binary grootte | Wanneer gebruiken |
|--------|----------|------------|-----------------|-------------------|
| Debug | Traag | Maximaal | Groot | Tijdens ontwikkeling |
| ReleaseSafe | Snel | Hoog | Gemiddeld | **Distributie (aanbevolen)** |
| ReleaseFast | Maximaal | Geen | Gemiddeld | Alleen als performance kritiek is |
| ReleaseSmall | Gemiddeld | Geen | Klein | Embedded of web (WASM) |

**Waarom ReleaseSafe voor initiële distributie?** Bij de eerste releases van je emulator zitten er gegarandeerd nog bugs in. `ReleaseSafe` vangt hele categorieën bugs op die anders tot mysterieuze crashes of corrupte emulatie leiden: integer overflows bij adresberekeningen, out-of-bounds geheugentoegangen, en meer. Zodra je emulator stabiel is en je vertrouwen hebt in de correctheid, kun je overwegen om naar `ReleaseFast` te schakelen voor de extra paar procent performance.

**Distributable archives maken:**

```bash
# Maak een zip per platform voor distributie
cd zig-out/release

# Windows
zip -r "gba-emu-v${VERSION}-windows-x64.zip" x86_64-windows/

# macOS (Apple Silicon)
tar -czf "gba-emu-v${VERSION}-macos-arm64.tar.gz" aarch64-macos/

# Linux
tar -czf "gba-emu-v${VERSION}-linux-x64.tar.gz" x86_64-linux/
```

> **Tip:** Voeg een versienummer toe aan je binary met build-time options. Je kunt dit doen via `b.option()` in `build.zig` of door een versie-string als compile-time constante mee te geven. Zo kunnen gebruikers altijd zien welke versie ze draaien, en kun jij bij bugrapporten snel achterhalen welke versie het betreft:
>
> ```zig
> // In build.zig: geef versie mee als build option
> const version = b.option([]const u8, "version", "Versienummer") orelse "dev";
> const options = b.addOptions();
> options.addOption([]const u8, "version", version);
> exe.root_module.addOptions("config", options);
>
> // In src/main.zig: gebruik de versie
> const config = @import("config");
> std.debug.print("GBA Emulator v{s}\n", .{config.version});
> ```

## Acceptatiecriteria

- `zig build -Dtarget=x86_64-windows` produceert een werkende `.exe`
- `zig build -Dtarget=aarch64-macos` produceert een werkende macOS binary
- SDL3 wordt automatisch meegebouwd (geen handmatige installatie nodig)
- Release builds zijn significant sneller dan debug builds
