# Story 6.1 — I/O register dispatcher

**Epic:** 06 - I/O Registers

**Doel:** Reads/writes naar 0x04000000+ doorsturen naar juiste component.

## Zig concepten

### Wat zijn I/O registers?

I/O registers zijn speciale geheugen-adressen waarmee de CPU communiceert met hardware-componenten (PPU, timers, DMA, etc.). Ze staan allemaal in het bereik 0x04000000 - 0x040003FF.

**Waarom memory-mapped I/O?** De GBA (en de meeste embedded systemen) gebruiken "memory-mapped I/O": hardware-registers zijn gewoon geheugen-adressen. De CPU hoeft geen speciale instructies te gebruiken — een gewone `ldr` (lees) of `str` (schrijf) naar het juiste adres communiceert direct met de hardware. Dit is simpel en efficiënt, maar het betekent dat de emulator elke geheugen-access moet controleren: "is dit een normaal RAM-adres, of een I/O register met speciaal gedrag?"

Als de CPU naar adres 0x04000000 schrijft, moet de emulator dat doorsturen naar de PPU (want dat is DISPCNT). Dit is wat de I/O dispatcher doet.

### Switch op adres

In Zig gebruik je een `switch` statement om op basis van het adres te bepalen welke component aangesproken wordt:

```zig
pub fn read16(self: *Io, address: u32) u16 {
    return switch (address) {
        // PPU registers
        0x04000000 => self.ppu.dispcnt,            // DISPCNT
        0x04000004 => self.ppu.dispstat,            // DISPSTAT
        0x04000006 => @as(u16, self.ppu.vcount),   // VCOUNT (read-only)

        // Input
        0x04000130 => self.keyinput,                // KEYINPUT (read-only)

        // Interrupts
        0x04000200 => self.irq.ie,                  // IE
        0x04000202 => self.irq.if_,                 // IF
        0x04000208 => @intFromBool(self.irq.ime),   // IME

        // Onbekend register
        else => 0,
    };
}
```

### Write-only en Read-only registers

Sommige registers kunnen alleen gelezen of alleen geschreven worden:

```zig
pub fn write16(self: *Io, address: u32, value: u16) void {
    switch (address) {
        // PPU registers (read/write)
        0x04000000 => self.ppu.dispcnt = value,

        // BG scroll registers (write-only!)
        // Je kunt deze schrijven maar niet teruglezen
        0x04000010 => self.ppu.bg0hofs = value & 0x1FF,  // 9-bit
        0x04000012 => self.ppu.bg0vofs = value & 0x1FF,

        // VCOUNT (0x04000006) is read-only, dus die staat hier NIET

        // IF register — speciaal: schrijf 1 om bit te CLEAREN
        0x04000202 => self.irq.if_ &= ~value,

        else => {},  // negeer onbekende writes
    }
}
```

### Bitmask voor schrijven — het `&= ~value` patroon

Het IF register is bijzonder: je schrijft een 1 om een bit te **clearen** (op 0 te zetten). Dit is een veelvoorkomend hardware-patroon:

```zig
// Stel IF = 0b0000_0101 (VBlank en Timer0 interrupt pending)
// Je wilt VBlank clearen, dus schrijf 0b0000_0001

self.irq.if_ &= ~value;
// ~ = bitwise NOT: ~0b0000_0001 = 0b1111_1110
// & = bitwise AND: 0b0000_0101 & 0b1111_1110 = 0b0000_0100
// Resultaat: alleen de Timer0 bit blijft staan
```

### Adres alignment

I/O registers zijn 16-bit of 32-bit aligned. Bij een 32-bit read lees je eigenlijk twee 16-bit registers. **Waarom?** De GBA's I/O registers zijn fundamenteel 16-bit breed (de ARM7TDMI heeft een 16-bit data bus voor I/O), maar de CPU kan 32-bit reads/writes uitvoeren. De hardware lost dit op door twee opeenvolgende 16-bit registers samen te voegen:

```zig
pub fn read32(self: *Io, address: u32) u32 {
    const lo: u32 = self.read16(address);
    const hi: u32 = self.read16(address + 2);
    return lo | (hi << 16);
}

pub fn write32(self: *Io, address: u32, value: u32) void {
    self.write16(address, @truncate(value));
    self.write16(address + 2, @truncate(value >> 16));
}
```

- `src/core/io.zig`

  - `pub fn write(address: u32, value: u8/u16/u32) void`

## Stap-voor-stap

### Stap 1: `src/core/io.zig` — Io struct aanmaken

Begin met een struct die referenties bevat naar alle subsystemen waarmee I/O registers communiceren:

```zig
const Io = struct {
    ppu: *Ppu,
    irq: *Irq,
    timers: *TimerController,
    dma: *DmaController,
    keyinput: u16 = 0x03FF, // alle knoppen losgelaten (active low)

    pub fn init(ppu: *Ppu, irq: *Irq, timers: *TimerController, dma: *DmaController) Io {
        return Io{
            .ppu = ppu,
            .irq = irq,
            .timers = timers,
            .dma = dma,
        };
    }
};
```

**Waarom pointers naar subsystemen?** De I/O dispatcher is een *doorgeefluik*: hij bezit de data niet, maar routeert reads/writes naar de juiste eigenaar. Pointers voorkomen dat je data dupliceert en zorgen ervoor dat een write naar DISPCNT direct het echte PPU-register wijzigt.

**Waarom `keyinput` wél direct in Io?** KEYINPUT is een eenvoudig register dat door de platform-laag (SDL) wordt geschreven en door de I/O dispatcher wordt gelezen. Het heeft geen eigen controller nodig — het is gewoon een `u16`.

### Stap 2: `src/core/io.zig` — `read16` switch-based dispatcher

De kern van de dispatcher is een grote `switch` op het adres:

```zig
pub fn read16(self: *Io, address: u32) u16 {
    return switch (address) {
        // PPU registers
        0x04000000 => self.ppu.dispcnt,
        0x04000004 => self.ppu.dispstat,
        0x04000006 => @as(u16, self.ppu.vcount),

        // Input
        0x04000130 => self.keyinput,

        // Timer counter values (read geeft huidige waarde)
        0x04000100 => self.timers.timers[0].counter,
        0x04000104 => self.timers.timers[1].counter,
        0x04000108 => self.timers.timers[2].counter,
        0x0400010C => self.timers.timers[3].counter,

        // Interrupts
        0x04000200 => self.irq.ie,
        0x04000202 => self.irq.if_,
        0x04000208 => @intFromBool(self.irq.ime),

        else => 0,
    };
}
```

**Waarom een `switch` en geen array/lookup-table?** Er zijn meerdere redenen:

1. **Sparse adressen**: Het I/O-bereik is 0x400 bytes (1024), maar slechts ~100 adressen zijn daadwerkelijk in gebruik. Een array zou 512 entries nodig hebben (voor 16-bit) waarvan de meeste leeg zijn.
2. **Heterogeen gedrag**: Elk register kan ander gedrag hebben. VCOUNT is read-only, KEYINPUT komt van de input-module, timer counters veranderen continu. Een array kan alleen opslag bieden, geen gedrag.
3. **Side effects**: Sommige registers hebben bijwerkingen bij lezen of schrijven. Een `switch` laat je per adres specifieke logica uitvoeren.

**Tip:** De `else => 0` branch is belangrijk. Games lezen soms van ongeïmplementeerde adressen. Door 0 terug te geven (in plaats van te crashen) blijft de emulator draaien. Later kun je hier een log toevoegen om te zien welke registers je nog mist.

### Stap 3: `src/core/io.zig` — `write16` met read-only/write-only logica

```zig
pub fn write16(self: *Io, address: u32, value: u16) void {
    switch (address) {
        // PPU registers (read/write)
        0x04000000 => self.ppu.dispcnt = value,
        0x04000004 => {
            // DISPSTAT: alleen bits 3-15 zijn schrijfbaar
            // Bits 0-2 zijn status flags (read-only)
            self.ppu.dispstat = (self.ppu.dispstat & 0x7) | (value & 0xFFF8);
        },

        // BG scroll registers (write-only)
        0x04000010 => self.ppu.bg0hofs = value & 0x1FF,
        0x04000012 => self.ppu.bg0vofs = value & 0x1FF,

        // Timer reload en control
        0x04000100 => self.timers.timers[0].reload = value,
        0x04000102 => self.timers.writeControl(0, value),

        // Interrupt registers
        0x04000200 => self.irq.ie = value,
        0x04000202 => self.irq.if_ &= ~value,  // schrijf 1 om te clearen!
        0x04000208 => self.irq.ime = (value & 1) == 1,

        // VCOUNT (0x04000006) staat hier NIET — is read-only
        // KEYINPUT (0x04000130) staat hier NIET — is read-only

        else => {},
    }
}
```

**Waarom geen gewoon `self.register = value`?**

Drie soorten registers vereisen speciale behandeling:

- **Read-only registers** (VCOUNT, KEYINPUT): staan niet in de `write16`-switch. Een write wordt gewoon genegeerd. De hardware zet deze waarden, software kan ze alleen lezen.
- **Write-only registers** (BG scroll): staan niet in de `read16`-switch. Je kunt ze schrijven, maar teruglezen geeft 0 of een stale waarde.
- **Mixed registers** (DISPSTAT): sommige bits zijn read-only, andere zijn schrijfbaar. Je moet de read-only bits behouden met een bitmask.
- **Side-effect registers** (IF): het schrijven van een 1-bit *clearet* die bit in plaats van hem te zetten. Dit `&= ~value` patroon is een hardwareconventie om race conditions te voorkomen (zie Story 8.1).

**Valkuil:** BG scroll registers zijn 9 bits breed, maar de write is 16-bit. Vergeet niet te masken met `& 0x1FF`, anders sla je ongeldige waarden op.

### Stap 4: `src/core/io.zig` — 32-bit reads/writes via twee 16-bit operaties

De GBA CPU kan 32-bit reads/writes doen, maar I/O registers zijn 16-bit georganiseerd. Splits ze op:

```zig
pub fn read32(self: *Io, address: u32) u32 {
    const lo: u32 = self.read16(address);
    const hi: u32 = self.read16(address + 2);
    return lo | (hi << 16);
}

pub fn write32(self: *Io, address: u32, value: u32) void {
    self.write16(address, @truncate(value));           // onderste 16 bits
    self.write16(address + 2, @truncate(value >> 16)); // bovenste 16 bits
}
```

**Waarom niet gewoon een 32-bit switch?** Omdat de twee helften van een 32-bit write naar *verschillende registers* kunnen gaan met *verschillend gedrag*. Bijvoorbeeld: een 32-bit write naar 0x04000100 schrijft timer 0's reload value (0x100) én control register (0x102). Door alles via `read16`/`write16` te laten lopen, hergebruik je alle speciale logica.

**8-bit reads/writes:** De GBA ondersteunt ook 8-bit I/O access. Voor reads kun je de 16-bit waarde lezen en de juiste byte selecteren. Voor writes is het complexer: sommige registers negeren 8-bit writes, andere spiegelen de byte naar beide helften. Begin eenvoudig (doorsturen als 16-bit) en verfijn later.

### Stap 5: `src/core/bus.zig` — integratie met de memory bus

De bus moet het adresbereik 0x04000000-0x040003FF doorsturen naar de I/O dispatcher:

```zig
pub fn read16(self: *Bus, address: u32) u16 {
    return switch (address >> 24) {
        0x00 => self.bios.read16(address),
        0x02 => self.ewram.read16(address),
        0x03 => self.iwram.read16(address),
        0x04 => self.io.read16(address),  // I/O registers!
        0x05 => self.ppu.palette.read16(address),
        0x06 => self.ppu.vram.read16(address),
        // ... etc
        else => 0,
    };
}
```

**Waarom `address >> 24`?** Het eerste byte van het adres bepaalt het geheugengebied op de GBA. 0x04 = I/O registers. Dit is een snelle manier om te dispatchen zonder het volledige adres te vergelijken.

**Tip:** Je kunt de Io struct geleidelijk uitbreiden. Begin met alleen DISPCNT, DISPSTAT en VCOUNT. Voeg registers toe naarmate je meer subsystemen implementeert (timers, DMA, interrupts). De `else => 0` / `else => {}` branches vangen alles op wat je nog niet hebt geïmplementeerd.

## Acceptatiecriteria

- DISPCNT, DISPSTAT, VCOUNT, KEYINPUT zijn leesbaar
- Writes naar read-only registers worden genegeerd
- 32-bit reads/writes werken correct via twee 16-bit operaties
