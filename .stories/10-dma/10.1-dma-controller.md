# Story 10.1 — DMA controller

**Epic:** 10 - DMA

**Doel:** 4 DMA kanalen implementeren.

## Zig concepten

### Wat is DMA?

DMA (Direct Memory Access) kopieert data van het ene geheugenadres naar het andere **zonder** dat de CPU elke byte handmatig moet verplaatsen. Dit is veel sneller en wordt gebruikt voor:
- Sprite/tile data naar VRAM kopiëren
- Audio samples naar FIFO sturen
- Scherm wissen (memory fill)

### DMA registers

Elk DMA kanaal heeft 4 registers:

```zig
// DMA0: 0x040000B0 - 0x040000BC
// DMA1: 0x040000BC - 0x040000C8
// DMA2: 0x040000C8 - 0x040000D4
// DMA3: 0x040000D4 - 0x040000E0

const DmaChannel = struct {
    source: u32 = 0,       // bronadres (waar data vandaan komt)
    dest: u32 = 0,         // doeladres (waar data naartoe gaat)
    count: u16 = 0,        // aantal transfers
    control: DmaControl = @bitCast(@as(u16, 0)),

    // Interne state (niet direct zichtbaar voor de game)
    internal_source: u32 = 0,
    internal_dest: u32 = 0,
    internal_count: u16 = 0,
};
```

### DMA control register

```zig
const DmaControl = packed struct(u16) {
    _unused: u5,
    dest_control: u2,    // 0=increment, 1=decrement, 2=fixed, 3=increment+reload
    source_control: u2,  // 0=increment, 1=decrement, 2=fixed, 3=forbidden
    repeat: u1,          // herhaal bij elke trigger?
    transfer_type: u1,   // 0=16-bit, 1=32-bit
    _unused2: u1,
    timing: u2,          // 0=immediate, 1=VBlank, 2=HBlank, 3=special
    irq_enable: u1,      // interrupt bij klaar?
    enabled: u1,         // DMA aan/uit
};
```

### Adres increment/decrement

Na elke transfer wordt het bron- en doeladres aangepast op basis van de control bits:

```zig
fn get_addr_delta(control: u2, transfer_size: u32) i32 {
    return switch (control) {
        0 => @intCast(transfer_size),   // increment: +2 of +4
        1 => -@as(i32, @intCast(transfer_size)),  // decrement: -2 of -4
        2 => 0,                          // fixed: adres verandert niet
        3 => @intCast(transfer_size),   // increment+reload (dest only)
    };
}

// transfer_size is 2 (16-bit) of 4 (32-bit)
```

### DMA transfer uitvoeren

```zig
fn execute_transfer(self: *DmaChannel, bus: *Bus) void {
    const size: u32 = if (self.control.transfer_type == 1) 4 else 2;
    const src_delta = get_addr_delta(self.control.source_control, size);
    const dst_delta = get_addr_delta(self.control.dest_control, size);

    var src = self.internal_source;
    var dst = self.internal_dest;

    for (0..self.internal_count) |_| {
        if (size == 4) {
            bus.write32(dst, bus.read32(src));
        } else {
            bus.write16(dst, bus.read16(src));
        }

        // Update adressen
        src = @intCast(@as(i64, src) + src_delta);
        dst = @intCast(@as(i64, dst) + dst_delta);
    }

    self.internal_source = src;
    self.internal_dest = dst;
}
```

### DMA timing — wanneer triggert de transfer?

```zig
const DmaTiming = enum(u2) {
    immediate = 0,  // meteen zodra DMA wordt enabled
    vblank = 1,     // bij elke VBlank
    hblank = 2,     // bij elke HBlank
    special = 3,    // verschilt per kanaal:
                    // DMA1/2: audio FIFO (wanneer FIFO samples nodig heeft)
                    // DMA3: video capture
};

// In je PPU code, bij VBlank:
fn on_vblank(self: *Ppu, dma: *DmaController) void {
    dma.check_and_transfer(.vblank);
}
```

### DMA prioriteit

Als meerdere DMA kanalen tegelijk willen transferren, wint het kanaal met het laagste nummer (DMA0 = hoogste prioriteit). Tijdens een DMA transfer is de CPU gepauzeerd.

```zig
fn check_and_transfer(self: *DmaController, timing: DmaTiming) void {
    // DMA0 heeft hoogste prioriteit, DMA3 laagste
    for (0..4) |i| {
        const ch = &self.channels[i];
        if (ch.control.enabled == 1 and
            @as(DmaTiming, @enumFromInt(ch.control.timing)) == timing) {
            ch.execute_transfer(self.bus);

            if (ch.control.repeat == 0) {
                ch.control.enabled = 0;  // eenmalig: disable na transfer
            }
        }
    }
}
```

## Context

- 4 DMA kanalen (DMA0-DMA3) met prioriteit (0 = hoogst) — **waarom prioriteit?** Meerdere hardware-subsystemen kunnen tegelijk DMA willen. Audio DMA (kanaal 1/2) en grafische DMA (kanaal 3) kunnen conflicteren. Prioriteit lost dit op: het laagste kanaalnummer gaat eerst.
- Elk kanaal: source address, destination address, word count, control
- **Waarom DMA in plaats van CPU-kopieën?** DMA is ~2x sneller dan een CPU-loop voor grote blokken data. De DMA-controller heeft directe toegang tot de geheugenbus en hoeft geen instructies te fetchen/decoderen.
- Start timing: immediate, VBlank, HBlank, special (audio DMA) — **waarom VBlank/HBlank timing?** Grafisch geheugen (VRAM, palette) wordt actief gelezen door de PPU tijdens het tekenen. DMA naar dit geheugen tijdens het tekenen veroorzaakt visuele glitches. VBlank/HBlank zijn de "veilige periodes" waarin de PPU niet leest.
- Transfer type: 16-bit of 32-bit
- Destination/source adres control: increment, decrement, fixed, increment/reload — **waarom fixed?** Voor audio DMA wil je steeds naar hetzelfde FIFO-adres schrijven (het doeladres verandert niet), terwijl het bronadres door de audiobuffer loopt.

## Stap-voor-stap

### Stap 1: `src/core/dma.zig` — DmaChannel struct aanmaken

Begin met de struct voor één DMA kanaal:

```zig
pub const DmaChannel = struct {
    // Door de game ingestelde waarden (via I/O registers)
    source: u32 = 0,
    dest: u32 = 0,
    count: u16 = 0,
    control: DmaControl = @bitCast(@as(u16, 0)),

    // Interne kopieën (door hardware gebruikt tijdens transfer)
    internal_source: u32 = 0,
    internal_dest: u32 = 0,
    internal_count: u16 = 0,
};
```

**Waarom aparte "interne" velden?** De game stelt source, dest en count in via de I/O registers. Maar tijdens de transfer worden de adressen aangepast (increment/decrement). Als je de originele waarden zou overschrijven, kan de game ze niet opnieuw gebruiken bij een repeat-transfer. De interne kopieën worden aangemaakt wanneer de DMA start, en de originelen blijven ongewijzigd.

**Waarom DMA überhaupt?** Direct Memory Access is een hardwaremechanisme dat data kopieert *zonder CPU-tussenkomst*. Op de GBA is dit essentieel omdat:
1. **Snelheid**: een DMA-transfer is ~2x sneller dan een CPU-loop die byte voor byte kopieert. De DMA-controller heeft een directe verbinding met de geheugenbus.
2. **CPU-vrijheid**: tijdens de transfer is de CPU gepauzeerd (hij hoeft niets te doen), maar de transfer loopt razendsnel via dedicated hardware.
3. **Timing-gevoeligheid**: sommige transfers moeten *precies* op het juiste moment plaatsvinden (tijdens HBlank/VBlank). De CPU kan dit niet nauwkeurig genoeg timen, maar de DMA-hardware reageert direct op hardware-signalen.

### Stap 2: `src/core/dma.zig` — DmaControl packed struct

```zig
pub const DmaControl = packed struct(u16) {
    _unused: u5 = 0,
    dest_control: u2 = 0,     // 0=inc, 1=dec, 2=fixed, 3=inc+reload
    source_control: u2 = 0,   // 0=inc, 1=dec, 2=fixed, 3=verboden
    repeat: u1 = 0,
    transfer_type: u1 = 0,    // 0=16-bit, 1=32-bit
    _unused2: u1 = 0,
    timing: u2 = 0,           // 0=immediate, 1=VBlank, 2=HBlank, 3=special
    irq_enable: u1 = 0,
    enabled: u1 = 0,
};
```

**Waarom dest_control waarde 3 (increment+reload)?** Bij repeat-DMA's (zoals HBlank DMA voor raster-effecten) wil je het doeladres elke keer terugzetten naar het begin. Increment+reload doet dit: tijdens de transfer incrementeert het adres normaal, maar bij de volgende trigger wordt het herlaad naar de originele waarde. Het bronadres heeft deze optie niet (waarde 3 is verboden voor source).

**Waarom 16-bit én 32-bit transfers?** 32-bit transfers zijn 2x zo snel (dubbel zoveel data per bus-cyclus), maar niet alle geheugen ondersteunt 32-bit access. VRAM en palette RAM zijn 16-bit breed — een 32-bit write wordt daar automatisch opgesplitst in twee 16-bit writes. De game kiest het type op basis van de bron en bestemming.

### Stap 3: `src/core/dma.zig` — transfer uitvoeren

De kernfunctie die data daadwerkelijk kopieert:

```zig
fn getAddrDelta(control: u2, transfer_size: u32) i32 {
    return switch (control) {
        0 => @intCast(transfer_size),                      // increment
        1 => -@as(i32, @intCast(transfer_size)),           // decrement
        2 => 0,                                             // fixed
        3 => @intCast(transfer_size),                      // inc+reload (dest only)
    };
}

fn executeTransfer(self: *DmaChannel, bus: *Bus) void {
    const size: u32 = if (self.control.transfer_type == 1) 4 else 2;
    const src_delta = getAddrDelta(self.control.source_control, size);
    const dst_delta = getAddrDelta(self.control.dest_control, size);

    var src = self.internal_source;
    var dst = self.internal_dest;
    const count: u32 = if (self.internal_count == 0)
        // count 0 betekent maximale lengte
        if (self.control.transfer_type == 1) 0x4000 else 0x10000
    else
        self.internal_count;

    for (0..count) |_| {
        if (size == 4) {
            bus.write32(dst, bus.read32(src));
        } else {
            bus.write16(dst, bus.read16(src));
        }
        src = @intCast(@as(i64, src) + src_delta);
        dst = @intCast(@as(i64, dst) + dst_delta);
    }

    self.internal_source = src;
    self.internal_dest = dst;
}
```

**Waarom `i64` voor de adresberekening?** Het bronadres is een `u32`, en de delta kan negatief zijn (`i32`). Om overflow te voorkomen bij de optelling (een `u32` + negatieve `i32` kan onder nul gaan in theorie), casten we tijdelijk naar `i64`. In de praktijk zijn GBA-adressen altijd geldig, maar Zig's strikte overflow-checks vereisen deze voorzichtigheid.

**Waarom count 0 = maximale lengte?** Dit is een hardware-eigenaardigheid: als de game 0 schrijft naar het count-register, betekent dit niet "kopieer niets" maar "kopieer het maximum". Voor DMA0-2 is dit 0x4000 (16K) 32-bit woorden, voor DMA3 is dit 0x10000 (64K). Dit is een veelvoorkomende GBA-conventie die je correct moet implementeren.

**Valkuil:** De adressen worden na de transfer bewaard in `internal_source`/`internal_dest`. Dit is belangrijk voor repeat-DMA's met `dest_control = 3` (increment+reload): het doeladres wordt herlaad, maar het bronadres gaat verder waar het gebleven was. Als je de interne state niet bijhoudt, wordt bij elke repeat dezelfde data gekopieerd.

### Stap 4: `src/core/dma.zig` — DmaController met prioriteit

```zig
pub const DmaTiming = enum(u2) {
    immediate = 0,
    vblank = 1,
    hblank = 2,
    special = 3,
};

pub const DmaController = struct {
    channels: [4]DmaChannel = [_]DmaChannel{.{}} ** 4,
    bus: *Bus,
    irq: *Irq,

    pub fn init(bus: *Bus, irq: *Irq) DmaController {
        return DmaController{ .bus = bus, .irq = irq };
    }

    pub fn checkAndTransfer(self: *DmaController, timing: DmaTiming) void {
        for (0..4) |i| {
            const ch = &self.channels[i];
            if (ch.control.enabled == 0) continue;
            if (@as(DmaTiming, @enumFromInt(ch.control.timing)) != timing) continue;

            ch.executeTransfer(self.bus);

            // Trigger IRQ als enabled
            if (ch.control.irq_enable == 1) {
                const irq_type: IrqType = switch (i) {
                    0 => .dma0,
                    1 => .dma1,
                    2 => .dma2,
                    3 => .dma3,
                    else => unreachable,
                };
                self.irq.request(irq_type);
            }

            // Eenmalige DMA: disable na transfer
            if (ch.control.repeat == 0) {
                ch.control.enabled = 0;
            } else if (ch.control.dest_control == 3) {
                // Repeat met inc+reload: herstel doeladres
                ch.internal_dest = ch.dest;
            }
        }
    }
};
```

**Waarom 4 kanalen met prioriteit?** De GBA heeft meerdere hardware-subsystemen die tegelijkertijd DMA willen uitvoeren:
- **DMA0**: hoogste prioriteit — meestal gereserveerd voor tijdkritische transfers
- **DMA1/DMA2**: audio FIFO — de geluidshardware vraagt continu nieuwe samples
- **DMA3**: laagste prioriteit — general-purpose, gebruikt voor ROM→RAM kopieën

De prioriteit (lager kanaalnummer = hogere prioriteit) lost conflicten op: als DMA0 en DMA3 tegelijk willen starten, gaat DMA0 eerst. De `for (0..4)` loop verwerkt ze automatisch in prioriteitsvolgorde.

**Waarom HBlank/VBlank timing?** Het scherm wordt opgebouwd in scanlines. Tijdens het *tekenen* van een scanline leest de PPU actief uit VRAM en palette RAM. Als je op dat moment data naar VRAM kopieert, krijg je visuele glitches (tearing, corrupte pixels). De oplossing:
- **HBlank**: de korte pauze tussen twee scanlines (~68 cycles). Genoeg voor kleine transfers (bijv. palette cycling per scanline voor kleureffecten).
- **VBlank**: de langere pauze na het laatste scanline (~83.776 cycles). Genoeg voor grote transfers (bijv. een volledig tilemap updaten).

Door DMA op deze momenten te triggeren, is de transfer "veilig" — de PPU leest op dat moment niet uit het geheugen dat wordt gewijzigd.

### Stap 5: `src/core/dma.zig` — writeControl voor DMA starten

Het schrijven naar het control register triggert potentieel een DMA:

```zig
pub fn writeControl(self: *DmaController, channel_id: usize, value: u16) void {
    const ch = &self.channels[channel_id];
    const old_enabled = ch.control.enabled;
    ch.control = @bitCast(value);
    const new_enabled = ch.control.enabled;

    // Bij het aanzetten: kopieer registers naar interne state
    if (old_enabled == 0 and new_enabled == 1) {
        ch.internal_source = ch.source;
        ch.internal_dest = ch.dest;
        ch.internal_count = ch.count;

        // Immediate DMA start meteen
        if (@as(DmaTiming, @enumFromInt(ch.control.timing)) == .immediate) {
            ch.executeTransfer(self.bus);

            if (ch.control.irq_enable == 1) {
                const irq_type: IrqType = switch (channel_id) {
                    0 => .dma0, 1 => .dma1, 2 => .dma2, 3 => .dma3,
                    else => unreachable,
                };
                self.irq.request(irq_type);
            }

            // Immediate DMA is altijd eenmalig
            ch.control.enabled = 0;
        }
    }
}
```

**Waarom interne kopieën bij enable?** Zodra de DMA wordt gestart, "snapt" de hardware de huidige waarden van source, dest en count. De game kan daarna de registers overschrijven voor de volgende DMA zonder de lopende transfer te verstoren. Dit is vooral belangrijk bij repeat-DMA's die getriggerd worden door VBlank/HBlank.

**Waarom immediate DMA direct uitvoeren?** Immediate DMA (timing = 0) start onmiddellijk wanneer de enable-bit wordt gezet. Er is geen trigger-event nodig. Dit wordt gebruikt voor eenmalige kopieën, zoals het laden van grafische data bij het starten van een level.

**Valkuil:** Vergeet niet om immediate DMA te disablen na uitvoering (`ch.control.enabled = 0`). Anders blijft het kanaal "actief" en zou het opnieuw kunnen triggeren. VBlank/HBlank DMA's met repeat=1 blijven wél actief — die worden bij elke trigger opnieuw uitgevoerd.

### Stap 6: `src/core/io.zig` — DMA registers koppelen

DMA source/dest zijn 32-bit, maar worden geschreven via twee 16-bit writes. Voeg toe aan de I/O dispatcher:

```zig
// In write16:
// DMA0 (0x040000B0 - 0x040000BE)
0x040000B0 => self.dma.channels[0].source = (self.dma.channels[0].source & 0xFFFF0000) | value,
0x040000B2 => self.dma.channels[0].source = (self.dma.channels[0].source & 0x0000FFFF) | (@as(u32, value) << 16),
0x040000B4 => self.dma.channels[0].dest = (self.dma.channels[0].dest & 0xFFFF0000) | value,
0x040000B6 => self.dma.channels[0].dest = (self.dma.channels[0].dest & 0x0000FFFF) | (@as(u32, value) << 16),
0x040000B8 => self.dma.channels[0].count = value,
0x040000BA => self.dma.writeControl(0, value),

// DMA1-3 volgen hetzelfde patroon met offsets +0x0C per kanaal
// DMA1: 0x040000BC - 0x040000CA
// DMA2: 0x040000C8 - 0x040000D6
// DMA3: 0x040000D4 - 0x040000E2
```

**Waarom 16-bit writes voor 32-bit adressen?** De I/O registers zijn 16-bit georganiseerd. Een 32-bit adres zoals `source` wordt in twee stukken geschreven: de onderste 16 bits (low halfword) en de bovenste 16 bits (high halfword). De `& 0xFFFF0000 | value` en `& 0x0000FFFF | (value << 16)` patronen zorgen ervoor dat alleen de juiste helft wordt overschreven.

**Tip:** Als je ook 32-bit I/O writes via `write32` ondersteunt (zie Story 6.1, stap 4), worden die automatisch opgesplitst in twee `write16` calls. Dit betekent dat een 32-bit write naar 0x040000B0 correct zowel de low als high helft van `source` zet — zonder extra code.

**Tip:** DMA is een van de moeilijkste onderdelen om correct te implementeren. Begin met alleen immediate DMA (timing=0) en increment-modus. Test met een ROM die een memcpy-achtige DMA doet. Voeg daarna VBlank/HBlank DMA toe, en pas als laatste audio DMA (special timing).
