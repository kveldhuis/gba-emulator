# Story 5.8 — Affine sprites

**Epic:** 05 - PPU (Graphics)

**Doel:** Sprites met rotatie en schaling renderen.

## Zig concepten

### Wat zijn affine sprites?

Normale sprites worden pixel-voor-pixel uit tile data gekopieerd. Affine sprites passen een rotatie/schaling **transformatiematrix** toe — dezelfde wiskunde als bij affine achtergronden (story 5.5), maar dan per sprite.

### Waarom affine sprites apart van normale sprites?

Normale sprites zijn relatief simpel: lees pixels uit tile data, pas optioneel flip toe, teken op de juiste positie. Affine sprites voegen een **matrixvermenigvuldiging** toe per pixel. Dit maakt de rendering significant complexer:
- Je moet affine parameters uit OAM lezen (die op een onhandige plek staan — verweven met andere sprite-attributen)
- De transformatiematrix moet per pixel worden toegepast (fixed-point rekenen)
- De double-size optie verandert het rendergebied en het middelpunt
- Flipping werkt anders (het zit verwerkt in de matrix, niet als losse bits)

Door affine sprites in een aparte story te behandelen, kun je eerst normale sprites volledig werkend krijgen (story 5.6) en dan de affine-complexiteit toevoegen zonder de basis te verstoren.

### Waarom de double-size optie?

Wanneer je een vierkante sprite 45° roteert, steken de hoeken buiten het oorspronkelijke sprite-vak uit. Zonder double-size worden deze hoeken afgesneden (clipping). De double-size optie verdubbelt het rendergebied (maar niet de sprite-data zelf), zodat de geroteerde sprite volledig zichtbaar is.

Het nadeel: een double-size 64×64 sprite neemt een rendervak van 128×128 pixels in — dat kost meer rendering-tijd. Games gebruiken double-size daarom alleen wanneer nodig.

### OAM Rotation/Scaling parameters

De GBA heeft 32 sets van 4 parameters (PA, PB, PC, PD) opgeslagen in de OAM, **tussen** de gewone sprite-attributen. Elke 4e OAM entry bevat een parameter:

```
OAM layout (8 bytes per entry × 128 entries):
Entry 0: attr0, attr1, attr2, rotation_param_PA_group0
Entry 1: attr0, attr1, attr2, rotation_param_PB_group0
Entry 2: attr0, attr1, attr2, rotation_param_PC_group0
Entry 3: attr0, attr1, attr2, rotation_param_PD_group0
Entry 4: attr0, attr1, attr2, rotation_param_PA_group1
...
```

In code:

```zig
const AffineParams = struct {
    pa: i16,  // dx  — horizontale schaling + rotatie component
    pb: i16,  // dmx — horizontale shear component
    pc: i16,  // dy  — verticale shear component
    pd: i16,  // dmy — verticale schaling + rotatie component
};

fn read_affine_params(bus: *Bus, group: u5) AffineParams {
    // Parameters zitten op byte 6 (attr3/padding) van elke 4e OAM entry
    const base: u32 = 0x07000000;
    return .{
        .pa = @bitCast(bus.read16(base + @as(u32, group) * 32 + 0x06)),
        .pb = @bitCast(bus.read16(base + @as(u32, group) * 32 + 0x0E)),
        .pc = @bitCast(bus.read16(base + @as(u32, group) * 32 + 0x16)),
        .pd = @bitCast(bus.read16(base + @as(u32, group) * 32 + 0x1E)),
    };
}
```

### Hoe de rotatie-matrix werkt

De parameters PA/PB/PC/PD vormen een 2×2 **inverse** transformatiematrix. Voor elke pixel op het scherm bereken je welke pixel uit de sprite-tile data je nodig hebt:

```
texture_x = PA * (screen_x - center_x) + PB * (screen_y - center_y) + half_width
texture_y = PC * (screen_x - center_x) + PD * (screen_y - center_y) + half_height
```

De waarden zijn **8.8 fixed-point** (8 bits integer, 8 bits fractie):

```zig
// Fixed-point: 1.0 = 256 (1 << 8)
// Geen rotatie/schaling: PA=256, PB=0, PC=0, PD=256 (= identity matrix)
// 90° rotatie: PA=0, PB=-256, PC=256, PD=0

fn apply_affine(params: AffineParams, sx: i32, sy: i32) struct { tx: i32, ty: i32 } {
    // sx, sy = offset van het midden van de sprite
    // Vermenigvuldig en corrigeer voor fixed-point (>> 8)
    const tx = (@as(i32, params.pa) * sx + @as(i32, params.pb) * sy) >> 8;
    const ty = (@as(i32, params.pc) * sx + @as(i32, params.pd) * sy) >> 8;
    return .{ .tx = tx, .ty = ty };
}
```

### Double-size rendering

Bij affine sprites kan een geroteerde sprite groter worden dan het originele sprite-vak (denk aan een vierkant dat 45° gedraaid is). De GBA heeft daarom een **double-size** optie:

```zig
// OAM attr0 bit 9: double-size flag (alleen bij affine sprites)
const double_size = (attr0 >> 9) & 1 == 1;

// Het rendergebied is 2× zo groot als de sprite-afmetingen
const render_width = if (double_size) sprite_width * 2 else sprite_width;
const render_height = if (double_size) sprite_height * 2 else sprite_height;

// Het middelpunt verschuift mee
const center_x = render_width / 2;
const center_y = render_height / 2;
```

Zonder double-size worden de hoeken van een geroteerde sprite afgesneden.

### Affine sprite rendering per scanline

```zig
fn render_affine_sprite(
    self: *Ppu,
    bus: *Bus,
    attr0: OamAttr0,
    attr1: OamAttr1,
    attr2: OamAttr2,
    y: u8,
) void {
    const params = read_affine_params(bus, @truncate(attr1.raw >> 9));  // group index

    // Sprite-afmetingen uit shape/size tabel (zie story 5.6)
    const width = sprite_sizes[attr0.shape][attr1.size][0];
    const height = sprite_sizes[attr0.shape][attr1.size][1];

    const double = attr0.double_size == 1;
    const render_w: i32 = if (double) @as(i32, width) * 2 else width;
    const render_h: i32 = if (double) @as(i32, height) * 2 else height;

    // Check of deze scanline de sprite raakt
    const sprite_y: i32 = @as(i32, attr0.y);
    const local_y = @as(i32, y) - sprite_y;
    if (local_y < 0 or local_y >= render_h) return;

    const half_w = render_w >> 1;
    const half_h = render_h >> 1;

    // Offset van het midden van de sprite
    const sy = local_y - half_h;

    // Bereken start texture-coördinaten (= begin van deze scanline)
    // Dan per pixel PA/PC optellen (zelfde optimalisatie als affine BG)
    var tex_x = @as(i32, params.pa) * (-half_w) + @as(i32, params.pb) * sy + (@as(i32, width) << 7);
    var tex_y = @as(i32, params.pc) * (-half_w) + @as(i32, params.pd) * sy + (@as(i32, height) << 7);

    const sprite_x: i32 = @as(i32, attr1.x);

    var screen_x: i32 = sprite_x;
    var px: i32 = 0;
    while (px < render_w) : (px += 1) {
        defer {
            tex_x += params.pa;
            tex_y += params.pc;
            screen_x += 1;
        }

        if (screen_x < 0 or screen_x >= 240) continue;

        // Converteer fixed-point naar pixel-coördinaten
        const tile_x = tex_x >> 8;
        const tile_y = tex_y >> 8;

        // Buiten sprite-grenzen? Skip (transparant)
        if (tile_x < 0 or tile_x >= width or tile_y < 0 or tile_y >= height) continue;

        // Lees pixel uit tile data (zelfde als normale sprite, story 5.6)
        const pixel = read_sprite_pixel(bus, attr2, @intCast(tile_x), @intCast(tile_y));
        if (pixel != 0) {  // 0 = transparant
            self.sprite_line[@intCast(screen_x)] = pixel;
        }
    }
}
```

### Waarom `<< 7` in plaats van `<< 8`?

Het middelpunt van de sprite in fixed-point (8.8) is `(width / 2) << 8`. Maar `(width / 2) << 8` = `width << 7`. Dit is een veelgebruikte optimalisatie: `(x / 2) * 256 = x * 128 = x << 7`.

### Per-pixel vs per-scanline optimalisatie

Net als bij affine achtergronden bereken je niet elke pixel opnieuw met de volledige formule. Je berekent de startpositie voor de scanline en telt dan per pixel alleen PA/PC op:

```
pixel 0: tex_x = PA * (-half_w) + PB * sy + center
pixel 1: tex_x += PA
pixel 2: tex_x += PA
...
```

Dit is veel sneller dan elke keer de volledige matrixvermenigvuldiging te doen.

## Context

- 32 rotatie/schaling parameter-groepen in OAM (PA, PB, PC, PD elk)
- Parameters zijn 8.8 fixed-point signed
- OAM attr0 bits [8:9] bepalen: 0=normaal, 1=affine, 2=verborgen, 3=affine+double-size
- attr1 bits [13:9] selecteren welke parameter-groep gebruikt wordt
- Double-size verdubbelt het rendergebied om afsnijding te voorkomen

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — affine parameters uit OAM lezen

De 32 sets affine parameters zijn opgeslagen op een ongebruikelijke manier: verweven tussen de normale sprite-attributen. Elke OAM entry is 8 bytes, en byte 6–7 (normaal padding) bevat één affine parameter. Vier opeenvolgende entries vormen samen één set (PA, PB, PC, PD).

```zig
const AffineParams = struct {
    pa: i16,  // dx  — horizontale schaling/rotatie
    pb: i16,  // dmx — horizontale shear
    pc: i16,  // dy  — verticale shear
    pd: i16,  // dmy — verticale schaling/rotatie
};

fn read_affine_params(bus: *Bus, group: u5) AffineParams {
    const base: u32 = 0x0700_0000;
    // Elke groep beslaat 4 OAM entries = 32 bytes
    // PA zit op offset 0x06 van entry 0 (= byte 6 van de groep)
    // PB zit op offset 0x06 van entry 1 (= byte 14)
    // PC zit op offset 0x06 van entry 2 (= byte 22)
    // PD zit op offset 0x06 van entry 3 (= byte 30)
    const g: u32 = @as(u32, group) * 32;
    return .{
        .pa = @bitCast(bus.read16(base + g + 0x06)),
        .pb = @bitCast(bus.read16(base + g + 0x0E)),
        .pc = @bitCast(bus.read16(base + g + 0x16)),
        .pd = @bitCast(bus.read16(base + g + 0x1E)),
    };
}
```

**Waarom deze bizarre layout?** De GBA-hardware leest OAM sequentieel. Door de affine parameters in de "padding" bytes van normale OAM entries te plaatsen, hoeft de hardware maar één keer door OAM te lopen om zowel sprite-attributen als affine-parameters te lezen. Het is een hardware-optimalisatie die voor software onhandig is.

**Waarom `u5` voor de groep?** Er zijn 32 groepen (0–31), wat precies in 5 bits past. Meerdere sprites kunnen dezelfde groep delen (bijv. alle frames van een geroteerde animatie gebruiken dezelfde rotatie-parameters).

**Valkuil:** De groep-index staat in attr1 bits [13:9], maar alleen bij affine sprites (attr0 bits [8:9] = 1 of 3). Bij normale sprites bevatten deze bits de flip-flags. Lees ze niet verkeerd!

### Stap 2: `src/core/ppu.zig` — double-size berekening

Bij het renderen van een affine sprite moet je eerst het rendergebied en het middelpunt bepalen:

```zig
fn get_affine_render_bounds(
    attr0: OamAttr0,
    attr1: OamAttr1,
) struct { width: u32, height: u32, render_w: u32, render_h: u32, half_w: i32, half_h: i32 } {
    const size = get_sprite_size(attr0.shape, attr1.size);
    const w: u32 = size.w;
    const h: u32 = size.h;

    // attr0 bits [8:9]: 0=normaal, 1=affine, 2=hidden, 3=affine+double-size
    const double = (attr0.raw >> 8) & 3 == 3;

    const render_w: u32 = if (double) w * 2 else w;
    const render_h: u32 = if (double) h * 2 else h;

    return .{
        .width = w,
        .height = h,
        .render_w = render_w,
        .render_h = render_h,
        .half_w = @intCast(render_w / 2),
        .half_h = @intCast(render_h / 2),
    };
}
```

**Waarom het middelpunt (`half_w`, `half_h`) berekenen?** De affine transformatie roteert om het middelpunt van het rendergebied. Bij double-size is dit middelpunt verschoven: een 32×32 sprite met double-size heeft een rendervak van 64×64, en het middelpunt is (32,32) in plaats van (16,16). De sprite-data zelf is nog steeds 32×32 — het extra rendervak geeft ruimte voor de geroteerde hoeken.

**Waarom `attr0 bits [8:9] == 3` voor double-size?** De GBA hergebruikt bit-combinaties: `00` = normaal, `01` = affine, `10` = verborgen (sprite niet tekenen), `11` = affine + double-size. Let op: als je `10` niet checkt, probeer je verborgen sprites als affine te renderen — een veelvoorkomende bug.

### Stap 3: `src/core/ppu.zig` — per-scanline affine sprite rendering

Dit is de kern van de affine sprite-logica. Voor elke pixel in het rendergebied bereken je welke texture-pixel erbij hoort via de affine matrix:

```zig
fn render_affine_sprite(
    self: *Ppu,
    bus: *Bus,
    index: u7,
    y: u8,
) void {
    const sprite = read_oam_entry(bus, index);
    const dispcnt = bus.read16(0x0400_0000);
    const is_1d = (dispcnt >> 6) & 1 == 1;

    // Haal render bounds en affine parameters op
    const bounds = get_affine_render_bounds(sprite.attr0, sprite.attr1);
    const group: u5 = @truncate(sprite.attr1.raw >> 9);
    const params = read_affine_params(bus, group);

    // Y-positie met wrap-around
    const sprite_y: i32 = blk: {
        const raw: i32 = @as(i32, sprite.attr0.y);
        break :blk if (raw >= 160) raw - 256 else raw;
    };

    const y32: i32 = @as(i32, y);
    const local_y = y32 - sprite_y;
    if (local_y < 0 or local_y >= @as(i32, bounds.render_h)) return;

    // Offset van het middelpunt
    const sy = local_y - bounds.half_h;

    // Bereken start texture-coördinaten voor deze scanline
    // De formule: tex_x = PA * (-half_w) + PB * sy + (width/2) * 256
    // (width/2 * 256 = width << 7, het middelpunt in 8.8 fixed-point)
    var tex_x: i32 = @as(i32, params.pa) * (-bounds.half_w) +
                     @as(i32, params.pb) * sy +
                     (@as(i32, @intCast(bounds.width)) << 7);
    var tex_y: i32 = @as(i32, params.pc) * (-bounds.half_w) +
                     @as(i32, params.pd) * sy +
                     (@as(i32, @intCast(bounds.height)) << 7);

    // X-positie met wrap-around
    const sprite_x: i32 = blk: {
        const raw: i32 = @as(i32, sprite.attr1.x);
        break :blk if (raw >= 240) raw - 512 else raw;
    };

    // Loop door alle pixels in het rendergebied op deze scanline
    var px: u32 = 0;
    while (px < bounds.render_w) : (px += 1) {
        defer {
            tex_x += @as(i32, params.pa);
            tex_y += @as(i32, params.pc);
        }

        const screen_x = sprite_x + @as(i32, @intCast(px));
        if (screen_x < 0 or screen_x >= 240) continue;

        // Converteer fixed-point naar pixel-coördinaten
        const tile_x = tex_x >> 8;
        const tile_y = tex_y >> 8;

        // Buiten sprite-grenzen? Skip (transparant)
        if (tile_x < 0 or tile_x >= @as(i32, bounds.width) or
            tile_y < 0 or tile_y >= @as(i32, bounds.height)) continue;

        // Lees pixel uit tile data (zelfde als normale sprite)
        const color = read_sprite_pixel(
            bus, sprite.attr2,
            @intCast(tile_x), @intCast(tile_y),
            sprite.attr0.color_mode, is_1d,
        );

        if (color != 0) {
            const sx: usize = @intCast(screen_x);
            self.sprite_line[sx] = color;
            self.sprite_priority[sx] = sprite.attr2.priority;
        }
    }
}
```

**Waarom `<< 7` in plaats van `<< 8`?** Het middelpunt van de sprite in texture-space is `(width/2, height/2)`. In 8.8 fixed-point is dat `(width/2) × 256 = width × 128 = width << 7`. Dit is een algebraïsche vereenvoudiging: `(x/2) × 256 = x × 128 = x << 7`. Het bespaart een deling.

**Waarom `defer` voor tex_x/tex_y?** Net als bij affine achtergronden (story 5.5) gebruiken we de per-scanline optimalisatie: bereken de startpositie één keer, tel dan per pixel alleen PA/PC op. De `defer` zorgt dat dit altijd gebeurt, ook bij `continue` (off-screen of transparante pixels).

**Waarom flipping niet apart afhandelen?** Bij affine sprites wordt flipping verwerkt door de matrix. Een horizontale flip is equivalent aan PA=-PA, PB=-PB. Er zijn geen aparte flip-bits in attr1 — die bits worden hergebruikt voor de affine groep-index.

### Stap 4: `src/core/ppu.zig` — affine sprites integreren in sprite renderer

Pas de sprite-rendering loop aan om zowel normale als affine sprites te ondersteunen:

```zig
fn render_all_sprites(self: *Ppu, bus: *Bus, y: u8) void {
    const dispcnt = bus.read16(0x0400_0000);

    // Reset buffers
    @memset(&self.sprite_line, 0);
    @memset(&self.sprite_priority, 4);

    // Loop omgekeerd (hogere index eerst = lagere prioriteit)
    var i: u8 = 128;
    while (i > 0) {
        i -= 1;
        const sprite = read_oam_entry(bus, @truncate(i));

        const obj_mode: u2 = @truncate(sprite.attr0.raw >> 8);

        switch (obj_mode) {
            0b00 => {
                // Normale sprite (story 5.6)
                self.render_normal_sprite(bus, @truncate(i), y);
            },
            0b01 => {
                // Affine sprite (zonder double-size)
                self.render_affine_sprite(bus, @truncate(i), y);
            },
            0b10 => {
                // Verborgen — skip
                continue;
            },
            0b11 => {
                // Affine sprite met double-size
                self.render_affine_sprite(bus, @truncate(i), y);
            },
        }
    }
}
```

**Waarom `obj_mode == 0b10` overslaan?** Bit-patroon `10` betekent "sprite is verborgen". Games gebruiken dit om sprites tijdelijk uit te schakelen zonder hun OAM-data te wissen. Het is essentieel dit te checken, anders render je "verborgen" sprites alsof ze affine zijn (want bit 9 = 1).

**Waarom dezelfde functie voor `0b01` en `0b11`?** Beide zijn affine sprites. Het verschil (double-size) wordt intern in `render_affine_sprite` afgehandeld via `get_affine_render_bounds`. De caller hoeft het onderscheid niet te maken.

### Stap 5: Testen met affine sprite effecten

Test met ROMs die affine sprites gebruiken. Voorbeelden: roterende items, schakelende vijanden, of een roterende spelercursor.

1. **Identiteitsmatrix test**: PA=256, PB=0, PC=0, PD=256. De sprite zou identiek moeten zijn aan een normale sprite op dezelfde positie.
2. **Rotatie test**: Draai een sprite 90°, 180°, 270°. Check dat de rotatie correct is en geen artefacten vertoont.
3. **Schaling test**: Vergroot (PA=128, PD=128 = 2× zoom) en verklein (PA=512, PD=512 = 0.5× zoom) een sprite.
4. **Double-size test**: Roteer een sprite 45° met en zonder double-size. Met double-size moeten de hoeken zichtbaar zijn.
5. **Mix test**: Normale en affine sprites naast elkaar. Check dat prioriteit en transparantie correct werken.

```
Veelvoorkomende bugs:
- Sprite staat op verkeerde positie → middelpunt-berekening is fout (half_w/half_h)
- Sprite is gespiegeld → PA/PC volgorde is verwisseld, of sign error in de matrix
- Hoeken afgesneden → double-size wordt niet correct gedetecteerd (check obj_mode bits)
- Sprite trilt/jittert → fixed-point precisie probleem (check >> 8 shifts)
- "Ghost sprites" verschijnen → obj_mode==0b10 (hidden) wordt niet overgeslagen
- Verkeerde affine parameters → groep-index wordt uit verkeerde bits gelezen
- Affine sprite heeft verkeerde kleuren → palette lookup gebruikt BG palette i.p.v. OBJ palette
```

**Tip:** Begin met een simpele test: maak een ROM die één sprite roteert met bekende parameters. Bereken handmatig waar elke pixel zou moeten verschijnen en vergelijk met je output. Dit is de snelste manier om matrix-fouten te vinden.

## Acceptatiecriteria

- Geroteerde sprites worden correct weergegeven
- Geschaalde sprites (groter en kleiner) werken
- Double-size voorkomt afsnijding bij rotatie
- Affine en normale sprites kunnen naast elkaar bestaan
