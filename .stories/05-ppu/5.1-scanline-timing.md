# Story 5.1 — PPU state en scanline timing

**Epic:** 05 - PPU (Graphics)

**Doel:** PPU timing modelleren op basis van scanlines.

## Zig concepten

### Grote arrays in structs

De framebuffer is 240×160 pixels, elk 16-bit (RGB555):

```zig
const Ppu = struct {
    framebuffer: [240 * 160]u16 = [_]u16{0} ** (240 * 160),
    scanline: u8 = 0,
    cycle: u16 = 0,
};
```

### Modulo/wrap met if

```zig
pub fn step(self: *Ppu, cycles: u32) void {
    self.cycle += @truncate(cycles);
    if (self.cycle >= 1232) {  // 308 dots × 4 cycles/dot
        self.cycle -= 1232;
        self.scanline += 1;
        if (self.scanline >= 228) {
            self.scanline = 0;
        }
    }
}
```

## Context — GBA display timing

### Waarom scanline-based rendering?

De GBA-hardware tekent het scherm **niet** als één compleet frame in één keer. In plaats daarvan werkt de PPU (Picture Processing Unit) **per scanline**: één horizontale lijn van 240 pixels tegelijk, van boven naar beneden. Dit is geen ontwerpkeuze van ons — het is hoe de echte hardware werkt. De CPU en PPU delen dezelfde klok, en de PPU heeft een strikt tijdschema: elke 1232 CPU-cycles is één scanline klaar.

Door dit in de emulator na te bootsen krijg je:
- **Correcte timing**: games lezen VCOUNT om te weten welke scanline actief is (bijv. voor raster-effecten)
- **IRQ timing**: VBlank en HBlank interrupts moeten op het juiste moment vuren
- **Mid-frame veranderingen**: sommige games veranderen registers halverwege een frame voor speciale effecten

### Waarom 228 scanlines (niet 160)?

Het scherm is 160 pixels hoog, maar de PPU heeft 228 scanlines per frame. De extra 68 scanlines (160–227) zijn de **VBlank periode** — het scherm wordt dan niet getekend. Games gebruiken VBlank om VRAM te updaten zonder visuele glitches, DMA transfers te starten, en gamelogica te draaien.

### Waarom het VCOUNT register belangrijk is

VCOUNT (0x04000006) geeft aan welke scanline de PPU momenteel verwerkt. Games lezen dit register constant:
- Om te weten of ze in VBlank zitten (veilig om VRAM te schrijven)
- Voor **raster-effecten**: palette of scroll registers veranderen op een specifieke scanline
- Om te synchroniseren met de PPU via de VCOUNT match feature in DISPSTAT

De GBA tekent het scherm lijn voor lijn (scanline rendering):

```
Scanline 0-159:    Zichtbare lijnen (HDraw + HBlank per lijn)
Scanline 160-227:  VBlank (scherm wordt niet getekend)

Per scanline:
├── HDraw (240 dots = 960 cycles) ──┤── HBlank (68 dots = 272 cycles) ──┤
                                     totaal: 308 dots = 1232 cycles

Totaal per frame: 228 scanlines × 1232 cycles = 280896 cycles ≈ 59.7 fps
```

### Relevante I/O registers

```
DISPCNT  (0x04000000) — display mode en welke lagen actief zijn
DISPSTAT (0x04000004) — VBlank/HBlank status flags + IRQ enables
VCOUNT   (0x04000006) — huidige scanline nummer (read-only)
```

DISPSTAT layout (packed struct):
```zig
const DispStat = packed struct(u16) {
    vblank: u1,         // bit 0: 1 als in VBlank (scanline 160-227)
    hblank: u1,         // bit 1: 1 als in HBlank
    vcounter: u1,       // bit 2: 1 als VCOUNT == LYC
    vblank_irq: u1,     // bit 3: trigger IRQ bij VBlank?
    hblank_irq: u1,     // bit 4: trigger IRQ bij HBlank?
    vcounter_irq: u1,   // bit 5: trigger IRQ bij VCOUNT match?
    _unused: u2,
    lyc: u8,            // bits [15:8]: scanline om op te matchen
};
```

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — Ppu struct definiëren

Begin met de basisstruct die alle PPU-state bijhoudt. De framebuffer is een array van 240×160 pixels in RGB555-formaat — dit is het "canvas" waar alle rendering naartoe schrijft, en dat uiteindelijk naar de SDL texture gekopieerd wordt.

```zig
const Ppu = struct {
    framebuffer: [240 * 160]u16 = [_]u16{0} ** (240 * 160),
    scanline: u9 = 0,    // u9 want max waarde is 227 (past in 8 bits, maar 9 bits geeft ruimte)
    cycle: u16 = 0,       // cycles binnen huidige scanline (max 1232)

    pub fn init() Ppu {
        return Ppu{};
    }
};
```

**Waarom `u9` voor scanline?** Hoewel 227 in een `u8` past, gebruiken we `u9` zodat tussenberekeningen (bijv. `scanline + 1`) niet overflown. Zig is strikt met integer overflow — een `u8` die 228 wordt is een runtime error in debug mode.

**Waarom de framebuffer als array in de struct?** De framebuffer is altijd precies 240×160×2 bytes. Door het in de struct te zetten is het eigendom duidelijk: de PPU beheert zijn eigen output. Later geeft `present()` in de renderer een pointer naar deze array aan SDL.

### Stap 2: `src/core/ppu.zig` — `step()` functie voor timing

De `step()` functie is het hart van de PPU-timing. De CPU roept deze functie aan na elke instructie, met het aantal verbruikte cycles. De PPU telt cycles op en bepaalt wanneer een scanline klaar is.

```zig
pub fn step(self: *Ppu, bus: *Bus, cycles: u32) void {
    var remaining = cycles;

    while (remaining > 0) {
        const to_consume = @min(remaining, @as(u32, 1232) - self.cycle);
        self.cycle += @as(u16, @truncate(to_consume));
        remaining -= to_consume;

        if (self.cycle >= 1232) {
            self.cycle -= 1232;
            self.end_scanline(bus);
        }
    }
}

fn end_scanline(self: *Ppu, bus: *Bus) void {
    self.scanline += 1;
    if (self.scanline >= 228) {
        self.scanline = 0;
    }
    self.update_io_registers(bus);
}
```

**Waarom een while-loop i.p.v. simpele optelling?** Een CPU-instructie kan meerdere cycles kosten (bijv. een multiply = 4 cycles). Als de PPU halverwege een instructie een scanline-grens passeert, moet je dat correct afhandelen. De while-loop verwerkt cycles in stukken: eerst de rest van de huidige scanline, dan eventueel de volgende.

**Waarom `@min()` gebruiken?** Dit voorkomt dat je per ongeluk voorbij de scanline-grens springt. Je consumeert maximaal het aantal cycles dat nodig is om de huidige scanline af te maken.

**Valkuil:** Vergeet niet dat `step()` meerdere keren `end_scanline()` kan aanroepen als `cycles` groot genoeg is (bijv. bij DMA transfers). De while-loop handelt dit correct af.

### Stap 3: `src/core/ppu.zig` — DISPSTAT en VCOUNT registers bijwerken

Na elke scanline-overgang moet je de I/O registers updaten. Games lezen deze registers om te bepalen waar de PPU is in het frame.

```zig
fn update_io_registers(self: *Ppu, bus: *Bus) void {
    // VCOUNT (0x04000006) — huidige scanline, read-only voor de CPU
    bus.io_write16(0x04000006, @as(u16, self.scanline));

    // DISPSTAT (0x04000004) — status flags bijwerken
    var dispstat = bus.io_read16(0x04000004);

    // Bit 0: VBlank flag — 1 als scanline >= 160
    if (self.scanline >= 160) {
        dispstat |= 0x0001;
    } else {
        dispstat &= ~@as(u16, 0x0001);
    }

    // Bit 2: VCounter match — 1 als VCOUNT == LYC (bits [15:8] van DISPSTAT)
    const lyc: u9 = @truncate(dispstat >> 8);
    if (self.scanline == lyc) {
        dispstat |= 0x0004;
    } else {
        dispstat &= ~@as(u16, 0x0004);
    }

    bus.io_write16(0x04000004, dispstat);
}
```

**Waarom DISPSTAT lezen en dan terugschrijven?** De bovenste byte van DISPSTAT (LYC en IRQ enable bits) wordt door de *game* geschreven. Je mag die bits niet overschrijven. Daarom lees je het register, pas je alleen de status-bits aan (bits 0–2), en schrijf je het terug.

**Waarom apart van step()?** Door register-updates in een aparte functie te zetten, blijft `step()` overzichtelijk. Bovendien kun je `update_io_registers()` later uitbreiden met HBlank timing zonder `step()` te wijzigen.

### Stap 4: `src/core/ppu.zig` — HBlank timing toevoegen

Elke scanline bestaat uit twee fasen: HDraw (960 cycles, pixels tekenen) en HBlank (272 cycles, pauze). Veel games gebruiken HBlank voor DMA transfers en register-updates.

```zig
pub fn step(self: *Ppu, bus: *Bus, cycles: u32) void {
    var remaining = cycles;

    while (remaining > 0) {
        const to_consume = @min(remaining, @as(u32, 1232) - self.cycle);
        self.cycle += @as(u16, @truncate(to_consume));
        remaining -= to_consume;

        // HBlank begint na 960 cycles (einde van HDraw)
        if (self.cycle >= 960 and !self.in_hblank) {
            self.in_hblank = true;
            self.on_hblank_start(bus);
        }

        if (self.cycle >= 1232) {
            self.cycle -= 1232;
            self.in_hblank = false;
            self.end_scanline(bus);
        }
    }
}

fn on_hblank_start(self: *Ppu, bus: *Bus) void {
    // Zet HBlank flag in DISPSTAT (bit 1)
    var dispstat = bus.io_read16(0x04000004);
    dispstat |= 0x0002;
    bus.io_write16(0x04000004, dispstat);

    // Render de scanline nu — dit is het moment dat de hardware klaar is
    // met tekenen en de pixels "vastliggen"
    if (self.scanline < 160) {
        self.render_scanline(bus);
    }
}
```

**Waarom renderen bij HBlank start?** Op de echte hardware is de scanline klaar met tekenen zodra HBlank begint. Door op dit moment te renderen, simuleer je het correcte tijdstip. Als een game registers verandert tijdens HDraw, worden die veranderingen meegenomen. Veranderingen tijdens HBlank zijn voor de *volgende* scanline.

**Waarom een `in_hblank` boolean?** Zonder deze flag zou je de HBlank-logica elke keer triggeren als `cycle >= 960`, ook als je al in HBlank zit. De boolean zorgt dat het precies één keer per scanline gebeurt.

### Stap 5: `src/core/ppu.zig` — IRQ triggering

De GBA kan interrupts genereren bij VBlank, HBlank en VCOUNT match. Dit is essentieel: veel games draaien hun hoofdloop in de VBlank IRQ handler.

```zig
fn check_irqs(self: *Ppu, bus: *Bus) void {
    const dispstat = bus.io_read16(0x04000004);

    var irq_flags: u16 = 0;

    // VBlank IRQ (DISPSTAT bit 3 = enable, check bij scanline 160)
    if (self.scanline == 160 and (dispstat & 0x0008) != 0) {
        irq_flags |= 0x0001;  // IF bit 0 = VBlank
    }

    // HBlank IRQ (DISPSTAT bit 4 = enable)
    if ((dispstat & 0x0010) != 0 and self.in_hblank) {
        irq_flags |= 0x0002;  // IF bit 1 = HBlank
    }

    // VCounter IRQ (DISPSTAT bit 5 = enable, bij VCOUNT match)
    if ((dispstat & 0x0020) != 0 and self.scanline == @as(u9, @truncate(dispstat >> 8))) {
        irq_flags |= 0x0004;  // IF bit 2 = VCounter
    }

    if (irq_flags != 0) {
        bus.request_irq(irq_flags);
    }
}
```

**Waarom IRQs apart checken?** IRQ-triggering is een side-effect van timing. Door het in een aparte functie te zetten, kun je het eenvoudig aan/uitzetten tijdens debugging. Roep `check_irqs()` aan vanuit `end_scanline()` (voor VBlank/VCounter) en `on_hblank_start()` (voor HBlank).

**Valkuil:** De VBlank IRQ moet alleen vuren bij de *overgang* naar scanline 160, niet op elke scanline 160–227. Gebruik een flag of check specifiek `self.scanline == 160`.

### Stap 6: `src/main.zig` — PPU integreren in de main loop

De main loop moet nu de PPU stappen laten maken synchroon met de CPU:

```zig
const Ppu = @import("core/ppu.zig").Ppu;
const platform = @import("platform/renderer.zig");

pub fn main() void {
    var renderer = platform.Renderer.init();
    defer renderer.deinit();

    var ppu = Ppu.init();

    while (platform.Renderer.processEvents()) {
        // Eén frame = 280896 cycles (228 scanlines × 1232 cycles)
        var frame_cycles: u32 = 0;
        while (frame_cycles < 280896) {
            // Later: const cpu_cycles = cpu.step();
            const cpu_cycles: u32 = 4;  // placeholder
            ppu.step(bus, cpu_cycles);
            frame_cycles += cpu_cycles;
        }

        renderer.present_frame(&ppu.framebuffer);
    }
}
```

**Waarom 280896 cycles per frame?** Dit is 228 scanlines × 1232 cycles/scanline = 280896 cycles. Bij een kloksnelheid van 16.78 MHz geeft dit ~59.7 fps — exact de GBA framerate.

**Waarom de framebuffer als pointer doorgeven?** De framebuffer is 76800 bytes (240×160×2). Kopiëren is duur. Door een pointer door te geven, kan SDL direct uit het PPU-geheugen lezen.
