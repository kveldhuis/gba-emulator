# Story 5.3 — Background Mode 4 (paletted bitmap)

**Epic:** 05 - PPU (Graphics)

**Doel:** 8-bit paletted bitmap mode.

## Zig concepten

### Waarom Mode 4?

Mode 4 lost twee beperkingen van Mode 3 op:
1. **Geheugengebruik**: In Mode 3 kost elke pixel 2 bytes (76800 bytes totaal). Dat is bijna alle VRAM (96KB). In Mode 4 kost elke pixel slechts 1 byte (38400 bytes) — de helft. De kleur wordt opgezocht in Palette RAM via een index.
2. **Double buffering**: Omdat een Mode 4 frame maar 38400 bytes kost, passen er twee frames in VRAM. Games tekenen op de onzichtbare buffer en wisselen dan — geen tearing of half-getekende frames zichtbaar.

Het nadeel: Mode 4 beperkt je tot 256 kleuren (uit een palette), terwijl Mode 3 alle 32768 kleuren direct kan gebruiken. Voor de meeste games is 256 kleuren ruim voldoende.

### Palette lookup — indirecte kleuren

In Mode 4 is elke pixel 1 byte (index) in plaats van 2 bytes (kleur). De index verwijst naar een kleur in Palette RAM.

```zig
fn render_mode4(self: *Ppu, bus: *Bus, y: u8) void {
    // DISPCNT bit 4 bepaalt welke buffer actief is
    const dispcnt = bus.read16(0x04000000);
    const base: u32 = if ((dispcnt >> 4) & 1 == 1) 0x0600A000 else 0x06000000;

    for (0..240) |x| {
        const pixel_offset = @as(u32, y) * 240 + @as(u32, @intCast(x));

        // Stap 1: lees palette index uit VRAM
        const index: u32 = bus.read8(base + pixel_offset);

        // Stap 2: zoek kleur op in Palette RAM (elke entry = 2 bytes)
        const color = bus.read16(0x05000000 + index * 2);

        self.framebuffer[pixel_offset] = color;
    }
}
```

### Double buffering

Mode 4 gebruikt de helft van de VRAM per frame (240×160×1 = 38400 bytes). Er passen twee frames in VRAM. DISPCNT bit 4 bepaalt welke actief is. Games wisselen dit bit om "page flipping" te doen — schrijf naar de onzichtbare buffer, flip, herhaal.

### Waarom page flipping?

Zonder double buffering tekent de game direct in het VRAM dat de PPU op dat moment leest. Als de game halverwege een frame is met tekenen wanneer de PPU die pixels leest, zie je een half-oud, half-nieuw beeld — dit heet **tearing**. Met double buffering schrijft de game naar een onzichtbare buffer terwijl de PPU de andere buffer toont. Na het tekenen wisselt de game de buffers (door DISPCNT bit 4 te flippen), en het nieuwe frame verschijnt in één keer. Dit geeft een schoon, vloeiend beeld.

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — Mode 4 toevoegen aan render_scanline()

Voeg Mode 4 toe aan de switch in `render_scanline()`. Het patroon is hetzelfde als Mode 3: per scanline, per pixel, maar nu met een extra stap (palette lookup).

```zig
pub fn render_scanline(self: *Ppu, bus: *Bus) void {
    const y = self.scanline;
    if (y >= 160) return;

    const dispcnt = bus.read16(0x04000000);
    const mode: u3 = @truncate(dispcnt & 0x7);

    switch (mode) {
        3 => self.render_mode3(bus, @truncate(y)),
        4 => self.render_mode4(bus, @truncate(y)),
        else => {},
    }
}
```

**Waarom dezelfde structuur als Mode 3?** Alle modes delen dezelfde rendering-pipeline: lees pixels per scanline, schrijf naar framebuffer. Alleen de manier waarop je van VRAM-data naar een kleur komt verschilt. Dit consistente patroon maakt het toevoegen van nieuwe modes eenvoudig.

### Stap 2: `src/core/ppu.zig` — render_mode4() met palette lookup

Het verschil met Mode 3: in plaats van direct een RGB555-kleur uit VRAM te lezen, lees je een 8-bit **index** die verwijst naar een kleur in Palette RAM.

```zig
fn render_mode4(self: *Ppu, bus: *Bus, y: u8) void {
    // DISPCNT bit 4 bepaalt welke buffer actief is
    const dispcnt = bus.read16(0x04000000);
    const page: u1 = @truncate(dispcnt >> 4);
    const base: u32 = if (page == 1) 0x0600_A000 else 0x0600_0000;

    const y32: u32 = @as(u32, y);

    for (0..240) |x| {
        const x32: u32 = @intCast(x);
        const pixel_offset: u32 = y32 * 240 + x32;

        // Stap 1: lees palette index (1 byte) uit VRAM
        const index: u32 = bus.read8(base + pixel_offset);

        // Stap 2: zoek kleur op in Palette RAM
        // Palette RAM start op 0x05000000
        // Elke kleur is 2 bytes (u16 in RGB555), dus index × 2
        const color = bus.read16(0x0500_0000 + index * 2);

        self.framebuffer[y32 * 240 + x32] = color;
    }
}
```

**Waarom `index * 2`?** De palette index (0–255) verwijst naar de n-de kleur. Elke kleur is 2 bytes (u16), dus het adres in Palette RAM is `base + index × 2`. Dit is hetzelfde principe als array-indexering: `array[i]` = `base + i * sizeof(element)`.

**Waarom `bus.read8()` voor de index?** In Mode 4 is elke pixel precies 1 byte (8 bits, 256 mogelijke kleuren). In Mode 3 was het `read16()` (2 bytes per pixel). Dit is het kernverschil: Mode 4 halveert VRAM-gebruik door indirectie via een palette.

**Valkuil:** Palette index 0 is typisch de transparante/achtergrondkleur bij sprites en tiled modes, maar bij Mode 4 backgrounds wordt index 0 gewoon als kleur getekend. Transparantie geldt hier niet.

### Stap 3: `src/core/ppu.zig` — DISPCNT bit 4 (page flipping) correct afhandelen

De page flip wordt bepaald door bit 4 van DISPCNT. Buffer 0 begint op `0x06000000`, buffer 1 op `0x0600A000`.

```zig
// De twee buffers in VRAM:
// Buffer 0: 0x06000000 – 0x06009600 (38400 bytes = 240 × 160)
// Buffer 1: 0x0600A000 – 0x06013600 (38400 bytes = 240 × 160)
//
// Merk op: er zit een "gat" van 0x400 bytes (1024 bytes) tussen de buffers.
// Dit is een hardware-eigenaardigheid van de GBA.
```

**Waarom 0x0600A000 en niet 0x06009600?** De GBA aligneert de tweede buffer op een 0xA000-grens (40960 bytes). Dit is waarschijnlijk voor eenvoudiger hardware-adressering: bit 15 van het VRAM-adres (na masking) bepaalt de pagina. De 0x400 bytes "gap" ertussen worden niet gebruikt.

**Tip:** Test page flipping met een ROM die afwisselend twee verschillende beelden toont. Als je altijd hetzelfde beeld ziet, wordt DISPCNT bit 4 niet correct gelezen.

### Stap 4: `src/core/ppu.zig` — Palette RAM als apart geheugen

Palette RAM (PRAM) is een apart geheugengebied van 1KB (1024 bytes = 512 kleuren). Het bevat:
- **BG palette** (0x05000000–0x050001FF): 256 kleuren voor achtergronden
- **OBJ palette** (0x05000200–0x050003FF): 256 kleuren voor sprites

```zig
// In je Bus/Memory struct:
palette_ram: [1024]u8 = [_]u8{0} ** 1024,

// Bij read:
fn read_palette(self: *Bus, addr: u32) u16 {
    const offset = addr & 0x3FF;  // mirror: PRAM herhaalt elke 1KB
    return @as(u16, self.palette_ram[offset]) |
           (@as(u16, self.palette_ram[offset + 1]) << 8);
}
```

**Waarom aparte palette RAM?** Op de echte GBA is Palette RAM fysiek apart geheugen met eigen bus-timing (geen waitstates). Door het apart te modelleren in je emulator, respecteer je de hardware-architectuur en kun je later correcte timing toevoegen.

### Stap 5: Testen met een Mode 4 ROM

Test met een homebrew ROM die Mode 4 gebruikt. Controleer:

1. **Kleuren kloppen:** De palette lookup moet correcte kleuren opleveren. Verkeerde kleuren betekent vaak dat de palette base-adres fout is (BG palette vs OBJ palette)
2. **Page flipping werkt:** Als de ROM page flipping doet (veel animatie-demos doen dit), moet je twee afwisselende beelden zien
3. **Geen off-by-one:** Het beeld moet 240×160 zijn zonder verschuiving. Een verschuiving van 1 pixel betekent dat je offset-berekening niet klopt

```
Veelvoorkomende bugs:
- Alle pixels dezelfde kleur → palette RAM is niet geïnitialiseerd (check ROM loading)
- Kleuren bijna goed maar net verkeerd → palette base-adres is 1 of 2 bytes verschoven
- Beeld lijkt "dubbel" → je leest 2 bytes per pixel i.p.v. 1 byte (Mode 3 code per ongeluk)
- Beeld flikkert → page flip werkt maar je rendert op het verkeerde moment
```
