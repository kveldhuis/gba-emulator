# Story 5.4 — Background Mode 0 (tiled, geen rotatie)

**Epic:** 05 - PPU (Graphics)

**Doel:** Tile-based rendering met 4 text-mode achtergrondlagen.

## Zig concepten

### Waarom tiles?

Bitmap modes (Mode 3/4) zijn simpel maar **duur** in geheugen: 240×160 pixels kosten 38–76KB, bijna alle VRAM. In de meeste games herhaalt veel van het beeld zich — gras, muren, lucht zijn allemaal variaties op een paar basispatronen.

Tile-based rendering lost dit op: je definieert kleine 8×8 pixel blokjes (tiles) en beschrijft vervolgens in een **tilemap** welke tile waar op het scherm staat. Een typisch Level met 256×256 pixels heeft een tilemap van 32×32 entries (2KB), plus misschien 100 unieke tiles (3.2–6.4KB). Totaal: ~5–8KB in plaats van 76KB. Dit is een factor 10–15 besparing.

Vrijwel alle commerciële GBA-games gebruiken tile modes. Mode 0 is de meest gebruikte: 4 achtergrondlagen, elk met hun eigen tilemap en scroll-positie.

### Waarom 4 lagen?

Mode 0 biedt 4 onafhankelijke achtergrondlagen (BG0–BG3). Games gebruiken dit voor **parallax scrolling**: de achterste laag (bijv. lucht) scrolt langzaam, de middelste (bergen) sneller, en de voorste (grond) het snelst. Dit geeft een illusie van diepte. Elke laag heeft eigen scroll-registers, dus je kunt ze onafhankelijk bewegen.

### Waarom tilemap en tile data gescheiden?

De tilemap ("welke tile staat waar") en de tile data ("hoe ziet elke tile eruit") staan op verschillende plekken in VRAM. Dit is bewust:
- Meerdere tilemaps kunnen dezelfde tile data delen (bijv. BG0 en BG1 gebruiken dezelfde tiles maar andere layouts)
- Je kunt tile data updaten zonder de layout te veranderen (animatie)
- De hardware kan efficiënter werken met gescheiden data-stromen

### Hoe tiled graphics werken

In plaats van elke pixel apart op te slaan, bestaat het beeld uit herbruikbare **tiles** (8×8 pixel blokjes). Een **tilemap** beschrijft welke tile waar staat.

```
Tilemap (32×32 entries):          Tile data (8×8 pixels):
┌───┬───┬───┬───┐                Tile 0: ████████
│ 0 │ 1 │ 0 │ 3 │                        ██    ██
├───┼───┼───┼───┤                        ██    ██
│ 2 │ 0 │ 1 │ 2 │                        ████████
└───┴───┴───┴───┘                         ...
Elk nummer verwijst naar een tile
```

### Tilemap entries (screen entries)

Elke tilemap entry is 16 bits:
```zig
const ScreenEntry = packed struct(u16) {
    tile_id: u10,      // welke tile (0-1023)
    h_flip: u1,        // horizontaal spiegelen
    v_flip: u1,        // verticaal spiegelen
    palette: u4,       // welke sub-palette (bij 4bpp)
};
```

### 4bpp vs 8bpp tiles

- **4bpp** (4 bits per pixel): 16 kleuren per tile, maar je kunt per tile een andere sub-palette kiezen. Elke tile = 32 bytes.
- **8bpp** (8 bits per pixel): 256 kleuren uit de volledige palette. Elke tile = 64 bytes.

```zig
// 4bpp: 2 pixels per byte
const byte = bus.read8(tile_base + tile_id * 32 + row * 4 + col / 2);
const pixel = if (col % 2 == 0) byte & 0xF else byte >> 4;
// pixel is een 4-bit palette index

// 8bpp: 1 pixel per byte
const pixel = bus.read8(tile_base + tile_id * 64 + row * 8 + col);
```

### BG control registers

Elke achtergrondlaag heeft een control register:
```
BG0CNT (0x04000008)
BG1CNT (0x0400000A)
BG2CNT (0x0400000C)
BG3CNT (0x0400000E)
```

```zig
const BgCnt = packed struct(u16) {
    priority: u2,          // 0=hoogste prioriteit
    tile_base: u2,         // base adres voor tile data (×16KB)
    _unused: u2,
    mosaic: u1,
    color_mode: u1,        // 0=4bpp, 1=8bpp
    map_base: u5,          // base adres voor tilemap (×2KB)
    _unused2: u1,
    screen_size: u2,       // 0=256x256, 1=512x256, 2=256x512, 3=512x512
};
```

### Scrolling

Elke BG laag heeft horizontale en verticale scroll registers:
```
BG0HOFS (0x04000010) — horizontale scroll offset
BG0VOFS (0x04000012) — verticale scroll offset
```

```zig
// Bereken welke pixel in de tilemap we nodig hebben:
const scroll_x = bus.read16(0x04000010) & 0x1FF;  // 9-bit
const scroll_y = bus.read16(0x04000012) & 0x1FF;

const map_x = (screen_x + scroll_x) % map_width;
const map_y = (screen_y + scroll_y) % map_height;

// Welke tile en welke pixel in die tile:
const tile_col = map_x / 8;
const tile_row = map_y / 8;
const pixel_x = map_x % 8;   // of: map_x & 7
const pixel_y = map_y % 8;
```

### Laag prioriteit

Meerdere lagen kunnen dezelfde pixel bedekken. Priority bepaalt welke bovenop tekent:
- Priority 0 = bovenop, Priority 3 = achteraan
- Bij gelijke priority: lagere BG nummer wint (BG0 > BG1 > BG2 > BG3)

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — BG control register parsen

Elke BG-laag wordt geconfigureerd door een BgCnt register. Begin met het parsen van dit register om te weten waar de tilemap en tile data staan, en welk kleurformaat gebruikt wordt.

```zig
const BgCnt = packed struct(u16) {
    priority: u2,          // 0=hoogste prioriteit, 3=laagste
    tile_base: u2,         // base adres voor tile data: waarde × 16KB
    _unused: u2,
    mosaic: u1,
    color_mode: u1,        // 0=4bpp (16 kleuren), 1=8bpp (256 kleuren)
    map_base: u5,          // base adres voor tilemap: waarde × 2KB
    _unused2: u1,
    screen_size: u2,       // 0=256×256, 1=512×256, 2=256×512, 3=512×512
};

fn read_bgcnt(bus: *Bus, bg: u2) BgCnt {
    const addr: u32 = 0x0400_0008 + @as(u32, bg) * 2;
    return @bitCast(bus.read16(addr));
}
```

**Waarom packed struct?** Zig's `packed struct` laat je de exacte bit-layout van een hardware-register beschrijven. Door `@bitCast` te gebruiken kun je een `u16` direct naar de struct converteren zonder handmatig bits te shiften en masken. Dit maakt de code leesbaarder en minder foutgevoelig.

**Waarom `tile_base × 16KB`?** De GBA adresseert tile data in blokken van 16KB (16384 bytes). Een `tile_base` waarde van 2 betekent dus dat de tile data begint op `0x06000000 + 2 × 16384 = 0x06008000`. Deze indirectie bespaart bits in het register.

### Stap 2: `src/core/ppu.zig` — tilemap entry lezen en decoderen

Voor elke pixel op het scherm moet je bepalen: welke tile staat hier, en welke pixel binnen die tile? Dit doe je door de screen-coördinaat om te rekenen naar tilemap-coördinaten.

```zig
fn read_screen_entry(bus: *Bus, map_base: u32, tile_col: u32, tile_row: u32, screen_size: u2) u16 {
    // Tilemap is 32×32 entries per "screen block" (2KB)
    // Bij grotere maps worden meerdere screen blocks gecombineerd:
    //   256×256 (size 0): 1 block
    //   512×256 (size 1): 2 blocks naast elkaar
    //   256×512 (size 2): 2 blocks boven elkaar
    //   512×512 (size 3): 4 blocks (2×2)

    var block_offset: u32 = 0;

    // Horizontale block: als tile_col >= 32, ga naar het volgende block
    if (tile_col >= 32) {
        block_offset += 0x800;  // 2KB = één screen block
    }

    // Verticale block: als tile_row >= 32
    if (tile_row >= 32) {
        block_offset += if (screen_size == 3) 0x1000 else 0x800;
    }

    const local_col = tile_col & 31;  // mod 32
    const local_row = tile_row & 31;
    const entry_offset = (local_row * 32 + local_col) * 2;

    return bus.read16(map_base + block_offset + entry_offset);
}
```

**Waarom screen blocks?** De GBA organiseert tilemaps in blokken van 32×32 entries (2KB). Een 256×256 map is precies één block. Grotere maps combineren meerdere blocks. Dit is een hardware-eigenaardigheid die je exact moet modelleren, anders worden grotere maps verkeerd getekend (bijv. de rechterhelft herhaalt de linker).

**Valkuil:** De block-layout bij 512×512 (size 3) is niet intuïtief. De volgorde is: linksboven, rechtsboven, linksonder, rechtsonder. Bij size 2 (256×512) is het: boven, onder. Let op de `block_offset` berekening — dit is een veelvoorkomende bron van bugs.

### Stap 3: `src/core/ppu.zig` — tile pixel decoderen (4bpp en 8bpp)

Nu je weet welke tile op een positie staat, moet je de juiste pixel uit de tile data lezen. De GBA ondersteunt twee formaten:

```zig
fn read_tile_pixel(
    bus: *Bus,
    tile_base: u32,
    tile_id: u10,
    pixel_x: u3,
    pixel_y: u3,
    color_mode: u1,
    palette_num: u4,
    h_flip: bool,
    v_flip: bool,
) u16 {
    // Pas flip toe
    const px: u3 = if (h_flip) 7 - pixel_x else pixel_x;
    const py: u3 = if (v_flip) 7 - pixel_y else pixel_y;

    if (color_mode == 0) {
        // 4bpp: elke tile = 32 bytes (8 rijen × 4 bytes per rij)
        // 2 pixels per byte (4 bits per pixel)
        const tile_addr = tile_base + @as(u32, tile_id) * 32;
        const byte_offset = @as(u32, py) * 4 + @as(u32, px) / 2;
        const byte = bus.read8(tile_addr + byte_offset);

        // Lage nibble = even pixel, hoge nibble = oneven pixel
        const index: u8 = if (px & 1 == 0) byte & 0x0F else byte >> 4;
        if (index == 0) return 0;  // transparant

        // Sub-palette: palette_num selecteert 1 van 16 sub-palettes (elk 16 kleuren)
        return bus.read16(0x0500_0000 + @as(u32, palette_num) * 32 + @as(u32, index) * 2);
    } else {
        // 8bpp: elke tile = 64 bytes (8 rijen × 8 bytes per rij)
        // 1 pixel per byte (8 bits per pixel)
        const tile_addr = tile_base + @as(u32, tile_id) * 64;
        const byte_offset = @as(u32, py) * 8 + @as(u32, px);
        const index: u32 = bus.read8(tile_addr + byte_offset);
        if (index == 0) return 0;  // transparant

        // Volledige 256-kleuren palette
        return bus.read16(0x0500_0000 + index * 2);
    }
}
```

**Waarom 4bpp en 8bpp?** 4bpp tiles zijn 32 bytes (half zo groot als 8bpp), maar beperkt tot 16 kleuren per tile. Het slimme is dat elke tile een *andere* sub-palette kan kiezen uit 16 mogelijkheden (totaal 16×16 = 256 kleuren in het spel). 8bpp tiles hebben direct 256 kleuren maar kosten meer VRAM. Games wisselen tussen beide afhankelijk van de scene.

**Waarom is index 0 transparant?** Dit is een hardware-conventie. Palette index 0 betekent "deze pixel is doorzichtig" — de achterliggende laag schijnt erdoorheen. Dit is essentieel voor het lagen-systeem: zonder transparantie zou elke laag de vorige volledig bedekken.

**Valkuil: nibble-volgorde bij 4bpp.** De *lage* nibble (bits 3–0) is de *eerste* pixel (even x), de *hoge* nibble (bits 7–4) is de *tweede* pixel (oneven x). Dit is counter-intuïtief als je gewend bent aan big-endian byte-volgorde.

### Stap 4: `src/core/ppu.zig` — scrolling implementeren

Elke BG-laag heeft horizontale en verticale scroll-registers. Scrolling verschuift het "kijkvenster" over de tilemap.

```zig
fn render_bg_scanline(self: *Ppu, bus: *Bus, bg: u2, y: u8) void {
    const bgcnt = read_bgcnt(bus, bg);
    const scroll_reg_base: u32 = 0x0400_0010 + @as(u32, bg) * 4;

    // Scroll offsets zijn 9-bit (0–511), wrappen automatisch
    const scroll_x: u32 = bus.read16(scroll_reg_base) & 0x1FF;
    const scroll_y: u32 = bus.read16(scroll_reg_base + 2) & 0x1FF;

    const tile_base: u32 = 0x0600_0000 + @as(u32, bgcnt.tile_base) * 0x4000;
    const map_base: u32 = 0x0600_0000 + @as(u32, bgcnt.map_base) * 0x800;

    // Bereken de map-grootte in pixels
    const map_w: u32 = if (bgcnt.screen_size & 1 != 0) 512 else 256;
    const map_h: u32 = if (bgcnt.screen_size & 2 != 0) 512 else 256;

    const y32: u32 = @as(u32, y);

    for (0..240) |screen_x| {
        const x32: u32 = @intCast(screen_x);

        // Pas scroll offset toe en wrap binnen map-grenzen
        const map_x = (x32 + scroll_x) % map_w;
        const map_y = (y32 + scroll_y) % map_h;

        // Welke tile en welke pixel binnen die tile
        const tile_col = map_x / 8;
        const tile_row = map_y / 8;
        const pixel_x: u3 = @truncate(map_x & 7);  // map_x % 8
        const pixel_y: u3 = @truncate(map_y & 7);

        // Lees tilemap entry
        const entry = read_screen_entry(bus, map_base, tile_col, tile_row, bgcnt.screen_size);
        const se: ScreenEntry = @bitCast(entry);

        // Lees pixel kleur
        const color = read_tile_pixel(
            bus, tile_base, se.tile_id, pixel_x, pixel_y,
            bgcnt.color_mode, se.palette, se.h_flip == 1, se.v_flip == 1,
        );

        if (color != 0) {
            // Sla op in een per-laag buffer (voor prioriteit-sorting later)
            self.bg_line[bg][x32] = color;
            self.bg_opaque[bg][x32] = true;
        }
    }
}
```

**Waarom `& 0x1FF` voor scroll?** Scroll-registers zijn 9 bits breed (0–511). De hardware negeert hogere bits. Door te masken simuleer je dit gedrag exact.

**Waarom `% map_w` voor wrapping?** Tilemap-rendering wrapt: als je voorbij de rechterrand scrollt, verschijnt de linkerrand weer. Dit is hoe hardware-scrolling werkt — de tilemap is een torus (herhalend in beide richtingen).

**Tip:** Gebruik `& 7` in plaats van `% 8` voor de pixel-offset binnen een tile. Beide geven hetzelfde resultaat (voor machten van 2), maar bitwise AND is sneller en maakt de intentie ("we willen de onderste 3 bits") duidelijker.

### Stap 5: `src/core/ppu.zig` — meerdere lagen compositen met prioriteit

Mode 0 heeft 4 lagen. Je moet ze in de juiste volgorde compositen (van achter naar voor):

```zig
fn composite_scanline(self: *Ppu, bus: *Bus, y: u8) void {
    const dispcnt = bus.read16(0x0400_0000);
    const y32: u32 = @as(u32, y);

    // Achtergrondkleur (palette entry 0) als default
    const backdrop = bus.read16(0x0500_0000);

    // Begin met backdrop voor elke pixel
    for (0..240) |x| {
        self.framebuffer[y32 * 240 + x] = backdrop;
    }

    // Sorteer lagen op prioriteit (laagste priority nummer = bovenop)
    // Bij gelijke prioriteit: lagere BG nummer wint
    const bg_order = sort_bgs_by_priority(bus);

    // Teken van achter naar voor (painter's algorithm)
    var i: usize = 3;
    while (true) : (i -= 1) {
        const bg = bg_order[i];

        // Check of deze laag actief is in DISPCNT
        if ((dispcnt >> (8 + @as(u4, bg))) & 1 == 0) {
            if (i == 0) break;
            continue;
        }

        // Render deze laag
        self.render_bg_scanline(bus, bg, y);

        // Overwrite framebuffer met niet-transparante pixels
        for (0..240) |x| {
            if (self.bg_opaque[bg][x]) {
                self.framebuffer[y32 * 240 + x] = self.bg_line[bg][x];
            }
        }

        if (i == 0) break;
    }
}
```

**Waarom van achter naar voor (painter's algorithm)?** Door eerst de achterste laag te tekenen en dan de bovenste lagen er overheen, hoef je geen complexe z-buffer te beheren. Een niet-transparante pixel van een hogere-prioriteit laag overschrijft simpelweg de vorige. Transparante pixels (index 0) worden overgeslagen, zodat de achterliggende laag zichtbaar blijft.

**Waarom DISPCNT bits 8–11 checken?** DISPCNT bits 8–11 zijn de "BG enable" flags. Een laag wordt alleen getekend als zijn enable-bit aan staat. Games schakelen lagen aan en uit voor verschillende schermen (bijv. alleen BG0 voor de titel, BG0+BG1+BG2 voor gameplay).

**Valkuil: prioriteit vs BG nummer.** Priority 0 is de *hoogste* prioriteit (tekent bovenop). Bij gelijke priority wint het *laagste* BG nummer (BG0 boven BG1). Dit is subtiel maar belangrijk — verwisseling hiervan geeft verkeerde laag-volgorde in complexe scenes.

### Stap 6: Testen met een tiled Mode 0 ROM

Test met een homebrew of commerciële ROM die Mode 0 gebruikt (de meeste GBA-games doen dit). Controleer:

1. **Tiles zijn herkenbaar:** Je zou duidelijke 8×8 patronen moeten zien die het beeld vormen
2. **Scrolling werkt:** Als de game scrollt, moet het beeld vloeiend meebewegen
3. **Meerdere lagen:** Als de game parallax scrolling heeft, moeten verschillende lagen op verschillende snelheden bewegen
4. **4bpp en 8bpp:** Test met ROMs die beide formaten gebruiken

```
Veelvoorkomende bugs:
- "Garbage tiles" (willekeurige patronen) → tile_base of map_base adres is verkeerd
- Tiles staan op de verkeerde plek → screen entry parsing is fout (check bit layout)
- Halve tiles verschoven → scroll offset wordt niet correct toegepast
- Kleuren per tile fout (maar sommige goed) → sub-palette selectie bij 4bpp is fout
- Lagen in verkeerde volgorde → priority sorting is omgekeerd
- Tiles gespiegeld die niet gespiegeld horen → h_flip/v_flip bits worden verkeerd gelezen
```

**Tip:** Gebruik een GBA debugger (bijv. mGBA's tile viewer) om te vergelijken wat de tiles *zouden* moeten zijn. Dit helpt enorm bij het debuggen van tile decoding.
