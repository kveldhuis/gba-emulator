# Story 5.5 — Background Mode 1 & 2 (affine/rotatie)

**Epic:** 05 - PPU (Graphics)

**Doel:** Achtergronden met rotatie en schaling.

## Zig concepten

### Waarom affine transformaties?

Affine achtergronden maken **rotatie, schaling en scheeftrekking** mogelijk. Dit is hoe klassieke "Mode 7"-achtige effecten werken (denk aan de vloer in Mario Kart of de wereldkaart in Final Fantasy). Zonder affine transformaties kun je alleen rechte, ongeroteerde achtergronden tonen.

De naam "affine" komt uit de wiskunde: een affine transformatie behoudt rechte lijnen en parallellisme. Rotatie, schaling, translatie en hun combinaties zijn allemaal affine. Wat *niet* kan: perspectief (dat vereist een projectieve transformatie). Games faken perspectief door de affine parameters per scanline te veranderen — een truc die verbluffend effectief is.

### Waarom fixed-point math?

De ARM7TDMI processor van de GBA heeft **geen floating-point unit** (FPU). Floating-point berekeningen in software zijn tientallen keren langzamer dan integer-berekeningen. Daarom gebruikt de GBA fixed-point wiskunde: integers die een decimale waarde voorstellen door een vast aantal bits als fractie te reserveren.

De GBA gebruikt 8.8 fixed-point voor affine parameters: 8 bits integer-deel, 8 bits fractie. De waarde 1.0 wordt opgeslagen als 256 (= 1 << 8). De waarde 0.5 = 128, 1.5 = 384, -1.0 = -256. Dit is exact genoeg voor smooth rotaties en schalingen, en kost slechts een shift-operatie (>> 8) om terug te converteren.

### Fixed-point wiskunde

Affine parameters zijn **fixed-point** getallen: integers die een decimale waarde voorstellen. De GBA gebruikt 8.8 fixed-point voor PA/PB/PC/PD (8 bits integer, 8 bits fractie):

```zig
// Fixed-point: de waarde 1.0 wordt opgeslagen als 256 (1 << 8)
// De waarde 0.5 = 128, de waarde 1.5 = 384
const pa: i16 = @bitCast(bus.read16(bg2pa_addr));

// Om een fixed-point vermenigvuldiging te doen:
const result = (@as(i32, pa) * @as(i32, x)) >> 8;
// >> 8 corrigeert voor het fixed-point formaat
```

### De affine transformatie

Voor elke pixel op het scherm bereken je welke pixel uit de tilemap je nodig hebt:

```
texture_x = PA * (screen_x - ref_x) + PB * (screen_y - ref_y) + ref_x
texture_y = PC * (screen_x - ref_x) + PD * (screen_y - ref_y) + ref_y
```

PA/PB/PC/PD vormen een 2×2 matrix. Door deze matrix aan te passen kun je roteren, schalen, en scheeftrekken.

```zig
// Per scanline: begin met referentiepunt, dan per pixel PA/PC optellen
var tex_x = ref_x;  // 28.4 fixed-point
var tex_y = ref_y;

for (0..240) |screen_x| {
    // Gebruik integer deel van tex_x/tex_y als tilemap coordinaten
    const map_x: u32 = @intCast(tex_x >> 8);
    const map_y: u32 = @intCast(tex_y >> 8);

    // Teken pixel...

    tex_x += pa;  // volgende pixel
    tex_y += pc;
}

// Na elke scanline: update referentiepunt
ref_x += pb;
ref_y += pd;
```

### Verschil affine tiles

Affine achtergronden gebruiken altijd 8bpp tiles en de tilemap entries zijn 8-bit (alleen tile index, geen flip/palette).

## Context

- Mode 1: BG0, BG1 = text mode (zie 5.4), BG2 = affine
- Mode 2: BG2, BG3 = beide affine
- Affine BG registers: PA, PB, PC, PD (rotatie/schaling matrix) + X, Y (referentiepunt)

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — affine registers lezen

Elke affine achtergrond (BG2 en BG3) heeft 6 registers: een 2×2 transformatiematrix (PA/PB/PC/PD) en een referentiepunt (X/Y). Het referentiepunt is 28.8 fixed-point (28 bits integer, 8 bits fractie) — nauwkeuriger dan de 8.8 matrix-parameters.

```zig
const AffineBgParams = struct {
    pa: i16,   // dx  — 8.8 fixed-point
    pb: i16,   // dmx — 8.8 fixed-point
    pc: i16,   // dy  — 8.8 fixed-point
    pd: i16,   // dmy — 8.8 fixed-point
    ref_x: i32, // 28.8 fixed-point (opgeslagen als 20.8 in registers, sign-extended)
    ref_y: i32, // 28.8 fixed-point
};

fn read_affine_bg_params(bus: *Bus, bg: u2) AffineBgParams {
    // BG2 parameters beginnen op 0x04000020, BG3 op 0x04000030
    const base: u32 = if (bg == 2) 0x0400_0020 else 0x0400_0030;

    return .{
        .pa = @bitCast(bus.read16(base + 0x00)),
        .pb = @bitCast(bus.read16(base + 0x02)),
        .pc = @bitCast(bus.read16(base + 0x04)),
        .pd = @bitCast(bus.read16(base + 0x06)),
        // Referentiepunt is 28-bit signed, opgeslagen in 32-bit register
        // Sign-extend van bit 27
        .ref_x = sign_extend_28(@bitCast(bus.read32(base + 0x08))),
        .ref_y = sign_extend_28(@bitCast(bus.read32(base + 0x0C))),
    };
}

fn sign_extend_28(val: i32) i32 {
    // Bit 27 is het sign bit — extend naar bit 31
    return (val << 4) >> 4;
}
```

**Waarom sign-extend?** Het referentiepunt is een 28-bit signed getal opgeslagen in een 32-bit register. De hardware negeert bits 28–31. Als bit 27 (het sign bit) gezet is, moet je de bovenste bits vullen met 1'en om een correct negatief getal te krijgen. De truc `(val << 4) >> 4` doet dit: eerst shift het sign bit naar bit 31, dan shift terug (arithmetic shift behoudt het sign).

**Waarom 28.8 voor het referentiepunt maar 8.8 voor de matrix?** Het referentiepunt moet een positie op het scherm (0–239) of in de tilemap (0–1023) met sub-pixel nauwkeurigheid kunnen beschrijven. 8 bits integer-deel is te weinig. De matrix-parameters beschrijven een *ratio* (hoeveel texture-pixels per scherm-pixel), waarvoor 8.8 voldoende is.

### Stap 2: `src/core/ppu.zig` — interne referentiepunt-tracking

**Belangrijk hardware-detail:** de referentiepunten (X/Y) worden door de hardware *intern* bijgewerkt na elke scanline. De game schrijft een startwaarde naar het register, en de PPU telt er elke scanline PB/PD bij op. Je moet dit intern bijhouden.

```zig
// In de Ppu struct:
internal_ref_x: [2]i32 = .{ 0, 0 },  // BG2 en BG3
internal_ref_y: [2]i32 = .{ 0, 0 },

// Bij het begin van VBlank (scanline 160→0 transitie):
fn on_vblank(self: *Ppu, bus: *Bus) void {
    // Herlaad interne referentiepunten vanuit registers
    const params2 = read_affine_bg_params(bus, 2);
    self.internal_ref_x[0] = params2.ref_x;
    self.internal_ref_y[0] = params2.ref_y;

    const params3 = read_affine_bg_params(bus, 3);
    self.internal_ref_x[1] = params3.ref_x;
    self.internal_ref_y[1] = params3.ref_y;
}

// Na elke zichtbare scanline (in end_scanline):
fn update_affine_refs(self: *Ppu, bus: *Bus) void {
    const params2 = read_affine_bg_params(bus, 2);
    self.internal_ref_x[0] += @as(i32, params2.pb);
    self.internal_ref_y[0] += @as(i32, params2.pd);

    const params3 = read_affine_bg_params(bus, 3);
    self.internal_ref_x[1] += @as(i32, params3.pb);
    self.internal_ref_y[1] += @as(i32, params3.pd);
}
```

**Waarom intern bijhouden?** Op de echte hardware wordt het referentiepunt-register niet teruggeschreven — de interne teller is apart. Als een game halverwege een frame het referentiepunt herschrijft (voor raster-effecten), moet je interne state resetten. Dit is een subtiel maar cruciaal detail voor correcte affine rendering.

**Valkuil:** Vergeet niet `update_affine_refs()` aan te roepen na elke zichtbare scanline (0–159), maar *niet* tijdens VBlank. En herlaad de referentiepunten bij de start van elk frame.

### Stap 3: `src/core/ppu.zig` — affine scanline renderer

De kernlogica: voor elke pixel op het scherm, bereken welke texture-coördinaat erbij hoort via de affine matrix. Gebruik de per-scanline optimalisatie (begin met referentiepunt, tel per pixel PA/PC op).

```zig
fn render_affine_bg(self: *Ppu, bus: *Bus, bg: u2, y: u8) void {
    const bgcnt = read_bgcnt(bus, bg);
    const params = read_affine_bg_params(bus, bg);
    const bg_idx: usize = if (bg == 2) 0 else 1;

    const tile_base: u32 = 0x0600_0000 + @as(u32, bgcnt.tile_base) * 0x4000;
    const map_base: u32 = 0x0600_0000 + @as(u32, bgcnt.map_base) * 0x800;

    // Map-grootte in tiles (affine BG: 16, 32, 64, of 128 tiles breed/hoog)
    const map_size: u32 = @as(u32, 16) << @as(u5, bgcnt.screen_size);
    const map_pixels: u32 = map_size * 8;  // in pixels

    // Start-positie voor deze scanline (van intern referentiepunt)
    var tex_x: i32 = self.internal_ref_x[bg_idx];
    var tex_y: i32 = self.internal_ref_y[bg_idx];

    for (0..240) |screen_x| {
        defer {
            tex_x += @as(i32, params.pa);
            tex_y += @as(i32, params.pc);
        }

        // Converteer fixed-point naar pixel-coördinaten
        const px = tex_x >> 8;
        const py = tex_y >> 8;

        // Buiten de map? Check wrap/clamp
        if (bgcnt.overflow_wrap == 1) {
            // Wrap: modulo map-grootte
            // (gebruik & (map_pixels-1) als map_pixels een macht van 2 is)
        } else {
            // Clamp: buiten de map = transparant
            if (px < 0 or px >= map_pixels or py < 0 or py >= map_pixels) continue;
        }

        const upx: u32 = @intCast(@mod(px, @as(i32, @intCast(map_pixels))));
        const upy: u32 = @intCast(@mod(py, @as(i32, @intCast(map_pixels))));

        // Affine tilemaps zijn 8-bit entries (alleen tile index, geen flip/palette)
        const tile_col = upx / 8;
        const tile_row = upy / 8;
        const tile_id: u32 = bus.read8(map_base + tile_row * map_size + tile_col);

        // Affine tiles zijn altijd 8bpp
        const pixel_x: u32 = upx & 7;
        const pixel_y: u32 = upy & 7;
        const index: u32 = bus.read8(tile_base + tile_id * 64 + pixel_y * 8 + pixel_x);

        if (index != 0) {
            const color = bus.read16(0x0500_0000 + index * 2);
            self.bg_line[bg][@intCast(screen_x)] = color;
            self.bg_opaque[bg][@intCast(screen_x)] = true;
        }
    }
}
```

**Waarom `defer` voor tex_x/tex_y update?** De `defer` zorgt dat de texture-coördinaten *altijd* worden bijgewerkt, ook als je `continue` gebruikt voor out-of-bounds pixels. Zonder `defer` zou je de update in de `continue` branch moeten dupliceren — foutgevoelig.

**Waarom altijd 8bpp bij affine?** De hardware-ontwerpers hebben de tilemap entries voor affine BGs versimpeld tot 8 bits (alleen tile index). Er is geen ruimte voor palette-bits, flip-bits etc. Dit beperkt je tot 256 tiles en 8bpp kleurmodus, maar vereenvoudigt de hardware.

### Stap 4: `src/core/ppu.zig` — wrap vs clamp bij out-of-bounds

Wanneer de affine transformatie een pixel buiten de tilemap plaatst, zijn er twee opties:

```zig
// BgCnt bit 13 (voor affine BGs): overflow/wraparound control
// 0 = transparant (clamp): pixels buiten de map worden niet getekend
// 1 = wraparound: de map herhaalt zich oneindig

// Clamp voorbeeld: een gedraaide achtergrond met een transparante rand eromheen
// Wrap voorbeeld: een oneindig herhalend patroon (bijv. een oceaan of vloer)
```

**Waarom is dit belangrijk?** Veel "Mode 7"-effecten (bijv. de grond in Mario Kart) draaien en schalen de achtergrond zo sterk dat grote delen buiten de originele map vallen. Met wrap herhaalt het patroon zich, wat een illusie van een oneindige vloer geeft. Met clamp zie je de achtergrond abrupt stoppen — gebruikt voor effecten waar dat gewenst is (bijv. een draaiende planeet).

### Stap 5: `src/core/ppu.zig` — Mode 1 en Mode 2 dispatch

Nu de affine renderer klaar is, integreer je hem in de mode-switch:

```zig
pub fn render_scanline(self: *Ppu, bus: *Bus) void {
    const y = self.scanline;
    if (y >= 160) return;

    const dispcnt = bus.read16(0x04000000);
    const mode: u3 = @truncate(dispcnt & 0x7);

    switch (mode) {
        0 => {
            // Alle 4 lagen zijn text-mode (story 5.4)
            self.render_text_bgs(bus, @truncate(y));
        },
        1 => {
            // BG0, BG1 = text mode; BG2 = affine
            self.render_text_bg(bus, 0, @truncate(y));
            self.render_text_bg(bus, 1, @truncate(y));
            self.render_affine_bg(bus, 2, @truncate(y));
            // BG3 is niet beschikbaar in Mode 1
        },
        2 => {
            // BG2 en BG3 zijn beide affine
            self.render_affine_bg(bus, 2, @truncate(y));
            self.render_affine_bg(bus, 3, @truncate(y));
            // BG0 en BG1 zijn niet beschikbaar in Mode 2
        },
        3 => self.render_mode3(bus, @truncate(y)),
        4 => self.render_mode4(bus, @truncate(y)),
        else => {},
    }

    // Compositing: combineer alle actieve lagen op prioriteit
    self.composite_scanline(bus, @truncate(y));
}
```

**Waarom is Mode 1 de meest gebruikte mode?** Mode 1 combineert het beste van twee werelden: 2 text-mode lagen voor de UI en level-layout (snelle, efficiënte tiles), plus 1 affine laag voor speciale effecten (draaiende achtergronden, zoom-effecten). De meeste commerciële GBA-games gebruiken Mode 1.

**Waarom heeft Mode 2 geen text-lagen?** Mode 2 biedt 2 affine lagen maar geen text-lagen. Dit is een hardware-beperking: de PPU heeft beperkte bandbreedte. Affine rendering vereist meer VRAM-reads per pixel (matrix-berekening + tile lookup), dus er is minder bandbreedte over voor extra lagen.

### Stap 6: Testen met affine effecten

Test met ROMs die affine achtergronden gebruiken. Goede test-cases:

1. **Rotatie:** Een achtergrond die draait. Check dat het beeld smooth roteert zonder visuele artefacten.
2. **Schaling:** In- en uitzoomen. Check dat de proportions correct blijven.
3. **Combinatie:** Rotatie + schaling tegelijk (de klassieke "Mode 7" vloer). Dit is de strengste test.
4. **Wrapping:** Een achtergrond die buiten de map valt en wrapt.

```
Veelvoorkomende bugs:
- Beeld trilt/jittert → fixed-point berekening verliest precisie (check bit-shifts)
- Beeld staat stil maar zou moeten roteren → interne referentiepunten worden niet bijgewerkt per scanline
- Beeld is gespiegeld/omgekeerd → PA/PB/PC/PD volgorde is verwisseld
- Beeld verschijnt op verkeerde plek → referentiepunt sign-extension is fout
- Zwart beeld maar text-lagen werken → affine renderer wordt niet aangeroepen (check mode switch)
```

**Tip:** Test eerst met een identiteitsmatrix (PA=256, PB=0, PC=0, PD=256, refX=0, refY=0). Dit zou exact hetzelfde beeld moeten geven als een ongeroteerde text-mode achtergrond. Als dit al fout is, zit het probleem in de basis-rendering, niet in de matrix-wiskunde.
