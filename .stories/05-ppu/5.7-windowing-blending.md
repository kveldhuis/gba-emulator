# Story 5.7 — Windowing en speciale effecten

**Epic:** 05 - PPU (Graphics)

**Doel:** Window masking en alpha blending.

## Zig concepten

### Waarom windowing?

Windows zijn rechthoekige gebieden op het scherm die bepalen welke lagen zichtbaar zijn. Dit klinkt simpel, maar is enorm krachtig:
- **UI-elementen**: Een statusbalk bovenaan die altijd BG0 toont, terwijl de rest van het scherm BG1+BG2 toont voor de gameplay
- **Spotlight-effecten**: Een circulair window (gesimuleerd door per-scanline de grenzen aan te passen) dat alleen een klein gebied verlicht
- **Split-screen**: Twee spelers elk hun eigen deel van het scherm

Zonder windowing zou je deze effecten moeten simuleren door lagen creatief te misbruiken, wat veel VRAM en processing kost. Windows zijn een hardware-versnelling voor masking.

### Waarom alpha blending?

Alpha blending mengt twee kleuren om transparantie-effecten te creëren:
- **Water**: Een semi-transparante blauwe laag over de achtergrond
- **Schaduwen**: Een donkere semi-transparante sprite onder een personage
- **Fade-effecten**: Geleidelijk naar zwart of wit faden bij schermovergangen
- **Rook/mist**: Overlappende semi-transparante sprites

De GBA biedt hardware-versnelde blending met instelbare coëfficiënten. Zonder dit zou je blending in software moeten doen, per pixel, per frame — te langzaam voor 60fps op de ARM7TDMI.

### Windows — gebieden op het scherm

De GBA heeft 2 rechthoekige windows (WIN0, WIN1) en een OBJ Window. Elk window bepaalt welke lagen (BG0-3, OBJ) zichtbaar zijn in dat gebied.

Window dimensies worden ingesteld met registers:

```zig
// WIN0H (0x04000040) — horizontale grenzen van Window 0
// Bits [15:8] = X1 (links), Bits [7:0] = X2 (rechts)
const win0h = bus.read16(0x04000040);
const x1: u8 = @truncate(win0h >> 8);   // linkerrand
const x2: u8 = @truncate(win0h);         // rechterrand

// WIN0V (0x04000044) — verticale grenzen van Window 0
const win0v = bus.read16(0x04000044);
const y1: u8 = @truncate(win0v >> 8);   // bovenrand
const y2: u8 = @truncate(win0v);         // onderrand
```

### Window control — welke lagen per window

WININ (0x04000048) en WINOUT (0x0400004A) bepalen welke lagen in/buiten de windows zichtbaar zijn:

```zig
const WinControl = packed struct(u8) {
    bg0: u1,       // BG0 zichtbaar in dit window?
    bg1: u1,
    bg2: u1,
    bg3: u1,
    obj: u1,       // sprites zichtbaar?
    effects: u1,   // color effects actief?
    _unused: u2,
};

// WININ bevat 2 window controls (WIN0 in lage byte, WIN1 in hoge byte)
const winin = bus.read16(0x04000048);
const win0_ctrl: WinControl = @bitCast(@as(u8, @truncate(winin)));
const win1_ctrl: WinControl = @bitCast(@as(u8, @truncate(winin >> 8)));
```

### Per-pixel window check

Bij het renderen van elke pixel moet je checken: in welk window zit deze pixel?

```zig
fn get_window_control(self: *Ppu, x: u16, y: u16) WinControl {
    // Check WIN0 eerst (hogere prioriteit)
    if (self.win0_enabled and x >= self.win0_x1 and x < self.win0_x2
        and y >= self.win0_y1 and y < self.win0_y2) {
        return self.win0_ctrl;
    }
    // Dan WIN1
    if (self.win1_enabled and x >= self.win1_x1 and x < self.win1_x2
        and y >= self.win1_y1 and y < self.win1_y2) {
        return self.win1_ctrl;
    }
    // Buiten alle windows
    return self.winout_ctrl;
}
```

### Alpha blending — kleuren mengen

BLDCNT (0x04000050) bepaalt welk blend effect actief is:

const BldCnt = packed struct(u16) {

### Alpha blend berekening

Bij alpha blending (mode 1) meng je twee kleuren:

```zig
// BLDALPHA (0x04000052): blend coefficiënten
const bldalpha = bus.read16(0x04000052);
const eva: u8 = @truncate(bldalpha & 0x1F);       // coëfficiënt voor first target (0-16)
const evb: u8 = @truncate((bldalpha >> 8) & 0x1F); // coëfficiënt voor second target (0-16)

// Blend twee RGB555 kleuren:
fn alpha_blend(color1: u16, color2: u16, eva: u8, evb: u8) u16 {
    const r1: u16 = color1 & 0x1F;
    const g1: u16 = (color1 >> 5) & 0x1F;
    const b1: u16 = (color1 >> 10) & 0x1F;

    const r2: u16 = color2 & 0x1F;
    const g2: u16 = (color2 >> 5) & 0x1F;
    const b2: u16 = (color2 >> 10) & 0x1F;

    // Formule: result = min(31, (color1 * EVA + color2 * EVB) / 16)
    const r = @min(31, (r1 * eva + r2 * evb) >> 4);
    const g = @min(31, (g1 * eva + g2 * evb) >> 4);
    const b = @min(31, (b1 * eva + b2 * evb) >> 4);

    return r | (g << 5) | (b << 10);
}
```

### Brightness veranderen

Bij mode 2 (brightness increase) en mode 3 (brightness decrease):

```zig
// BLDY (0x04000054): brightness factor (0-16)
const evy: u8 = @truncate(bus.read16(0x04000054) & 0x1F);

// Brightness increase: color + (31 - color) * EVY / 16
// Brightness decrease: color - color * EVY / 16
fn brightness_increase(color: u16, evy: u8) u16 {
    const r: u16 = color & 0x1F;
    const g: u16 = (color >> 5) & 0x1F;
    const b: u16 = (color >> 10) & 0x1F;

    const new_r = r + ((31 - r) * evy >> 4);
    const new_g = g + ((31 - g) * evy >> 4);
    const new_b = b + ((31 - b) * evy >> 4);

    return new_r | (new_g << 5) | (new_b << 10);
}
```

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — window registers lezen

Begin met het lezen van de window-dimensies en control registers. De GBA heeft twee rechthoekige windows (WIN0, WIN1) en een OBJ window (gedefinieerd door sprite-pixels).

```zig
const WindowState = struct {
    win0_enabled: bool,
    win1_enabled: bool,
    obj_win_enabled: bool,
    // WIN0 grenzen
    win0_x1: u8, win0_x2: u8,
    win0_y1: u8, win0_y2: u8,
    // WIN1 grenzen
    win1_x1: u8, win1_x2: u8,
    win1_y1: u8, win1_y2: u8,
    // Per-window layer controls
    win0_ctrl: WinControl,
    win1_ctrl: WinControl,
    obj_ctrl: WinControl,
    out_ctrl: WinControl,  // voor pixels buiten alle windows
};

fn read_window_state(bus: *Bus) WindowState {
    const dispcnt = bus.read16(0x0400_0000);

    // DISPCNT bits 13-15 enablen de windows
    const win0_en = (dispcnt >> 13) & 1 == 1;
    const win1_en = (dispcnt >> 14) & 1 == 1;
    const obj_win = (dispcnt >> 15) & 1 == 1;

    // Window dimensies
    const win0h = bus.read16(0x0400_0040);
    const win1h = bus.read16(0x0400_0042);
    const win0v = bus.read16(0x0400_0044);
    const win1v = bus.read16(0x0400_0046);

    // Window controls
    const winin = bus.read16(0x0400_0048);
    const winout = bus.read16(0x0400_004A);

    return .{
        .win0_enabled = win0_en,
        .win1_enabled = win1_en,
        .obj_win_enabled = obj_win,
        .win0_x1 = @truncate(win0h >> 8),
        .win0_x2 = @truncate(win0h),
        .win0_y1 = @truncate(win0v >> 8),
        .win0_y2 = @truncate(win0v),
        .win1_x1 = @truncate(win1h >> 8),
        .win1_x2 = @truncate(win1h),
        .win1_y1 = @truncate(win1v >> 8),
        .win1_y2 = @truncate(win1v),
        .win0_ctrl = @bitCast(@as(u8, @truncate(winin))),
        .win1_ctrl = @bitCast(@as(u8, @truncate(winin >> 8))),
        .obj_ctrl = @bitCast(@as(u8, @truncate(winout >> 8))),
        .out_ctrl = @bitCast(@as(u8, @truncate(winout))),
    };
}
```

**Waarom alle state in één struct lezen?** Window-evaluatie moet per pixel gebeuren. Door alles in één struct te cachen, vermijd je 240 keer per scanline de bus te lezen. De registers veranderen niet tijdens een scanline (op de echte hardware), dus één keer lezen per scanline is correct.

**Valkuil: x1 > x2 of y1 > y2.** Als de rechterrand links van de linkerrand staat (x2 < x1), wrapt het window: het is actief van x1 tot het einde van de scanline, EN van het begin tot x2. Dezelfde logica geldt verticaal. Vergeet dit niet, anders werken sommige games niet correct.

### Stap 2: `src/core/ppu.zig` — per-pixel window evaluatie

Voor elke pixel moet je bepalen in welk window die valt, en welke lagen zichtbaar zijn:

```zig
fn evaluate_window(self: *Ppu, ws: *const WindowState, x: u16, y: u16) WinControl {
    // Als geen enkel window actief is, zijn alle lagen zichtbaar
    if (!ws.win0_enabled and !ws.win1_enabled and !ws.obj_win_enabled) {
        return WinControl{ .bg0 = 1, .bg1 = 1, .bg2 = 1, .bg3 = 1, .obj = 1, .effects = 1, ._unused = 0 };
    }

    // WIN0 heeft hoogste prioriteit
    if (ws.win0_enabled and in_window(x, y, ws.win0_x1, ws.win0_x2, ws.win0_y1, ws.win0_y2)) {
        return ws.win0_ctrl;
    }

    // Dan WIN1
    if (ws.win1_enabled and in_window(x, y, ws.win1_x1, ws.win1_x2, ws.win1_y1, ws.win1_y2)) {
        return ws.win1_ctrl;
    }

    // Dan OBJ window (gedefinieerd door sprite-pixels die als "obj window" gemarkeerd zijn)
    if (ws.obj_win_enabled and self.obj_window_line[@intCast(x)]) {
        return ws.obj_ctrl;
    }

    // Buiten alle windows
    return ws.out_ctrl;
}

fn in_window(x: u16, y: u16, x1: u8, x2: u8, y1: u8, y2: u8) bool {
    // Horizontale check met wrap-around
    const h_in = if (x1 <= x2)
        x >= x1 and x < x2
    else
        x >= x1 or x < x2;  // wrap

    // Verticale check met wrap-around
    const v_in = if (y1 <= y2)
        y >= y1 and y < y2
    else
        y >= y1 or y < y2;  // wrap

    return h_in and v_in;
}
```

**Waarom prioriteit WIN0 > WIN1 > OBJ > WINOUT?** Dit is de hardware-volgorde. Als meerdere windows overlappen, wint WIN0 altijd. Deze prioriteit is niet configureerbaar. Games ontwerpen hun windows met deze volgorde in gedachten.

**Waarom `x < x2` (strict less than)?** Het window omvat pixels van x1 *tot* x2, maar x2 zelf valt er net buiten. Dit is consistent met hoe ranges werken in de meeste systemen (half-open interval). Een window van x1=0 tot x2=240 omvat het hele scherm.

### Stap 3: `src/core/ppu.zig` — window-aware rendering integreren

Pas de compositing aan om window-controls te respecteren:

```zig
fn composite_scanline_with_windows(self: *Ppu, bus: *Bus, y: u8) void {
    const ws = read_window_state(bus);
    const y32: u32 = @as(u32, y);
    const backdrop = bus.read16(0x0500_0000);

    for (0..240) |x| {
        const win = self.evaluate_window(&ws, @intCast(x), @as(u16, y));

        var best_color: u16 = backdrop;
        var best_priority: u3 = 7;

        // Check BG lagen — alleen als het window ze toelaat
        for (0..4) |bg| {
            // Check of deze BG zichtbaar is in het huidige window
            const bg_visible = switch (bg) {
                0 => win.bg0 == 1,
                1 => win.bg1 == 1,
                2 => win.bg2 == 1,
                3 => win.bg3 == 1,
                else => false,
            };
            if (!bg_visible) continue;
            if (!self.bg_opaque[bg][x]) continue;

            const bg_priority = self.bg_priority[bg];
            if (bg_priority < best_priority) {
                best_priority = bg_priority;
                best_color = self.bg_line[bg][x];
            }
        }

        // Check sprites — alleen als window OBJ toelaat
        if (win.obj == 1 and self.sprite_line[x] != 0) {
            const sprite_pri: u3 = self.sprite_priority[x];
            if (sprite_pri <= best_priority) {
                best_color = self.sprite_line[x];
            }
        }

        self.framebuffer[y32 * 240 + x] = best_color;
    }
}
```

**Waarom per-pixel window evaluatie?** Windowing is per pixel. Een pixel in de linkerhelft van het scherm kan een heel ander set lagen tonen dan een pixel in de rechterhelft. Je kunt dit niet vereenvoudigen tot "per laag" of "per scanline" — het moet echt per pixel.

### Stap 4: `src/core/ppu.zig` — BLDCNT en blend-mode selectie

De GBA ondersteunt drie speciale effecten: alpha blending, brightness increase, en brightness decrease. Het BLDCNT register bepaalt welk effect actief is en welke lagen als "first target" en "second target" fungeren.

```zig
const BlendMode = enum(u2) {
    none = 0,
    alpha = 1,
    brightness_increase = 2,
    brightness_decrease = 3,
};

fn get_blend_mode(bus: *Bus) BlendMode {
    const bldcnt = bus.read16(0x0400_0050);
    return @enumFromInt(@as(u2, @truncate((bldcnt >> 6) & 3)));
}

fn is_first_target(bldcnt: u16, layer: u4) bool {
    return (bldcnt >> layer) & 1 == 1;
}

fn is_second_target(bldcnt: u16, layer: u4) bool {
    return (bldcnt >> (8 + layer)) & 1 == 1;
}
// layer: 0=BG0, 1=BG1, 2=BG2, 3=BG3, 4=OBJ, 5=backdrop
```

**Waarom "first target" en "second target"?** Alpha blending mengt twee kleuren: de *first target* is de bovenste zichtbare pixel, de *second target* is de pixel direct eronder. Je moet bijhouden welke laag de bovenste pixel levert en welke de op-één-na-bovenste, om te bepalen of blending van toepassing is.

**Valkuil:** Blending vindt alleen plaats als (1) het effect is ingeschakeld in BLDCNT, (2) de bovenste pixel een "first target" is, en (3) de tweede pixel een "second target" is. Als één van deze voorwaarden niet klopt, wordt de bovenste pixel ongeblend getekend.

### Stap 5: `src/core/ppu.zig` — alpha blend implementatie

Bij alpha blending meng je twee RGB555 kleuren met instelbare coëfficiënten:

```zig
fn apply_alpha_blend(color1: u16, color2: u16, bus: *Bus) u16 {
    // BLDALPHA (0x04000052): blend coëfficiënten
    const bldalpha = bus.read16(0x0400_0052);
    const eva: u16 = @min(16, bldalpha & 0x1F);       // first target coëfficiënt (0-16)
    const evb: u16 = @min(16, (bldalpha >> 8) & 0x1F); // second target coëfficiënt (0-16)

    // Splits beide kleuren in RGB componenten
    const r1: u16 = color1 & 0x1F;
    const g1: u16 = (color1 >> 5) & 0x1F;
    const b1: u16 = (color1 >> 10) & 0x1F;

    const r2: u16 = color2 & 0x1F;
    const g2: u16 = (color2 >> 5) & 0x1F;
    const b2: u16 = (color2 >> 10) & 0x1F;

    // Formule: result = min(31, (color1 × EVA + color2 × EVB) / 16)
    const r = @min(@as(u16, 31), (r1 * eva + r2 * evb) >> 4);
    const g = @min(@as(u16, 31), (g1 * eva + g2 * evb) >> 4);
    const b = @min(@as(u16, 31), (b1 * eva + b2 * evb) >> 4);

    return r | (g << 5) | (b << 10);
}
```

**Waarom `@min(16, ...)`?** De coëfficiënten zijn 5 bits (0–31), maar waarden boven 16 worden door de hardware behandeld als 16. `EVA=16, EVB=0` betekent 100% van de eerste kleur, `EVA=8, EVB=8` is 50/50 mix, `EVA=0, EVB=16` is 100% tweede kleur.

**Waarom `>> 4` (delen door 16)?** De coëfficiënten lopen van 0 tot 16 (niet 0 tot 1). Een coëfficiënt van 16 = 100%. De formule `(color × coef) / 16` is de integer-versie van `color × (coef/16)`. De shift `>> 4` is efficiënter dan een deling.

**Waarom `@min(31, ...)`?** Als EVA + EVB > 16 (bijv. EVA=12, EVB=12), kan het resultaat boven 31 uitkomen. Clamping naar 31 voorkomt overflow — de kleurcomponent kan niet meer dan 5 bits zijn.

### Stap 6: `src/core/ppu.zig` — brightness increase/decrease

De andere twee blend-modes veranderen de helderheid van een enkel pixel:

```zig
fn apply_brightness_increase(color: u16, bus: *Bus) u16 {
    const evy: u16 = @min(16, bus.read16(0x0400_0054) & 0x1F);

    const r: u16 = color & 0x1F;
    const g: u16 = (color >> 5) & 0x1F;
    const b: u16 = (color >> 10) & 0x1F;

    // Formule: result = color + (31 - color) × EVY / 16
    // Bij EVY=16: result = color + (31-color) = 31 (= wit)
    // Bij EVY=0: result = color (= onveranderd)
    const new_r = r + ((31 - r) * evy >> 4);
    const new_g = g + ((31 - g) * evy >> 4);
    const new_b = b + ((31 - b) * evy >> 4);

    return new_r | (new_g << 5) | (new_b << 10);
}

fn apply_brightness_decrease(color: u16, bus: *Bus) u16 {
    const evy: u16 = @min(16, bus.read16(0x0400_0054) & 0x1F);

    const r: u16 = color & 0x1F;
    const g: u16 = (color >> 5) & 0x1F;
    const b: u16 = (color >> 10) & 0x1F;

    // Formule: result = color - color × EVY / 16
    // Bij EVY=16: result = 0 (= zwart)
    // Bij EVY=0: result = color (= onveranderd)
    const new_r = r - (r * evy >> 4);
    const new_g = g - (g * evy >> 4);
    const new_b = b - (b * evy >> 4);

    return new_r | (new_g << 5) | (new_b << 10);
}
```

**Waarom twee aparte functies?** De formules zijn subtiel anders: increase *voegt* kleur toe (richting wit), decrease *haalt* kleur weg (richting zwart). Ze delen dezelfde EVY parameter maar het resultaat is tegengesteld. Twee functies zijn duidelijker dan één functie met een direction-flag.

**Waarom worden deze gebruikt?** Brightness effects zijn essentieel voor fade-in/fade-out schermovergangen. Bijna elke GBA-game fade naar zwart bij schermwissels. Games animeren dit door EVY geleidelijk van 0 naar 16 te veranderen, elk frame een stapje.

### Stap 7: `src/core/ppu.zig` — blend-effecten integreren in compositing

Pas de compositing-pipeline aan om blending toe te passen op het eindresultaat:

```zig
fn composite_final(self: *Ppu, bus: *Bus, y: u8) void {
    const bldcnt = bus.read16(0x0400_0050);
    const blend_mode = get_blend_mode(bus);
    const ws = read_window_state(bus);
    const y32: u32 = @as(u32, y);

    for (0..240) |x| {
        const win = self.evaluate_window(&ws, @intCast(x), @as(u16, y));

        // Vind de bovenste twee niet-transparante pixels en hun laag-ID
        const top = self.find_top_pixel(x);
        const bottom = self.find_second_pixel(x, top.layer);

        var final_color = top.color;

        // Blending alleen als window het toelaat (effects bit)
        if (win.effects == 1) {
            switch (blend_mode) {
                .alpha => {
                    if (is_first_target(bldcnt, top.layer) and
                        is_second_target(bldcnt, bottom.layer))
                    {
                        final_color = apply_alpha_blend(top.color, bottom.color, bus);
                    }
                },
                .brightness_increase => {
                    if (is_first_target(bldcnt, top.layer)) {
                        final_color = apply_brightness_increase(top.color, bus);
                    }
                },
                .brightness_decrease => {
                    if (is_first_target(bldcnt, top.layer)) {
                        final_color = apply_brightness_decrease(top.color, bus);
                    }
                },
                .none => {},
            }
        }

        self.framebuffer[y32 * 240 + x] = final_color;
    }
}
```

**Waarom de "second pixel" bijhouden?** Alpha blending mengt de bovenste pixel met de pixel direct eronder. Zonder de tweede pixel te kennen, kun je niet blenden. Dit is waarom de compositing-pipeline twee "winnaars" moet bijhouden, niet één.

**Waarom het `effects` bit in het window checken?** Elk window heeft een `effects` bit (bit 5 in WinControl). Als dit bit uit staat, worden geen speciale effecten toegepast in dat window-gebied — zelfs niet als BLDCNT anders zegt. Dit geeft games fijne controle over waar blending actief is.

**Valkuil: semi-transparante sprites.** Sprites met `obj_mode == 1` (semi-transparent) forceren alpha blending, ongeacht BLDCNT. Dit is een speciale hardware-modus: de sprite wordt altijd geblend met de achtergrond. Vergeet dit niet!

### Stap 8: Testen met windowing en blending effecten

```
Veelvoorkomende bugs bij windowing:
- Lagen verdwijnen volledig → window control bits zijn verkeerd (alle bits 0)
- Windows werken niet → DISPCNT window enable bits (13-15) worden niet gelezen
- Window grenzen verkeerd → x1/x2 of y1/y2 zijn verwisseld in het register
- Window wrap werkt niet → de x1>x2 case wordt niet afgehandeld

Veelvoorkomende bugs bij blending:
- Geen zichtbaar effect → blend mode is 0 (none) of geen first/second targets geselecteerd
- Kleuren te helder/donker → EVA/EVB waarden worden niet correct gelezen of geclamped
- Fade werkt per frame maar niet smooth → EVY wordt niet elke frame bijgewerkt door de game (check of de game's VBlank IRQ handler draait)
- Semi-transparante sprites werken niet → obj_mode==1 check ontbreekt
```

**Tip:** Test brightness decrease met EVY=16 — het hele scherm zou zwart moeten zijn. Test brightness increase met EVY=16 — het hele scherm zou wit moeten zijn. Dit zijn extreme gevallen die makkelijk visueel te verifiëren zijn.
