# Story 5.6 — Sprites (OBJ rendering)

**Epic:** 05 - PPU (Graphics)

**Doel:** Sprites tekenen vanuit OAM (Object Attribute Memory).

## Zig concepten

### Waarom sprites apart van backgrounds?

Backgrounds zijn grote, statische lagen die het level vormen. Maar een personage, vijanden, kogels — die moeten **onafhankelijk** kunnen bewegen. Sprites (in GBA-terminologie "OBJ" = objects) zijn precies hiervoor ontworpen:
- Ze hebben hun eigen positie (x, y) die onafhankelijk van scroll-registers verandert
- Ze worden vanuit apart geheugen (OAM + VRAM) getekend
- Ze hebben eigen prioriteit-levels voor z-ordering (wat staat voor/achter wat)
- Je kunt er 128 tegelijk op het scherm hebben

Op de echte hardware worden sprites door een apart stuk logica in de PPU afgehandeld, parallel aan de achtergrondlagen. Dit is efficiënter dan alles in dezelfde rendering-pipeline te proppen.

### Waarom OAM apart geheugen is

OAM (Object Attribute Memory, 0x07000000) is een klein geheugengebied van 1KB, fysiek gescheiden van VRAM en Palette RAM. Dit is een bewuste hardware-keuze:
- **Snelheid**: OAM heeft eigen bus-connectie, zodat de PPU sprite-attributen kan lezen zonder VRAM-toegang te blokkeren
- **Timing**: OAM kan alleen gelezen/geschreven worden tijdens HBlank en VBlank (op de echte hardware). Tijdens HDraw heeft de PPU exclusieve toegang
- **Structuur**: 128 entries van 8 bytes = exact 1024 bytes. Klein genoeg voor dedicated SRAM.

### Waarom een prioriteit systeem?

Sprites moeten samenwerken met achtergrondlagen. Een personage loopt *achter* een boom (achtergrond) maar *voor* de lucht (andere achtergrond). Het priority-veld (0–3) bepaalt waar een sprite tekent ten opzichte van BG-lagen met dezelfde priority-waarde. Priority 0 = meest naar voren.

Bij sprites met dezelfde priority wint de sprite met het **laagste OAM index**. Sprite 0 is altijd bovenop sprite 1, etc. Games plaatsen het belangrijkste object (speler) daarom vaak in de eerste OAM slots.

### OAM entries parsen met packed structs

Elke sprite heeft 3 attributen (elk 16-bit) + 1 rotatie parameter:

```zig
const OamAttr0 = packed struct(u16) {
    y: u8,                // Y positie op scherm
    obj_mode: u2,         // 0=normal, 1=semi-transparent, 2=obj window, 3=forbidden
    gfx_mode: u1,         // 0=normal, 1=affine
    mosaic: u1,
    color_mode: u1,       // 0=4bpp, 1=8bpp
    shape: u2,            // 0=square, 1=horizontal, 2=vertical
    _unused: u1,
};

const OamAttr1 = packed struct(u16) {
    x: u9,                // X positie op scherm
    _unused: u3,
    h_flip: u1,           // (alleen bij niet-affine)
    v_flip: u1,           // (alleen bij niet-affine)
    size: u2,             // samen met shape bepaalt dit de sprite-grootte
};

const OamAttr2 = packed struct(u16) {
    tile_id: u10,         // welke tile(s)
    priority: u2,         // 0-3, relatief t.o.v. BG lagen
    palette: u4,          // sub-palette (bij 4bpp)
};
```

### Sprite grootte tabel

Shape + Size samen bepalen de sprite-dimensies:

```zig
const sprite_sizes = [3][4][2]u8{
    // Square        Horizontal    Vertical
    .{ .{8,8},   .{16,8},   .{8,16}  },   // size 0
    .{ .{16,16}, .{32,8},   .{8,32}  },   // size 1
    .{ .{32,32}, .{32,16},  .{16,32} },   // size 2
    .{ .{64,64}, .{64,32},  .{32,64} },   // size 3
};
```

### Sprites en prioriteit

Sprites hebben een priority veld (0-3) dat bepaalt waar ze ten opzichte van BG lagen tekenen. Bovendien: bij sprites met dezelfde priority wint de sprite met het laagste OAM index (sprite 0 is bovenop).

### Sprite rendering per scanline

```zig
fn render_sprites(self: *Ppu, bus: *Bus, y: u8) void {
    // Loop door alle 128 sprites (omgekeerd: hogere index = lagere prioriteit)
    var i: u8 = 127;
    while (true) : (i -= 1) {
        const attr0: OamAttr0 = @bitCast(bus.read16(0x07000000 + @as(u32, i) * 8));
        const attr1: OamAttr1 = @bitCast(bus.read16(0x07000000 + @as(u32, i) * 8 + 2));
        const attr2: OamAttr2 = @bitCast(bus.read16(0x07000000 + @as(u32, i) * 8 + 4));

        // Check of sprite op deze scanline zichtbaar is
        // Teken zichtbare pixels...

        if (i == 0) break;
    }
}
```

## Context

- OAM op 0x07000000, 128 entries van 8 bytes
- Sprites kunnen 4bpp of 8bpp zijn
- Affine sprites gebruiken een rotatie/schaling matrix uit OAM
- Sprite tile data staat in VRAM 0x06010000 (in bitmap modes) of 0x06010000 (in tile modes)

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — OAM entries parsen

Begin met het lezen en decoderen van de 128 sprite-entries uit OAM. Elke entry is 8 bytes (3 attributen van 16 bits + 2 bytes padding/affine data).

```zig
const Sprite = struct {
    attr0: OamAttr0,
    attr1: OamAttr1,
    attr2: OamAttr2,
};

fn read_oam_entry(bus: *Bus, index: u7) Sprite {
    const base: u32 = 0x0700_0000 + @as(u32, index) * 8;
    return .{
        .attr0 = @bitCast(bus.read16(base + 0)),
        .attr1 = @bitCast(bus.read16(base + 2)),
        .attr2 = @bitCast(bus.read16(base + 4)),
        // byte 6-7: affine parameter (zie story 5.8) of ongebruikt
    };
}
```

**Waarom `u7` voor de index?** Er zijn precies 128 sprites (0–127), wat past in 7 bits. Door `u7` te gebruiken, voorkomt Zig compile-time dat je per ongeluk een te hoog index doorgeeft. Type-veiligheid als documentatie.

**Valkuil:** De Y-positie in attr0 is een `u8` (0–255), maar het scherm is 160 pixels hoog. Y-waarden boven 160 betekenen dat de sprite "boven" het scherm begint (wrap-around). Een sprite met Y=200 verschijnt als Y=200-256=-56, dus 56 pixels boven het scherm. Dit is hoe games grote sprites laten inschuiven van bovenaf.

### Stap 2: `src/core/ppu.zig` — sprite-grootte bepalen

De sprite-grootte wordt bepaald door *twee* velden samen: `shape` (attr0) en `size` (attr1). Dit geeft 12 mogelijke combinaties:

```zig
const SpriteSize = struct { w: u8, h: u8 };

const sprite_sizes = [3][4]SpriteSize{
    //                size=0      size=1       size=2       size=3
    /* square     */ .{ .{8,8},   .{16,16},    .{32,32},    .{64,64}  },
    /* horizontal */ .{ .{16,8},  .{32,8},     .{32,16},    .{64,32}  },
    /* vertical   */ .{ .{8,16},  .{8,32},     .{16,32},    .{32,64}  },
};

fn get_sprite_size(shape: u2, size: u2) SpriteSize {
    return sprite_sizes[shape][size];
}
```

**Waarom shape en size apart?** De GBA-hardware codeert de grootte in 4 bits (2+2) in plaats van een simpele lookup met 4 bits. Dit geeft 12 combinaties (3 shapes × 4 sizes) met maar 4 bits, terwijl een directe codering 16 mogelijkheden zou geven waarvan de helft ongebruikt. Het is een ruimte-efficiënte hardware-encoding.

### Stap 3: `src/core/ppu.zig` — sprite tile data lezen

Sprite tile data staat in VRAM, maar op een ander adres dan achtergrond-tiles:

```zig
fn read_sprite_pixel(
    bus: *Bus,
    attr2: OamAttr2,
    pixel_x: u32,
    pixel_y: u32,
    color_mode: u1,
    is_1d_mapping: bool,
) u16 {
    // Sprite tile data begint op 0x06010000 (in tile modes)
    const tile_base: u32 = 0x0601_0000;
    const tile_id: u32 = @as(u32, attr2.tile_id);

    // Welke tile binnen de sprite bevat deze pixel?
    const tile_col = pixel_x / 8;
    const tile_row = pixel_y / 8;
    const local_x: u3 = @truncate(pixel_x & 7);
    const local_y: u3 = @truncate(pixel_y & 7);

    // Tile mapping: 1D of 2D
    var actual_tile: u32 = undefined;
    if (is_1d_mapping) {
        // 1D: tiles zijn sequentieel in geheugen
        // De tile-stride hangt af van het kleurformaat
        const tiles_per_row = if (color_mode == 0) tile_col else tile_col * 2;
        actual_tile = tile_id + tile_row * (if (color_mode == 0)
            @as(u32, get_sprite_size(attr2.shape, attr2.size).w / 8)
        else
            @as(u32, get_sprite_size(attr2.shape, attr2.size).w / 8) * 2) + tiles_per_row;
    } else {
        // 2D: tiles staan in een 32-tile breed raster
        actual_tile = tile_id + tile_row * 32 + tile_col;
    }

    if (color_mode == 0) {
        // 4bpp
        const addr = tile_base + actual_tile * 32 + @as(u32, local_y) * 4 + @as(u32, local_x) / 2;
        const byte = bus.read8(addr);
        const index: u8 = if (local_x & 1 == 0) byte & 0x0F else byte >> 4;
        if (index == 0) return 0;  // transparant
        return bus.read16(0x0500_0200 + @as(u32, attr2.palette) * 32 + @as(u32, index) * 2);
    } else {
        // 8bpp
        const addr = tile_base + actual_tile * 64 + @as(u32, local_y) * 8 + @as(u32, local_x);
        const index: u32 = bus.read8(addr);
        if (index == 0) return 0;
        return bus.read16(0x0500_0200 + index * 2);
    }
}
```

**Waarom 1D vs 2D mapping?** DISPCNT bit 6 bepaalt hoe sprite-tiles in VRAM georganiseerd zijn:
- **1D mapping**: tiles staan sequentieel. Een 32×32 sprite (4×4 tiles) gebruikt tile IDs 0,1,2,3,4,5,...15 achter elkaar. Eenvoudiger te begrijpen.
- **2D mapping**: tiles staan in een virtueel 32-tile breed raster. Dezelfde sprite zou tiles 0,1,2,3,32,33,34,35,64,... gebruiken. Dit is hoe de SNES het deed.

De meeste GBA-games gebruiken 1D mapping. 2D mapping is er voor backward-compatibiliteit.

**Waarom OBJ palette (0x05000200)?** Sprites gebruiken de *tweede* helft van Palette RAM. De eerste 256 kleuren (0x05000000) zijn voor achtergronden, de volgende 256 (0x05000200) voor sprites. Dit geeft sprites hun eigen kleurenpalet, onafhankelijk van de achtergrond.

### Stap 4: `src/core/ppu.zig` — per-scanline sprite rendering met flipping

Nu combineer je alles: loop door alle sprites, check of ze op de huidige scanline zichtbaar zijn, en teken ze.

```zig
fn render_sprites(self: *Ppu, bus: *Bus, y: u8) void {
    const dispcnt = bus.read16(0x0400_0000);
    const is_1d = (dispcnt >> 6) & 1 == 1;

    // Reset sprite line buffer
    @memset(&self.sprite_line, 0);
    @memset(&self.sprite_priority, 4);  // 4 = geen sprite

    // Loop omgekeerd: hogere index = lagere prioriteit (wordt eerst getekend)
    var i: u8 = 128;
    while (i > 0) {
        i -= 1;
        const sprite = read_oam_entry(bus, @truncate(i));

        // Skip verborgen sprites (obj_mode == 2 = hidden, gfx_mode == 1 = affine → story 5.8)
        if (sprite.attr0.obj_mode == 2) continue;
        if (sprite.attr0.gfx_mode == 1) continue;  // affine sprites later

        const size = get_sprite_size(sprite.attr0.shape, sprite.attr1.size);

        // Y-positie met wrap-around
        const sprite_y: i32 = @as(i32, sprite.attr0.y);
        const adjusted_y: i32 = if (sprite_y >= 160) sprite_y - 256 else sprite_y;

        const y32: i32 = @as(i32, y);
        const local_y = y32 - adjusted_y;
        if (local_y < 0 or local_y >= size.h) continue;  // niet op deze scanline

        // Vertical flip
        const actual_y: u32 = if (sprite.attr1.v_flip == 1)
            @intCast(size.h - 1 - @as(u8, @intCast(local_y)))
        else
            @intCast(local_y);

        const sprite_x: i32 = blk: {
            const raw: i32 = @as(i32, sprite.attr1.x);
            break :blk if (raw >= 240) raw - 512 else raw;  // 9-bit signed
        };

        for (0..@as(usize, size.w)) |px| {
            const screen_x = sprite_x + @as(i32, @intCast(px));
            if (screen_x < 0 or screen_x >= 240) continue;

            // Horizontal flip
            const actual_x: u32 = if (sprite.attr1.h_flip == 1)
                size.w - 1 - @as(u8, @intCast(px))
            else
                @intCast(px);

            const color = read_sprite_pixel(
                bus, sprite.attr2, actual_x, actual_y,
                sprite.attr0.color_mode, is_1d,
            );

            if (color != 0) {
                const sx: usize = @intCast(screen_x);
                // Alleen overschrijven als deze sprite hogere prioriteit heeft
                // (lagere OAM index is automatisch hogere prioriteit door omgekeerde loop)
                self.sprite_line[sx] = color;
                self.sprite_priority[sx] = sprite.attr2.priority;
            }
        }
    }
}
```

**Waarom omgekeerd loopen (127→0)?** Sprites met lager OAM-index hebben hogere prioriteit. Door omgekeerd te loopen (hoog→laag), tekent sprite 127 eerst en sprite 0 laatst. Sprite 0 overschrijft dus alles eronder — precies het juiste gedrag. Dit is de eenvoudigste manier om OAM-prioriteit correct te implementeren.

**Waarom X wrap-around bij 240?** De X-positie is 9 bits (0–511). Waarden boven 240 betekenen dat de sprite links van het scherm begint. Een sprite op X=500 verschijnt als X=500-512=-12, dus 12 pixels links van het scherm. Zonder deze correctie zouden sprites niet van links het scherm in kunnen schuiven.

**Valkuil:** De Y wrap-around grens is 160 (schermhoogte), maar de X wrap-around grens is 240+sprite_breedte (want een sprite kan gedeeltelijk zichtbaar zijn). Vergeet niet dat sprites gedeeltelijk buiten het scherm kunnen staan — check per pixel, niet per sprite.

### Stap 5: `src/core/ppu.zig` — sprite-BG compositing

Sprites moeten samengevlochten worden met achtergrondlagen op basis van het priority-veld:

```zig
fn composite_with_sprites(self: *Ppu, bus: *Bus, y: u8) void {
    const y32: u32 = @as(u32, y);
    const backdrop = bus.read16(0x0500_0000);

    for (0..240) |x| {
        var best_color: u16 = backdrop;
        var best_priority: u3 = 7;  // laagste prioriteit

        // Check alle actieve BG lagen
        for (0..4) |bg| {
            if (!self.bg_active[bg]) continue;
            if (!self.bg_opaque[bg][x]) continue;

            const bg_priority = self.bg_priority[bg];
            if (bg_priority < best_priority) {
                best_priority = bg_priority;
                best_color = self.bg_line[bg][x];
            }
        }

        // Check sprite op deze pixel
        if (self.sprite_line[x] != 0) {
            const sprite_pri: u3 = self.sprite_priority[x];
            // Sprite tekent BOVEN BG lagen met zelfde of lagere prioriteit
            if (sprite_pri <= best_priority) {
                best_color = self.sprite_line[x];
            }
        }

        self.framebuffer[y32 * 240 + x] = best_color;
    }
}
```

**Waarom sprites apart renderen en dan compositen?** Door eerst alle sprites in een apart buffer te renderen (`sprite_line`) en daarna met BG lagen te combineren, vermijd je complexe nested loops. Het is ook hoe de hardware werkt: de sprite-engine en BG-engine draaien parallel, en de output wordt daarna gecombineerd.

**Waarom `<=` bij sprite vs BG prioriteit?** Bij gelijke priority tekent een sprite *boven* een achtergrondlaag. Dit is hardware-gedrag: sprites winnen bij gelijke prioriteit. Als je `<` gebruikt i.p.v. `<=`, verdwijnen sprites achter achtergronden met dezelfde priority — een subtiele maar zichtbare bug.

### Stap 6: Testen met sprite-rijke ROMs

Test met ROMs die sprites gebruiken. Vrijwel elke GBA-game gebruikt sprites voor de speler, vijanden, en UI-elementen.

```
Veelvoorkomende bugs:
- Sprites staan op verkeerde plek → Y of X wrap-around is fout
- Sprites zijn vervormd (verkeerde grootte) → shape/size lookup tabel is verkeerd
- Sprite kleuren fout → OBJ palette (0x05000200) wordt niet gelezen, of BG palette wordt per ongeluk gebruikt
- Sprites achter achtergronden die ze zouden moeten overlappen → priority compositing is fout
- Sprites zijn niet gespiegeld terwijl ze dat moeten zijn → h_flip/v_flip check is omgekeerd
- Helft van sprite pixels ontbreken → 1D/2D tile mapping is verwisseld
```

**Tip:** mGBA heeft een OAM viewer die laat zien waar elke sprite staat en hoe groot die is. Vergelijk dit met je eigen output om snel te vinden waar het misgaat.
