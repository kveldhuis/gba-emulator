# Story 3.3 — ARM instructie decodering

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** 32-bit ARM instructies decoderen naar het juiste type.

## Zig concepten

### Bits extraheren — het decodeerpatroon

Een ARM instructie is 32 bits. Verschillende bitgroepen vertellen je wat de instructie doet:

```
31..28  27..20  19..16  15..12  11..8  7..4  3..0
[cond ] [opcode/type  ] [Rn  ] [Rd  ] [    operand2    ]
```

Om bits [27:20] te pakken:
```zig
const bits_27_20 = (instruction >> 20) & 0xFF;  // shift 20, pak 8 bits
```

Om bits [7:4] te pakken:
```zig
const bits_7_4 = (instruction >> 4) & 0xF;  // shift 4, pak 4 bits
```

### Lookup tables met comptime

In plaats van een enorme `switch` bij elke instructie, kun je een tabel voorberekenen **tijdens compilatie**. De CPU hoeft dan runtime alleen een array-index op te zoeken.

```zig
// Type voor een instructie handler functie
const InstructionHandler = *const fn (*Cpu, *Bus, u32) void;

// Bouw de tabel bij compilatie
const lut = comptime blk: {
    var table: [4096]InstructionHandler = undefined;
    for (0..4096) |i| {
        // i bevat bits [27:20] (hoge 8) en [7:4] (lage 4)
        table[i] = decode(@intCast(i));
    }
    break :blk table;
};

// Runtime: zoek de handler op
pub fn execute(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    const index = ((instruction >> 16) & 0xFF0) | ((instruction >> 4) & 0xF);
    lut[index](cpu, bus, instruction);
}
```

### comptime uitgelegd

`comptime` betekent "bereken dit tijdens het compileren". De compiler voert de code uit en bakt het resultaat in je programma. Runtime is het dan gewoon een constante tabel — supersnel.

```zig
// Dit wordt TIJDENS compilatie berekend:
const table = comptime blk: {
    var t: [256]u8 = undefined;
    for (0..256) |i| {
        t[i] = @intCast(i * 2);
    }
    break :blk t;  // "break" uit het blok met de waarde t
};

// Runtime is table gewoon een array met 256 voorberekende waarden
```

### Function pointers

Een function pointer is een variabele die naar een functie wijst. Je kunt hem aanroepen alsof het een gewone functie is.

```zig
fn add_handler(cpu: *Cpu, bus: *Bus, instr: u32) void { ... }
fn sub_handler(cpu: *Cpu, bus: *Bus, instr: u32) void { ... }

// Een variabele die naar een functie wijst:
const handler: InstructionHandler = add_handler;
handler(cpu, bus, instruction);  // roept add_handler aan
```

### Condition codes

Elke ARM instructie heeft een 4-bit condition code op bits [31:28]. De CPU checkt deze VOOR uitvoering — als de conditie niet klopt, wordt de instructie overgeslagen.

```zig
fn check_condition(cpu: *const Cpu, instruction: u32) bool {
    const cond: u4 = @truncate(instruction >> 28);
    return switch (cond) {
        0x0 => cpu.cpsr.z == 1,           // EQ: equal (Z set)
        0x1 => cpu.cpsr.z == 0,           // NE: not equal (Z clear)
        0x2 => cpu.cpsr.c == 1,           // CS: carry set
        0x3 => cpu.cpsr.c == 0,           // CC: carry clear
        0x4 => cpu.cpsr.n == 1,           // MI: minus/negative
        0x5 => cpu.cpsr.n == 0,           // PL: plus/positive
        0x6 => cpu.cpsr.v == 1,           // VS: overflow set
        0x7 => cpu.cpsr.v == 0,           // VC: overflow clear
        0x8 => cpu.cpsr.c == 1 and cpu.cpsr.z == 0,  // HI: unsigned higher
        0x9 => cpu.cpsr.c == 0 or cpu.cpsr.z == 1,   // LS: unsigned lower or same
        0xA => cpu.cpsr.n == cpu.cpsr.v,              // GE: signed >=
        0xB => cpu.cpsr.n != cpu.cpsr.v,              // LT: signed <
        0xC => cpu.cpsr.z == 0 and cpu.cpsr.n == cpu.cpsr.v,  // GT: signed >
        0xD => cpu.cpsr.z == 1 or cpu.cpsr.n != cpu.cpsr.v,   // LE: signed <=
        0xE => true,                      // AL: always (voer altijd uit)
        0xF => true,                      // reserved (behandel als always)
    };
}
```

### @truncate — veilig naar kleiner type

```zig
const big: u32 = 0xABCD1234;
const small: u4 = @truncate(big >> 28);  // pakt onderste 4 bits na shift = 0xA
```

`@truncate` zegt tegen Zig: "ik weet dat ik bits weggooi, dat is oké". Zonder `@truncate` geeft Zig een compileer-error als je een u32 in een u4 probeert te stoppen.

## Context — ARM instructie categorieën

**Waarom deze indeling?** ARM instructies zijn ontworpen voor efficiënte hardware-decodering. De bits [27:20] geven de CPU snel (in één clock-cyclus) genoeg informatie om te weten welk hardware-pad geactiveerd moet worden: de ALU, de geheugenbus, de branch-unit, etc. Dit is de kracht van RISC: uniformiteit maakt de hardware simpeler en sneller.

| Bits [27:20] patroon | Instructietype |
|----------------------|----------------|
| `000xxxxx` + bits [7:4] | Data Processing / Multiply / Halfword transfer |
| `001xxxxx` | Data Processing (immediate) |
| `010xxxxx` | Single Data Transfer (LDR/STR) |
| `011xxxxx` | Single Data Transfer (register offset) |
| `100xxxxx` | Block Data Transfer (LDM/STM) |
| `101xxxxx` | Branch (B/BL) |
| `1111xxxx` | Software Interrupt (SWI) |

## Stap-voor-stap

### Stap 1: `src/core/arm.zig` — Bestand opzetten en InstructionHandler type

Maak een nieuw bestand `src/core/arm.zig` aan voor alle ARM-specifieke decoderingslogica.

**Waarom een apart bestand?** ARM decodering is complex (4096-entry lookup table, condition checks, stub handlers). Door het te scheiden van `cpu.zig` houd je de CPU-state gescheiden van de instructie-logica. Dit maakt testen ook makkelijker: je kunt ARM-functies testen zonder de hele CPU op te zetten.

```zig
const std = @import("std");
const Cpu = @import("cpu.zig").Cpu;
const Bus = @import("../bus.zig").Bus;

/// Elke ARM instructie-handler heeft dezelfde signature:
/// - cpu: de CPU state om te muteren
/// - bus: geheugen om te lezen/schrijven
/// - instruction: de ruwe 32-bit instructie
const InstructionHandler = *const fn (*Cpu, *Bus, u32) void;
```

**Waarom function pointers?** De lookup table bevat 4096 entries — voor elke mogelijke combinatie van de decodeer-bits een function pointer naar de juiste handler. Runtime is het dan één array-lookup + één indirect function call. Geen `switch`, geen `if`-chain, geen runtime overhead.

### Stap 2: `src/core/arm.zig` — De decode() functie schrijven

De `decode()` functie vertaalt een 12-bit index (gecombineerd uit bits [27:20] en [7:4] van een instructie) naar de juiste handler-functie.

**Waarom bits [27:20] en [7:4]?** ARM instructies zijn slim ingedeeld. Bits 27-20 bepalen de brede categorie (data processing, load/store, branch, etc.). Bits 7-4 verfijnen dit voor bepaalde categorieën (multiply vs. halfword transfer vs. reguliere ALU). Samen geven deze 12 bits genoeg informatie om elke instructie eenduidig te classificeren.

**Waarom 4096 entries?** 8 bits (27:20) + 4 bits (7:4) = 12 bits = 2^12 = 4096 mogelijke combinaties. Elke combinatie mapt naar precies één handler.

```zig
fn decode(i: u12) InstructionHandler {
    const bits_27_20: u8 = @truncate(i >> 4);  // hoge 8 bits van de index
    const bits_7_4: u4 = @truncate(i);          // lage 4 bits van de index

    // Bit 27-26 geeft de hoofdcategorie
    const category: u2 = @truncate(bits_27_20 >> 6);

    return switch (category) {
        0b00 => {
            // Data Processing, Multiply, Halfword transfers
            // Bits [7:4] = 1001 en bit 24-21 specifiek → Multiply
            if (bits_7_4 == 0b1001) {
                if (bits_27_20 & 0b11111100 == 0b00000000)
                    return multiply;
                if (bits_27_20 & 0b11111000 == 0b00001000)
                    return multiplyLong;
                // Swap instructies
                if (bits_27_20 & 0b11111011 == 0b00010000)
                    return singleDataSwap;
            }

            // Halfword/signed transfer: bits [7:4] = 1011, 1101, of 1111
            if (bits_7_4 & 0b1001 == 0b1001 and bits_7_4 & 0b0110 != 0) {
                return halfwordTransfer;
            }

            // PSR transfer: MRS/MSR
            if (bits_27_20 & 0b11111011 == 0b00010000 and bits_7_4 == 0b0000)
                return mrs;
            if (bits_27_20 & 0b11111011 == 0b00010010 and bits_7_4 & 0b1111 == 0b0000)
                return msr;
            if (bits_27_20 & 0b11111011 == 0b00110010)
                return msrImmediate;

            // BX: branch and exchange
            if (bits_27_20 == 0b00010010 and bits_7_4 == 0b0001)
                return branchExchange;

            // Anders: gewone data processing
            return dataProcessing;
        },
        0b01 => {
            // Single Data Transfer (LDR/STR) en undefined
            if (bits_27_20 & 0b10_0 == 0 and bits_7_4 & 0b0001 == 1) {
                return undefinedInstruction;
            }
            return singleDataTransfer;
        },
        0b10 => {
            // Bit 25 bepaalt: block data transfer of branch
            if (bits_27_20 & 0b00100000 != 0) return branch;
            return blockDataTransfer;
        },
        0b11 => {
            // Software Interrupt (SWI) als bits [27:24] = 1111
            if (bits_27_20 >> 4 == 0b1111) return softwareInterrupt;
            // Coprocessor (niet relevant voor GBA)
            return undefinedInstruction;
        },
    };
}
```

**Valkuil:** De decodering is het meest foutgevoelige deel van de emulator. Gebruik de ARM7TDMI technical reference manual (ARM DDI 0029E) als bron. De decode-prioriteiten zijn belangrijk: multiply moet vóór data processing gecheckt worden, want ze overlappen in het bit-patroon.

**Tip:** Begin simpel! Start met alleen `dataProcessing`, `branch`, `singleDataTransfer` en `undefinedInstruction`. Voeg multiply, halfword transfers, en PSR transfers later toe als je ze nodig hebt.

### Stap 3: `src/core/arm.zig` — Comptime lookup table genereren

Nu combineer je de `decode()` functie met Zig's `comptime` om de hele tabel te bouwen bij compilatie.

**Waarom comptime?** De lookup table verandert nooit — elke bit-combinatie heeft altijd dezelfde handler. Door de tabel bij compilatie te berekenen, wordt het een constante in je binary. Runtime is het dan O(1): één array-lookup, geen if/switch overhead. Dit is essentieel voor performance omdat `execute()` miljoenen keren per seconde wordt aangeroepen.

```zig
/// De lookup table: 4096 function pointers, berekend bij compilatie
const lut: [4096]InstructionHandler = comptime blk: {
    @setEvalBranchQuota(5000);  // compiletime loops hebben een budget
    var table: [4096]InstructionHandler = undefined;
    for (0..4096) |i| {
        table[i] = decode(@intCast(i));
    }
    break :blk table;
};
```

**Waarom `@setEvalBranchQuota`?** Zig beperkt hoeveel "stappen" comptime code mag uitvoeren (standaard 1000) om oneindige loops te voorkomen. Met 4096 iteraties en logica in elke iteratie heb je meer nodig. 5000 is ruim genoeg.

**Waarom `break :blk`?** Dit is Zig's manier om een waarde uit een gelabeld blok te "returnen". Het `comptime blk:` label markeert het blok, en `break :blk table` zegt "het resultaat van dit blok is `table`".

### Stap 4: `src/core/arm.zig` — checkCondition() implementeren

**Waarom condition codes?** Dit is een uniek ARM-kenmerk: ELKE ARM instructie kan conditioneel zijn. De bovenste 4 bits (31:28) bevatten een conditie. De CPU evalueert de conditie tegen de huidige CPSR flags en slaat de instructie over als de conditie niet klopt. Dit maakt veel branches overbodig:

```zig
// In plaats van:
//   CMP R0, R1
//   BEQ skip
//   MOV R2, #1
//   skip:
// Kun je schrijven:
//   CMP R0, R1
//   MOVNE R2, #1    ← voert MOV alleen uit als Z=0 (not equal)
```

```zig
pub fn checkCondition(cpu: *const Cpu, instruction: u32) bool {
    const cond: u4 = @truncate(instruction >> 28);
    const n = cpu.cpsr.n;
    const z = cpu.cpsr.z;
    const c = cpu.cpsr.c;
    const v = cpu.cpsr.v;

    return switch (cond) {
        0x0 => z == 1,                       // EQ — equal
        0x1 => z == 0,                       // NE — not equal
        0x2 => c == 1,                       // CS/HS — carry set / unsigned >=
        0x3 => c == 0,                       // CC/LO — carry clear / unsigned <
        0x4 => n == 1,                       // MI — minus (negative)
        0x5 => n == 0,                       // PL — plus (positive or zero)
        0x6 => v == 1,                       // VS — overflow set
        0x7 => v == 0,                       // VC — overflow clear
        0x8 => c == 1 and z == 0,            // HI — unsigned higher
        0x9 => c == 0 or z == 1,             // LS — unsigned lower or same
        0xA => n == v,                       // GE — signed >=
        0xB => n != v,                       // LT — signed <
        0xC => z == 0 and n == v,            // GT — signed >
        0xD => z == 1 or n != v,             // LE — signed <=
        0xE => true,                         // AL — always (veruit de meest gebruikte)
        0xF => true,                         // NV — never (deprecated, behandel als always)
    };
}
```

**Waarom is 0xE (always) de meest voorkomende?** De meeste instructies hoeven niet conditioneel te zijn. De assembler zet dan conditie 0xE ("altijd uitvoeren"). Je ziet dit in raw instructies: `0xE...` begint bijna elke instructie.

**Waarom 0xF als "always" behandelen?** Conditie 0xF was oorspronkelijk "never" (voer nooit uit), maar in latere ARM versies wordt het hergebruikt voor speciale instructies. Op de ARM7TDMI is het veilig om het als "always" te behandelen.

**Tip:** De signed vergelijkingen (GE, LT, GT, LE) vergelijken N met V. De logica is: als N==V, dan is het resultaat positief (of nul); als N!=V, dan is het resultaat negatief. Dit komt door hoe two's complement overflow werkt.

### Stap 5: `src/core/arm.zig` — execute() functie (runtime entry point)

Dit is de functie die vanuit de CPU's `armStep()` wordt aangeroepen:

```zig
pub fn execute(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    // Bouw de 12-bit index: bits [27:20] als hoge 8, bits [7:4] als lage 4
    const index: u12 = @truncate(
        ((instruction >> 16) & 0xFF0) | ((instruction >> 4) & 0xF)
    );

    // Lookup en aanroepen
    lut[index](cpu, bus, instruction);
}
```

**Waarom `(instruction >> 16) & 0xFF0`?** Dit pakt bits [27:20] en verschuift ze naar positie [11:4] van de index. `(instruction >> 4) & 0xF` pakt bits [7:4] en zet ze op positie [3:0]. Samen vormen ze de 12-bit index.

**Waarom geen condition check hier?** De condition check zit in `armStep()` (story 3.2), vóór `execute()` wordt aangeroepen. Als de conditie faalt, wordt `execute()` helemaal niet aangeroepen. Dit is efficiënter dan elke handler individueel de conditie te laten checken.
