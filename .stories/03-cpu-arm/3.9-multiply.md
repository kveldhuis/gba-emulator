# Story 3.9 — ARM Multiply instructies

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** MUL, MLA, en long multiply varianten.

## Zig concepten

### Vermenigvuldiging en truncatie

32-bit × 32-bit kan een 64-bit resultaat geven. Voor MUL wil je alleen de onderste 32 bits:

```zig
// MUL: Rd = (Rm * Rs) afgekapt tot 32 bits
const result = cpu.r[rm] *% cpu.r[rs];  // *% = wrapping multiply
cpu.r[rd] = result;
```

### 64-bit resultaat met @mulWithOverflow of widening

Voor UMULL/SMULL heb je het volledige 64-bit resultaat nodig:

```zig
// UMULL (unsigned): RdHi:RdLo = Rm * Rs
const a: u64 = cpu.r[rm];
const b: u64 = cpu.r[rs];
const result: u64 = a * b;
cpu.r[rd_lo] = @truncate(result);          // onderste 32 bits
cpu.r[rd_hi] = @truncate(result >> 32);    // bovenste 32 bits

// SMULL (signed): zelfde maar signed
const a: i64 = @as(i32, @bitCast(cpu.r[rm]));  // sign-extend naar i64
const b: i64 = @as(i32, @bitCast(cpu.r[rs]));
const result: i64 = a * b;
const unsigned: u64 = @bitCast(result);
cpu.r[rd_lo] = @truncate(unsigned);
cpu.r[rd_hi] = @truncate(unsigned >> 32);
```

### Accumulate varianten (MLA, UMLAL, SMLAL)

"Accumulate" = tel er nog een waarde bij op:
```zig
// MLA: Rd = Rm * Rs + Rn
const result = cpu.r[rm] *% cpu.r[rs] +% cpu.r[rn];

// UMLAL: RdHi:RdLo += Rm * Rs
const product: u64 = @as(u64, cpu.r[rm]) * @as(u64, cpu.r[rs]);
const existing: u64 = (@as(u64, cpu.r[rd_hi]) << 32) | cpu.r[rd_lo];
const result: u64 = existing +% product;
```

### Register velden bij multiply

Let op: de register velden zitten op andere plekken dan bij data processing:
```
Bits [19:16] = Rd (of RdHi bij long multiply)
Bits [15:12] = Rn (accumulate register, of RdLo)
Bits [11:8]  = Rs
Bits [3:0]   = Rm
```

## Context

**Waarom staan multiply instructies apart van data processing?** Hoewel multiply-instructies in dezelfde encoding-ruimte vallen als data processing (bits [27:26] = `00`), hebben ze een heel ander formaat. Ze worden herkend aan het bitpatroon `1001` in bits [7:4]. De reden voor een aparte groep is dat vermenigvuldiging **anders werkt in de hardware**: de ALU kan twee getallen in één cyclus optellen, maar vermenigvuldiging vereist meerdere cycli (de ARM7TDMI heeft een iteratieve multiplier). Dit verschil in pipeline-timing betekent dat multiply instructies apart gedecodeerd en getimed moeten worden.

**Waarom zijn long multiply varianten (UMULL/SMULL) nodig?** GBA games gebruiken veel fixed-point wiskunde: in plaats van floating point (wat de ARM7TDMI niet heeft) gebruiken ze getallen als "16.16 fixed point" — 16 bits voor het gehele deel en 16 bits voor de fractie. Als je twee 16.16 getallen vermenigvuldigt, krijg je een 32.32 resultaat — 64 bits. Zonder long multiply zou je dit in meerdere stappen moeten doen, wat veel langzamer is.

**Waarom de Rd ≠ Rm restrictie?** Op de ARM7TDMI mag het destination register (Rd) niet hetzelfde zijn als Rm. Dit is een hardware-beperking van de vroege iteratieve multiplier: het leest Rm in dezelfde cyclus als het naar Rd schrijft, en als ze hetzelfde register zijn, gaat de deelresultaat-schrijf de bronwaarde overschrijven halverwege de berekening. Later ARM versies hebben deze restrictie niet meer, maar voor correcte GBA-emulatie moet je het respecteren (of op z'n minst een waarschuwing loggen).

## Stap-voor-stap

### Stap 1: `src/core/arm.zig` — instructieformaat decoderen

De multiply instructies hebben een uniek formaat dat verschilt van data processing. **Let op:** de registervelden zitten op andere posities dan je gewend bent!

```
Bits [31:28] = condition
Bits [27:22] = 000000 (MUL/MLA) of 00001U (long multiply)
Bit  [21]    = A (Accumulate: 0=multiply only, 1=multiply-accumulate)
Bit  [20]    = S (Set condition flags)
Bits [19:16] = Rd (MUL/MLA) of RdHi (long multiply)
Bits [15:12] = Rn (MUL/MLA accumulate) of RdLo (long multiply)
Bits [11:8]  = Rs (tweede operand)
Bits [7:4]   = 1001 (multiply identificatie)
Bits [3:0]   = Rm (eerste operand)
```

```zig
fn multiply(cpu: *Cpu, instruction: u32) void {
    const rd = @as(u4, @truncate(instruction >> 16));
    const rn = @as(u4, @truncate(instruction >> 12));
    const rs = @as(u4, @truncate(instruction >> 8));
    const rm = @as(u4, @truncate(instruction));

    const set_flags = (instruction >> 20) & 1 == 1;  // S bit
    const accumulate = (instruction >> 21) & 1 == 1; // A bit
}
```

**Waarom zitten Rm en Rs op andere posities dan bij data processing?** Bij data processing zit de tweede operand (Operand2) in bits [11:0] en Rn in bits [19:16]. Bij multiply is Rm (bits [3:0]) de eerste factor, Rs (bits [11:8]) de tweede, en Rn (bits [15:12]) de accumulate-waarde. Deze andere layout komt doordat de hardware-multiplier een ander intern datapad gebruikt dan de ALU.

### Stap 2: `src/core/arm.zig` — MUL en MLA implementeren

**Waarom wrapping multiply (`*%`)?** Een 32×32 vermenigvuldiging kan een resultaat tot 64 bits geven. MUL neemt alleen de onderste 32 bits — het overflow-deel wordt weggegooid. Zig's `*%` operator doet precies dit: vermenigvuldigen met wrapping, geen panic bij overflow.

```zig
fn multiply(cpu: *Cpu, instruction: u32) void {
    // ... (decodering uit stap 1) ...

    // Onderscheid 32-bit vs 64-bit multiply op basis van bit [23]
    const long = (instruction >> 23) & 1 == 1;

    if (!long) {
        // === 32-bit multiply (MUL / MLA) ===
        var result: u32 = cpu.r[rm] *% cpu.r[rs];

        if (accumulate) {
            // MLA: tel Rn erbij op
            result +%= cpu.r[rn];
        }

        cpu.r[rd] = result;

        if (set_flags) {
            cpu.cpsr.z = if (result == 0) 1 else 0;
            cpu.cpsr.n = @truncate(result >> 31);
            // C flag is "onvoorspelbaar" (destroyed) bij multiply
            // V flag is ongewijzigd
        }
    }
}
```

**Valkuil — Carry flag bij multiply:** De ARM specificatie zegt dat de C flag "onvoorspelbaar" (UNPREDICTABLE) is na een MUL/MLA met S=1. Op de ARM7TDMI wordt hij specifiek "meaningless" — sommige implementaties zetten hem op een waarde gerelateerd aan de interne multiplier-staat. De veiligste aanpak is om C ongewijzigd te laten of op 0 te zetten. Test-ROMs verwachten vaak dat C niet verandert.

### Stap 3: `src/core/arm.zig` — UMULL en SMULL (long multiply)

**Waarom naar 64-bit promoten vóór de vermenigvuldiging?** Als je twee u32-waarden vermenigvuldigt in Zig, krijg je een u32 resultaat (met wrapping). Om het volledige 64-bit resultaat te krijgen, moet je de operanden eerst naar u64 (of i64 voor signed) promoten. Dan doet Zig de vermenigvuldiging in 64-bit precisie.

```zig
if (long) {
    const is_signed = (instruction >> 22) & 1 == 1;  // U bit (0=unsigned, 1=signed)

    if (!is_signed) {
        // === UMULL / UMLAL (unsigned) ===
        const a: u64 = cpu.r[rm];     // zero-extend naar u64
        const b: u64 = cpu.r[rs];
        var result: u64 = a * b;

        if (accumulate) {
            // UMLAL: tel bestaande RdHi:RdLo waarde erbij op
            const existing: u64 = (@as(u64, cpu.r[rd]) << 32) | cpu.r[rn];
            result +%= existing;
        }

        // Sla 64-bit resultaat op in twee registers
        cpu.r[rn] = @truncate(result);         // RdLo = onderste 32 bits
        cpu.r[rd] = @truncate(result >> 32);   // RdHi = bovenste 32 bits
    }
}
```

**Let op de registernamen bij long multiply:** Bij long multiply is `Rd` (bits [19:16]) eigenlijk **RdHi** en `Rn` (bits [15:12]) eigenlijk **RdLo**. Dit is verwarrend omdat bij MUL/MLA `Rn` de accumulate-bron is, maar bij long multiply is het een destination register. Gebruik comments om dit duidelijk te maken in je code.

### Stap 4: `src/core/arm.zig` — SMULL en SMLAL (signed long multiply)

**Waarom is signed anders dan unsigned?** Bij unsigned multiply interpreteer je de bits als positieve getallen. Bij signed multiply moet je de bits als two's complement interpreteren: `0xFFFFFFFF` is niet 4.294.967.295 maar -1. Als je -1 × -1 doet, moet het resultaat +1 zijn (64-bit), niet het unsigned product.

```zig
if (is_signed) {
    // === SMULL / SMLAL (signed) ===
    // Stap 1: interpreteer als signed (i32)
    const a_signed: i32 = @bitCast(cpu.r[rm]);
    const b_signed: i32 = @bitCast(cpu.r[rs]);

    // Stap 2: promoot naar i64 — Zig doet sign extension automatisch
    const a: i64 = a_signed;
    const b: i64 = b_signed;

    // Stap 3: vermenigvuldig in signed 64-bit
    var result: i64 = a * b;

    if (accumulate) {
        // SMLAL: bestaande waarde als signed 64-bit interpreteren
        const existing_unsigned: u64 = (@as(u64, cpu.r[rd]) << 32) | cpu.r[rn];
        const existing: i64 = @bitCast(existing_unsigned);
        result +%= existing;
    }

    // Stap 4: terug naar unsigned voor opslag
    const unsigned_result: u64 = @bitCast(result);
    cpu.r[rn] = @truncate(unsigned_result);        // RdLo
    cpu.r[rd] = @truncate(unsigned_result >> 32);   // RdHi
}
```

**Waarom de omweg via @bitCast?** Zig's type systeem onderscheidt strikt tussen signed en unsigned. Je kunt niet zomaar een `u32` als `i32` gebruiken. `@bitCast` zegt tegen de compiler: "dezelfde bits, ander type." Dit is een zero-cost operatie (geen runtime overhead), maar het maakt je intentie expliciet — je *wilt* de bits als signed interpreteren.

**Concreet voorbeeld:**
```
Rm = 0xFFFFFFFF (unsigned: 4294967295, signed: -1)
Rs = 0x00000002 (unsigned: 2, signed: 2)

UMULL: 4294967295 × 2 = 8589934590 = 0x00000001_FFFFFFFE
SMULL: -1 × 2 = -2 = 0xFFFFFFFF_FFFFFFFE

Zelfde input bits, compleet ander 64-bit resultaat!
```

### Stap 5: `src/core/arm.zig` — flags zetten bij long multiply

```zig
if (set_flags) {
    // Voor long multiply: flags gebaseerd op het 64-bit resultaat
    const unsigned_result: u64 = (@as(u64, cpu.r[rd]) << 32) | cpu.r[rn];
    cpu.cpsr.z = if (unsigned_result == 0) 1 else 0;
    cpu.cpsr.n = @as(u1, @truncate(cpu.r[rd] >> 31));  // bit 63 van het resultaat
    // C en V zijn onvoorspelbaar bij long multiply
}
```

**Waarom N gebaseerd op bit 63?** De N (negative) flag geeft aan of het resultaat negatief is in two's complement. Voor een 64-bit resultaat is dat bit 63, wat overeenkomt met bit 31 van RdHi.

### Stap 6: `src/core/arm.zig` — Rd ≠ Rm restrictie en testen

**Waarom moet je de Rd ≠ Rm restrictie kennen (maar niet per se afdwingen)?** De ARM7TDMI documentatie zegt dat het resultaat UNPREDICTABLE is als Rd == Rm. In de praktijk negeren de meeste emulators deze check — geen enkel correct gecompileerd spel zal dit doen, en het afdwingen kost een extra vergelijking per multiply. Maar het is goed om een debug-assert of log-waarschuwing toe te voegen zodat je het opmerkt als het toch gebeurt:

```zig
if (rd == rm) {
    std.log.warn("Multiply met Rd == Rm (undefined behavior)", .{});
}
```

**Test scenario's:**

```zig
test "MUL basis" {
    // R0 = 7 * 3 = 21
    // MUL R0, R1, R2 met R1=7, R2=3
    // Verwacht: R0 = 21
}

test "MUL overflow wraps" {
    // 0x80000000 * 2 = 0x100000000 → afgekapt tot 0x00000000
    // Verwacht: R0 = 0, Z flag = 1 (als S=1)
}

test "SMULL signed multiply" {
    // -1 (0xFFFFFFFF) * 2 (0x00000002) = -2
    // Verwacht: RdHi = 0xFFFFFFFF, RdLo = 0xFFFFFFFE
}

test "UMLAL accumulate" {
    // 3 * 4 = 12, bestaand = 100
    // Verwacht: RdHi:RdLo = 112
}
```
