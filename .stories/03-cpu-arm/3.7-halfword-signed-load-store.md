# Story 3.7 — ARM Halfword & Signed Load/Store

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** LDRH, STRH, LDRSH, LDRSB implementeren.

## Zig concepten

### Sign extension revisited — voor bytes en halfwords

Bij LDRSB (load signed byte) lees je 1 byte en breidt die uit naar 32 bits met behoud van teken:

```zig
// Unsigned load: 0xFF → 0x000000FF (255)
const unsigned: u32 = bus.read8(addr);

// Signed load: 0xFF → 0xFFFFFFFF (-1)
const byte: u8 = bus.read8(addr);
const signed: i8 = @bitCast(byte);       // interpreteer als signed
const extended: i32 = signed;             // Zig breidt het teken uit
const result: u32 = @bitCast(extended);   // terug naar u32
```

Zelfde principe voor LDRSH (signed halfword, 16→32 bits):
```zig
const half: u16 = bus.read16(addr);
const signed: i16 = @bitCast(half);
const extended: i32 = signed;
const result: u32 = @bitCast(extended);
```

### Split immediate — offset verdeeld over twee velden

Bij deze instructies is de 8-bit immediate offset niet aaneengesloten maar gesplitst:

```
Bits [11:8] = hoge 4 bits van offset
Bits [3:0]  = lage 4 bits van offset
```

Om ze samen te voegen:
```zig
const hi = (instruction >> 4) & 0xF0;  // bits [11:8] naar positie [7:4]
const lo = instruction & 0xF;           // bits [3:0]
const offset = hi | lo;                 // combineer tot 8-bit offset
```

### Bits [6:5] bepalen het type

```zig
const op = (instruction >> 5) & 0x3;  // bits [6:5]
switch (op) {
    0b01 => {}, // LDRH/STRH (unsigned halfword)
    0b10 => {}, // LDRSB (signed byte)
    0b11 => {}, // LDRSH (signed halfword)
    else => {}, // SWP instructie (apart geval)
}
```

## Context

**Waarom een aparte instructiegroep naast LDR/STR?** De originele ARM architectuur (ARMv1) had alleen 32-bit word loads en 8-bit byte loads. Halfword (16-bit) ondersteuning werd later toegevoegd in ARMv4, maar er was geen ruimte meer in het bestaande instructie-encoding formaat. Daarom kregen halfword en signed transfers een compleet eigen encoding, met een smaller offset-veld (8-bit i.p.v. 12-bit) en de split immediate truc.

- LDRH: load unsigned 16-bit halfword — voor `unsigned short` in C
- STRH: store 16-bit halfword — voor schrijven naar 16-bit hardware registers (GBA I/O is 16-bit)
- LDRSH: load signed 16-bit (sign-extend naar 32-bit) — voor `short` / `s16` in C
- LDRSB: load signed 8-bit (sign-extend naar 32-bit) — voor `signed char` / `s8` in C
- Offset is register of 8-bit immediate (split over twee velden)
- Op de GBA worden deze instructies veel gebruikt: VRAM en I/O registers zijn 16-bit, en veel game-data gebruikt signed types voor coördinaten en snelheden

## Stap-voor-stap

### Stap 1: `src/core/arm.zig` — instructieformaat begrijpen en bits decoderen

**Waarom staan halfword/signed loads apart van story 3.6?** De "gewone" LDR/STR uit story 3.6 gebruiken instructie-encoding categorie `01` (bits [27:26]), maar halfword en signed transfers gebruiken een compleet ander formaat dat valt onder categorie `00` — dezelfde categorie als data processing! Ze worden onderscheiden door een specifiek bitpatroon: bits [7] en [4] zijn beide `1`, en bits [6:5] zijn niet `00`. Dit andere formaat betekent dat de offset-velden, de opcode-bits en de operatie-selectie allemaal anders werken dan bij 3.6. Vandaar een aparte story.

De instructie-layout ziet er zo uit:
```
Bits [31:28] = condition
Bits [27:25] = 000
Bit  [24]    = P (pre/post index)
Bit  [23]    = U (up/down: optellen of aftrekken)
Bit  [22]    = I (0=register offset, 1=immediate offset)
Bit  [21]    = W (write-back)
Bit  [20]    = L (0=store, 1=load)
Bits [19:16] = Rn (base register)
Bits [15:12] = Rd (destination/source register)
Bits [11:8]  = offset hoge nibble (als I=1) of SBZ
Bits [7:4]   = 1SH1 (S en H bepalen het type operatie)
Bits [3:0]   = offset lage nibble (als I=1) of Rm (als I=0)
```

Begin met het decoderen van de control bits — deze zijn hetzelfde patroon als bij 3.6:

```zig
fn halfwordTransfer(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    const pre      = (instruction >> 24) & 1 == 1;  // P bit
    const up       = (instruction >> 23) & 1 == 1;  // U bit
    const imm      = (instruction >> 22) & 1 == 1;  // I bit (let op: OMGEKEERD t.o.v. 3.6!)
    const wb       = (instruction >> 21) & 1 == 1;  // W bit
    const load     = (instruction >> 20) & 1 == 1;  // L bit

    const rn = @as(u4, @truncate(instruction >> 16));
    const rd = @as(u4, @truncate(instruction >> 12));
}
```

**Valkuil — de I-bit is omgekeerd!** Bij gewone LDR/STR (3.6) betekent `I=0` immediate offset en `I=1` register offset. Bij halfword transfers is het **andersom**: `I=1` = immediate, `I=0` = register. Dit is een veelvoorkomende bron van bugs. De ARM documentatie is hier verwarrend — check altijd de encoding tabel.

### Stap 2: `src/core/arm.zig` — offset berekenen (register vs split immediate)

**Waarom is de immediate offset gesplitst over twee velden?** In de gewone LDR/STR heb je 12 bits voor de offset (bits [11:0]). Maar bij halfword transfers worden bits [7:4] gebruikt voor het opcode-patroon (`1SH1`), dus is er geen ruimte voor een aaneengesloten offset. ARM loste dit op door de 8-bit offset te splitsen: de hoge nibble in bits [11:8] en de lage nibble in bits [3:0].

```zig
const offset: u32 = if (imm) blk: {
    // Immediate offset: samengesteld uit twee 4-bit velden
    const hi: u32 = (instruction >> 4) & 0xF0;  // bits [11:8] → positie [7:4]
    const lo: u32 = instruction & 0xF;           // bits [3:0]
    break :blk hi | lo;                          // combineer tot 8-bit offset
} else blk: {
    // Register offset: Rm (bits [3:0]) bevat de offset
    const rm = @as(u4, @truncate(instruction));
    break :blk cpu.r[rm];
};
```

**Tip:** De `(instruction >> 4) & 0xF0` truc verschuift bits [11:8] naar positie [7:4] in één operatie. Je zou ook `((instruction >> 8) & 0xF) << 4` kunnen schrijven, maar dat is minder efficiënt. Controleer met een voorbeeld: als bits [11:8] = `0b1010` en bits [3:0] = `0b0011`, dan is de offset `0b10100011` = 0xA3.

### Stap 3: `src/core/arm.zig` — adresberekening (pre/post index)

De adresberekening werkt identiek aan story 3.6. Hergebruik hetzelfde patroon:

```zig
const base = cpu.r[rn];

// Pre-index: bereken adres VOOR de transfer
const addr: u32 = if (pre)
    if (up) base +% offset else base -% offset
else
    base;

// ... doe de load/store op addr ...

// Post-index of write-back: update base register
// Bij post-index (P=0) wordt write-back altijd gedaan
if (!pre or wb) {
    cpu.r[rn] = if (up) base +% offset else base -% offset;
}
```

**Waarom `+%` en `-%` (wrapping arithmetic)?** Adresberekeningen kunnen in theorie overflown (bijv. 0xFFFFFFFF + 4). Op de echte hardware wrapt dit gewoon rond. Met Zig's `+%` operator emuleer je dat correct zonder runtime panics.

### Stap 4: `src/core/arm.zig` — operatietype bepalen via bits [6:5]

**Waarom bepalen bits [6:5] het type?** In het vaste bitpatroon `1SH1` (bits [7:4]) staan S (bit 6) en H (bit 5) voor "Signed" en "Halfword". De combinatie van deze twee bits, samen met de L-bit (load/store), bepaalt welke operatie uitgevoerd wordt:

```zig
const op = @as(u2, @truncate(instruction >> 5));  // bits [6:5] = SH

switch (op) {
    0b01 => {
        // H=1, S=0: unsigned halfword
        if (load) {
            // LDRH: laad unsigned 16-bit halfword
            cpu.r[rd] = bus.read16(addr);
        } else {
            // STRH: sla onderste 16 bits op
            bus.write16(addr, @truncate(cpu.r[rd]));
        }
    },
    0b10 => {
        // H=0, S=1: signed byte (LDRSB)
        // Alleen load — store variant bestaat niet
        const byte: u8 = bus.read8(addr);
        const signed: i8 = @bitCast(byte);
        const extended: i32 = signed;           // Zig doet sign extension
        cpu.r[rd] = @bitCast(extended);
    },
    0b11 => {
        // H=1, S=1: signed halfword (LDRSH)
        // Alleen load — store variant bestaat niet
        const half: u16 = bus.read16(addr);
        const signed: i16 = @bitCast(half);
        const extended: i32 = signed;
        cpu.r[rd] = @bitCast(extended);
    },
    0b00 => {
        // SWP instructie — dit pad zou niet via deze handler
        // bereikt moeten worden (apart gedecodeerd)
    },
}
```

**Waarom bestaan STRSB en STRSH niet?** Bij een store schrijf je altijd de relevante bytes — er is geen verschil tussen signed en unsigned bij schrijven. Het teken is alleen relevant bij het *laden* (uitbreiden naar 32 bits). Daarom bestaan er alleen STRH (store halfword) en geen signed store varianten.

### Stap 5: `src/core/arm.zig` — sign extension begrijpen

**Waarom is sign extension nodig?** ARM registers zijn altijd 32 bits breed. Als je een signed byte (-1 = `0xFF`) in een register laadt zonder sign extension, krijg je `0x000000FF` = 255 — een totaal verkeerde waarde. Sign extension kopieert het teken-bit (de hoogste bit van de bronwaarde) naar alle hogere bits, zodat de numerieke waarde behouden blijft.

Dit is cruciaal voor correcte emulatie van C code op de GBA. In C zijn `signed char` en `short` veelgebruikte types. De compiler genereert LDRSB voor `signed char` en LDRSH voor `short`. Zonder correcte sign extension zou alle signed arithmetic fout gaan.

**Zig maakt sign extension elegant:** Door te casten van een kleiner signed type naar een groter signed type doet Zig automatisch sign extension:

```zig
// Voorbeeld: 0x80 = -128 als i8
const byte: u8 = 0x80;
const signed: i8 = @bitCast(byte);     // = -128
const extended: i32 = signed;           // = -128 = 0xFFFFFF80 ✓
const result: u32 = @bitCast(extended); // = 0xFFFFFF80

// Zonder sign extension zou je krijgen:
// 0x00000080 = 128 (FOUT voor signed waarden!)
```

**Valkuil — unaligned halfword reads:** Op de echte GBA hardware geeft een LDRH op een oneven adres (niet deelbaar door 2) een geroteerd resultaat, vergelijkbaar met unaligned LDR in story 3.6. Voor LDRSH op een oneven adres wordt het resultaat zelfs een sign-extended byte in plaats van een halfword. Dit is obscuur gedrag dat sommige games misbruiken. Begin met een simpele implementatie en voeg dit later toe als tests het vereisen.

### Stap 6: `src/core/arm.zig` — testen

Test elke variant apart. De meest voorkomende bugs zitten in:
1. De omgekeerde I-bit (verwisseld register/immediate)
2. Foutieve sign extension (vergeten of verkeerd type)
3. De split immediate verkeerd samengevoegd

```zig
test "LDRH laadt unsigned halfword" {
    // Setup: zet 0x8000 op adres 0x100 in geheugen
    // LDRH R0, [R1] met R1 = 0x100
    // Verwacht: R0 = 0x00008000 (geen sign extension)
}

test "LDRSH sign-extends halfword" {
    // Setup: zet 0x8000 op adres 0x100
    // LDRSH R0, [R1] met R1 = 0x100
    // Verwacht: R0 = 0xFFFF8000 (sign-extended, want bit 15 = 1)
}

test "LDRSB sign-extends byte" {
    // Setup: zet 0xFF op adres 0x100
    // LDRSB R0, [R1] met R1 = 0x100
    // Verwacht: R0 = 0xFFFFFFFF (-1 sign-extended)
}

test "split immediate offset" {
    // Instructie met bits [11:8] = 0x2, bits [3:0] = 0x4
    // Verwacht offset = 0x24
}
```
