# Story 3.10 — ARM Software Interrupt (SWI)

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** SWI instructie die naar BIOS functies springt.

## Zig concepten

### Exception handling in de CPU — een gestructureerd patroon

Een "exception" in ARM is niet hetzelfde als een Zig error — het is een hardware-event dat de CPU in een speciale mode zet. SWI, IRQ, en Abort zijn allemaal exceptions.

Het patroon is altijd hetzelfde:

```zig
fn exception(cpu: *Cpu, mode: Mode, vector: u32) void {
    // 1. Sla huidige CPSR op in SPSR van de nieuwe mode
    cpu.spsr[@intFromEnum(mode)] = cpu.cpsr;

    // 2. Switch naar de nieuwe mode
    cpu.switch_mode(mode);

    // 3. Sla return address op in LR van de nieuwe mode
    cpu.r[14] = cpu.r[15] - 4;  // (verschilt per exception type)

    // 4. Disable IRQs
    cpu.cpsr.i = 1;

    // 5. Switch naar ARM mode
    cpu.cpsr.t = 0;

    // 6. Spring naar de exception vector
    cpu.r[15] = vector;
    // + pipeline refill
}
```

Voor SWI specifiek:
- Mode = Supervisor (0x13)
- Vector = 0x00000008
- LR = adres van instructie na de SWI

### @intFromEnum — enum naar getal

```zig
const mode = Mode.supervisor;
const index = @intFromEnum(mode);  // geeft de numerieke waarde van de enum
```

Handig als index in arrays (bijv. `spsr[index]`).

## Context

**Waarom is SWI de manier voor games om BIOS functies aan te roepen?** De GBA heeft een ingebouwd BIOS (Basic Input/Output System) in ROM op adres 0x00000000-0x00003FFF. Dit BIOS biedt handige functies: wachtlussen (Halt, VBlankIntrWait), wiskundige functies (Div, Sqrt), decompressie (LZ77, Huffman), en audio mixing. Games kunnen deze functies niet direct aanroepen (het BIOS is beschermd geheugen), dus gebruiken ze SWI — een **software interrupt**, vergelijkbaar met een syscall in Linux. De SWI instructie schakelt de CPU naar Supervisor mode (een geprivilegieerde mode) en springt naar het BIOS-ingangspunt op adres 0x08, waar de BIOS-code het SWI-nummer leest en de juiste functie uitvoert.

**Waarom is het comment field onbelangrijk voor de hardware maar nuttig voor debugging?** De SWI instructie heeft een 24-bit "comment field" (bits [23:0]) dat de hardware **volledig negeert** — de CPU springt altijd naar adres 0x08 ongeacht de waarde. Het is de BIOS-software die het SWI-nummer leest (via het geheugen op het return-adres) om te bepalen welke functie aangeroepen wordt. Voor een emulator is dit veld wel relevant: als je later BIOS HLE (High Level Emulation) implementeert, lees je het SWI-nummer om de juiste functie direct in je emulator te simuleren, zonder het echte BIOS te draaien.

- SWI schakelt naar Supervisor mode
- Slaat CPSR op in SPSR_svc, slaat return address op in LR_svc
- Springt naar adres 0x00000008 (BIOS SWI handler)
- Comment field (bits [23:0]) bevat SWI nummer — wordt later door BIOS HLE gebruikt
- Op de GBA is het SWI-nummer in bits [23:16] (de bovenste byte), niet [7:0] zoals op andere ARM systemen

## Stap-voor-stap

### Stap 1: `src/core/cpu.zig` — generieke exception handler bouwen

**Waarom een generieke helper?** SWI is niet de enige exception — later komen IRQ (hardware interrupts, story 5.x) en mogelijk Undefined Instruction exceptions. Het exception-mechanisme is bij allemaal identiek: bewaar CPSR, switch mode, zet LR, disable IRQs, spring naar vector. Door dit in één functie te zetten voorkom je code-duplicatie en bugs.

Het exception-patroon is altijd dezelfde 6 stappen. De volgorde is cruciaal — als je bijvoorbeeld eerst van mode switcht en dan pas CPSR opslaat in SPSR, sla je de verkeerde (nieuwe) CPSR op!

```zig
pub fn exception(self: *Cpu, bus: *Bus, vector: u32, mode: Mode, return_offset: u32) void {
    // Stap 1: Sla huidige CPSR op in SPSR van de nieuwe mode
    // Dit MOET vóór de mode switch, anders sla je de verkeerde waarde op!
    const spsr_idx = spsrIndex(mode);
    self.spsr[spsr_idx] = self.cpsr;

    // Stap 2: Switch naar de exception mode
    // Dit wisselt de banked registers (SP, LR worden die van de nieuwe mode)
    self.switchMode(mode);

    // Stap 3: Sla return address op in LR van de nieuwe mode
    // LR_mode = PC - return_offset
    // Voor SWI: LR = adres van de instructie NA de SWI (PC - 4 in ARM, na pipeline)
    self.r[14] = self.r[15] -% return_offset;

    // Stap 4: Disable IRQs (voorkom dat een interrupt de exception handler verstoort)
    self.cpsr.i = 1;

    // Stap 5: Switch naar ARM mode (exceptions worden altijd in ARM mode afgehandeld)
    self.cpsr.t = 0;

    // Stap 6: Spring naar de exception vector
    self.r[15] = vector;

    // Pipeline moet opnieuw gevuld worden (net als bij een branch)
    self.pipelineFlush(bus);
}
```

**Waarom `return_offset` als parameter?** Verschillende exceptions hebben een ander return-adres:
- **SWI**: LR = adres van instructie ná de SWI → `PC - 4` (we willen doorgaan ná de SWI)
- **IRQ**: LR = adres van de onderbroken instructie + 4 → `PC - 4` (maar de handler moet `SUBS PC, LR, #4` doen)
- **Data Abort**: LR = instructie + 8 → `PC - 4` (retry na abort)

Door de offset als parameter mee te geven, kan elke exception-caller de juiste waarde specificeren.

**Waarom worden IRQs disabled?** Als een exception handler zelf weer onderbroken zou worden door een IRQ, zonder dat de eerste exception volledig afgehandeld is, raakt de CPU-state corrupt (SPSR wordt overschreven). Vandaar dat de hardware automatisch IRQs uitschakelt. De exception handler kan ze later weer aanzetten als dat veilig is.

**Waarom altijd ARM mode?** Exception vectors staan op vaste adressen (0x00-0x1C) en bevatten ARM instructies. Zelfs als de CPU in Thumb mode was toen de exception optrad, wordt er geswitcht naar ARM. De T-bit in de opgeslagen SPSR bewaart de oorspronkelijke mode, zodat de return-instructie (`MOVS PC, LR`) correct terug kan switchen.

### Stap 2: `src/core/arm.zig` — SWI handler implementeren

De SWI instructie zelf is heel simpel — al het echte werk zit in de generieke `exception()` helper:

```zig
fn softwareInterrupt(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    _ = instruction;  // comment field wordt (nog) niet gebruikt

    // SWI: spring naar vector 0x08 in Supervisor mode
    cpu.exception(bus, 0x08, .supervisor, 4);
}
```

**Waarom is de SWI handler zo kort?** Alle complexiteit zit in `exception()`. De SWI instructie hoeft alleen maar te zeggen: "doe een exception naar Supervisor mode, vector 0x08, met return-offset 4." De eenvoud van deze handler bevestigt dat het exception-mechanisme goed geabstraheerd is.

**Waarom offset 4?** Na het uitvoeren van de SWI-instructie moet de CPU terugkeren naar de instructie *ná* de SWI. Door de ARM pipeline wijst PC al naar de instructie die 2 stappen vooruit is (PC+8). De return-offset van 4 zorgt ervoor dat LR = PC - 4 = het adres van de volgende instructie na de SWI.

### Stap 3: `src/core/arm.zig` — SWI nummer extraheren (voor toekomstige BIOS HLE)

**Waarom dit nu al voorbereiden?** Hoewel je nu nog geen BIOS HLE implementeert, is het goed om de extractie alvast klaar te hebben. Veel emulators voegen later een "intercept" toe die het SWI-nummer checkt en de BIOS-functie direct emuleert in plaats van het echte BIOS te draaien.

```zig
fn softwareInterrupt(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    // Het SWI-nummer zit in bits [23:0] van de instructie.
    // Op de GBA kijkt het BIOS naar bits [23:16] (de bovenste byte)
    // om te bepalen welke functie aangeroepen wordt.
    const swi_number: u8 = @truncate(instruction >> 16);

    // TODO: BIOS HLE — als je later BIOS functies wilt emuleren:
    // switch (swi_number) {
    //     0x00 => biosHalt(),           // SoftReset
    //     0x02 => biosHalt(),           // Halt
    //     0x05 => biosVBlankWait(),     // VBlankIntrWait
    //     0x06 => biosDiv(),            // Div
    //     0x0B => biosCpuSet(),         // CpuSet
    //     0x0C => biosCpuFastSet(),     // CpuFastSet
    //     else => {},
    // }
    _ = swi_number;

    cpu.exception(bus, 0x08, .supervisor, 4);
}
```

**Waarom bits [23:16] en niet [7:0]?** Dit is GBA-specifiek. Op andere ARM systemen zit het SWI-nummer in de onderste bits. Maar het GBA BIOS leest het nummer door naar het geheugen te kijken op `LR - 2` (voor Thumb) of `LR - 4` (voor ARM), en pakt dan de bovenste byte. Dit betekent dat SWI 0x050000 en SWI 0x050001 dezelfde functie aanroepen (VBlankIntrWait). In de praktijk gebruikt iedereen het formaat `SWI #0xNN0000` (ARM) of `SWI #0xNN` (Thumb).

### Stap 4: `src/core/cpu.zig` — exception vectors documenteren

Voeg de vector-adressen toe als constanten voor duidelijkheid:

```zig
// ARM Exception Vectors — vaste adressen in het BIOS
pub const VECTOR_RESET: u32 = 0x00000000;  // Reset
pub const VECTOR_UND:   u32 = 0x00000004;  // Undefined Instruction
pub const VECTOR_SWI:   u32 = 0x00000008;  // Software Interrupt
pub const VECTOR_PABT:  u32 = 0x0000000C;  // Prefetch Abort
pub const VECTOR_DABT:  u32 = 0x00000010;  // Data Abort
pub const VECTOR_IRQ:   u32 = 0x00000018;  // IRQ (normale interrupt)
pub const VECTOR_FIQ:   u32 = 0x0000001C;  // FIQ (snelle interrupt)
```

**Waarom vaste adressen?** Dit is hardwired in de ARM CPU — bij een exception laadt de hardware het vectoradres in de PC. Elke vector heeft precies 4 bytes ruimte (één ARM instructie), dus staat er typisch een branch-instructie die naar de eigenlijke handler springt. Op de GBA wijzen deze vectoren naar het BIOS ROM.

**Tip:** Adres 0x00000014 ontbreekt in de lijst — dat is gereserveerd (niet gebruikt). IRQ staat op 0x18 en FIQ op 0x1C. Dit is een veelgemaakte fout: 0x14 vergeten en denken dat IRQ op 0x14 zit.

### Stap 5: testen

```zig
test "SWI schakelt naar Supervisor mode" {
    var cpu = Cpu.init();
    cpu.cpsr.mode = @intFromEnum(Mode.system);
    cpu.r[15] = 0x08000010;  // huidige PC

    // Simuleer SWI instructie
    cpu.exception(&bus, 0x08, .supervisor, 4);

    // Check mode switch
    try std.testing.expectEqual(@intFromEnum(Mode.supervisor), cpu.cpsr.mode);

    // Check dat CPSR opgeslagen is in SPSR_svc
    const spsr = cpu.spsr[spsrIndex(.supervisor)];
    try std.testing.expectEqual(@intFromEnum(Mode.system), spsr.mode);

    // Check LR = return address
    try std.testing.expectEqual(@as(u32, 0x08000010 - 4), cpu.r[14]);

    // Check PC = vector
    // (na pipeline flush, PC bevat vector + 8 of vector afhankelijk van je implementatie)

    // Check IRQs disabled
    try std.testing.expectEqual(@as(u1, 1), cpu.cpsr.i);

    // Check ARM mode
    try std.testing.expectEqual(@as(u1, 0), cpu.cpsr.t);
}
```

**Tip:** Test ook dat een SWI vanuit Thumb mode correct werkt: de T-bit moet in de SPSR bewaard worden en de CPU moet naar ARM mode switchen.

## Acceptatiecriteria

- Mode switch naar Supervisor werkt
- CPSR wordt opgeslagen in SPSR_svc
- PC springt naar 0x08
- IRQs worden disabled
