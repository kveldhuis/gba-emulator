# Story 3.8 — ARM Block Data Transfer (LDM/STM)

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** Push/pop van meerdere registers tegelijk.

## Zig concepten

### Bitmask als register-lijst

De onderste 16 bits van de instructie vormen een bitmask: elke bit staat voor een register.

```
Bit 0  = R0
Bit 1  = R1
...
Bit 13 = R13 (SP)
Bit 14 = R14 (LR)
Bit 15 = R15 (PC)
```

Voorbeeld: `0b0100000000001111` = R0, R1, R2, R3, en R14 (LR).

### Door bits itereren

Om te checken welke registers in de lijst zitten:

```zig
const reg_list: u16 = @truncate(instruction);

var i: u5 = 0;
while (i < 16) : (i += 1) {
    if (reg_list & (@as(u16, 1) << @as(u4, @truncate(i))) != 0) {
        // Register i zit in de lijst
    }
}
```

Uitleg van `(@as(u16, 1) << @as(u4, @truncate(i)))`:
- `@as(u16, 1)` = het getal 1 als u16 type
- `@as(u4, @truncate(i))` = cast i (u5) naar u4 als shift-hoeveelheid
- `<< ...` = shift die 1 naar positie i
- Als die bit in `reg_list` ook 1 is, is de `&` niet-nul

**Let op het type van `i`:** We gebruiken `u5` (0-31) in plaats van `u4` (0-15), omdat het continue-deel `i += 1` anders overloopt wanneer i=15. Een `u4` kan maximaal 15 bevatten — `15 + 1` zou een overflow-panic geven in Zig. Met `u5` is er ruimte: na de laatste iteratie (i=15) wordt i=16, de conditie `i < 16` faalt, en de loop stopt netjes.

### While met "after" expressie

```zig
var i: u5 = 0;
while (i < 16) : (i += 1) {
    // ...
}
```

Het deel na `:` wordt na elke iteratie uitgevoerd (vergelijkbaar met de derde deel van een for-loop in C). Het loop-type is `u5` zodat `i += 1` bij i=15 niet overloopt (u4 zou paniken).

### @popCount — bits tellen

```zig
const reg_list: u16 = @truncate(instruction);
const count = @popCount(reg_list);  // aantal registers in de lijst
```

Handig om te weten hoeveel bytes de transfer beslaat (count × 4).

### De 4 adresseringsmodi

```
IB (Increment Before):  adres += 4 VOOR elke transfer
IA (Increment After):   adres += 4 NA elke transfer  
DB (Decrement Before):  adres -= 4 VOOR elke transfer
DA (Decrement After):   adres -= 4 NA elke transfer
```

Bits [24:23] (P en U) bepalen welke mode:
```zig
const pre = (instruction >> 24) & 1 == 1;  // P bit
const up  = (instruction >> 23) & 1 == 1;  // U bit

// Up + Pre = IB,  Up + Post = IA
// Down + Pre = DB, Down + Post = DA
```

Voorbeeld STMDB (push op stack):
```
SP = 0x03007F00, registers: R0, R1, R2

SP -= 4 → schrijf R2 op 0x03007EFC
SP -= 4 → schrijf R1 op 0x03007EF8
SP -= 4 → schrijf R0 op 0x03007EF4
SP = 0x03007EF4 (als write-back aan staat)
```

## Context

**Waarom bestaan LDM/STM?** In ARM assembly moet je bij een functie-aanroep registers saven op de stack en ze later herstellen. Zonder LDM/STM zou je voor elk register een aparte STR/LDR nodig hebben — bij 5 registers is dat 5 instructies. LDM/STM doen het in één instructie, wat veel sneller is: minder instructie-fetches, en de bus kan burst-transfers doen (opeenvolgende adressen zijn sneller op de GBA).

**Waarom een 16-bit bitmask als register-lijst?** Met een bitmask kun je een variabel aantal registers specificeren in een vast instructieformaat. Elke bit correspondeert met een register (bit 0 = R0, bit 15 = R15). Dit is compacter dan individuele registernummers opsommen en laat de hardware eenvoudig door de bits itereren.

**Waarom 4 adresseringsmodi?** De 4 modi (IB, IA, DB, DA) corresponderen met veelgebruikte stack-conventies:
- **STMDB/LDMIA** = "Full Descending" stack (de GBA-conventie): STMDB = push (stack groeit omlaag), LDMIA = pop
- **STMIA/LDMDB** = "Full Ascending" stack (minder gebruikelijk)
- De assembler heeft aliassen: `STMFD` = `STMDB`, `LDMFD` = `LDMIA`, wat het makkelijker maakt om push/pop te schrijven

- Register list is een 16-bit bitmask (elke bit = een register)
- 4 adresseringsmodi: IB, IA, DB, DA
- Optionele write-back van base register
- Speciaal geval: als R15 (PC) in de lijst zit bij LDM → branch

## Stap-voor-stap

### Stap 1: `src/core/arm.zig` — instructie-bits decoderen

De block data transfer instructie heeft dit formaat:
```
Bits [31:28] = condition
Bits [27:25] = 100
Bit  [24]    = P (Pre/Post: adres aanpassen voor of na transfer)
Bit  [23]    = U (Up/Down: incrementeren of decrementeren)
Bit  [22]    = S (PSR & force user mode — speciaal gedrag)
Bit  [21]    = W (Write-back: base register updaten)
Bit  [20]    = L (0=STM store, 1=LDM load)
Bits [19:16] = Rn (base register)
Bits [15:0]  = register list (bitmask)
```

```zig
fn blockDataTransfer(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    const pre  = (instruction >> 24) & 1 == 1;  // P bit
    const up   = (instruction >> 23) & 1 == 1;  // U bit
    const s    = (instruction >> 22) & 1 == 1;  // S bit (voorlopig negeren)
    const wb   = (instruction >> 21) & 1 == 1;  // W bit
    const load = (instruction >> 20) & 1 == 1;  // L bit

    const rn = @as(u4, @truncate(instruction >> 16));
    const reg_list: u16 = @truncate(instruction);
}
```

**Waarom de S-bit voorlopig negeren?** De S-bit heeft complex gedrag: bij LDM met R15 in de lijst kopieert hij SPSR naar CPSR (return from exception), en zonder R15 forceert hij user-mode register access. Dit is edge-case gedrag dat je later kunt toevoegen. Begin simpel.

### Stap 2: `src/core/arm.zig` — startadres berekenen met @popCount

**Waarom moet je het totale aantal registers van tevoren weten?** Bij "decrement" modi (DB/DA) moet je het startadres berekenen vóór je begint met transfers. Je moet weten hoeveel registers er zijn om te bepalen waar het laagste adres ligt. Bij "increment" modi is dit niet strikt nodig, maar het is het makkelijkst om altijd van laag naar hoog adres te werken — dat is hoe de hardware het doet (laagste register op laagste adres).

```zig
const count: u32 = @popCount(reg_list);  // aantal '1' bits = aantal registers
const total_bytes = count * 4;           // elke register = 4 bytes

const base = cpu.r[rn];

// Bereken het startadres afhankelijk van de adresseringsmodus
// We werken altijd van laag naar hoog adres
var addr: u32 = switch (@as(u2, @truncate(
    (@as(u2, @intFromBool(pre)) << 1) | @as(u2, @intFromBool(up))
))) {
    // !pre, !up = DA (Decrement After)
    0b00 => base -% total_bytes +% 4,
    // !pre, up  = IA (Increment After)
    0b01 => base,
    // pre, !up  = DB (Decrement Before)
    0b10 => base -% total_bytes,
    // pre, up   = IB (Increment Before)
    0b11 => base +% 4,
};
```

**Waarom altijd van laag naar hoog werken?** De ARM specificatie zegt dat registers altijd in oplopende volgorde worden overgedragen, met het laagste registernummer op het laagste adres. Dit geldt ongeacht de adresseringsmodus. De modus bepaalt alleen *waar* het blok begint relatief tot de base, niet de volgorde van transfers.

**Alternatieve aanpak — per-register berekenen:** Een simpeler (maar minder efficiënte) aanpak is om per register het adres te berekenen:

```zig
var current_addr = base;

var i: u5 = 0;
while (i < 16) : (i += 1) {
    if (reg_list & (@as(u16, 1) << @as(u4, @truncate(i))) != 0) {
        if (pre) {
            current_addr = if (up) current_addr +% 4 else current_addr -% 4;
        }

        // ... transfer op current_addr ...

        if (!pre) {
            current_addr = if (up) current_addr +% 4 else current_addr -% 4;
        }
    }
}
```

Beide aanpakken werken. De eerste is dichter bij hoe de hardware het doet; de tweede is makkelijker te begrijpen. Kies wat voor jou het duidelijkst is.

### Stap 3: `src/core/arm.zig` — door de register list itereren en transfers uitvoeren

**Waarom van bit 0 naar bit 15?** Het laagste registernummer moet altijd op het laagste adres staan. Door van R0 naar R15 te itereren en het adres steeds te verhogen, garandeer je deze volgorde automatisch.

```zig
var i: u5 = 0;
while (i < 16) : (i += 1) {
    if (reg_list & (@as(u16, 1) << @as(u4, @truncate(i))) != 0) {
        const reg = @as(u4, @truncate(i));

        if (load) {
            // LDM: lees 32-bit woord van geheugen naar register
            cpu.r[reg] = bus.read32(addr);
        } else {
            // STM: schrijf register naar geheugen
            bus.write32(addr, cpu.r[reg]);
        }

        addr +%= 4;  // volgende register op volgend adres
    }
}
```

**Waarom u5 voor de loop counter?** Zoals uitgelegd in de Zig concepten sectie: `u4` kan maximaal 15 bevatten, en `i += 1` bij i=15 zou een overflow-panic geven. Met `u5` (bereik 0-31) is er ruimte: na i=15 wordt i=16, de conditie `i < 16` faalt, en de loop stopt netjes.

### Stap 4: `src/core/arm.zig` — R15 (PC) speciaal behandelen

**Waarom is R15 speciaal bij LDM?** Als R15 (PC) in de register-lijst zit bij een LDM, laad je een nieuw adres in de program counter — dit is effectief een **branch**. De CPU moet dan de pipeline flushen (net als bij een gewone branch in story 3.5). Dit wordt veel gebruikt voor return from subroutine: `LDMFD SP!, {R4-R11, PC}` popt de saved registers én returnt in één instructie.

```zig
if (load) {
    cpu.r[reg] = bus.read32(addr);

    if (reg == 15) {
        // PC is geladen → pipeline moet geflusht worden
        cpu.pipeline_flush(bus);
        // Opmerking: als S-bit aan staat, kopieer SPSR → CPSR
        // (return from exception). Dat voegen we later toe.
    }
} else {
    // Bij STM van R15: de opgeslagen waarde is PC+12
    // (implementatie-specifiek op ARM7TDMI)
    const value = if (reg == 15) cpu.r[15] +% 4 else cpu.r[reg];
    bus.write32(addr, value);
}
```

**Valkuil — STM met R15:** Bij STM is de waarde die voor R15 geschreven wordt implementatie-specifiek. Op de ARM7TDMI is het PC+12 (huidige instructie + 12). Sommige emulators schrijven PC+8 of PC+12 — test dit met een ROM die het specifiek controleert. Begin met PC+12 (wat neerkomt op `cpu.r[15] + 4` als je PC al met 8 hebt verhoogd voor de pipeline).

### Stap 5: `src/core/arm.zig` — write-back van base register

**Waarom is de volgorde van write-back belangrijk?** Als het base register (Rn) ook in de register-lijst staat, ontstaat er een conflict. De ARM specificatie zegt:
- Bij **STM**: als Rn het **eerste** (laagste) register in de lijst is, wordt de **originele** waarde geschreven. Anders is het onvoorspelbaar.
- Bij **LDM**: als write-back aan staat én Rn in de lijst zit, wint de geladen waarde (write-back wordt genegeerd).

```zig
// Na alle transfers: update base register als write-back aan staat
if (wb) {
    // Bereken het nieuwe base adres
    cpu.r[rn] = if (up)
        base +% total_bytes
    else
        base -% total_bytes;
}
```

**Waarom `base +% total_bytes` en niet `addr`?** Het uiteindelijke `addr` na de loop bevat het adres ná de laatste transfer. Maar het correcte write-back adres is altijd `base ± (count × 4)`, ongeacht de pre/post modus. Gebruik de originele `base` waarde voor de berekening.

**Tip:** Een veelvoorkomende optimalisatie in ARM assembly is `STMDB SP!, {R4-R11, LR}` bij functie-entry (push registers) en `LDMIA SP!, {R4-R11, PC}` bij return (pop + return). Het `!` achter SP betekent write-back. Dit is het standaard functie-proloog/epiloog patroon dat de GCC compiler genereert.

### Stap 6: `src/core/arm.zig` — edge case: lege register list

**Waarom dit testen?** De ARM specificatie zegt dat een lege register list (alle bits 0) onvoorspelbaar gedrag geeft. In de praktijk laden/storen sommige ARM implementaties R15 en verhogen de base met 0x40. Dit is obscuur, maar sommige test-ROMs controleren het.

```zig
// Aan het begin van de functie:
if (reg_list == 0) {
    // Ongedefinieerd gedrag — ARM7TDMI specifiek:
    // Transfereert R15, base ± 0x40
    if (load) {
        cpu.r[15] = bus.read32(base);
        cpu.pipeline_flush(bus);
    } else {
        bus.write32(base, cpu.r[15] +% 4);
    }
    if (wb) {
        cpu.r[rn] = if (up) base +% 0x40 else base -% 0x40;
    }
    return;
}
```

**Tip:** Dit edge-case gedrag is ARM7TDMI-specifiek. Andere ARM versies doen iets anders. Implementeer het als je test-ROMs (zoals jsmolka's arm.gba) het vereisen; anders kun je het voorlopig overslaan en een `log.warn` gebruiken.
