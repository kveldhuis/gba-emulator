# Story 3.2 — CPU fetch-decode-execute loop

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** De basiscyclus implementeren: instructie ophalen → decoderen → uitvoeren.

## Zig concepten

### If/else als expressie

In Zig kan `if` een waarde teruggeven:

```zig
const instr_size: u32 = if (cpsr.t == 1) 2 else 4;
// Thumb = 2 bytes, ARM = 4 bytes
```

### De pipeline uitgelegd

**Waarom een pipeline?** Zonder pipeline zou de CPU voor elke instructie drie stappen achter elkaar moeten doen: fetch, decode, execute. Met een pipeline doet de CPU deze drie stappen *tegelijk* voor drie verschillende instructies — dit verdrievoudigt de doorvoer. Het nadeel is complexiteit, vooral bij branches (de pipeline moet "geflusht" worden).

De ARM7TDMI heeft een 3-stage pipeline:

```
Stap 1: FETCH    — haal volgende instructie op uit geheugen
Stap 2: DECODE   — ontcijfer wat de instructie is
Stap 3: EXECUTE  — voer de instructie uit
```

Dit betekent dat terwijl instructie N wordt uitgevoerd, instructie N+1 wordt gedecodeerd, en instructie N+2 wordt opgehaald. Het gevolg: **PC wijst altijd 2 instructies vooruit**.

```
In ARM mode (32-bit):   PC = adres van huidige instructie + 8
In Thumb mode (16-bit): PC = adres van huidige instructie + 4
```

Dit lijkt raar, maar het is hoe de hardware werkt. Als je PC leest in een instructie, krijg je niet het adres van die instructie maar het adres + 8 (of +4 in Thumb).

### Simpele step functie

```zig
pub fn step(self: *Cpu, bus: *Bus) void {
    if (self.cpsr.t == 1) {
        // Thumb mode: lees 16-bit instructie
        const instruction = bus.read16(self.r[15] - 4);  // PC - pipeline offset
        // decode en execute...
        self.r[15] += 2;
    } else {
        // ARM mode: lees 32-bit instructie
        const instruction = bus.read32(self.r[15] - 8);  // PC - pipeline offset
        // decode en execute...
        self.r[15] += 4;
    }
}
```

**Let op:** er zijn meerdere manieren om de pipeline te modelleren. Bovenstaande is een simpele variant. Je kunt ook de "huidige instructie" apart bijhouden los van PC.

## Stap-voor-stap

### Stap 1: `src/core/cpu.zig` — step() functie skelet

De CPU werkt als een klok: elke "tick" voert één instructie uit. De `step()` functie is die tick.

**Waarom een 3-stage pipeline?** De ARM7TDMI haalt een instructie op (fetch), ontcijfert de vorige (decode), en voert de instructie daarvoor uit (execute) — allemaal tegelijk. Dit maakt de CPU sneller, maar heeft een belangrijk gevolg: de PC wijst altijd 2 instructies vooruit ten opzichte van wat er nu uitgevoerd wordt.

**Waarom is PC+8 zo belangrijk?** Als een instructie op adres 0x100 wordt **uitgevoerd**, is de CPU al bezig met **fetchen** op adres 0x108 (twee instructies verder). Dus PC = 0x108. Dit betekent dat als de instructie PC leest, hij 0x108 ziet, niet 0x100. Dit pipeline-effect is vastgelegd in de ARM specificatie en games rekenen erop.

```zig
pub fn step(self: *Cpu, bus: *Bus) void {
    if (self.cpsr.t == 1) {
        self.thumbStep(bus);
    } else {
        self.armStep(bus);
    }
}
```

**Waarom ARM en Thumb splitsen?** ARM instructies zijn 32-bit, Thumb zijn 16-bit. Ze hebben verschillende decodering, verschillende pipeline-offsets (+8 vs +4), en lezen verschillende hoeveelheden data van de bus. Door ze te splitsen houd je de code overzichtelijk.

### Stap 2: `src/core/cpu.zig` — ARM fetch implementeren

De fetch-stap leest de instructie uit het geheugen op het adres waar de CPU "denkt" dat de huidige instructie staat.

**Waarom `self.r[15] - 8` als fetch-adres?** Er zijn twee gangbare strategieën om de pipeline te modelleren:

1. **PC loopt vooruit (aanbevolen):** PC bevat altijd de fetch-positie (= huidige instructie + 8). Bij fetch lees je op `PC - 8` om de huidige instructie te krijgen.
2. **PC volgt:** PC bevat het adres van de huidige instructie en je past +8 aan bij het lezen.

Strategie 1 is gangbaarder in emulators omdat het overeenkomt met wat de hardware doet:

```zig
fn armStep(self: *Cpu, bus: *Bus) void {
    // Fetch: lees de 32-bit instructie op de "werkelijke" uitvoerpositie
    const pc = self.r[15];
    const instruction = bus.read32(pc -% 8);

    // PC alvast verhogen voor de volgende cyclus
    self.r[15] +%= 4;

    // Condition check (story 3.3): sla instructie over als conditie niet klopt
    if (!checkCondition(self, instruction)) return;

    // Decode + execute (story 3.3): via lookup table
    arm.execute(self, bus, instruction);
}
```

**Waarom PC verhogen VOOR execute?** Als de instructie PC leest (bijv. `MOV R0, PC`), moet hij de pipeline-waarde zien (PC+8). Door PC al te verhogen vóór uitvoering, heeft `self.r[15]` automatisch de juiste waarde. Een branch-instructie overschrijft PC daarna gewoon.

**Valkuil:** Gebruik `+%` (wrapping add) en `-%` (wrapping subtract), niet `+` en `-`. Hoewel adressen in de praktijk niet overflown, crasht Zig anders als het theoretisch zou kunnen.

### Stap 3: `src/core/cpu.zig` — Thumb fetch (voorbereiding)

Thumb mode werkt hetzelfde maar met andere waardes:

```zig
fn thumbStep(self: *Cpu, bus: *Bus) void {
    const pc = self.r[15];
    const instruction = bus.read16(pc -% 4);  // Thumb pipeline offset = 4

    self.r[15] +%= 2;  // Thumb instructies zijn 2 bytes

    // Thumb instructies hebben GEEN condition code (behalve conditionele branches)
    // Decode + execute via Thumb lookup table (aparte epic)
    thumb.execute(self, bus, instruction);
}
```

**Waarom PC+4 in Thumb mode?** Thumb instructies zijn 16 bits (2 bytes) breed. De pipeline is nog steeds 3 stages, maar elke stage verwerkt 2 bytes. Dus PC = huidige instructie + 2 × 2 = +4.

**Tip:** Implementeer Thumb pas volledig in een latere epic. Voor nu is een stub genoeg die een warning logt.

### Stap 4: `src/core/cpu.zig` — Pipeline flush bij branches

Wanneer een branch PC verandert, moet de pipeline "geflusht" worden — de instructies die al gefetcht en gedecodeerd zijn, zijn niet meer geldig.

**Waarom is dit belangrijk?** Normaal gesproken wordt PC elke stap met 4 verhoogd (ARM) of 2 (Thumb). Maar bij een branch springt PC naar een heel ander adres. De al-verhoogde PC uit stap 2 is dan verkeerd.

```zig
pub fn set_reg(self: *Cpu, r: u4, value: u32) void {
    self.r[r] = value;
    if (r == 15) {
        // Pipeline flush: PC is veranderd, pipeline moet opnieuw gevuld worden
        // De nieuwe PC moet "vooruit" staan, dus voeg pipeline offset toe
        self.r[15] = value +% if (self.cpsr.t == 1) @as(u32, 4) else @as(u32, 8);
    }
}
```

**Waarom pipeline offset optellen bij een flush?** De step-functie verwacht dat PC altijd 8 (ARM) of 4 (Thumb) vooruit wijst. Als een branch naar adres 0x200 springt, moet PC op 0x208 gezet worden, zodat de volgende fetch correct op `0x208 - 8 = 0x200` uitkomt.

**Alternatieve aanpak:** Sommige emulators gebruiken een boolean `pipeline_flushed` flag. Na een branch wordt deze flag gezet, en de step-functie slaat dan de normale PC-verhoging over. Beide methodes werken; kies de variant die je het duidelijkst vindt.

### Stap 5: `src/main.zig` — CPU integratie testen

Voordat je verdergaat met decodering (story 3.3), test dat de basis-loop werkt:

```zig
// In main.zig of een test:
var cpu = Cpu.init();
var bus = Bus.init();

// Schrijf een simpele NOP-achtige instructie op het startadres
bus.write32(0x08000000, 0xE1A00000);  // MOV R0, R0 (NOP in ARM)

// Eén stap uitvoeren
cpu.step(&bus);

// PC moet 4 bytes verder zijn (van 0x08000008 naar 0x0800000C intern)
// Extern: de *volgende* instructie die uitgevoerd wordt staat op 0x08000004
```

**Waarom MOV R0, R0 als test?** Dit is de standaard ARM NOP (No Operation). De instructie doet niets behalve PC verhogen, waardoor je de basis-loop kunt testen zonder dat je complexe instructies hoeft te implementeren.

**Tip:** Begin met een stub `arm.execute()` die alleen een log print:
```zig
pub fn execute(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    _ = bus;
    std.log.warn("Onbekende ARM instructie: 0x{X:0>8} op PC=0x{X:0>8}", .{
        instruction, cpu.r[15] -% 12,  // -12 want PC is al verhoogd + pipeline
    });
}
```
