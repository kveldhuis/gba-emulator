# Story 3.6 — ARM Load/Store instructies

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** LDR, STR, LDRB, STRB implementeren (single data transfer).

## Zig concepten

### Bits uitlezen om instructievarianten te bepalen

Een ARM load/store instructie bevat meerdere vlaggen als losse bits:

```
Bit 25: I — 0=immediate offset, 1=register offset
Bit 24: P — 0=post-index, 1=pre-index
Bit 23: U — 0=offset aftrekken, 1=offset optellen
Bit 22: B — 0=word (32-bit), 1=byte
Bit 21: W — 0=geen write-back, 1=write base register terug
Bit 20: L — 0=store, 1=load
```

Om een enkele bit te testen:
```zig
const is_load  = (instruction >> 20) & 1 == 1;  // bit 20
const is_byte  = (instruction >> 22) & 1 == 1;  // bit 22
const add      = (instruction >> 23) & 1 == 1;  // bit 23
const pre      = (instruction >> 24) & 1 == 1;  // bit 24
```

### Pre-index vs post-index

```
Pre-index (P=1):   adres = base + offset, dan laad/sla op
                   optioneel: base = adres (als W=1, "write-back")

Post-index (P=0):  adres = base, laad/sla op, dan base = base + offset
                   (write-back altijd aan)
```

In code:
```zig
const base = cpu.r[rn];
const addr = if (pre) 
    if (add) base +% offset else base -% offset
else 
    base;

// Doe de load of store op addr...

// Post-index of write-back: update base register
if (!pre or write_back) {
    cpu.r[rn] = if (add) base +% offset else base -% offset;
}
```

### Unaligned reads — rotatie bij LDR

Op de GBA, als je een 32-bit LDR doet op een adres dat niet deelbaar is door 4, krijg je een **geroteerd** resultaat. Dit is raar maar belangrijk:

```zig
const aligned_addr = addr & 0xFFFFFFFC;  // rond af naar beneden op 4
const value = bus.read32(aligned_addr);
const rotation = (addr & 0x3) * 8;  // 0, 8, 16, of 24 bits
const result = std.math.rotr(u32, value, rotation);
// rotr = rotate right
```

### std.math.rotr — rotate right

```zig
// Rotate right: bits die rechts eruit vallen komen links terug
std.math.rotr(u32, 0x12345678, 8)  // = 0x78123456
```

## Context

**Waarom zijn load/store instructies zo belangrijk?** De ARM7TDMI is een "load/store architectuur" — dit betekent dat de ALU alleen op registers kan werken, NIET direct op geheugen. Om een waarde uit het geheugen te bewerken, moet je: (1) laden naar een register, (2) bewerken, (3) terugschrijven. Load/store instructies zijn dus de brug tussen de CPU en het geheugen.

- Adres = base register +/- offset (immediate 12-bit of shifted register)
- Pre/post indexing, write-back optioneel — dit maakt pointer-arithmetiek in één instructie mogelijk
- LDR: laad 32-bit woord van geheugen naar register
- STR: schrijf register naar geheugen
- LDRB/STRB: byte varianten (voor karakter-data, pixel-bytes, etc.)

## Bestanden

  - Adresberekening: base +/- offset, pre/post index
  - LDR doet een geroteerde read bij unaligned adres

## Stap-voor-stap

### Stap 1: `src/core/arm.zig` — Instructie-bits decoderen

Eerst moet je de vele vlag-bits uit de instructie halen. Een ARM load/store instructie bevat 6 losse bits die het gedrag bepalen.

**Waarom zoveel bits?** ARM probeert maximale flexibiliteit in één instructie te proppen. Met deze 6 bits kun je in één instructie: laden OF opslaan, een byte OF een word, met optelling OF aftrekking van de offset, vóór OF na de geheugen-operatie, met OF zonder write-back naar het basisregister. Dat zijn 64 combinaties in één instructie-formaat.

```zig
fn singleDataTransfer(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    // Decodeer alle vlag-bits
    const i_bit  = (instruction >> 25) & 1 == 1;  // bit 25: 0=immediate, 1=register offset
    const pre    = (instruction >> 24) & 1 == 1;  // bit 24: pre/post indexing
    const add    = (instruction >> 23) & 1 == 1;  // bit 23: offset optellen of aftrekken
    const byte   = (instruction >> 22) & 1 == 1;  // bit 22: byte of word transfer
    const wb     = (instruction >> 21) & 1 == 1;  // bit 21: write-back
    const load   = (instruction >> 20) & 1 == 1;  // bit 20: load of store

    const rn: u4 = @truncate(instruction >> 16);   // base register
    const rd: u4 = @truncate(instruction >> 12);   // source/dest register

    // ... (zie volgende stappen)
}
```

**Waarom is de I-bit "omgekeerd"?** Bij data processing is bit 25 = 1 voor immediate. Bij load/store is bit 25 = 1 voor **register** offset. Dit is verwarrend maar het staat zo in de ARM specificatie. Let goed op bij het decoderen!

### Stap 2: `src/core/arm.zig` — Offset berekenen

De offset kan een immediate waarde of een geshifted register zijn.

**Waarom twee vormen?** Immediate is de meest voorkomende: `LDR R0, [R1, #4]` laadt van adres R1+4. Register offset is krachtig voor array-indexering: `LDR R0, [R1, R2, LSL #2]` laadt van adres R1 + R2×4 (handig voor een array van u32's).

```zig
    // Bereken de offset
    const offset: u32 = if (!i_bit) blk: {
        // Immediate: 12-bit unsigned waarde (bits [11:0])
        break :blk instruction & 0xFFF;
    } else blk: {
        // Register: Rm (bits [3:0]) geshifted
        const rm: u4 = @truncate(instruction);
        const shift_type: u2 = @truncate(instruction >> 5);
        const shift_amount: u5 = @truncate(instruction >> 7);
        const rm_value = cpu.get_reg(rm);
        const result = barrelShift(rm_value, shift_type, shift_amount, cpu.cpsr.c);
        break :blk result.result;
    };
```

**Waarom geen register-specified shift amount?** Anders dan bij data processing, ondersteunt load/store alleen een immediate shift amount (5-bit, bits [11:7]). Register-specified shifts (bit 4 = 1) zijn niet beschikbaar bij load/store. Dit vereenvoudigt de hardware.

### Stap 3: `src/core/arm.zig` — Adresberekening met pre/post-indexing

Dit is waar de P-bit en U-bit samenkomen.

**Waarom pre/post indexing?** Dit is essentieel voor efficiënte stack-operaties en array-traversal:
- **Pre-index** (`LDR R0, [R1, #4]`): bereken adres eerst, gebruik dan. Perfect voor struct-velden: `R1` wijst naar de struct, `#4` is de offset van het veld.
- **Post-index** (`LDR R0, [R1], #4`): gebruik het basisadres, verhoog dan. Perfect voor array-traversal: laad element, schuif pointer op.

```zig
    const base = cpu.get_reg(rn);

    // Pre-index: pas offset toe VOOR de geheugen-operatie
    const addr: u32 = if (pre)
        if (add) base +% offset else base -% offset
    else
        base;  // Post-index: gebruik ongewijzigd basisadres
```

**Waarom wrapping arithmetic (+%, -%)?** Adressen zijn unsigned 32-bit waarden. In theorie zou een aftrekking onder 0 kunnen duiken, wat op hardware gewoon wrapt. We gebruiken `+%` en `-%` om dit gedrag exact te matchen.

### Stap 4: `src/core/arm.zig` — LDR en STR uitvoeren

Nu de daadwerkelijke geheugen-operatie:

**Waarom roteren bij unaligned LDR?** Dit is specifiek ARM7TDMI hardware-gedrag. Als je `LDR R0, [0x03]` doet (adres niet deelbaar door 4), leest de hardware wél 4 bytes vanaf het aligned adres (0x00), maar **roteert** het resultaat zodat de byte op het gevraagde adres in de onderste positie komt. Dit is GEEN fout — games gebruiken dit soms bewust.

Andere architecturen (x86) zouden gewoon de bytes op het juiste adres laden. MIPS zou een exception geven. ARM koos voor deze rare rotatie.

```zig
    if (load) {
        if (byte) {
            // LDRB: lees 1 byte, zero-extend naar 32 bits
            const value: u32 = bus.read8(addr);
            cpu.set_reg(rd, value);
        } else {
            // LDR: lees 32-bit word, met rotatie bij unaligned
            const aligned_addr = addr & 0xFFFFFFFC;  // rond af naar 4-byte grens
            var value = bus.read32(aligned_addr);

            // Unaligned rotatie: draai de bytes zodat de gevraagde byte onderaan staat
            const rotation = (addr & 0x3) * 8;  // 0, 8, 16, of 24 bits
            if (rotation != 0) {
                value = std.math.rotr(u32, value, @as(u32, rotation));
            }

            cpu.set_reg(rd, value);
        }
    } else {
        // STR / STRB
        const value = cpu.get_reg(rd);
        if (byte) {
            // STRB: schrijf onderste byte
            bus.write8(addr, @truncate(value));
        } else {
            // STR: schrijf 32-bit word (adres wordt afgekapt op 4-byte grens)
            bus.write32(addr & 0xFFFFFFFC, value);
        }
    }
```

**Waarom zero-extend bij LDRB?** LDRB leest 1 byte en plaatst het in de onderste 8 bits van het register. De bovenste 24 bits worden met nullen gevuld. Dit is "zero extension" — het getal wordt als unsigned behandeld. (Voor signed extension heb je LDRSB nodig, dat in een latere story komt via halfword transfers.)

**Valkuil:** Bij `STR` met `Rd = R15` (PC), is de geschreven waarde PC+12 (in plaats van PC+8). Dit is een eigenaardigheid van de pipeline. Sommige ARM referenties zeggen +8, sommige +12 — check de ARM7TDMI specifieke documentatie.

### Stap 5: `src/core/arm.zig` — Write-back en post-indexing

Na de geheugen-operatie moet het basisregister mogelijk worden bijgewerkt:

**Waarom write-back?** Write-back is een optimalisatie: in plaats van apart een ADD/SUB te doen om de pointer te verschuiven, doet de load/store instructie dit in één keer mee. Dit bespaart één instructie (en dus clock cycles) bij elke array-toegang.

```zig
    // Write-back: update base register
    if (!pre) {
        // Post-index: write-back is altijd actief
        cpu.set_reg(rn, if (add) base +% offset else base -% offset);
    } else if (wb) {
        // Pre-index met W-bit: write-back expliciet gevraagd
        cpu.set_reg(rn, addr);
    }
```

**Waarom is post-index altijd write-back?** Bij post-indexing wordt het basisadres eerst gebruikt (zonder offset) en daarna bijgewerkt. Het zou nutteloos zijn om de offset te berekenen en dan niet toe te passen — de W-bit wordt bij post-index genegeerd (effectief altijd 1).

**Valkuil:** Als `Rn == Rd` bij een load met write-back, wat heeft voorrang? De ARM specificatie zegt dat het resultaat "UNPREDICTABLE" is. In de praktijk overschrijft de load het register en gaat de write-back verloren. Test dit geval en kies één gedrag.

### Stap 6: `src/core/arm.zig` — Testen schrijven

```zig
test "LDR R0, [R1] — simpele load" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    // Schrijf testwaarde in geheugen
    bus.write32(0x03000000, 0xDEADBEEF);
    cpu.r[1] = 0x03000000;
    // E5910000 = LDR R0, [R1] (pre, add, word, no write-back)
    singleDataTransfer(&cpu, &bus, 0xE5910000);
    try std.testing.expectEqual(@as(u32, 0xDEADBEEF), cpu.r[0]);
}

test "STR R0, [R1, #4] — store met offset" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    cpu.r[0] = 0x12345678;
    cpu.r[1] = 0x03000000;
    // E5810004 = STR R0, [R1, #4]
    singleDataTransfer(&cpu, &bus, 0xE5810004);
    try std.testing.expectEqual(@as(u32, 0x12345678), bus.read32(0x03000004));
}

test "LDR unaligned roteert" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    bus.write32(0x03000000, 0x04030201);
    cpu.r[1] = 0x03000001;  // unaligned: 1 byte verschoven
    // E5910000 = LDR R0, [R1]
    singleDataTransfer(&cpu, &bus, 0xE5910000);
    // Verwacht: geroteerd met 8 bits → 0x01040302
    try std.testing.expectEqual(@as(u32, 0x01040302), cpu.r[0]);
}

test "LDR post-index update base" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    bus.write32(0x03000000, 0xAAAAAAAA);
    cpu.r[1] = 0x03000000;
    // E4910004 = LDR R0, [R1], #4 (post-index, add offset 4)
    singleDataTransfer(&cpu, &bus, 0xE4910004);
    try std.testing.expectEqual(@as(u32, 0xAAAAAAAA), cpu.r[0]);
    try std.testing.expectEqual(@as(u32, 0x03000004), cpu.r[1]);  // base updated
}

test "LDRB zero-extends" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    bus.write8(0x03000000, 0xFF);
    cpu.r[1] = 0x03000000;
    // E5D10000 = LDRB R0, [R1]
    singleDataTransfer(&cpu, &bus, 0xE5D10000);
    try std.testing.expectEqual(@as(u32, 0x000000FF), cpu.r[0]);
}
```

**Tip:** Unaligned access is een veelvoorkomende bron van subtiele bugs. Test alle 4 de alignment-offsets (0, 1, 2, 3) om er zeker van te zijn dat de rotatie correct is.

**Tip:** Gebruik de GBA testROMs (zoals jsmolka's arm-tests) om je load/store implementatie te valideren tegen de echte hardware.
