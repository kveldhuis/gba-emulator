## Zig concepten

### Packed structs — bits exact uitleggen

Een packed struct vertelt Zig precies welke bits bij welk veld horen. Ideaal voor hardware registers.

```zig
const PSR = packed struct(u32) {
    mode: u5,        // bits [4:0]   — processor mode
    t: u1,           // bit [5]      — 0=ARM, 1=Thumb
    f: u1,           // bit [6]      — FIQ disable
    i: u1,           // bit [7]      — IRQ disable
    _reserved: u20,  // bits [27:8]  — niet gebruikt
    v: u1,           // bit [28]     — overflow flag
    c: u1,           // bit [29]     — carry flag
    z: u1,           // bit [30]     — zero flag
    n: u1,           // bit [31]     — negative flag
};
```

De velden worden van laag naar hoog ingevuld: `mode` zit op de onderste 5 bits, `n` op de bovenste bit. De totale breedte is precies 32 bits.

Je kunt een packed struct casten van/naar het onderliggende type:
```zig
var cpsr = PSR{ .mode = 0x1F, .t = 0, .z = 0, .n = 0, ... };
cpsr.z = 1;                        // zet zero flag
const raw: u32 = @bitCast(cpsr);   // als gewone u32 behandelen
const back: PSR = @bitCast(raw);   // en weer terug
```

### Enums — benoemde waarden

```zig
const Mode = enum(u5) {
    user       = 0b10000,  // 0x10
    fiq        = 0b10001,  // 0x11
    irq        = 0b10010,  // 0x12
    supervisor = 0b10011,  // 0x13
    abort      = 0b10111,  // 0x17
    undefined  = 0b11011,  // 0x1B
    system     = 0b11111,  // 0x1F
};
```

Een enum met `(u5)` is exact 5 bits breed — past in het `mode` veld van de PSR.

### Arrays initialiseren

```zig
// Array van 16 u32's, allemaal 0
var registers: [16]u32 = [_]u32{0} ** 16;

// [_] = "leid de grootte af"
// {0} = de waarde
// ** 16 = herhaal 16 keer
```

### Banked registers uitgelegd

Sommige CPU modes hebben hun eigen versie van bepaalde registers. Als je naar IRQ mode switcht, worden R13 en R14 vervangen door aparte R13_irq en R14_irq. De oude waarden worden bewaard en hersteld als je terug switcht.

```zig
const Cpu = struct {
    r: [16]u32,                  // actieve registers
    cpsr: PSR,
    spsr: [5]PSR,                // saved PSR per mode (niet User/System)
    banked_sp: [6]u32,           // opgeslagen SP per mode
    banked_lr: [6]u32,           // opgeslagen LR per mode
    // FIQ heeft ook banked R8-R12:
    banked_fiq: [5]u32,          // R8_fiq t/m R12_fiq
    banked_fiq_old: [5]u32,      // R8-R12 die FIQ vervangt
};
```

## Context — ARM7TDMI registers

De ARM7TDMI is de CPU in de Game Boy Advance. Het is een 32-bit RISC processor, wat betekent dat de instructies eenvoudig en uniform zijn (in tegenstelling tot x86 die CISC is). "TDMI" staat voor: **T**humb (16-bit instructie-set), **D**ebug support, **M**ultiplier (ingebouwde vermenigvuldiger), **I**CE (in-circuit emulator).

- 16 general-purpose registers: R0-R15
  - R0-R12 — vrij te gebruiken voor berekeningen
  - R13 = Stack Pointer (SP) — wijst naar de top van de stack (conventie, niet hardware-afgedwongen)
  - R14 = Link Register (LR) — bewaart return address bij functie-aanroep (BL instructie)
  - R15 = Program Counter (PC) — adres van de huidige instructie + pipeline offset
- CPSR (Current Program Status Register):
  - N (negative), Z (zero), C (carry), V (overflow) — rekenresultaat flags, gebruikt door conditionele instructies
  - T bit — 0=ARM mode (32-bit instructies), 1=Thumb mode (16-bit instructies)
  - I en F bits — IRQ en FIQ interrupt disable (1=uit, 0=aan)
  - Mode bits [4:0] — in welke processor mode je zit (bepaalt welke banked registers actief zijn)
- SPSR (Saved Program Status Register) — kopie van CPSR bij een mode switch, zodat de originele state hersteld kan worden
- Banked registers per processor mode (elk mode heeft eigen SP, LR, en SPSR — dit maakt snelle context switches mogelijk zonder software overhead)

## Stap-voor-stap

### Stap 1: `src/core/cpu.zig` — PSR packed struct definiëren

De CPSR (Current Program Status Register) is een 32-bit hardware register waar elke bit een specifieke betekenis heeft. Een packed struct is hiervoor de perfecte Zig-constructie.

**Waarom een packed struct en geen gewone struct?**
Een gewone struct in Zig mag padding toevoegen tussen velden voor alignment. Een `packed struct(u32)` garandeert dat de velden exact de bits innemen die je opgeeft, zonder padding. Dit is cruciaal omdat je de PSR moet kunnen casten naar/van een `u32` — dat is hoe de hardware het register behandelt. Met `@bitCast` kun je naadloos wisselen tussen de struct-weergave (voor individuele velden) en de u32-weergave (voor het hele register).

```zig
const PSR = packed struct(u32) {
    mode: u5 = 0x1F,    // bits [4:0]   — processor mode (System default)
    t: u1 = 0,          // bit [5]      — 0=ARM, 1=Thumb
    f: u1 = 0,          // bit [6]      — FIQ disable
    i: u1 = 0,          // bit [7]      — IRQ disable
    _reserved: u20 = 0, // bits [27:8]  — niet gebruikt op GBA
    v: u1 = 0,          // bit [28]     — overflow flag
    c: u1 = 0,          // bit [29]     — carry flag
    z: u1 = 0,          // bit [30]     — zero flag
    n: u1 = 0,          // bit [31]     — negative flag
};
```

**Waarom default waarden meegeven?** Dit maakt `PSR{}` bruikbaar als initialisatie (alle velden 0 behalve mode). Zonder defaults moet je elk veld handmatig invullen bij het aanmaken.

**Valkuil:** De volgorde van velden in een packed struct bepaalt de bit-positie. Het eerste veld (`mode`) zit op de laagste bits [4:0], het laatste veld (`n`) op de hoogste bit [31]. Dit is de omgekeerde richting van hoe je het op papier leest!

**Tip:** Test dat `@bitCast` correct werkt met een simpele controle:
```zig
var psr = PSR{};
psr.z = 1;
const raw: u32 = @bitCast(psr);
// raw moet 0x4000001F zijn: bit 30 (Z) + mode bits
std.debug.assert(raw == 0x4000001F);
```

### Stap 2: `src/core/cpu.zig` — Mode enum definiëren

**Waarom een enum met (u5)?** Het mode-veld in de PSR is exact 5 bits breed. Door de enum ook 5 bits te maken, past hij direct in het `mode`-veld van de PSR zonder conversie. Zig garandeert dat de enum-waarden in 5 bits passen.

```zig
pub const Mode = enum(u5) {
    user       = 0b10000,  // 0x10 — normaal programma
    fiq        = 0b10001,  // 0x11 — snelle interrupt
    irq        = 0b10010,  // 0x12 — normale interrupt
    supervisor = 0b10011,  // 0x13 — software interrupt (SWI)
    abort      = 0b10111,  // 0x17 — geheugen fout
    undefined  = 0b11011,  // 0x1B — onbekende instructie
    system     = 0b11111,  // 0x1F — bevoorrecht, zelfde registers als user
};
```

**Waarom een 5-bit mode veld?** Dit is een ARM hardware-keuze. Met 5 bits kun je 32 modes adresseren, maar ARM gebruikt er maar 7. De specifieke bitpatronen (allemaal met bit 4 aan = `1xxxx`) zijn gekozen zodat illegale mode-waarden makkelijk detecteerbaar zijn.

**Waarom deze specifieke modes?** Elke mode correspondeert met een andere "reden" waarom de CPU is onderbroken:
- **User/System:** normaal programma — System is dezelfde registers maar met privileges
- **IRQ/FIQ:** hardware interrupts — FIQ ("Fast") heeft extra banked registers voor snellere afhandeling
- **Supervisor:** software interrupt (SWI instructie) — gebruikt door het BIOS
- **Abort:** geheugen fout — komt op GBA zelden voor
- **Undefined:** onbekende instructie — ook zeldzaam op GBA

**Valkuil:** Niet alle 5-bit waarden zijn geldige modes. Als je een u5 uit de PSR leest, kan het een illegale waarde zijn. Gebruik `@enumFromInt` met foutafhandeling:
```zig
const mode = std.meta.intToEnum(Mode, psr.mode) catch {
    @panic("Ongeldige CPU mode");
};
```

### Stap 3: `src/core/cpu.zig` — Cpu struct met banked registers

**Waarom banked registers?** De ARM hardware heeft een slimme truc: bij een mode-switch hoeft de CPU niet alle registers te saven en restoren. In plaats daarvan heeft elke mode **eigen fysieke registers** voor SP (R13) en LR (R14). Als je naar IRQ mode switcht, worden R13 en R14 "onzichtbaar" vervangen door R13_irq en R14_irq. De oude waarden staan veilig in hun eigen fysieke registers.

Dit is waarom interrupt handling zo snel is op ARM: de hardware doet de context switch automatisch voor SP en LR.

```zig
pub const Cpu = struct {
    /// De 16 actieve registers (R0-R15). R15 = PC.
    r: [16]u32,

    /// Current Program Status Register
    cpsr: PSR,

    /// Saved PSR per exception mode (niet User/System)
    /// Index: 0=FIQ, 1=IRQ, 2=Supervisor, 3=Abort, 4=Undefined
    spsr: [5]PSR,

    /// Banked SP (R13) per mode
    /// Index: 0=User/System, 1=FIQ, 2=IRQ, 3=Supervisor, 4=Abort, 5=Undefined
    banked_sp: [6]u32,

    /// Banked LR (R14) per mode
    banked_lr: [6]u32,

    /// FIQ heeft ook banked R8-R12 (voor extra snelheid)
    banked_fiq_r8_r12: [5]u32,
    /// De "gewone" R8-R12 die FIQ tijdelijk vervangt
    banked_usr_r8_r12: [5]u32,
};
```

**Waarom heeft FIQ extra banked registers (R8-R12)?** FIQ staat voor "Fast Interrupt Request". Het idee is dat een FIQ handler 5 extra registers heeft die niet gesaved hoeven te worden, waardoor de interrupt nóg sneller afgehandeld wordt. Op de GBA wordt FIQ nauwelijks gebruikt, maar je moet het wel correct implementeren.

**Waarom 6 entries voor banked_sp/banked_lr?** Eén per mode: User/System (delen dezelfde registers), FIQ, IRQ, Supervisor, Abort, Undefined. User en System delen dezelfde registers — dat is het hele punt van System mode (privileged maar dezelfde register bank als User).

**Tip:** Maak een helper functie om van Mode naar array-index te converteren:
```zig
fn modeToIndex(mode: Mode) u3 {
    return switch (mode) {
        .user, .system => 0,
        .fiq => 1,
        .irq => 2,
        .supervisor => 3,
        .abort => 4,
        .undefined => 5,
    };
}
```

### Stap 4: `src/core/cpu.zig` — init() functie

**Waarom begint de GBA op 0x08000000?** Na het uitvoeren van het BIOS (dat begint op 0x00000000) springt de GBA naar 0x08000000, het begin van het Game Pak ROM. Als je het BIOS overslaat (wat de meeste emulators doen), moet je de CPU direct in deze staat initialiseren.

**Waarom System mode?** Na het BIOS draait de GBA in System mode, wat privileged is maar dezelfde registers als User mode gebruikt. Dit is de "standaard" toestand na boot.

```zig
pub fn init() Cpu {
    var cpu = Cpu{
        .r = [_]u32{0} ** 16,
        .cpsr = PSR{},                      // mode = 0x1F (System), T=0 (ARM)
        .spsr = [_]PSR{PSR{}} ** 5,
        .banked_sp = [_]u32{0} ** 6,
        .banked_lr = [_]u32{0} ** 6,
        .banked_fiq_r8_r12 = [_]u32{0} ** 5,
        .banked_usr_r8_r12 = [_]u32{0} ** 5,
    };
    // GBA staat na BIOS: PC op ROM start, SP's geïnitialiseerd
    cpu.r[15] = 0x08000000;
    // De GBA BIOS initialiseert de stack pointers voor elke mode:
    cpu.banked_sp[modeToIndex(.irq)] = 0x03007FA0;
    cpu.banked_sp[modeToIndex(.supervisor)] = 0x03007FE0;
    cpu.r[13] = 0x03007F00;  // SP voor User/System mode
    return cpu;
}
```

**Waarom die specifieke SP-waarden?** Dit zijn de adressen die het GBA BIOS instelt. IWRAM (intern werkgeheugen) loopt van 0x03000000 tot 0x03007FFF, en de stacks groeien naar beneden (van hoog naar laag adres). De BIOS reserveert de bovenste bytes van IWRAM voor de stacks van verschillende modes.

**Valkuil:** Vergeet niet dat PC (R15) in ARM mode altijd 8 bytes vooruit wijst door de pipeline. Maar bij initialisatie zet je PC op het startadres — de pipeline-offset wordt pas relevant als je instructies gaat uitvoeren (story 3.2).

### Stap 5: `src/core/cpu.zig` — get_reg en set_reg

**Waarom aparte functies voor register access?** R15 (PC) is speciaal: als je PC leest in een instructie, geeft de hardware PC+8 terug (in ARM mode) door de pipeline. Door register access via functies te laten lopen, kun je dit op één plek afhandelen. Direct `cpu.r[15]` lezen zou de pipeline-offset missen.

```zig
pub fn get_reg(self: *const Cpu, r: u4) u32 {
    if (r == 15) {
        // PC lezen: pipeline effect — instructie ziet PC+8 (ARM) of PC+4 (Thumb)
        return self.r[15];  // PC bevat al het "vooruit" adres
    }
    return self.r[r];
}

pub fn set_reg(self: *Cpu, r: u4, value: u32) void {
    self.r[r] = value;
    if (r == 15) {
        // PC is aangepast — dit veroorzaakt een pipeline flush
        // De fetch-decode-execute loop moet hiermee rekening houden
    }
}
```

**Waarom u4 als register index?** ARM heeft 16 registers (R0-R15), genummerd 0 t/m 15. Dat past precies in 4 bits (u4). Zig garandeert dan op type-niveau dat je nooit een ongeldig register-nummer kunt gebruiken — geen runtime check nodig.

**Tip:** De pipeline-offset is een bron van veel bugs in emulators. Een veelgebruikte aanpak is om bij het begin van elke instructie PC al te verhogen, zodat `get_reg(15)` automatisch de juiste waarde geeft. Dit wordt verder uitgewerkt in story 3.2.

### Stap 6: `src/core/cpu.zig` — switch_mode functie

Dit is het meest complexe deel: bij een mode-switch moeten de banked registers gewisseld worden.

**Waarom deze volgorde?** Je moet eerst de huidige registers **opslaan** in de bank van de oude mode, en dan de registers **laden** uit de bank van de nieuwe mode. Als je de volgorde omdraait, overschrijf je de oude waarden.

```zig
pub fn switchMode(self: *Cpu, new_mode: Mode) void {
    const old_mode = std.meta.intToEnum(Mode, self.cpsr.mode) catch return;
    if (old_mode == new_mode) return;  // geen switch nodig

    const old_idx = modeToIndex(old_mode);
    const new_idx = modeToIndex(new_mode);

    // 1. Sla huidige SP en LR op in de bank van de OUDE mode
    self.banked_sp[old_idx] = self.r[13];
    self.banked_lr[old_idx] = self.r[14];

    // 2. Laad SP en LR van de NIEUWE mode
    self.r[13] = self.banked_sp[new_idx];
    self.r[14] = self.banked_lr[new_idx];

    // 3. FIQ speciale behandeling: R8-R12 wisselen
    if (old_mode == .fiq) {
        // Verlaat FIQ: sla FIQ's R8-R12 op, herstel gewone R8-R12
        for (0..5) |j| {
            self.banked_fiq_r8_r12[j] = self.r[8 + j];
            self.r[8 + j] = self.banked_usr_r8_r12[j];
        }
    } else if (new_mode == .fiq) {
        // Ga naar FIQ: sla gewone R8-R12 op, laad FIQ's R8-R12
        for (0..5) |j| {
            self.banked_usr_r8_r12[j] = self.r[8 + j];
            self.r[8 + j] = self.banked_fiq_r8_r12[j];
        }
    }

    // 4. Update mode in CPSR
    self.cpsr.mode = @intFromEnum(new_mode);
}
```

**Waarom FIQ apart behandelen?** Alle modes banken alleen R13 (SP) en R14 (LR). Maar FIQ bankt óók R8 t/m R12 — vijf extra registers. Dit betekent dat je bij het wisselen naar/van FIQ mode extra werk moet doen. Let op dat je dit alleen doet als de oude OF nieuwe mode FIQ is, niet bij een switch tussen bijv. IRQ en Supervisor.

**Valkuil:** Vergeet niet de SPSR (Saved Program Status Register). Bij een exception slaat de hardware de huidige CPSR op als SPSR van de nieuwe mode. Dit hoeft niet in `switchMode` zelf (de exception handler doet dat), maar houd er rekening mee dat de SPSR bij het verlaten van een exception mode teruggezet moet worden naar CPSR.

**Test je mode switch:**
```zig
test "mode switch behoudt registers" {
    var cpu = Cpu.init();
    cpu.r[13] = 0xAAAAAAAA;  // SP in System mode
    cpu.switchMode(.irq);
    // SP is nu IRQ's SP (0x03007FA0 na init)
    try std.testing.expectEqual(@as(u32, 0x03007FA0), cpu.r[13]);
    cpu.switchMode(.system);
    // SP is weer System's SP
    try std.testing.expectEqual(@as(u32, 0xAAAAAAAA), cpu.r[13]);
}
```
