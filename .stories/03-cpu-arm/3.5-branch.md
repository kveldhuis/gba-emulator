# Story 3.5 — ARM Branch instructies

**Epic:** 03 - CPU — ARM7TDMI Basis

**Doel:** B, BL, en BX implementeren.

## Zig concepten

### Sign extension — een klein signed getal groter maken

Een branch offset in ARM is 24 bits breed, maar signed (kan negatief zijn voor backward jumps). Je moet het uitbreiden naar 32 bits met behoud van het teken.

**Hoe werkt signed in binair?**
Het bovenste bit is het "teken-bit": 0 = positief, 1 = negatief.

```
8-bit voorbeeld:
0b00000101 = +5
0b11111011 = -5   (two's complement)
```

Als je een 8-bit signed getal naar 16 bits wilt uitbreiden, moet je het teken-bit herhalen:

```
+5:  0b00000101 → 0b0000000000000101  (nullen erbij)
-5:  0b11111011 → 0b1111111111111011  (enen erbij)
```

In Zig:
```zig
// Methode 1: via @as en @intCast met signed types
const offset_24: u24 = @truncate(instruction);     // pak 24 bits
const signed: i24 = @bitCast(offset_24);            // interpreteer als signed
const extended: i32 = signed;                        // Zig extend het teken automatisch
const result: u32 = @bitCast(extended);              // terug naar unsigned voor adresberekening

// Methode 2: handmatig (voor begrip)
// Als bit 23 (teken-bit van 24-bit getal) == 1, vul bovenste 8 bits met 1en
var offset: u32 = instruction & 0x00FFFFFF;
if (offset & 0x800000 != 0) {
    offset |= 0xFF000000;  // vul bovenste bits met 1en
}
```

### Shift left voor vermenigvuldiging

`<< 2` is hetzelfde als vermenigvuldigen met 4. ARM branch offsets worden altijd met 4 vermenigvuldigd omdat instructies op 4-byte grenzen staan:

```zig
// offset van 24-bit signed, shift left 2 (×4)
const jump: u32 = sign_extended_offset << 2;
// PC = PC + jump
```

### Bitwise AND met complement voor alignment

BX springt naar een adres in een register. Bit 0 bepaalt ARM/Thumb, maar het adres zelf moet aligned zijn:

```zig
const addr = cpu.r[rm];
cpu.cpsr.t = @truncate(addr);      // bit 0 → Thumb flag
cpu.r[15] = addr & 0xFFFFFFFE;     // bit 0 wegmasken (= ~1)
// 0xFFFFFFFE = alle bits behalve bit 0
```

## Context

**Waarom drie soorten branches?** Elke variant bedient een ander scenario:
- `B offset` — PC-relatieve sprong (offset is signed, 24-bit, <<2). Gebruikt voor if/else, loops, en simpele jumps. De offset is relatief aan PC, wat position-independent code mogelijk maakt (de code werkt op elk adres).
- `BL offset` — zelfde maar slaat return address op in LR (R14) voor terugkeren. Dit is de standaard functie-aanroep instructie. De "L" staat voor "Link" — het linkt het return-adres.
- `BX Rm` — springt naar adres in Rm, bit 0 bepaalt ARM/Thumb mode switch. Dit is de enige manier om tussen ARM en Thumb mode te wisselen. De GBA BIOS gebruikt BX om van ARM naar Thumb te schakelen.

## Bestanden

- `src/core/arm.zig` — branch handlers
  - B/BL: sign-extend 24-bit offset, shift left 2, tel op bij PC
  - BX: zet T-bit in CPSR op basis van Rm[0], PC = Rm & ~1

## Stap-voor-stap

### Stap 1: `src/core/arm.zig` — B (Branch) implementeren

De eenvoudigste branch: spring naar een ander adres, relatief aan de huidige PC.

**Waarom sign extension bij de offset?** De branch offset is een 24-bit signed getal. "Signed" betekent dat het ook negatief kan zijn (voor backward jumps, bijv. loops). Maar PC-berekeningen werken op 32-bit. Als je een negatief 24-bit getal simpelweg in een u32 zet, wordt het een groot positief getal (bijv. `0x00FFFFFF` in plaats van `-1`). Sign extension herhaalt het teken-bit (bit 23) naar bits 24-31, zodat de negatieve waarde correct bewaard blijft.

**Waarom shift left 2?** ARM instructies zijn altijd 4 bytes (32-bit) aligned. Het heeft dus geen zin om de onderste 2 bits van het adres op te slaan (die zijn altijd 00). Door de offset te shiften met 2 (= ×4), krijg je een bereik van ±32MB in plaats van ±8MB. Slimme encoding!

```zig
fn branch(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    _ = bus;
    const link = (instruction >> 24) & 1 == 1;  // bit 24: L=1 → BL

    // Stap 1: Pak de 24-bit offset
    const offset_24: u24 = @truncate(instruction);

    // Stap 2: Sign extend naar 32 bits
    const signed_offset: i32 = @as(i32, @as(i24, @bitCast(offset_24)));

    // Stap 3: Shift left 2 (×4 voor word-alignment)
    const jump: i32 = signed_offset << 2;

    // Stap 4: BL slaat return address op in LR
    if (link) {
        // LR = adres van de instructie NA de BL (= PC - 4, want PC is al verhoogd)
        cpu.r[14] = cpu.r[15] -% 4;
    }

    // Stap 5: Pas PC aan
    // PC bevat al de pipeline-waarde, dus gewoon optellen
    const new_pc: u32 = @bitCast(@as(i32, @bitCast(cpu.r[15])) +% jump);
    cpu.r[15] = new_pc;
}
```

**Waarom `LR = PC - 4`?** De PC is al verhoogd in de step-functie (story 3.2). Op het moment van uitvoering wijst PC naar de instructie NA de branch + pipeline offset. `PC - 4` geeft het adres van de instructie direct na de BL, wat exact is waar de functie naar moet terugkeren.

**Valkuil:** De sign extension chain is lang: `u24 → i24 → i32`. Elke stap is noodzakelijk:
1. `@truncate` naar u24: pak de 24 bits uit de instructie
2. `@bitCast` naar i24: interpreteer als signed (bit 23 wordt teken-bit)
3. `@as(i32, ...)`: Zig extend het teken automatisch naar 32 bits

### Stap 2: `src/core/arm.zig` — BL (Branch with Link) begrijpen

BL is identiek aan B maar slaat eerst het return-adres op in R14 (Link Register).

**Waarom is BL nodig?** BL is hoe ARM functie-aanroepen doet. Na een `BL subroutine` staat het terugkeer-adres in LR. De subroutine kan terugkeren met `BX LR` of `MOV PC, LR`. Zonder BL zou je handmatig het return adres moeten berekenen en opslaan.

```
Voorbeeld: functie aanroep
    0x100: BL 0x200     ← spring naar 0x200, LR = 0x104
    0x104: ...          ← hier gaat het verder na return

    0x200: PUSH {LR}    ← sla LR op (als functie andere functies aanroept)
    ...
    0x220: POP {PC}     ← spring terug naar 0x104
```

**Tip:** In de bovenstaande implementatie in stap 1 is BL al ingebouwd via de `link`-check. B en BL delen dezelfde handler — het enige verschil is het opslaan van LR.

### Stap 3: `src/core/arm.zig` — BX (Branch and Exchange) implementeren

BX is fundamenteel anders dan B/BL: het springt naar een adres in een **register** en kan wisselen tussen ARM en Thumb mode.

**Waarom checkt BX de T-bit?** De ARM7TDMI ondersteunt twee instructie-sets: ARM (32-bit) en Thumb (16-bit). BX is het mechanisme om tussen de twee te wisselen. Bit 0 van het doel-adres bepaalt de mode:
- Bit 0 = 0 → ARM mode (instructies op 4-byte grenzen)
- Bit 0 = 1 → Thumb mode (instructies op 2-byte grenzen)

Het adres zelf moet aligned zijn, dus bit 0 wordt niet meegenomen als adres.

```zig
fn branchExchange(cpu: *Cpu, bus: *Bus, instruction: u32) void {
    _ = bus;
    const rm: u4 = @truncate(instruction);  // bits [3:0] = register met doeladres

    const addr = cpu.get_reg(rm);

    // Bit 0 bepaalt ARM/Thumb mode
    cpu.cpsr.t = @truncate(addr);  // bit 0 → T flag

    if (cpu.cpsr.t == 1) {
        // Thumb mode: halfword aligned, pipeline offset = 4
        cpu.r[15] = (addr & 0xFFFFFFFE) +% 4;
    } else {
        // ARM mode: word aligned, pipeline offset = 8
        cpu.r[15] = (addr & 0xFFFFFFFC) +% 8;
    }
}
```

**Waarom masken we bit 0 (en bit 1)?** Bit 0 is het mode-signaal, niet onderdeel van het werkelijke adres. In Thumb mode masken we alleen bit 0 (adressen zijn halfword-aligned, elke 2 bytes). In ARM mode masken we bits 0-1 (adressen zijn word-aligned, elke 4 bytes).

**Waarom voegen we de pipeline offset toe?** De rest van de emulator verwacht dat PC altijd de pipeline-offset bevat (PC = uitvoer-adres + offset). Na een BX naar 0x200 in ARM mode moet PC dus 0x208 worden, niet 0x200.

**Valkuil:** Als `rm` register 15 (PC) is, is de waarde PC+8. Een BX naar PC springt effectief naar het huidige adres + 8, wat in de praktijk een NOP is (je bent al daar). Dit is zeldzaam maar sommige software doet het.

### Stap 4: `src/core/arm.zig` — Testen schrijven

```zig
test "B forward jump" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    // EA000002 = B +8 (offset=2, want <<2 = 8 bytes)
    // Huidige PC (na step-verhoging) bevat pipeline-adres
    cpu.r[15] = 0x08000008;  // simuleer pipeline: uitvoer-adres is 0x08000000
    branch(&cpu, &bus, 0xEA000002);
    // Verwacht: PC = 0x08000008 + (2 << 2) = 0x08000010
    try std.testing.expectEqual(@as(u32, 0x08000010), cpu.r[15]);
}

test "B backward jump (negatieve offset)" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    cpu.r[15] = 0x08000108;
    // EAFFFFFE = B -8 (offset=0xFFFFFE sign extended = -2, <<2 = -8)
    branch(&cpu, &bus, 0xEAFFFFFE);
    // Verwacht: PC = 0x08000108 + (-8) = 0x08000100
    try std.testing.expectEqual(@as(u32, 0x08000100), cpu.r[15]);
}

test "BL slaat return address op" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    cpu.r[15] = 0x08000108;
    // EB000010 = BL +64 (offset=0x10, <<2 = 0x40)
    branch(&cpu, &bus, 0xEB000010);
    // LR = PC - 4 = 0x08000104
    try std.testing.expectEqual(@as(u32, 0x08000104), cpu.r[14]);
}

test "BX naar Thumb mode" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    cpu.r[0] = 0x08000001;  // bit 0 = 1 → Thumb mode
    branchExchange(&cpu, &bus, 0xE12FFF10);  // BX R0
    try std.testing.expectEqual(@as(u1, 1), cpu.cpsr.t);
    // Adres = 0x08000000 + 4 (Thumb pipeline)
    try std.testing.expectEqual(@as(u32, 0x08000004), cpu.r[15]);
}

test "BX naar ARM mode" {
    var cpu = Cpu.init();
    var bus = Bus.init();
    cpu.cpsr.t = 1;  // start in Thumb
    cpu.r[0] = 0x08000000;  // bit 0 = 0 → ARM mode
    branchExchange(&cpu, &bus, 0xE12FFF10);  // BX R0
    try std.testing.expectEqual(@as(u1, 0), cpu.cpsr.t);
    try std.testing.expectEqual(@as(u32, 0x08000008), cpu.r[15]);
}
```

**Tip:** Backward branches (negatieve offsets) zijn essentieel voor loops. Test altijd met zowel positieve als negatieve offsets. De meest voorkomende bug is dat sign extension niet correct werkt, waardoor negatieve branches als grote forward jumps worden geïnterpreteerd.

## Acceptatiecriteria

- Forward en backward branches werken
- BL slaat correct return address op
