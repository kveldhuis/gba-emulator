# Story 16.1 — RTC (Real-Time Clock)

**Epic:** 16 - Accuracy — Medium Impact

**Doel:** De Real-Time Clock in de cartridge emuleren voor tijdsafhankelijke games.

## Waarom belangrijk?

Pokémon Ruby/Sapphire/Emerald en enkele andere games bevatten een RTC-chip in de cartridge. Zonder RTC-emulatie werken dag/nacht cycli niet, groeien berries niet, en verschijnen time-based events niet.

**Waarom zit de RTC in de cartridge en niet in de GBA?** De GBA zelf heeft geen klok-chip — Nintendo hield de console goedkoop. Alleen cartridges die tijdsfuncties nodig hadden, bevatten een RTC-chip (de S3511A) met een eigen knoopcel-batterij. Dit betekent dat niet alle games een RTC hebben — je emulator moet detecteren of de geladen ROM er een verwacht.

**Waarom is het GPIO-protocol complex?** De RTC communiceert via een serieel protocol over slechts 3 draden (clock, data, chip-select). De game stuurt bit-voor-bit een commando, en de RTC antwoordt bit-voor-bit met de tijd. Dit vereist een state machine in je emulator die elke bit-wissel bijhoudt — je kunt niet simpelweg "geef me de tijd" implementeren als één functie-aanroep.

## Context

De RTC communiceert via **GPIO** (General Purpose I/O) pinnen op de cartridge. Het protocol is serieel — bit-voor-bit, gestuurd door de game.

```
GPIO registers:
0x080000C4  — Data register (lezen/schrijven van bits)
0x080000C6  — Direction register (welke pinnen input/output zijn)
0x080000C8  — Control register (GPIO enable)
```

## Zig concepten

### GPIO en RTC struct

```zig
const Gpio = struct {
    data: u4 = 0,
    direction: u4 = 0,  // 0=input (lezen), 1=output (schrijven)
    control: u1 = 0,    // 0=GPIO uit, 1=GPIO aan

    rtc: Rtc = .{},

    pub fn read(self: *Gpio, address: u32) u8 {
        return switch (address) {
            0x080000C4 => self.rtc.read_data(self.direction),
            0x080000C6 => self.direction,
            0x080000C8 => self.control,
            else => 0,
        };
    }

    pub fn write(self: *Gpio, address: u32, value: u8) void {
        switch (address) {
            0x080000C4 => self.rtc.write_data(value, self.direction),
            0x080000C6 => self.direction = @truncate(value),
            0x080000C8 => self.control = @truncate(value),
            else => {},
        }
    }
};
```

### RTC serieel protocol

De RTC chip (S3511A) communiceert via 3 lijnen:
- **SCK** (bit 0): clock signal
- **SIO** (bit 1): data in/out
- **CS** (bit 2): chip select

```zig
const Rtc = struct {
    state: RtcState = .idle,
    command: u8 = 0,
    bit_count: u4 = 0,
    data_byte: u8 = 0,
    buffer: [8]u8 = [_]u8{0} ** 8,
    buffer_pos: u4 = 0,

    const RtcState = enum {
        idle,
        receiving_command,
        sending_data,
        receiving_data,
    };

    fn write_data(self: *Rtc, value: u8, direction: u4) void {
        const cs  = (value >> 2) & 1;
        const sck = (value >> 0) & 1;
        const sio = (value >> 1) & 1;

        if (cs == 0) {
            self.state = .idle;
            return;
        }

        // Lees bits op de rising edge van SCK
        if (sck == 1) {
            switch (self.state) {
                .idle => {
                    self.state = .receiving_command;
                    self.bit_count = 0;
                    self.command = 0;
                },
                .receiving_command => {
                    self.command |= @as(u8, @truncate(sio)) << self.bit_count;
                    self.bit_count += 1;
                    if (self.bit_count >= 8) {
                        self.process_command();
                    }
                },
                // ... sending_data, receiving_data
                else => {},
            }
        }
    }
};
```

### Tijd ophalen van het host-systeem

```zig
const std = @import("std");

fn get_current_time() struct { year: u8, month: u8, day: u8, hour: u8, min: u8, sec: u8, weekday: u8 } {
    const timestamp = std.time.timestamp();
    const epoch = std.time.epoch.EpochSeconds{ .secs = @intCast(timestamp) };
    const day_seconds = epoch.getDaySeconds();
    const year_day = epoch.getEpochDay().calculateYearDay();

    return .{
        .year = @intCast(year_day.year - 2000),  // RTC: jaar relatief aan 2000
        .month = year_day.calculateMonthDay().month.numeric(),
        .day = year_day.calculateMonthDay().day_index + 1,
        .hour = day_seconds.getHoursIntoDay(),
        .min = day_seconds.getMinutesIntoHour(),
        .sec = day_seconds.getSecondsIntoMinute(),
        .weekday = epoch.getEpochDay().calculateDayOfWeek().numeric(),
    };
}
```

### BCD encoding — Binary Coded Decimal

De RTC retourneert waarden in BCD-formaat: elke nibble (4 bits) is een decimaal cijfer.

```zig
fn to_bcd(value: u8) u8 {
    // Voorbeeld: 23 → 0x23
    // 23 / 10 = 2, 23 % 10 = 3 → 0x23
    return (value / 10) << 4 | (value % 10);
}

// Tijd als BCD:
// 14:30:45 → { 0x14, 0x30, 0x45 }
```

## Stap-voor-stap

### Stap 1: `src/core/bus.zig` — GPIO adressen routeren

De GPIO-registers zitten in het ROM-adresbereik (0x080000C4-C8). Dit is ongebruikelijk — normaal is ROM read-only. Je bus moet writes naar deze specifieke adressen detecteren en doorsturen naar de GPIO-module.

```zig
fn write16(self: *Bus, address: u32, value: u16) void {
    const region = (address >> 24) & 0xF;
    switch (region) {
        0x08 => {
            // GPIO registers zitten in ROM-adresruimte
            if (address >= 0x080000C4 and address <= 0x080000C8) {
                self.gpio.write(address, @truncate(value));
            }
            // Andere ROM writes worden genegeerd
        },
        // ... andere regio's ...
    }
}
```

**Waarom zitten GPIO-registers in ROM-adresruimte?** De GPIO-pinnen zitten fysiek op de cartridge-connector, die gemapped is op de ROM-adressen. Nintendo hergebruikte ongebruikte adressen in het ROM-bereik voor GPIO. Dit is slim maar verwarrend: je moet een uitzondering maken in je bus-logica voor een adresregio die normaal read-only is.

**Valkuil:** Niet alle ROMs hebben GPIO. Als een game geen RTC of andere GPIO-hardware heeft, moeten writes naar deze adressen gewoon genegeerd worden. Controleer of GPIO ingeschakeld is (control register 0x080000C8) voordat je writes verwerkt.

### Stap 2: `src/core/gpio.zig` — GPIO struct met data/direction/control registers

Implementeer de drie GPIO-registers die de interface vormen tussen de game en de cartridge-hardware.

```zig
const Gpio = struct {
    data: u4 = 0,       // actuele pin-waarden
    direction: u4 = 0,  // per pin: 0=input (GBA leest), 1=output (GBA schrijft)
    control: u1 = 0,    // 0=GPIO uit (registers onzichtbaar), 1=GPIO aan

    rtc: Rtc = .{},

    pub fn read(self: *Gpio, address: u32) u8 {
        if (self.control == 0) return 0;  // GPIO uitgeschakeld
        return switch (address) {
            0x080000C4 => blk: {
                // Lees alleen de pinnen die als "input" geconfigureerd zijn
                var result: u4 = self.data;
                const input_mask = ~self.direction;  // input = direction bit 0
                result &= input_mask;
                // RTC levert zijn output-bit via SIO (bit 1)
                result |= @as(u4, self.rtc.output_bit()) << 1;
                break :blk result;
            },
            0x080000C6 => self.direction,
            0x080000C8 => @as(u8, self.control),
            else => 0,
        };
    }

    pub fn write(self: *Gpio, address: u32, value: u8) void {
        switch (address) {
            0x080000C4 => {
                self.data = @truncate(value);
                self.rtc.write_pins(self.data, self.direction);
            },
            0x080000C6 => self.direction = @truncate(value),
            0x080000C8 => self.control = @truncate(value),
            else => {},
        }
    }
};
```

**Waarom een direction register?** GPIO-pinnen zijn bidirectioneel: de game kan ze als output configureren (om commando's te sturen naar de RTC) of als input (om data van de RTC te lezen). Het direction register bepaalt per pin welke richting actief is. Dit is standaard embedded-programmering — dezelfde aanpak die je vindt in microcontrollers.

**Waarom het input-mask?** Wanneer de game het data register leest, moet je alleen de pinnen retourneren die als input geconfigureerd zijn. De output-pinnen bevatten de waarde die de game zelf geschreven heeft — die hoeft niet "gelezen" te worden van de hardware. Het RTC antwoord komt binnen via de SIO-pin (bit 1) wanneer die als input staat.

### Stap 3: `src/core/rtc.zig` — RTC state machine implementeren

Het seriële protocol werkt met drie signalen: SCK (clock), SIO (data), CS (chip select). De game stuurt een 8-bit commando, en de RTC antwoordt met 1-7 bytes aan data.

```zig
const Rtc = struct {
    state: State = .idle,
    command: u8 = 0,
    bit_count: u4 = 0,
    buffer: [8]u8 = [_]u8{0} ** 8,
    buffer_len: u4 = 0,
    buffer_pos: u4 = 0,
    current_bit: u1 = 0,

    last_sck: u1 = 0,  // voor edge-detectie

    const State = enum { idle, receiving_command, sending_data, receiving_data };

    fn write_pins(self: *Rtc, data: u4, direction: u4) void {
        const cs  = @as(u1, @truncate(data >> 2));
        const sck = @as(u1, @truncate(data >> 0));
        const sio = @as(u1, @truncate(data >> 1));

        // CS laag = reset
        if (cs == 0) {
            self.state = .idle;
            self.last_sck = sck;
            return;
        }

        // Detecteer rising edge van SCK (0 → 1)
        const rising_edge = (self.last_sck == 0 and sck == 1);
        self.last_sck = sck;

        if (!rising_edge) return;

        switch (self.state) {
            .idle => {
                self.state = .receiving_command;
                self.command = 0;
                self.bit_count = 0;
            },
            .receiving_command => {
                self.command |= @as(u8, sio) << self.bit_count;
                self.bit_count += 1;
                if (self.bit_count >= 8) {
                    self.process_command();
                }
            },
            .sending_data => {
                // RTC stuurt data bit-voor-bit naar de game
                if (self.buffer_pos < self.buffer_len) {
                    const byte = self.buffer[self.buffer_pos];
                    self.current_bit = @truncate(byte >> self.bit_count);
                    self.bit_count += 1;
                    if (self.bit_count >= 8) {
                        self.bit_count = 0;
                        self.buffer_pos += 1;
                    }
                }
            },
            .receiving_data => {
                // Game stuurt data naar RTC (bijv. tijd instellen)
                // Minder belangrijk voor basic emulatie
            },
        }
    }

    fn output_bit(self: *Rtc) u1 {
        return if (self.state == .sending_data) self.current_bit else 1;
    }
};
```

**Waarom rising edge detectie?** Het seriële protocol is clock-gestuurd: data wordt gelezen op het moment dat SCK van 0 naar 1 gaat (de "rising edge"). Dit is standaard SPI-communicatie. Als je op elke write naar het data register zou reageren (in plaats van alleen op clock-flanken), zou je bits dubbel tellen.

**Waarom een state machine?** Het protocol heeft vier fasen: idle (wachten), commando ontvangen (8 bits), data sturen (variabel), data ontvangen (variabel). Elke fase reageert anders op binnenkomende bits. Een state machine is de meest natuurlijke manier om dit te modelleren.

### Stap 4: `src/core/rtc.zig` — Commando's verwerken en tijd ophalen

Na het ontvangen van een 8-bit commando, vul de buffer met de gevraagde data.

```zig
fn process_command(self: *Rtc) void {
    // RTC commando format: bits 4-6 = register, bit 0 = read/write
    const register = (self.command >> 4) & 0x7;
    const is_read = self.command & 1 == 1;

    switch (register) {
        0x2 => {  // Date/Time register (7 bytes)
            if (is_read) {
                const time = get_current_time();
                self.buffer[0] = to_bcd(time.year);
                self.buffer[1] = to_bcd(time.month);
                self.buffer[2] = to_bcd(time.day);
                self.buffer[3] = to_bcd(time.weekday);
                self.buffer[4] = to_bcd(time.hour);
                self.buffer[5] = to_bcd(time.min);
                self.buffer[6] = to_bcd(time.sec);
                self.buffer_len = 7;
                self.state = .sending_data;
            }
        },
        0x4 => {  // Time register (3 bytes, alleen HH:MM:SS)
            if (is_read) {
                const time = get_current_time();
                self.buffer[0] = to_bcd(time.hour);
                self.buffer[1] = to_bcd(time.min);
                self.buffer[2] = to_bcd(time.sec);
                self.buffer_len = 3;
                self.state = .sending_data;
            }
        },
        0x1 => {  // Control/Status register
            if (is_read) {
                self.buffer[0] = 0x40;  // 24-uurs modus
                self.buffer_len = 1;
                self.state = .sending_data;
            }
        },
        else => {},
    }
    self.bit_count = 0;
    self.buffer_pos = 0;
}
```

**Waarom BCD-encoding?** De RTC-chip retourneert waarden in Binary Coded Decimal: elke nibble (4 bits) stelt een decimaal cijfer voor. Dus 23 uur = 0x23, niet 0x17 (wat 23 in hex is). Dit is een conventie uit de klokwereld — BCD maakt het makkelijk om de waarde direct op een display te tonen zonder conversie.

```zig
fn to_bcd(value: u8) u8 {
    return (value / 10) << 4 | (value % 10);
}
```

**Waarom de systeemtijd gebruiken?** De GBA RTC loopt op een echte klok met batterij. De meest natuurlijke emulatie is om de systeemtijd van de host-computer te gebruiken. Pokémon checkt bij het opstarten de RTC en vergelijkt met de opgeslagen tijd om te bepalen hoeveel "game-tijd" er verstreken is.

### Stap 5: `src/core/bus.zig` — Cartridge-type detectie

Niet elke ROM heeft een RTC. Je moet detecteren of de geladen ROM GPIO gebruikt.

```zig
fn detect_gpio_type(rom: []const u8) GpioType {
    // Zoek naar bekende strings in de ROM
    if (std.mem.indexOf(u8, rom, "FLASH_V") != null or
        std.mem.indexOf(u8, rom, "FLASH512_V") != null) {
        // Flash save type — kan GPIO hebben
    }

    // Check de ROM game code voor bekende RTC-games
    const game_code = rom[0xAC..0xB0];
    if (std.mem.eql(u8, game_code, "AXVE") or  // Pokémon Ruby
        std.mem.eql(u8, game_code, "AXPE") or  // Pokémon Sapphire
        std.mem.eql(u8, game_code, "BPEE"))     // Pokémon Emerald
    {
        return .rtc;
    }

    // Alternatief: zoek naar GPIO register writes in de ROM code
    // (als de ROM naar 0x080000C4-C8 schrijft, heeft het waarschijnlijk GPIO)
    return .none;
}
```

**Waarom detectie nodig?** Als je altijd GPIO inschakelt, kunnen games die geen GPIO verwachten rare waarden lezen van adres 0x080000C4 (dat normaal ROM-data bevat). Door te detecteren welke games GPIO nodig hebben, voorkom je incompatibiliteiten.

**Tip:** Een robuustere methode is om in de ROM te zoeken naar code die naar de GPIO-adressen schrijft. Als ergens in de ROM een instructie staat die naar 0x080000C4-C8 schrijft, is het veilig om GPIO in te schakelen. mGBA gebruikt een vergelijkbare heuristiek.
