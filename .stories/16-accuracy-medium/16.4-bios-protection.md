# Story 16.4 — BIOS Read Protection

**Epic:** 16 - Accuracy — Medium Impact

**Doel:** Het BIOS beschermen tegen lezen van buiten de BIOS regio.

## Waarom belangrijk?

Sommige games en anti-piracy routines proberen het BIOS te lezen. De GBA retourneert alleen het laatst door BIOS-code gelezen woord — niet de daadwerkelijke inhoud. Dit is een simpele maar belangrijke beveiligingsmaatregel.

**Waarom beschermde Nintendo het BIOS?** Het BIOS bevat Nintendo's copyrightmelding, het opstartlogo, en de implementatie van alle SWI (software interrupt) routines. Als het BIOS vrij leesbaar was, zou iedereen het kunnen dumpen en gebruiken in onofficiële software — wat Nintendo wilde voorkomen. De bescherming maakt het BIOS effectief "write-only" vanuit het perspectief van game-code.

**Waarom is dit relevant voor emulatie?** Sommige games (en met name anti-piracy checks) lezen bewust van de BIOS-regio om te controleren wat er terugkomt. Op echte hardware krijgen ze de latch-waarde (voorspelbaar), maar op een naïeve emulator die gewoon de BIOS-data retourneert, krijgen ze de daadwerkelijke BIOS-inhoud — wat een emulatie-detectie mogelijk maakt. Door het latch-gedrag te emuleren, passeren deze checks.

## Implementatie

```zig
const Bus = struct {
    bios_latch: u32 = 0,  // laatst door BIOS gelezen waarde

    fn read_bios(self: *Bus, address: u32, pc: u32) u32 {
        if (pc < 0x4000) {
            // CPU draait BIOS code → echt lezen, latch updaten
            const val = read_from_bios_data(address);
            self.bios_latch = val;
            return val;
        }
        // CPU is buiten BIOS → retourneer latch
        return self.bios_latch;
    }
};
```

Na een SWI-call (die de CPU tijdelijk in de BIOS laat draaien) bevat de latch het laatste woord dat de BIOS-routine las. Dit is een bekende waarde die games soms controleren.

## Stap-voor-stap

### Stap 1: `src/core/bus.zig` — BIOS latch veld toevoegen aan Bus struct

Voeg een `bios_latch` veld toe dat de laatst door BIOS-code gelezen waarde bijhoudt.

```zig
const Bus = struct {
    bios_data: []const u8,    // de geladen BIOS ROM (16KB)
    bios_latch: u32 = 0,      // laatst gelezen waarde terwijl PC in BIOS was

    // ... overige velden ...
};
```

**Waarom een apart veld en niet gewoon `last_fetched_opcode` hergebruiken?** De BIOS latch en de open bus waarde (story 16.2) zijn twee verschillende dingen. De open bus waarde verandert bij elke instruction fetch. De BIOS latch verandert alleen wanneer de CPU daadwerkelijk BIOS-code uitvoert en een read doet. Ze kunnen dus op elk moment verschillende waarden bevatten.

**Waarom initialiseren op 0?** Bij een cold boot (zonder BIOS emulatie) is er geen bekende latch-waarde. Als je het BIOS-bootproces wél emuleert, zal de latch na het opstarten een bekende waarde bevatten (typisch `0xE129F000`). Door op 0 te initialiseren, werk je correct in beide scenario's.

### Stap 2: `src/core/bus.zig` — Read protection logica in `read_bios()`

De kern van de bescherming: controleer of de CPU zich in de BIOS-regio bevindt vóór je de BIOS-data retourneert.

```zig
fn read_bios(self: *Bus, address: u32, pc: u32) u32 {
    if (pc < 0x4000) {
        // CPU draait code in BIOS (0x0000-0x3FFF) → lezen is toegestaan
        const aligned_addr = address & 0x3FFC;  // forceer 32-bit alignment
        const val = std.mem.readInt(u32, self.bios_data[aligned_addr..][0..4], .little);
        self.bios_latch = val;  // update de latch bij elke BIOS read
        return val;
    }

    // CPU is buiten BIOS → retourneer de latch-waarde
    return self.bios_latch;
}
```

**Waarom `pc < 0x4000` als check?** Het BIOS is 16KB groot en begint op adres 0x00000000. Het eindigt op 0x00003FFF. Als de program counter (PC) binnen dit bereik valt, voert de CPU BIOS-code uit en mag hij de BIOS-data lezen. In alle andere gevallen wordt de latch geretourneerd.

**Waarom `address & 0x3FFC`?** Twee redenen: (1) mask met 0x3FFF om binnen de 16KB BIOS te blijven (hogere bits negeren), en (2) mask met 0xFFFC om 32-bit alignment af te dwingen. De GBA bus levert altijd 32-bit aligned data; het gevraagde byte/halfword wordt uit het 32-bit woord geselecteerd.

**Valkuil:** De PC die je controleert moet de *huidige* PC zijn op het moment van de read, niet de PC van de volgende instructie. In de ARM7TDMI pipeline is de PC typisch 8 bytes (ARM) of 4 bytes (Thumb) vooruit. Je moet de PC van de executie-fase gebruiken, niet de fetch-fase.

### Stap 3: `src/core/bus.zig` — Latch update bij SWI-aanroepen

Wanneer een game een SWI (Software Interrupt) uitvoert, springt de CPU naar het BIOS. Tijdens het uitvoeren van de SWI-routine leest de BIOS-code zelf data uit het BIOS, en de latch wordt bijgewerkt. Na de SWI-return bevat de latch het laatste woord dat de BIOS-routine las.

```zig
// Geen speciale code nodig! De read_bios() functie uit stap 2 handelt dit
// automatisch af: wanneer de CPU SWI-code uitvoert (PC < 0x4000),
// worden reads normaal afgehandeld en de latch bijgewerkt.

// Maar: als je BIOS-emulatie via HLE (High-Level Emulation) doet,
// moet je de latch handmatig bijwerken na elke HLE SWI:
fn handle_swi_hle(self: *Bus, swi_number: u8) void {
    switch (swi_number) {
        0x01 => { // RegisterRamReset
            // ... HLE implementatie ...
            // De echte BIOS leest als laatste instructie: BX LR (0xE12FFF1E)
            // maar het laatste DATA-woord dat gelezen wordt, hangt af van de routine
        },
        // ...
        else => {},
    }
    // Na elke SWI is de bekende latch-waarde:
    self.bios_latch = 0xE3A02004;  // "mov r2, #4" — typisch laatste BIOS read
}
```

**Waarom is HLE een complicatie?** Als je SWI-routines emuleert via High-Level Emulation (de functie in Zig reimplementeren in plaats van de BIOS-code uit te voeren), wordt `read_bios()` nooit aangeroepen — de CPU komt nooit in de BIOS-regio. Je moet de latch-waarde dan handmatig instellen op de waarde die de echte BIOS-routine zou hebben gelezen.

**Tip:** De exacte latch-waarde na een SWI verschilt per SWI-routine. Voor de meeste games maakt het niet uit welke waarde het is, zolang het maar consistent is. Alleen specifieke anti-piracy checks controleren de exacte waarde. Een veelgebruikte default is `0xE3A02004`.

**Valkuil:** Vergeet niet dat de BIOS latch ook bijgewerkt wordt tijdens de BIOS-boot (het Nintendo-logo, de jingle, etc.). Als je het bootproces overslaat, begint de latch op 0 — wat anders is dan op echte hardware. De meeste games hebben hier geen last van, maar het is goed om te weten.

## Acceptatiecriteria

- BIOS reads van buiten de BIOS retourneren de latch-waarde, niet 0
- Na SWI bevat de latch de juiste waarde
