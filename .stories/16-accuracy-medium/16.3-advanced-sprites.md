# Story 16.3 — Advanced Sprite Rendering

**Epic:** 16 - Accuracy — Medium Impact

**Doel:** Semi-transparante sprites, pixel limiet, en forced blank.

## Waarom belangrijk?

Deze drie features zijn elk relatief klein, maar samen verbeteren ze de visuele correctheid van veel games significant.

**Waarom semi-transparante sprites?** Veel games gebruiken sprites met een semi-transparant effect — denk aan schaduwen, magie-effecten, water-reflecties, en ghost-achtige vijanden. Zonder correcte semi-transparantie worden deze sprites volledig opaak getekend, wat er visueel fout uitziet. Het bijzondere is dat semi-transparante sprites de normale blending-regels omzeilen: ze forceren alpha blending ongeacht de BLDCNT-instellingen.

**Waarom de sprite pixel limiet?** De GBA PPU heeft een fysieke beperking: het kan maximaal ~1210 sprite-pixels per scanline berekenen (minder als er affine sprites bij zijn). Games die veel sprites tonen (bijv. bullet hell of RPG-gevechten) raken deze limiet, en sprites voorbij de limiet worden simpelweg niet getekend. Als je emulator deze limiet niet respecteert, zie je sprites die op echte hardware onzichtbaar zouden zijn.

**Waarom forced blank?** Games zetten DISPCNT bit 7 aan terwijl ze VRAM vullen, om te voorkomen dat de PPU halfgeschreven data rendert. Zonder forced blank emulatie zou je corrupte frames kunnen zien tijdens laden.

## Semi-transparante sprites

Een sprite met `obj_mode = 1` (semi-transparent) gedraagt zich alsof het altijd de "first target" is voor alpha blending, ongeacht de BLDCNT instellingen:

```zig
fn composite_pixel(self: *Ppu, bus: *Bus, x: u16, y: u16) u16 {
    // ... bepaal top twee lagen op basis van prioriteit ...

    if (top_is_sprite and sprite_is_semitransparent) {
        // Semi-transparant: forceer alpha blend met laag eronder
        if (has_second_target(bottom_layer)) {
            return alpha_blend(top_color, bottom_color, eva, evb);
        }
        // Geen geldige second target? Toon gewoon de sprite-kleur
        return top_color;
    }

    // Normale blending logica (BLDCNT)...
}
```

**Let op:** semi-transparante sprites werken ALLEEN als de laag eronder een "second target" is in BLDCNT. Als er geen second target is, wordt de sprite gewoon opaak getekend.

## Sprite pixel limiet

De GBA kan maximaal **1210 sprite-pixels per scanline** renderen (of 954 als er affine sprites zijn). Daarna worden overige sprites op die scanline onzichtbaar:

```zig
fn render_sprites_scanline(self: *Ppu, bus: *Bus, y: u8) void {
    var pixels_rendered: u32 = 0;
    const pixel_limit: u32 = if (self.has_affine_sprite_on_line) 954 else 1210;

    var i: u8 = 0;
    while (i < 128) : (i += 1) {
        if (pixels_rendered >= pixel_limit) break;

        const attr0 = read_oam_attr0(bus, i);
        // ... check visibility ...

        const width = get_sprite_width(attr0, attr1);
        pixels_rendered += width;  // elke zichtbare sprite kost width pixels

        // Render sprite pixels...
    }
}
```

## Forced blank (DISPCNT bit 7)

Als bit 7 van DISPCNT gezet is, stuurt de PPU een wit scherm. Dit wordt vaak gebruikt tijdens VRAM setup:

```zig
fn render_scanline(self: *Ppu, bus: *Bus) void {
    const dispcnt = bus.read16(0x04000000);

    if ((dispcnt >> 7) & 1 == 1) {
        // Forced blank: hele scanline wit
        const y: u32 = self.scanline;
        for (0..240) |x| {
            self.framebuffer[y * 240 + x] = 0x7FFF;  // wit in RGB555
        }
        return;
    }

    // Normale rendering...
}
```

## Stap-voor-stap

### Stap 1: `src/core/ppu.zig` — Semi-transparante sprite detectie toevoegen

Bij het renderen van sprites moet je het `obj_mode` veld in OAM attribute 0 controleren. Mode 1 betekent semi-transparant.

```zig
fn render_sprites_scanline(self: *Ppu, bus: *Bus, y: u8) void {
    for (0..128) |i| {
        const attr0 = read_oam_attr0(bus, @intCast(i));
        const obj_mode = (attr0 >> 10) & 0x3;

        if (obj_mode == 2) continue;  // mode 2 = onzichtbaar (gebruikt als OBJ window)

        // ... bestaande visibility checks ...

        const is_semi_transparent = (obj_mode == 1);

        // Sla het semi-transparant flag op per pixel in een aparte buffer
        if (is_semi_transparent) {
            for (sprite_start_x..sprite_end_x) |x| {
                if (sprite_has_pixel_at(x)) {
                    self.sprite_attr[x] = .semi_transparent;
                }
            }
        }
    }
}
```

**Waarom een aparte attributen-buffer?** Je hebt per pixel twee stukken informatie nodig: de kleur (in de framebuffer) en of het een semi-transparante sprite is (voor de blending-stap). Een aparte buffer van 240 bytes (één per pixel op de scanline) is de meest efficiënte manier om dit bij te houden.

**Waarom is mode 2 onzichtbaar?** OBJ mode 2 is het "OBJ window" — de sprite wordt niet getekend, maar zijn niet-transparante pixels definiëren een window-regio. Dit is een apart feature (windowing) dat hier alleen relevant is om te filteren.

### Stap 2: `src/core/ppu.zig` — Blending logica aanpassen voor semi-transparante sprites

Semi-transparante sprites omzeilen de normale BLDCNT-regels. Ze gedragen zich altijd alsof ze "first target" zijn voor alpha blending, ongeacht of sprites als first target in BLDCNT staan.

```zig
fn composite_pixel(self: *Ppu, x: u16) u16 {
    const top_layer = self.get_top_layer(x);
    const top_color = self.get_layer_color(top_layer, x);

    // Check of de bovenste laag een semi-transparante sprite is
    if (top_layer == .obj and self.sprite_attr[x] == .semi_transparent) {
        // Forceer alpha blend — maar ALLEEN als de laag eronder
        // een "second target" is in BLDCNT
        const bottom_layer = self.get_second_layer(x);
        if (self.is_blend_second_target(bottom_layer)) {
            const bottom_color = self.get_layer_color(bottom_layer, x);
            return alpha_blend(top_color, bottom_color, self.eva, self.evb);
        }
        // Geen geldige second target? Sprite wordt gewoon opaak getekend
        return top_color;
    }

    // Normale blending logica (via BLDCNT)
    return self.apply_normal_blending(x, top_layer, top_color);
}
```

**Waarom moet de laag eronder een second target zijn?** Dit is een subtiliteit die veel emulators fout hebben. Semi-transparante sprites forceren alpha blend als first target, maar de second target moet wél correct geconfigureerd zijn in BLDCNT. Als de achtergrondlaag niet als second target staat, vindt er geen blending plaats en wordt de sprite gewoon opaak getekend. Dit voorkomt dat sprites "per ongeluk" blenden met lagen die de game niet bedoeld heeft.

**Valkuil:** De EVA/EVB (alpha blend coefficiënten) uit het BLDALPHA register worden wél gebruikt voor semi-transparante sprites. Alleen de "mode" en "first target" selectie uit BLDCNT worden genegeerd. Als EVA=16 en EVB=0, is het resultaat volledig opaak — technisch "geblend" maar visueel identiek aan niet-blended.

### Stap 3: `src/core/ppu.zig` — Sprite pixel limiet implementeren

De GBA PPU heeft een hardware-limiet op het aantal sprite-pixels dat het per scanline kan verwerken. Dit komt door de beperkte tijd die beschikbaar is tijdens de draw-periode.

```zig
fn render_sprites_scanline(self: *Ppu, bus: *Bus, y: u8) void {
    var cycles_used: u32 = 0;

    for (0..128) |i| {
        const attr0 = read_oam_attr0(bus, @intCast(i));
        if (!sprite_visible(attr0, y)) continue;

        const is_affine = (attr0 >> 8) & 1 == 1;
        const width = get_sprite_width(attr0, attr1);

        // Affine sprites kosten meer rendering-cycles
        const cost = if (is_affine) width * 2 + 10 else width;

        if (cycles_used + cost > 1210) break;  // limiet bereikt
        cycles_used += cost;

        // Render deze sprite
        self.draw_sprite(bus, @intCast(i), y);
    }
}
```

**Waarom 1210 cycles?** De PPU heeft precies 1210 "dot cycles" beschikbaar om sprites te renderen op een scanline. Elke niet-affine sprite-pixel kost 1 cycle. Affine sprites zijn duurder: ze kosten `width * 2 + 10` cycles vanwege de matrix-transformatie die per pixel berekend moet worden. Sommige bronnen noemen 954 cycles als limiet voor affine sprites — dat is het maximum bij uitsluitend affine sprites.

**Waarom OAM-volgorde (0-127)?** De GBA rendert sprites in OAM-volgorde, niet in prioriteitsvolgorde. Sprite 0 wordt als eerste verwerkt. Als de pixel limiet bereikt wordt, worden de hogere sprite-nummers afgekapt. Games die veel sprites gebruiken, plaatsen de belangrijkste sprites (player character) bewust op lage OAM-indices.

**Tip:** Sommige games roteren de OAM-indices elk frame ("OAM cycling") om te voorkomen dat altijd dezelfde sprites afgekapt worden. Dit geeft een "flicker" effect in plaats van sprites die volledig verdwijnen — vergelijkbaar met het flickeren op de NES.

### Stap 4: `src/core/ppu.zig` — Forced blank implementeren

Forced blank is de eenvoudigste van de drie features: als DISPCNT bit 7 gezet is, render je een volledig wit scherm.

```zig
fn render_scanline(self: *Ppu, bus: *Bus) void {
    const dispcnt: u16 = bus.read16(0x04000000);

    if ((dispcnt >> 7) & 1 == 1) {
        // Forced blank: hele scanline wit (0x7FFF = wit in RGB555)
        const y: u32 = self.scanline;
        @memset(self.framebuffer[y * 240 ..][0..240], 0x7FFF);
        return;
    }

    // Normale rendering pipeline...
    self.render_backgrounds(bus);
    self.render_sprites(bus);
    self.composite_scanline(bus);
}
```

**Waarom wit en niet zwart?** Op echte hardware stuurt de PPU in forced blank modus een "blanking" signaal naar het LCD, wat resulteert in een wit scherm. Dit is een hardware-eigenschap van het GBA LCD-paneel. Sommige emulators gebruiken zwart, maar wit is technisch correcter.

**Waarom gebruiken games forced blank?** Tijdens het laden van VRAM-data (tiles, tilemaps, palettes) wil je niet dat de PPU half-geschreven data rendert — dat zou corrupte pixels tonen. Door forced blank aan te zetten, kan de game in alle rust VRAM vullen. Bovendien: tijdens forced blank zijn er geen PPU-contentie cycles, dus VRAM-writes zijn sneller.

**Tip:** Gebruik `@memset` in Zig om efficiënt een hele scanline te vullen. Dit is sneller dan een for-loop en het maakt de intentie duidelijk.
