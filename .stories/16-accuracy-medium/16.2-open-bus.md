# Story 16.2 — Open Bus Behavior

**Epic:** 16 - Accuracy — Medium Impact

**Doel:** Correcte waarden retourneren bij reads van niet-gemapped geheugen.

## Waarom belangrijk?

Als de CPU leest van een adres dat nergens naartoe wijst (unmapped memory), retourneert de GBA niet gewoon 0. In plaats daarvan krijg je de **laatste waarde die op de bus stond** — typisch de laatst gefetchte instructie. Sommige games (en anti-piracy checks) leunen hierop.

**Waarom retourneert de GBA niet gewoon 0?** Dit is geen bewuste keuze, maar een eigenschap van de hardware. De databus is een fysieke set van 32 draden. Als geen enkel geheugen-IC antwoordt op een adres, blijven de draden op hun vorige spanning staan — de waarde van de laatste succesvolle bustransfer. Dit heet "open bus" of "floating bus" gedrag.

**Waarom hangen sommige games hiervan af?** Meestal per ongeluk. Een game leest van een ongeldig adres door een bug (off-by-one, verkeerde pointer), maar crasht niet op echte hardware omdat de open bus toevallig een bruikbare waarde retourneert. Als je emulator 0 retourneert bij ongeldige reads, kan die waarde (0) een crash of freeze veroorzaken die op echte hardware nooit optreedt. Sommige anti-piracy routines lezen bewust van unmapped adressen om te detecteren of ze op echte hardware draaien.

## Context

```
Open bus retourneert het laatst gefetchte opcode:
- In ARM mode: de 32-bit instructie die net opgehaald is
- In Thumb mode: de 16-bit instructie, gedupliceerd als 32-bit
- BIOS regio (0x00-0x03FFF): retourneert het laatst gelezen BIOS woord (alleen na SWI)

Specifiek per regio:
- 0x00000000-0x00003FFF  BIOS: speciaal gedrag (zie story 16.4)
- 0x10000000+             Volledig unmapped: open bus
- Unused bits in I/O:     0 of open bus (registerafhankelijk)
```

## Zig concepten

### De last_fetched_opcode bijhouden

```zig
const Bus = struct {
    // ...bestaande velden...
    last_fetched_opcode: u32 = 0,

    pub fn read32(self: *Bus, address: u32) u32 {
        const region = (address >> 24) & 0xF;
        return switch (region) {
            0x02 => self.read_ewram_32(address),
            0x03 => self.read_iwram_32(address),
            // ... alle bekende regio's ...
            else => self.open_bus_value(address),
        };
    }

    fn open_bus_value(self: *Bus, address: u32) u32 {
        // Retourneer de laatst gefetchte instructie
        // Sommige emulators retourneren dit geroteerd op basis van alignment
        _ = address;
        return self.last_fetched_opcode;
    }
};
```

### Update bij elke fetch

```zig
// In cpu.step():
const instruction = bus.read32(self.r[15]);
bus.last_fetched_opcode = instruction;  // update open bus waarde
```

### BIOS regio — speciale open bus

De BIOS is alleen leesbaar terwijl de CPU code uit de BIOS uitvoert. Op alle andere momenten retourneert een BIOS-read het laatst door de BIOS gelezen woord:

```zig
bios_last_read: u32 = 0,

fn read_bios(self: *Bus, address: u32, cpu_pc: u32) u32 {
    if (cpu_pc < 0x4000) {
        // CPU zit in BIOS → gewoon lezen
        const val = std.mem.readInt(u32, self.bios[address & 0x3FFF ..][0..4], .little);
        self.bios_last_read = val;
        return val;
    } else {
        // CPU zit buiten BIOS → retourneer laatst gelezen BIOS waarde
        return self.bios_last_read;
    }
}
```

## Stap-voor-stap

### Stap 1: `src/core/bus.zig` — `last_fetched_opcode` veld toevoegen

Voeg een veld toe aan de Bus struct dat altijd de laatst gefetchte instructie bijhoudt. Dit is de waarde die open bus retourneert.

```zig
const Bus = struct {
    // ... bestaande velden ...
    last_fetched_opcode: u32 = 0,

    // ... bestaande methoden ...
};
```

**Waarom de laatst *gefetchte* instructie en niet de laatst gelezen waarde?** Op de GBA is de instruction fetch de meest recente bus-activiteit op het moment dat de CPU een data-read doet. De ARM7TDMI pipeline werkt zo: fetch → decode → execute. Tijdens de execute-fase (waarin een LDR-instructie de data-read doet) bevat de bus nog de waarde van de meest recente fetch. Dit is een subtiel maar belangrijk verschil.

### Stap 2: `src/core/cpu.zig` — Update bij elke instruction fetch

Elke keer dat de CPU een instructie fetched, update de open bus waarde.

```zig
pub fn step(self: *Cpu, bus: *Bus) u32 {
    // Fetch de instructie
    const instruction = if (self.cpsr.t == 1)
        @as(u32, bus.read16(self.r[15]))
    else
        bus.read32(self.r[15]);

    // Update open bus waarde
    bus.last_fetched_opcode = if (self.cpsr.t == 1)
        instruction | (instruction << 16)  // Thumb: gedupliceerd als 32-bit
    else
        instruction;

    // ... decode en execute ...
}
```

**Waarom wordt een Thumb-instructie gedupliceerd?** De databus is 32-bit breed. Een 16-bit Thumb-instructie neemt maar de helft van de bus in beslag. Op echte hardware staat dezelfde 16-bit waarde op beide helften van de bus: `0x4770` wordt `0x47704770`. Dit is omdat de bus-driver de 16-bit waarde op beide helften plaatst. Als een game een 32-bit read doet van unmapped geheugen terwijl de CPU in Thumb mode is, krijgt het deze gedupliceerde waarde.

### Stap 3: `src/core/bus.zig` — Open bus waarde retourneren bij unmapped reads

Pas de read-functies aan om `last_fetched_opcode` te retourneren in plaats van 0 bij onbekende adressen.

```zig
pub fn read32(self: *Bus, address: u32) u32 {
    const region = (address >> 24) & 0xF;
    return switch (region) {
        0x00 => self.read_bios(address),
        0x02 => self.read_ewram_32(address),
        0x03 => self.read_iwram_32(address),
        0x04 => self.read_io_32(address),
        0x05 => self.read_palette_32(address),
        0x06 => self.read_vram_32(address),
        0x07 => self.read_oam_32(address),
        0x08...0x0D => self.read_rom_32(address),
        0x0E, 0x0F => self.read_sram(address),
        else => self.open_bus_value(address),  // unmapped → open bus
    };
}

fn open_bus_value(self: *Bus, address: u32) u32 {
    _ = address;
    return self.last_fetched_opcode;
}
```

**Waarom geen specifieke waarde per regio?** In de meest simpele implementatie retourneer je altijd de laatst gefetchte opcode. Dit is voor 99% van de gevallen correct. Geavanceerdere emulators houden ook rekening met de alignment van het adres (de open bus waarde kan geroteerd zijn), maar dat is zelden nodig voor game-compatibiliteit.

**Tip:** Sommige regio's (bijv. unused I/O registers) retourneren ook open bus. Als je I/O read-handler een `else` branch heeft voor onbekende registers, gebruik dan ook `self.last_fetched_opcode` in plaats van 0.

### Stap 4: `src/core/bus.zig` — BIOS speciale open bus

De BIOS-regio heeft een uniek open bus gedrag: het retourneert niet de laatst gefetchte opcode, maar het laatst *door de BIOS gelezen* woord.

```zig
bios_latch: u32 = 0,  // laatst gelezen waarde terwijl CPU in BIOS was

fn read_bios(self: *Bus, address: u32) u32 {
    if (self.cpu_pc < 0x4000) {
        // CPU draait BIOS code → normaal lezen, latch updaten
        const val = std.mem.readInt(u32, self.bios[address & 0x3FFC ..][0..4], .little);
        self.bios_latch = val;
        return val;
    }
    // CPU is buiten BIOS → retourneer de latch
    return self.bios_latch;
}
```

**Waarom een aparte latch voor BIOS?** Dit is een bewuste beveiligingsmaatregel van Nintendo. Het BIOS bevat Nintendo's copyright en vertrouwelijke code (SWI routines). Door het BIOS alleen leesbaar te maken terwijl de CPU er code uit uitvoert, wordt het moeilijker om het BIOS te dumpen. De latch bevat altijd het laatste woord dat de BIOS-routine zelf las — typisch de return-waarde van een SWI, wat een bekende waarde is.

**Waarom is dit gekoppeld aan open bus?** Omdat de BIOS-regio dezelfde hardware-eigenschap heeft: als het geheugen niet antwoordt (omdat de CPU geen toegang heeft), retourneert de bus de laatst bekende waarde. Maar in dit geval is die waarde specifiek de laatst door BIOS gelezen waarde, niet de laatst gefetchte opcode. De hardware heeft een apart latch-register voor BIOS-reads.

**Valkuil:** De initiële waarde van `bios_latch` (bij power-on zonder BIOS boot) moet 0 zijn. Maar als je het BIOS-bootproces emuleert, bevat de latch na het opstarten typisch `0xE129F000` — het laatste woord dat de BIOS-startup-routine leest. Sommige games controleren deze waarde.

## Acceptatiecriteria

- Unmapped reads retourneren de laatst gefetchte opcode
- BIOS is onleesbaar van buiten de BIOS regio
- Anti-piracy checks in games passeren
