# Story 12.1 — Cycle-accurate event scheduler

**Epic:** 12 - Event Scheduler

**Doel:** Centraal systeem voor het plannen van timed events.

## Zig concepten

### Waarom een scheduler?

Zonder scheduler zou je elke CPU-cycle alle componenten moeten updaten: PPU, timers, DMA, audio. Dat is verspilling — de PPU hoeft pas iets te doen bij de volgende scanline (elke ~1232 cycles), en een timer misschien pas over 1024 cycles. In de praktijk betekent dit dat je bij een "poll alles elke cycle"-aanpak meer dan 99% van de tijd niets nuttigs doet: de PPU verandert pas na ~1232 cycles, een timer misschien pas na 1024 cycles, en audio na ~512 cycles. Al die tussenliggende checks zijn pure verspilling.

Met een scheduler plan je events vooruit: "over 1232 cycles → volgende scanline". De CPU draait vrij tot het eerste event, handelt dat af, en gaat door.

**Waarom cycle-based en niet frame-based?** Een frame-based aanpak (eerst alle CPU-instructies, dan PPU, dan audio) is eenvoudiger maar mist cruciale details. De GBA ondersteunt mid-scanline effecten: een game kan halverwege een scanline registers wijzigen om visuele trucs te bereiken. Ook audio-synchronisatie vereist nauwkeurige timing — als samples niet op het juiste cycle worden gegenereerd, hoor je clicks en pops. Een cycle-accurate scheduler is daarom de standaardaanpak in emulator-ontwikkeling: vrijwel elke serieuze GBA/SNES/GB emulator gebruikt dit patroon.

### Event struct

```zig
const EventType = enum {

    ppu_vblank,      // PPU gaat naar VBlank

    timer1_overflow,
    timer2_overflow,
    timer3_overflow,
    apu_sample,      // audio sample moet gegenereerd worden
};

const Event = struct {
    timestamp: u64,      // op welk cycle-nummer moet dit event gebeuren?
    event_type: EventType,
};
```

### Gesorteerde lijst als eenvoudige priority queue

De simpelste manier om events bij te houden is een gesorteerde array. Het event met de laagste timestamp staat vooraan.

**Waarom een gesorteerde array en geen heap of linked list?** In een emulator zijn er typisch minder dan 20 actieve events tegelijk (een paar PPU-events, 4 timers, audio, misschien DMA). Bij zo weinig elementen is een gesorteerde array de beste keuze:
- **Cache-friendly:** alle events staan aaneengesloten in het geheugen, wat snelle iteratie geeft.
- **O(n) insert is verwaarloosbaar:** met n < 20 is het verschuiven van een paar elementen sneller dan de overhead van een heap met pointer-indirecties.
- **Eenvoudig te debuggen:** je kunt de hele array in één keer printen en zien of de volgorde klopt.
- **O(1) lookup van het eerstvolgende event:** het staat altijd op index 0.

Een min-heap zou theoretisch O(log n) insert geven, maar bij n < 20 is het verschil niet meetbaar en voegt het complexiteit toe.

```zig
const Scheduler = struct {
    events: [64]Event = undefined,  // max 64 events tegelijk
    count: u8 = 0,
    current_time: u64 = 0,          // huidige cycle-teller

    fn schedule(self: *Scheduler, event_type: EventType, cycles_from_now: u64) void {
        const timestamp = self.current_time + cycles_from_now;

        // Vind de juiste positie (gesorteerd op timestamp)
        var insert_pos: u8 = self.count;
        for (0..self.count) |i| {
            if (timestamp < self.events[i].timestamp) {
                insert_pos = @intCast(i);
                break;
            }
        }

        // Schuif latere events op om ruimte te maken
        var j: u8 = self.count;
        while (j > insert_pos) : (j -= 1) {
            self.events[j] = self.events[j - 1];
        }

        // Plaats het nieuwe event
        self.events[insert_pos] = .{
            .timestamp = timestamp,
            .event_type = event_type,
        };
        self.count += 1;
    }
};
```

### Events verwerken

```zig
fn process_events(self: *Scheduler, ppu: *Ppu, irq: *Irq) void {
    while (self.count > 0 and self.events[0].timestamp <= self.current_time) {
        const event = self.events[0];

        // Verwijder het event (schuif alles naar voren)
        for (0..self.count - 1) |i| {
            self.events[i] = self.events[i + 1];
        }
        self.count -= 1;

        // Handel het event af
        switch (event.event_type) {
            .ppu_hblank => {
                ppu.enter_hblank();
                // Plan het volgende event
                self.schedule(.ppu_scanline, 272);  // HBlank duurt 272 cycles
            },
            .ppu_scanline => {
                ppu.render_scanline();
                self.schedule(.ppu_hblank, 960);  // visible part = 960 cycles
            },
            .ppu_vblank => {
                ppu.enter_vblank();
                irq.request(.vblank);
            },
            // ... andere events
            else => {},
        }
    }
}
```

### De main loop met scheduler

**Waarom werkt dit "run to next event" patroon?** Elke ARM7TDMI-instructie neemt een bekend aantal cycles in beslag (1 cycle voor simpele ALU-operaties, 3+ voor geheugentoegangen, etc.). Dat betekent dat je de CPU kunt laten draaien tot de `current_time` de timestamp van het volgende event bereikt of passeert. Op dat moment pauzeer je de CPU, handel je het event af (wat mogelijk nieuwe events plant), en ga je verder. Dit is efficiënt omdat de CPU-loop geen enkele check hoeft te doen voor componenten — alleen de simpele vergelijking `current_time < target`.

```zig
fn run_frame(cpu: *Cpu, scheduler: *Scheduler) void {
    const frame_end = scheduler.current_time + 280896;  // cycles per frame

    while (scheduler.current_time < frame_end) {
        // Hoeveel cycles kan de CPU draaien tot het volgende event?
        const target = scheduler.next_event_time();
        
        // CPU draait tot het volgende event
        while (scheduler.current_time < target) {
            const cycles = cpu.step();  // voer 1 instructie uit
            scheduler.current_time += cycles;
        }

        // Verwerk alle events die nu moeten gebeuren
        scheduler.process_events();
    }
}
```

## Context

- In plaats van elke component elke cycle te updaten, plan events vooruit
- Voorbeelden: "PPU VBlank over 57344 cycles", "Timer 0 overflow over 1024 cycles"
- Priority queue gesorteerd op timestamp
- CPU runt tot het eerstvolgende event, dan wordt dat event afgehandeld

## Bestanden

- `src/core/scheduler.zig`
  - `Event` struct: `timestamp: u64`, `type: EventType`
  - `pub fn schedule(event: Event) void` — voeg event toe
  - `pub fn next_event_time() u64` — wanneer is het volgende event?
  - `pub fn process_events(current_time: u64) void` — voer verlopen events uit

## Stap-voor-stap

### Stap 1: `src/core/scheduler.zig` — EventType en Event structs definiëren

Begin met het definiëren van de event types die je scheduler moet ondersteunen. Elk type vertegenwoordigt een specifiek moment waarop een component aandacht nodig heeft:

```zig
const std = @import("std");

pub const EventType = enum {
    ppu_hblank,      // PPU gaat naar HBlank (horizontale blanking-periode)
    ppu_vblank,      // PPU gaat naar VBlank (verticale blanking-periode)
    ppu_scanline,    // volgende zichtbare scanline renderen
    timer0_overflow, // hardware timer 0 loopt over
    timer1_overflow, // hardware timer 1 loopt over
    timer2_overflow, // hardware timer 2 loopt over
    timer3_overflow, // hardware timer 3 loopt over
    apu_sample,      // audio sample moet gegenereerd worden
};

pub const Event = struct {
    timestamp: u64,      // op welk cycle-nummer moet dit event gebeuren?
    event_type: EventType,
};
```

**Waarom deze specifieke event types?**
- **PPU events** (`ppu_hblank`, `ppu_vblank`, `ppu_scanline`): de PPU doorloopt een vast patroon per frame. Elke scanline heeft een zichtbaar deel (960 cycles) en een HBlank-deel (272 cycles). Na 160 zichtbare scanlines volgen 68 VBlank-scanlines. Deze events sturen dat ritme aan.
- **Timer events** (`timer0_overflow` t/m `timer3_overflow`): de GBA heeft 4 hardware timers die op configureerbare snelheden tellen. Wanneer een timer overloopt (van 0xFFFF naar 0), kan dat een interrupt triggeren. De scheduler plant het overflow-moment vooruit.
- **APU event** (`apu_sample`): audio moet op regelmatige intervallen samples genereren (typisch elke ~512 cycles voor 32768 Hz). Te laat of te vroeg → hoorbare glitches.

**Waarom `u64` voor de timestamp?** De GBA-CPU draait op 16.78 MHz (16,777,216 cycles per seconde). Een `u32` kan maximaal ~4.29 miljard tellen, wat slechts ~256 seconden (~4 minuten) aan gameplay dekt voordat het overloopt. Een `u64` geeft je ruim 34.000 jaar aan cycles — meer dan genoeg.

### Stap 2: `src/core/scheduler.zig` — Scheduler struct met gesorteerde array

Nu de kern van de scheduler: een struct die events beheert in een gesorteerde array.

```zig
pub const Scheduler = struct {
    events: [64]Event = undefined,  // max 64 events tegelijk
    count: u8 = 0,
    current_time: u64 = 0,          // huidige cycle-teller

    pub fn schedule(self: *Scheduler, event_type: EventType, cycles_from_now: u64) void {
        std.debug.assert(self.count < 64);  // voorkom overflow

        const timestamp = self.current_time + cycles_from_now;

        // Vind de juiste positie (gesorteerd op timestamp)
        var insert_pos: u8 = self.count;
        for (0..self.count) |i| {
            if (timestamp < self.events[i].timestamp) {
                insert_pos = @intCast(i);
                break;
            }
        }

        // Schuif latere events op om ruimte te maken
        var j: u8 = self.count;
        while (j > insert_pos) : (j -= 1) {
            self.events[j] = self.events[j - 1];
        }

        // Plaats het nieuwe event
        self.events[insert_pos] = .{
            .timestamp = timestamp,
            .event_type = event_type,
        };
        self.count += 1;
    }
};
```

**Waarom een vaste array van 64?**
- Er zijn nooit meer dan ~15-20 actieve events tegelijk in een GBA-emulator (3 PPU-events, 4 timers, audio, eventueel DMA). 64 is ruim voldoende.
- Een vaste array heeft geen allocator nodig — geen `malloc`, geen faalpad, geen fragmentatie. Dit maakt de scheduler voorspelbaar en eenvoudig.
- De `count` variabele houdt bij hoeveel entries geldig zijn. Alles na index `count - 1` is `undefined`.

**Waarom gesorteerde insertie?** Door bij elke `schedule()` het event op de juiste plek in te voegen, is de array altijd gesorteerd. Het eerstvolgende event staat altijd op `events[0]`, wat O(1) lookup geeft — en die lookup gebeurt elke paar CPU-instructies in de main loop, dus dat moet snel zijn.

**Valkuil:** Vergeet niet te checken of `count < 64` voordat je een event toevoegt! Zonder die check schrijf je voorbij de array-grenzen, wat in Zig een panic veroorzaakt (in debug mode) of undefined behavior (in release mode). De `std.debug.assert` hierboven vangt dit op tijdens ontwikkeling.

### Stap 3: `src/core/scheduler.zig` — Events opvragen en verwijderen

Voeg functies toe om het eerstvolgende event op te vragen en events te verwijderen na verwerking:

```zig
pub fn next_event_time(self: *Scheduler) u64 {
    if (self.count == 0) return std.math.maxInt(u64);
    return self.events[0].timestamp;  // eerste = vroegste
}
```

**Waarom `maxInt(u64)` bij een lege scheduler?** Dit is een slimme truc: de main loop vergelijkt `current_time < target` om te bepalen hoelang de CPU mag draaien. Door `maxInt(u64)` terug te geven bij een lege scheduler, draait de CPU gewoon door tot het einde van het frame — zonder speciale `if (scheduler.empty())` checks in de loop. Eén vergelijking volstaat altijd.

Het verwijderen van een verwerkt event werkt door alle elementen één positie naar voren te schuiven:

```zig
fn pop_first_event(self: *Scheduler) Event {
    std.debug.assert(self.count > 0);
    const event = self.events[0];

    // Schuif alles naar voren
    for (0..self.count - 1) |i| {
        self.events[i] = self.events[i + 1];
    }
    self.count -= 1;

    return event;
}
```

**Waarom het hele array verschuiven?** Dit houdt de gesorteerde volgorde in stand en is bij kleine arrays (n < 20) extreem snel — het past in één of twee cache lines. De CPU-kosten zijn verwaarloosbaar vergeleken met wat de event handler zelf doet.

**Tip:** Voor betere performance kun je later overstappen op een min-heap of een circulaire buffer, maar start simpel. Optimaliseer pas als profiling aantoont dat de scheduler een bottleneck is — en dat zal vrijwel nooit het geval zijn.

### Stap 4: `src/core/scheduler.zig` — Event dispatch met switch

De `process_events` functie verwerkt alle events waarvan de timestamp bereikt is en plant periodieke events opnieuw in:

```zig
pub fn process_events(self: *Scheduler, ppu: *Ppu, irq: *Irq) void {
    while (self.count > 0 and self.events[0].timestamp <= self.current_time) {
        const event = self.pop_first_event();

        switch (event.event_type) {
            .ppu_hblank => {
                ppu.enter_hblank();
                // Na HBlank komt de volgende scanline (of VBlank)
                self.schedule(.ppu_scanline, 272);  // HBlank duurt 272 cycles
            },
            .ppu_scanline => {
                ppu.render_scanline();
                // Na het zichtbare deel komt HBlank
                self.schedule(.ppu_hblank, 960);  // zichtbaar deel = 960 cycles
            },
            .ppu_vblank => {
                ppu.enter_vblank();
                irq.request(.vblank);  // VBlank interrupt triggeren
            },
            .timer0_overflow => {
                // Timer afhandelen en opnieuw plannen
                const next_overflow = handle_timer_overflow(0);
                self.schedule(.timer0_overflow, next_overflow);
            },
            .apu_sample => {
                // Audio sample genereren en volgende plannen
                generate_audio_sample();
                self.schedule(.apu_sample, 512);  // ~32768 Hz sample rate
            },
            else => {},
        }
    }
}
```

**Waarom plannen events zichzelf opnieuw in?** PPU-events zijn periodiek: na elke scanline komt een HBlank, na elke HBlank komt de volgende scanline, enzovoort. Door aan het einde van elke event handler het *volgende* event te plannen, ontstaat een zelfonderhoudende keten. Je hoeft de scheduler niet van buitenaf te voeden — eenmaal gestart loopt het vanzelf.

Hetzelfde geldt voor timers en audio: een timer die overloopt, herstart zichzelf en plant het volgende overflow-moment. Audio genereert een sample en plant de volgende sample.

**Tip:** Sommige events zijn eenmalig en hoeven niet herplannen. Denk aan DMA-transfers: die worden extern getriggerd (door een I/O register schrijven) en zijn klaar na één uitvoering. Het is belangrijk om in je event handler bewust te kiezen: herplan ik dit event, of niet?

### Stap 5: `src/core/scheduler.zig` — Event annuleren (deschedule)

Niet alle geplande events hoeven daadwerkelijk uitgevoerd te worden. Als een game een timer uitschakelt, moet het bijbehorende overflow-event geannuleerd worden. Zonder deze functie zou een uitgeschakelde timer alsnog "afgaan":

```zig
pub fn deschedule(self: *Scheduler, event_type: EventType) void {
    var i: u8 = 0;
    while (i < self.count) {
        if (self.events[i].event_type == event_type) {
            // Verwijder dit event door alles erna naar voren te schuiven
            var j: u8 = i;
            while (j < self.count - 1) : (j += 1) {
                self.events[j] = self.events[j + 1];
            }
            self.count -= 1;
            // Niet i verhogen — het volgende element is nu op positie i
        } else {
            i += 1;
        }
    }
}
```

**Waarom is dit nodig?** Stel dat timer 0 een overflow-event heeft gepland over 500 cycles. De game schrijft naar het timer-controleregister en schakelt timer 0 uit. Zonder `deschedule` zou het event na 500 cycles alsnog afgevuurd worden, wat een onterechte interrupt veroorzaakt — en mogelijk een crash in de game.

Typische scenario's waarin je `deschedule` nodig hebt:
- Een timer wordt uitgeschakeld via het controleregister
- Een timer wordt opnieuw geconfigureerd (annuleer het oude event, plan een nieuw event)
- DMA wordt afgebroken

**Valkuil:** Vergeet niet events te annuleren wanneer een timer of component wordt uitgeschakeld! Dit is een veelvoorkomende bug in emulators: alles lijkt te werken totdat een game een timer uitschakelt en onverwachte interrupts krijgt.

### Stap 6: `src/main.zig` — Main loop integreren met scheduler

Verbind de scheduler met de CPU in de main loop:

```zig
fn init_scheduler(scheduler: *Scheduler) void {
    // Plan de eerste events — zonder dit doet de scheduler niets!
    scheduler.schedule(.ppu_scanline, 0);     // begin direct met eerste scanline
    scheduler.schedule(.apu_sample, 512);     // eerste audio sample na 512 cycles
}

fn run_frame(cpu: *Cpu, scheduler: *Scheduler) void {
    const frame_end = scheduler.current_time + 280896;  // cycles per frame

    while (scheduler.current_time < frame_end) {
        // Bepaal tot wanneer de CPU mag draaien
        const target = scheduler.next_event_time();

        // CPU draait instructies uit tot het volgende event
        while (scheduler.current_time < target) {
            const cycles = cpu.step();  // voer 1 instructie uit
            scheduler.current_time += cycles;
        }

        // Verwerk alle events die nu (of eerder) moeten gebeuren
        scheduler.process_events();
    }
}
```

**Waarom 280896 cycles per frame?** Dit is 228 scanlines × 1232 cycles per scanline. De GBA heeft 160 zichtbare scanlines + 68 VBlank-scanlines = 228 totaal. Elke scanline duurt 1232 cycles (960 zichtbaar + 272 HBlank). Dit geeft exact 280896 cycles per frame, wat bij 16.78 MHz neerkomt op ~59.73 frames per seconde — de native GBA framerate.

**Waarom `init_scheduler` apart?** De scheduler begint leeg. Als je vergeet de eerste events te plannen, zal `next_event_time()` altijd `maxInt(u64)` teruggeven en de CPU draait het hele frame zonder ooit een event te verwerken. Het resultaat: een zwart scherm en geen geluid. Door de initialisatie expliciet te maken in een aparte functie, is het duidelijk dat dit een vereiste opstartstap is.

**Tip:** Plan de eerste events bij `init()`, anders doet de scheduler helemaal niets! Dit is een veelgemaakte fout bij het implementeren van een scheduler. Test vroeg door te loggen welke events verwerkt worden — als je geen output ziet, heb je waarschijnlijk vergeten de initiële events te plannen.

## Acceptatiecriteria

- Events worden in juiste volgorde afgehandeld
- Scheduler overhead is minimaal
- PPU timing via scheduler komt overeen met directe cycle-counting
