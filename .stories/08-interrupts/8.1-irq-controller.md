# Story 8.1 — Interrupt controller

**Epic:** 08 - Interrupts

**Doel:** IRQ afhandeling implementeren.

## Zig concepten

### Wat zijn interrupts?

Een interrupt is een signaal van hardware naar de CPU: "er is iets gebeurd, reageer nu!" Bijvoorbeeld: het scherm is klaar met tekenen (VBlank), of een timer is afgelopen. Zonder interrupts zou de CPU constant moeten controleren of er iets is veranderd (polling). Met interrupts kan de CPU gewoon doorwerken en wordt hij "onderbroken" wanneer nodig.

### De drie interrupt-registers

```zig
const Irq = struct {
    ie: u16 = 0,      // Interrupt Enable — welke interrupts MAG de CPU ontvangen?
    if_: u16 = 0,     // Interrupt Flags — welke interrupts ZIJN er gebeurd?
    ime: bool = false, // Interrupt Master Enable — staan interrupts überhaupt aan?
};

// Naamgeving: "if_" met underscore omdat "if" een keyword is in Zig
```

### Interrupt types als bits

Elke interrupt heeft een eigen bit in IE en IF:

```zig
const IrqType = enum(u16) {
    vblank    = 1 << 0,   // bit 0: scherm klaar met tekenen
    hblank    = 1 << 1,   // bit 1: scanline klaar
    vcount    = 1 << 2,   // bit 2: specifieke scanline bereikt
    timer0    = 1 << 3,   // bit 3: timer 0 overflow
    timer1    = 1 << 4,
    timer2    = 1 << 5,
    timer3    = 1 << 6,
    serial    = 1 << 7,
    dma0      = 1 << 8,
    dma1      = 1 << 9,
    dma2      = 1 << 10,
    dma3      = 1 << 11,
    keypad    = 1 << 12,
    gamepak   = 1 << 13,
};
```

### Een interrupt aanvragen (request)

Wanneer hardware een event genereert, zet je het corresponderende bit in IF:

```zig
fn request(self: *Irq, irq_type: IrqType) void {
    // Zet het bit met OR
    self.if_ |= @intFromEnum(irq_type);
}

// Voorbeeld: PPU is klaar met frame
irq.request(.vblank);
// if_ was 0b0000, wordt 0b0001
```

### Checken of CPU een IRQ moet nemen

Een IRQ wordt alleen getriggerd als:
1. IME aan staat (master switch)
2. Er een interrupt is die zowel in IE als IF staat

```zig
fn check_irq(self: *Irq) bool {
    return self.ime and (self.ie & self.if_) != 0;
}
// ie & if_ : alleen bits die in BEIDE registers staan
// Als ie = 0b0001 (VBlank enabled) en if_ = 0b0001 (VBlank happened) → 0b0001 != 0 → true
// Als ie = 0b0000 (niets enabled) en if_ = 0b0001 → 0b0000 = 0 → false
```

### IRQ exception — wat doet de CPU?

Wanneer een IRQ wordt getriggerd, doet de CPU automatisch:

```zig
fn handle_irq(self: *Cpu) void {
    // 1. Sla huidige status op
    self.regs_irq[0] = self.pc;      // R14_irq = return address
    self.spsr_irq = self.cpsr;       // SPSR_irq = huidige CPSR

    // 2. Switch naar IRQ mode
    self.cpsr.mode = .irq;

    // 3. Disable verdere IRQs
    self.cpsr.irq_disable = true;

    // 4. Switch naar ARM mode
    self.cpsr.thumb = false;

    // 5. Spring naar IRQ vector
    self.pc = 0x00000018;
}
```

### IF clearen — het "schrijf 1 om te clearen" patroon

Dit is een veelvoorkomend hardware-patroon: om een bit in IF te clearen, schrijf je een 1 naar dat bit:

```zig
// Game schrijft naar IF register:
fn write_if(self: *Irq, value: u16) void {
    self.if_ &= ~value;
    // ~ = bitwise NOT, & = AND
    // Voorbeeld: if_ = 0b0101, value = 0b0001
    // ~0b0001 = 0b1110
    // 0b0101 & 0b1110 = 0b0100 (bit 0 is gecleard)
}
```

## Context

- IE (0x04000200) — Interrupt Enable: welke interrupts actief zijn. De game zet deze bits bij initialisatie.
- IF (0x04000202) — Interrupt Flags: welke interrupts pending zijn. **Schrijf 1 om te clearen** — dit hardware-patroon voorkomt race conditions waarbij interrupts verloren gaan (zie stap 5 voor uitleg).
- IME (0x04000208) — Interrupt Master Enable: globale on/off. **Waarom een aparte master switch?** Zodat de game tijdelijk alle interrupts kan uitschakelen (voor critical sections) zonder IE te hoeven opslaan en herstellen.
- Als (IE & IF) != 0 en IME = 1 → trigger IRQ exception
- CPU springt naar 0x00000018 (IRQ vector) in ARM mode — dit adres is hardware-bepaald en niet configureerbaar
- De CPU schakelt naar IRQ mode met **banked registers** — aparte R13/R14 zodat de interrupt handler de return address niet kan kwijtraken

## Bestanden

- `src/core/irq.zig`
  - `Irq` struct met `ie: u16`, `if_: u16`, `ime: bool`
  - `pub fn request(interrupt: IrqType) void` — zet bit in IF

- `src/core/cpu.zig` — IRQ exception handling

## Stap-voor-stap

### Stap 1: `src/core/irq.zig` — Irq struct met registers

Begin met de drie interrupt-registers:

```zig
pub const Irq = struct {
    ie: u16 = 0,       // Interrupt Enable
    if_: u16 = 0,      // Interrupt Flags
    ime: bool = false,  // Interrupt Master Enable

    pub fn init() Irq {
        return Irq{};
    }
};
```

**Waarom drie aparte registers in plaats van één?** Elk register heeft een andere rol in de interrupt-keten:

- **IE (Interrupt Enable)**: de game bepaalt *welke* interrupts hij wil ontvangen. Dit verandert zelden — meestal alleen bij initialisatie.
- **IF (Interrupt Flags)**: de hardware zet bits wanneer events plaatsvinden. Dit verandert continu — elke VBlank, elke timer overflow.
- **IME (Interrupt Master Enable)**: een globale aan/uit-schakelaar.

**Waarom IME als master switch?** IME stelt de game in staat om *alle* interrupts tijdelijk uit te schakelen zonder IE te hoeven wijzigen. Dit is cruciaal voor critical sections: code die niet onderbroken mag worden. Zonder IME zou je IE moeten opslaan, op 0 zetten, je werk doen, en IE herstellen — meer werk en foutgevoeliger.

**Waarom `if_` met underscore?** `if` is een gereserveerd keyword in Zig. De conventie is om een underscore toe te voegen. Dit is geen Zig-specifiek probleem — dezelfde naamconflicten komen voor in C/C++ emulators.

### Stap 2: `src/core/irq.zig` — IrqType enum en request functie

Definieer de interrupt-types als bitmasks en een functie om interrupts aan te vragen:

```zig
pub const IrqType = enum(u16) {
    vblank  = 1 << 0,
    hblank  = 1 << 1,
    vcount  = 1 << 2,
    timer0  = 1 << 3,
    timer1  = 1 << 4,
    timer2  = 1 << 5,
    timer3  = 1 << 6,
    serial  = 1 << 7,
    dma0    = 1 << 8,
    dma1    = 1 << 9,
    dma2    = 1 << 10,
    dma3    = 1 << 11,
    keypad  = 1 << 12,
    gamepak = 1 << 13,
};

pub fn request(self: *Irq, irq_type: IrqType) void {
    self.if_ |= @intFromEnum(irq_type);
}
```

**Waarom `enum(u16)` met bitmask-waarden?** Elke interrupt-type komt overeen met exact één bit in de IE/IF registers. Door de enum-waarden als bitmasks te definiëren (`1 << 0`, `1 << 1`, etc.) kun je ze direct gebruiken in bitwise operaties zonder extra conversie. Vergelijk: `self.if_ |= @intFromEnum(.vblank)` zet bit 0 — schoon en duidelijk.

**Waarom `request` zo simpel?** De `request`-functie is een *fire-and-forget*: hardware meldt dat er iets is gebeurd, punt. Of de CPU er iets mee doet hangt af van IE en IME — dat is niet de verantwoordelijkheid van `request`. Deze scheiding houdt de code modulair.

### Stap 3: `src/core/irq.zig` — check_irq logica

De check-functie bepaalt of de CPU daadwerkelijk een interrupt moet afhandelen:

```zig
pub fn check_irq(self: *Irq) bool {
    return self.ime and (self.ie & self.if_) != 0;
}
```

**Waarom `ie & if_`?** Dit is de kern van het interrupt-systeem. De AND-operatie zoekt bits die in *beide* registers staan:

```
IE  = 0b0000_0000_0001  (alleen VBlank enabled)
IF  = 0b0000_0000_0101  (VBlank én VCount pending)
AND = 0b0000_0000_0001  (alleen VBlank matcht)
```

VCount is pending maar niet enabled → wordt genegeerd. Alleen VBlank triggert de interrupt. Dit geeft de game volledige controle over welke interrupts daadwerkelijk de CPU onderbreken.

**Wanneer roep je `check_irq` aan?** Na elke CPU-instructie. Dit klinkt duur, maar de check is slechts twee condities (een bool en een bitwise AND). In de praktijk is dit verwaarloosbaar.

**Valkuil:** Vergeet niet om `check_irq` ook te controleren wanneer de CPU in HALT-modus staat. HALT zet de CPU in slaap totdat een interrupt binnenkomt — zonder deze check blijft de CPU eeuwig slapen.

### Stap 4: `src/core/cpu.zig` — IRQ exception handling

Wanneer `check_irq()` true retourneert, moet de CPU naar IRQ-modus schakelen:

```zig
fn handle_irq(self: *Cpu) void {
    // 1. Bewaar return address in R14_irq (banked register)
    self.regs_irq[0] = self.pc;

    // 2. Bewaar huidige CPSR in SPSR_irq
    self.spsr_irq = self.cpsr;

    // 3. Switch naar IRQ mode
    self.cpsr.mode = .irq;

    // 4. Disable verdere IRQs (voorkom oneindige nesting)
    self.cpsr.irq_disable = true;

    // 5. Switch naar ARM mode (IRQ handler is altijd ARM)
    self.cpsr.thumb = false;

    // 6. Spring naar IRQ vector
    self.pc = 0x00000018;
}
```

**Waarom mode switch naar IRQ mode?** De ARM7TDMI heeft "banked registers" — aparte kopieën van R13 (stack pointer) en R14 (link register) per CPU-mode. Door naar IRQ mode te schakelen, gebruikt de CPU automatisch een aparte stack en link register. Dit beschermt de return address: als de IRQ handler een functie aanroept, wordt het oorspronkelijke return address niet overschreven.

**Waarom IRQs disablen?** Zonder dit zou een tweede interrupt midden in de afhandeling van de eerste kunnen binnenkomen, waardoor SPSR_irq en R14_irq overschreven worden. De interrupt handler kan later IRQs weer enablen als hij nesting wil toestaan (dit is wat de GBA BIOS doet).

**Waarom altijd ARM mode?** De exception vector table op adres 0x00-0x1C bevat ARM-instructies. Zelfs als de CPU in Thumb-modus draaide, schakelt hij over naar ARM voor de exception handler. De BIOS-code op 0x18 is in ARM geschreven.

**Waarom adres 0x00000018?** De ARM7TDMI heeft een vaste exception vector table:
- `0x00`: Reset
- `0x08`: Undefined instruction
- `0x10`: SWI (software interrupt)
- `0x18`: IRQ
- `0x1C`: FIQ (niet gebruikt op GBA)

Dit zijn hardware-gedefinieerde adressen — je kunt ze niet veranderen.

### Stap 5: `src/core/io.zig` — IF clearen via I/O writes

Koppel de interrupt-registers aan de I/O dispatcher:

```zig
// In read16:
0x04000200 => self.irq.ie,
0x04000202 => self.irq.if_,
0x04000208 => @intFromBool(self.irq.ime),

// In write16:
0x04000200 => self.irq.ie = value,
0x04000202 => self.irq.if_ &= ~value,  // schrijf 1 om te clearen!
0x04000208 => self.irq.ime = (value & 1) == 1,
```

**Waarom "schrijf 1 om te clearen" voor IF?** Dit is een klassiek hardware-patroon dat race conditions voorkomt. Stel je het alternatief voor (schrijf 0 om te clearen):

1. CPU leest IF: `0b0101` (VBlank + Timer0 pending)
2. CPU wil VBlank clearen, schrijft `0b0100` (alleen Timer0 behouden)
3. **Maar** tussen stap 1 en 2 komt er een HBlank interrupt → IF wordt `0b0111`
4. De write overschrijft IF met `0b0100` → HBlank interrupt is verloren!

Met "schrijf 1 om te clearen":
1. CPU schrijft `0b0001` (clear VBlank)
2. Hardware doet `IF &= ~0b0001` = clear alleen bit 0
3. Eventuele nieuwe interrupts die ertussen kwamen, blijven behouden

**Valkuil:** Dit is de meest voorkomende fout bij het implementeren van interrupts in een emulator. Als je `self.irq.if_ = value` schrijft in plaats van `self.irq.if_ &= ~value`, zullen games vastlopen omdat interrupts nooit correct worden gecleard.

### Stap 6: `src/core/cpu.zig` — HALT modus integreren

De CPU moet na elke instructie controleren op interrupts, en HALT correct afhandelen:

```zig
pub fn step(self: *Cpu) void {
    if (self.halted) {
        // CPU slaapt — doe niets tot er een interrupt komt
        if (self.irq.check_irq()) {
            self.halted = false;
            self.handle_irq();
        }
        return;
    }

    // Voer één instructie uit
    self.execute_next_instruction();

    // Check interrupts na elke instructie
    if (self.irq.check_irq()) {
        self.handle_irq();
    }
}
```

**Waarom HALT?** Veel GBA-games gebruiken een patroon waarbij ze HALT aanroepen na het enablen van VBlank-interrupts. De CPU stopt met stroom te verbruiken (op echte hardware) tot de VBlank interrupt binnenkomt. In de emulator kun je HALT simuleren door de CPU-stap over te slaan en alleen de interrupt-check te doen.

**Tip:** Zelfs in HALT-modus moeten timers, PPU en DMA blijven draaien. Alleen de CPU stopt — de rest van de hardware gaat gewoon door. Vergeet dus niet om `ppu.step()`, `timers.step()` etc. aan te roepen ook als de CPU in HALT staat.

## Acceptatiecriteria

- VBlank interrupt wekt CPU uit HALT
- Interrupt handler wordt correct aangeroepen
- IF bits worden correct gecleard bij schrijven
