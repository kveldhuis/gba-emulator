# Story 9.1 — Timer controller

**Epic:** 09 - Timers

**Doel:** 4 hardware timers implementeren.

## Zig concepten

### Wat doen timers?

Timers zijn hardware-tellers die automatisch optellen. Ze worden gebruikt voor:
- Audio sample rate (hoe snel samples worden afgespeeld)
- Game timing (wachten op een bepaalde tijd)
- Frequentiemeting

Elke timer telt op van een startwaarde tot 0xFFFF. Bij overflow (voorbij 0xFFFF) wordt de timer herladen met een reload-waarde en kan optioneel een interrupt worden getriggerd.

### Timer registers

Elke timer heeft twee registers:

```zig
// Timer 0: TM0CNT_L (0x04000100) en TM0CNT_H (0x04000102)
// Timer 1: TM1CNT_L (0x04000104) en TM1CNT_H (0x04000106)
// Timer 2: TM2CNT_L (0x04000108) en TM2CNT_H (0x0400010A)
// Timer 3: TM3CNT_L (0x0400010C) en TM3CNT_H (0x0400010E)

const TimerControl = packed struct(u16) {
    prescaler: u2,    // 0=1x, 1=64x, 2=256x, 3=1024x
    count_up: u1,     // cascade mode (telt alleen bij overflow vorige timer)
    _unused: u3,
    irq_enable: u1,   // trigger interrupt bij overflow?
    enabled: u1,      // timer aan/uit
    _unused2: u8,
};
```

### Prescaler — snelheid vertragen

De prescaler bepaalt hoeveel CPU-cycles nodig zijn voor één timer-tick:

```zig
const prescaler_values = [4]u16{ 1, 64, 256, 1024 };

// Voorbeeld: prescaler = 2 (256x)
// De timer telt 1 keer per 256 CPU-cycles
// CPU draait op ~16.78 MHz, dus deze timer tikt ~65.5 kHz
```

### Timer struct

```zig
const Timer = struct {
    counter: u16 = 0,       // huidige tellerwaarde
    reload: u16 = 0,        // herlaadwaarde bij overflow
    control: TimerControl = @bitCast(@as(u16, 0)),
    cycles: u16 = 0,        // interne cycle-teller voor prescaler

    fn step(self: *Timer, cpu_cycles: u32) bool {
        if (!self.control.enabled or self.control.count_up == 1) return false;

        self.cycles += @intCast(cpu_cycles);
        const prescaler = prescaler_values[self.control.prescaler];

        var overflowed = false;
        while (self.cycles >= prescaler) {
            self.cycles -= prescaler;

            // Tel op, check overflow
            const result = @addWithOverflow(self.counter, 1);
            self.counter = result[0];
            if (result[1] == 1) {  // overflow!
                self.counter = self.reload;
                overflowed = true;
            }
        }
        return overflowed;
    }
};
```

### @addWithOverflow — veilig overflow detecteren

In Zig kun je niet zomaar voorbij het maximum van een integer tellen (dat is een fout). `@addWithOverflow` geeft een tuple terug met het resultaat én een overflow-flag:

```zig
const result = @addWithOverflow(@as(u16, 0xFFFF), @as(u16, 1));
// result[0] = 0x0000 (de overgelopen waarde)
// result[1] = 1      (er was een overflow)

const result2 = @addWithOverflow(@as(u16, 100), @as(u16, 1));
// result2[0] = 101
// result2[1] = 0     (geen overflow)
```

### Cascade mode — timers aan elkaar koppelen

In count-up mode telt een timer niet op basis van CPU-cycles, maar telt hij alleen als de vorige timer overloopt:

```zig
fn step_all_timers(self: *TimerController, cpu_cycles: u32) void {
    var overflow = self.timers[0].step(cpu_cycles);

    // Timer 1-3 kunnen in cascade mode staan
    for (1..4) |i| {
        if (self.timers[i].control.count_up == 1) {
            // Cascade: tel alleen als vorige timer overflowed
            if (overflow) {
                const result = @addWithOverflow(self.timers[i].counter, 1);
                self.timers[i].counter = result[0];
                overflow = result[1] == 1;
                if (overflow) {
                    self.timers[i].counter = self.timers[i].reload;
                }
            } else {
                overflow = false;
            }
        } else {
            overflow = self.timers[i].step(cpu_cycles);
        }

        // Trigger IRQ bij overflow als enabled
        if (overflow and self.timers[i].control.irq_enable == 1) {
            irq.request(switch (i) {
                0 => .timer0,
                1 => .timer1,
                2 => .timer2,
                3 => .timer3,
                else => unreachable,
            });
        }
    }
}
```

## Context

- 4 timers (TM0-TM3), elk met:
  - TM0CNT_L (counter value, read) / reload value (write) — **waarom dubbel gedrag?** De hardware gebruikt hetzelfde adres voor twee functies: lezen geeft de huidige stand, schrijven stelt de herlaadwaarde in. Dit spaart adresruimte.
  - TM0CNT_H (control): prescaler, count-up mode, IRQ enable, start/stop
- Prescaler: 1, 64, 256, 1024 CPU cycles per tick — **waarom prescaler?** Dezelfde timer kan op verschillende snelheden draaien. Audio heeft een snelle timer nodig (~16 MHz), game-timing een langzamere (~16 kHz). De prescaler maakt deze flexibiliteit mogelijk met minimale hardware.
- Count-up mode: timer telt alleen als vorige timer overloopt (cascade) — **waarom?** Door timers te koppelen ontstaat effectief een 32-bit of 48-bit teller, nodig voor langere meetperiodes die een enkele 16-bit timer niet aankan.
- Bij overflow (0xFFFF → 0x0000): herlaad reload value, optioneel IRQ — **waarom IRQ bij overflow?** Dit maakt timer-gebaseerde events mogelijk: stel een timer in op een bepaalde frequentie, en de interrupt handler wordt automatisch op die frequentie aangeroepen (essentieel voor audio playback).

## Stap-voor-stap

### Stap 1: `src/core/timer.zig` — Timer struct aanmaken

Begin met een struct voor één individuele timer:

```zig
pub const Timer = struct {
    counter: u16 = 0,
    reload: u16 = 0,
    control: TimerControl = @bitCast(@as(u16, 0)),
    cycles: u16 = 0,  // interne prescaler-teller
};
```

**Waarom `counter` en `reload` apart?** Het TMxCNT_L register heeft een dubbele persoonlijkheid:
- **Lezen** geeft de huidige tellerwaarde (`counter`)
- **Schrijven** zet de reload-waarde (`reload`), niet de teller zelf

Dit is een hardware-eigenaardigheid: de game kan de startwaarde instellen via reload, maar kan de teller niet direct overschrijven. De teller wordt alleen geladen vanuit reload bij een overflow of wanneer de timer opnieuw wordt gestart.

**Waarom een `cycles` teller?** De prescaler werkt als een vertrager. Als prescaler = 64, moet de timer pas 1 keer tikken na 64 CPU-cycles. De `cycles`-teller houdt bij hoeveel CPU-cycles er verstreken zijn sinds de laatste tick. Wanneer `cycles >= prescaler`, tikt de timer en resetten we de teller.

### Stap 2: `src/core/timer.zig` — TimerControl packed struct

```zig
pub const TimerControl = packed struct(u16) {
    prescaler: u2 = 0,    // 0=1, 1=64, 2=256, 3=1024
    count_up: u1 = 0,     // cascade mode
    _unused: u3 = 0,
    irq_enable: u1 = 0,
    enabled: u1 = 0,
    _unused2: u8 = 0,
};

pub const prescaler_values = [4]u16{ 1, 64, 256, 1024 };
```

**Waarom een prescaler?** De GBA CPU draait op ~16.78 MHz. Zonder prescaler zou een timer 16.78 miljoen keer per seconde tikken — veel te snel voor de meeste toepassingen. De prescaler verdeelt de kloksnelheid:

| Prescaler | CPU-cycles per tick | Effectieve frequentie | Typisch gebruik |
|-----------|--------------------|-----------------------|-----------------|
| 0 (1x)   | 1                  | 16.78 MHz            | Audio samples   |
| 1 (64x)  | 64                 | 262.2 kHz            | Snelle timing   |
| 2 (256x) | 256                | 65.5 kHz             | Audio timing    |
| 3 (1024x)| 1024               | 16.4 kHz             | Langzame events |

**Waarom `packed struct(u16)`?** Het control register is een 16-bit waarde waar elk bit een specifieke betekenis heeft. Met `packed struct` kun je direct casten tussen de struct en een `u16`: `@bitCast(control)` geeft de ruwe 16-bit waarde, en `@bitCast(value)` maakt er een struct van. Dit is essentieel voor I/O register reads en writes.

### Stap 3: `src/core/timer.zig` — step functie met prescaler

De `step` functie is het hart van elke timer:

```zig
pub fn step(self: *Timer, cpu_cycles: u32) bool {
    // Niet-actieve timers en cascade timers tikken niet zelf
    if (self.control.enabled == 0 or self.control.count_up == 1) return false;

    self.cycles += @intCast(cpu_cycles);
    const prescaler = prescaler_values[self.control.prescaler];

    var overflowed = false;
    while (self.cycles >= prescaler) {
        self.cycles -= prescaler;

        const result = @addWithOverflow(self.counter, 1);
        self.counter = result[0];
        if (result[1] == 1) {
            self.counter = self.reload;
            overflowed = true;
        }
    }
    return overflowed;
}
```

**Waarom een `while`-loop in plaats van `if`?** Als de CPU meerdere cycles tegelijk uitvoert (bijvoorbeeld: een instructie kost 4 cycles) en de prescaler is 1, dan moeten er 4 timer-ticks plaatsvinden. De while-loop handelt alle opgestapelde ticks af in één keer.

**Waarom `@addWithOverflow` in plaats van `if (self.counter == 0xFFFF)`?** Twee redenen:
1. **Correctheid**: `@addWithOverflow` is de idiomatische Zig-manier om overflow te detecteren. In Zig is integer overflow undefined behavior in debug mode — een gewone `+= 1` op `0xFFFF` zou crashen.
2. **Duidelijkheid**: de tuple `(result, overflow_bit)` maakt de intentie expliciet. Je kunt niet per ongeluk vergeten de overflow te checken.

**Waarom geeft `step` een `bool` terug?** De retourwaarde geeft aan of de timer is overflowed. Dit is cruciaal voor twee systemen:
- **Cascade mode**: de volgende timer telt alleen als de vorige overflowed (zie stap 4)
- **Interrupts**: een overflow kan een interrupt triggeren (zie stap 5)

### Stap 4: `src/core/timer.zig` — TimerController met cascade mode

Maak een controller die alle 4 timers beheert:

```zig
pub const TimerController = struct {
    timers: [4]Timer = [_]Timer{.{}} ** 4,
    irq: *Irq,

    pub fn init(irq: *Irq) TimerController {
        return TimerController{ .irq = irq };
    }

    pub fn step(self: *TimerController, cpu_cycles: u32) void {
        // Timer 0 tikt altijd zelfstandig
        var overflow = self.timers[0].step(cpu_cycles);
        if (overflow and self.timers[0].control.irq_enable == 1) {
            self.irq.request(.timer0);
        }

        // Timers 1-3 kunnen in cascade staan
        for (1..4) |i| {
            if (self.timers[i].control.enabled == 0) {
                overflow = false;
                continue;
            }

            if (self.timers[i].control.count_up == 1) {
                // Cascade: tel alleen als vorige timer overflowed
                if (overflow) {
                    const result = @addWithOverflow(self.timers[i].counter, 1);
                    self.timers[i].counter = result[0];
                    overflow = result[1] == 1;
                    if (overflow) {
                        self.timers[i].counter = self.timers[i].reload;
                    }
                } else {
                    overflow = false;
                }
            } else {
                overflow = self.timers[i].step(cpu_cycles);
            }

            if (overflow and self.timers[i].control.irq_enable == 1) {
                const irq_type: IrqType = switch (i) {
                    1 => .timer1,
                    2 => .timer2,
                    3 => .timer3,
                    else => unreachable,
                };
                self.irq.request(irq_type);
            }
        }
    }
};
```

**Waarom cascade mode?** Een enkele 16-bit timer kan maximaal 65.536 ticks tellen voordat hij overloopt. Met cascade mode koppel je timers aan elkaar tot een langere keten. Twee timers in cascade vormen effectief een 32-bit teller (tot ~4,3 miljard ticks). Dit is essentieel voor:
- **Audio**: DMA-gebaseerde audio gebruikt Timer 0 of 1 als sample rate. Bij hoge sample rates met prescaler 1 zou een enkele timer te snel overlopen.
- **Lange timings**: sommige games gebruiken gekoppelde timers om langere periodes te meten (secondes in plaats van microseconden).

**Waarom verwerkt Timer 0 altijd zelfstandig?** Timer 0 heeft geen "vorige timer" — er is niets om mee te cascaden. Daarom negeert Timer 0 de count_up-bit. Timers 1-3 kijken naar de overflow van hun directe voorganger.

**Valkuil:** De volgorde van verwerking is cruciaal. Timer 0 → Timer 1 → Timer 2 → Timer 3. Als je Timer 3 vóór Timer 1 verwerkt, mist Timer 3 de overflow van Timer 2. De `for (1..4)` loop garandeert de juiste volgorde.

### Stap 5: `src/core/timer.zig` — writeControl voor start/stop logica

Het schrijven naar het control register heeft side effects:

```zig
pub fn writeControl(self: *TimerController, timer_id: usize, value: u16) void {
    const old_enabled = self.timers[timer_id].control.enabled;
    self.timers[timer_id].control = @bitCast(value);
    const new_enabled = self.timers[timer_id].control.enabled;

    // Bij het aanzetten van een timer: herlaad counter vanuit reload
    if (old_enabled == 0 and new_enabled == 1) {
        self.timers[timer_id].counter = self.timers[timer_id].reload;
        self.timers[timer_id].cycles = 0;
    }
}
```

**Waarom herladen bij enable?** De GBA-specificatie zegt: wanneer een timer wordt gestart (enabled bit gaat van 0 naar 1), wordt de counter geladen met de reload-waarde. Dit zorgt ervoor dat de timer altijd begint te tellen vanaf het punt dat de game heeft ingesteld, niet vanaf een willekeurige waarde.

**Waarom de oude enabled-waarde opslaan?** We willen de reload alleen uitvoeren bij een 0→1 transitie, niet bij elke write naar het control register. Als de timer al draait en de game het control register herschrijft (bijvoorbeeld om de prescaler te veranderen), moet de counter niet gereset worden.

### Stap 6: `src/core/io.zig` — timer registers koppelen

Voeg de timer-registers toe aan de I/O dispatcher:

```zig
// In read16:
0x04000100 => self.timers.timers[0].counter,
0x04000102 => @bitCast(self.timers.timers[0].control),
0x04000104 => self.timers.timers[1].counter,
0x04000106 => @bitCast(self.timers.timers[1].control),
0x04000108 => self.timers.timers[2].counter,
0x0400010A => @bitCast(self.timers.timers[2].control),
0x0400010C => self.timers.timers[3].counter,
0x0400010E => @bitCast(self.timers.timers[3].control),

// In write16:
0x04000100 => self.timers.timers[0].reload = value,
0x04000102 => self.timers.writeControl(0, value),
0x04000104 => self.timers.timers[1].reload = value,
0x04000106 => self.timers.writeControl(1, value),
0x04000108 => self.timers.timers[2].reload = value,
0x0400010A => self.timers.writeControl(2, value),
0x0400010C => self.timers.timers[3].reload = value,
0x0400010E => self.timers.writeControl(3, value),
```

**Waarom `@bitCast` voor control reads?** De control register is opgeslagen als een `TimerControl` packed struct, maar de game leest het als een ruwe `u16`. `@bitCast` converteert de packed struct terug naar een u16 zonder enige berekening — het is puur een type-conversie.

**Tip:** Merk op dat schrijven naar TMxCNT_L (de counter/reload adressen) de *reload* waarde zet, niet de counter. Maar lezen van hetzelfde adres geeft de *counter* waarde. Dit dubbele gedrag is een veelvoorkomend patroon bij GBA I/O registers.
